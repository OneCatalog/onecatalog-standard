# OneCatalog Import — стандарт интеграции (platform-agnostic)

Документ описывает **общие принципы, внутреннюю логику и обязательные элементы**
интеграции с OneCatalog, в форме, пригодной для построения такого же модуля на
**любой CMS/платформе** (WooCommerce — эталонная реализация; Bitrix24, PrestaShop,
OpenCart, Magento, кастомные backend — по этому стандарту).

Стандарт покрывает **два независимых сценария** (внедряются по отдельности):
1. **Импорт каталога** из Wiki API — структура и контент товара (§1–§12). Цену
   импорт НЕ задаёт (§5.6).
2. **Синхронизация цен и остатков** из B2B-фида ритейлера (§13) — живой фид,
   перезаписывающий цену/остаток. Отдельный API и авторизация, отдельная философия:
   владение ценой/остатком, change-detection (чтобы не нагружать сайт).

Эталонная реализация: WordPress + WooCommerce, namespace `OneCatalog\Import`
(см. `README.md`). Здесь — абстракция над ней.

Документация OneCatalog: <https://docs.onecatalog.net/>

---

## 1. Назначение и философия

**Задача модуля:** перенести выбранные товары из централизованного каталога
OneCatalog в товарную систему конкретного сайта вместе со всеми зависимостями
(категории, характеристики, бренды, коллекции, изображения), идемпотентно и в
фоне, не ломая оформление сайта.

Четыре принципа, обязательные на любой платформе:

1. **Тонкое универсальное ядро + сайтовый слой через события.** Модуль импорта
   НЕ пишет оформительских/тематических полей конкретного сайта. Всё, что
   специфично для витрины, навешивается снаружи через хуки/события
   (см. §8). Так один и тот же модуль переносится между проектами без правок.
2. **Идемпотентность по стабильному внешнему ключу.** Повторный импорт обновляет,
   а не дублирует (§5.1).
3. **Источник истины — внешний; чего в нём нет — не выдумывать.** OneCatalog Wiki
   API — единственный источник; локально храним проекцию + ссылки на исходные
   сущности. Отсутствующие в источнике поля (напр. цены) НЕ синтезировать (§5.6).
4. **Деградация без падений.** Один битый товар/недоступный эндпоинт не
   прерывает пакет; всё логируется (§5.5).

### Порядок прототипирования (рекомендуемый)

```
0. Разведка API: payload товара, коллекции, изображения, языки, пагинация.
1. Ядро импорта одного товара по идентификатору (CLI/скрипт, без UI).
   → доказать маппинг и идемпотентность на 1–3 товарах.
2. Зависимости: категории → характеристики (вкл. boolean) → бренд → страна →
   теги → коллекция → изображения.
3. Контроллер (REST/endpoint) приёма списка идентификаторов.
4. UI выбора (виджет-пикер) + кнопка на странице товаров.
5. Фоновая очередь + прогресс/лог.
6. Настройки: токен, язык, шаг; статус новых; коллекция/бренд/страна — «тип объекта
   + цель»; теги вкл/выкл; ручной маппинг характеристик (+ boolean true/false).
7. Трекинг качества + дедуп изображений; политика артикулов; нормализация
   (цена не задаётся, статус только новым, boolean=false).
8. Точки расширения (хуки/события), локализация, миграции, версионирование/CHANGELOG.
```

Каждый шаг — самостоятельно проверяемый инкремент.

## 2. Внешние контракты (не зависят от CMS)

### 2.1 Wiki API

- **Базовый URL:** `https://api.onecatalog.net/wiki/v1` (прод).
  Должен переопределяться (env/настройка/фильтр).
- **Аутентификация:** заголовок `X-API-Key: <token>`. Без токена доступен
  публичный каталог, но изображения отдаются только в низком качестве (`min`).
- **Язык:** query-параметр `lang` добавляется ко ВСЕМ запросам. Поддерживаются
  `ru, en, ar, zh, kk` (источник — OpenAPI-спека). По умолчанию `en`.
- **Формат ответа:** `{ "data": <object|array>, "success": <bool> }`;
  у списков — `{ "data": [...], "meta": { "counts": <int> } }`, где `counts` —
  ОБЩЕЕ число записей. Списки/справочники пагинируются (`?start&limit`):
  справочник (напр. `/specifications/`) брать **одним запросом с заведомо высоким
  `limit`** — иначе дефолтный лимит инстанса молча обрежет выдачу; при
  необходимости дочитывать страницами, сверяясь с `counts`.
- **Эндпоинты (GET):**

  | Эндпоинт | Назначение |
  |---|---|
  | `/products/?start&limit&lang&filter_by&search` | список товаров (пагинация) |
  | `/products_ids/` | только public_id (лёгкий список) |
  | `/products/{public_id_or_slug}/` | один товар (полный payload) |
  | `/collections/`, `/collections/{id_or_slug}/` | коллекции |
  | `/brands/`, `/categories/`, `/specifications/`, `/product-shapes/`, `/countries/`, `/measurement-units/` | справочники |
  | `/search/` | сквозной поиск |
  | `/media_files/{hash}` | бинарный файл изображения |

  ⚠️ На текущем инстансе `/collections/{slug}/` отдаёт 500 — **не полагайтесь на
  него**: все поля коллекции уже приходят в эмбеде `collections[]` товара.

- **Идентификация товара:** `public_id` (например `OC.IND.1`) — стабильный
  бизнес-ключ. Это НЕ артикул производителя (см. §5.2).

### 2.2 Структура payload товара (полная)

```
id, public_id, slug, menutitle (название),
article (артикул производителя; часто пуст),
in_stock (bool), status_publish,
measurement_unit { id, slug, name },                         // единица измерения (м²/шт/…)
brand   { id, menutitle, slug, description_text, foundation_year, website, country, images_urls, images_data },
country { id, slug, menutitle, country_code },
description_text (часто null),
product_shape { id, menutitle, slug },                       // вид/тип товара
collections [ { id, menutitle, slug, link_3d, link_official_site, images_urls, images_data } ],
categories  [ { id, menutitle, slug } ],
tags        [ { id, title, slug } ],
options     [ { specification_id, specification_label, specification_type(text|boolean|numeric),
                slug, specification_option_id, specification_option_name,
                bool_option, numeric_option } ],             // ← характеристики
files       [ { id, slug, name, alt, description, category(images), file_type, urls{min,middle,max} } ],
sizes       { weight, length, width, height, length_unit, weight_unit },
areas_per_package, upsells[], cross_sells[],
images_urls { min, middle, max },                            // обложка
images_data { name, attr_title, alt, description, category }
```

- ⚠️ **boolean-характеристики** приходят без текстового значения:
  `specification_option_name = null`, фактическое значение — в
  `bool_option (true|false)`. `numeric`-характеристики — в `numeric_option`.
  Отсутствие / `null` / `false` булева трактовать как **false** (§5.6). Без явного
  правила «true/false → значение терма» bool-характеристика текстового значения не
  имеет (иначе молча теряется).
- **Многие поля payload не имеют прямого аналога в товаре** (ед. изм., площадь
  упаковки `areas_per_package`, вид `product_shape`, alt/описание картинок, детали
  бренда, upsells/cross_sells). Ядро их НЕ трогает — их заполняет сайт через
  событие `product_imported` (§8). Это граница «универсальное ↔ проектное».

### 2.3 Изображения

- URL вида `…/media_files/<base64hash>` — **без расширения**. Многие платформенные
  загрузчики (напр. WP `media_sideload_image`) такие отвергают → **скачивать
  вручную и определять MIME по содержимому**.
- Каждый файл доступен в трёх размерах `min | middle | max`. С токеном доступен
  `max`; без токена — только `min`/превью.
- В base64-префиксе хеша закодирован `size|path|token` — можно восстановить
  размер из URL (нужно для миграции качества, §5.3).

### 2.4 Виджет выбора товаров (Picker)

Отдельный контракт UI-выбора (опционален — можно импортировать просто по списку ID):

- Загрузчик встраивает кнопку → по клику открывает **iframe** на
  `https://tools.onecatalog.net/picker.html?token=…&parentOrigin=…`.
- Связь родитель↔iframe — `postMessage`:
  - `{ type: 'ONECATALOG_SELECTED', productIds, productPublicIds }` — выбор подтверждён;
  - `{ type: 'ONECATALOG_CLOSE' }` — закрыть.
- **Импортировать по `productPublicIds`** (стабильные ключи), не по числовым id.
- Подводные камни: iframe sandbox `allow-scripts allow-forms allow-same-origin`;
  origin виджета задавать явно (не `window.location.origin`); проверять
  `event.origin`.

## 3. Доменная модель и маппинг сущностей

Импорт — это маппинг абстрактных сущностей OneCatalog на сущности платформы.
Обязательно спроектировать **таблицу соответствий** до кода:

| OneCatalog | Абстракция | WooCommerce | Bitrix24 | PrestaShop |
|---|---|---|---|---|
| product | Товар | `product` (CPT) | Элемент инфоблока «Каталог» | `Product` |
| public_id | Внешний ключ | мета `_onecatalog_public_id` | внешний код элемента (`XML_ID`) | `reference`/своё поле |
| article | Артикул произв. | `SKU` | артикул (свойство) | `reference`/`supplier_reference` |
| options[] | Характеристика | глоб. атрибут `pa_*` + термины (поиск по **имени**; опц. ручной маппинг `specification_id`→атрибут) | свойство инфоблока (список) | `Feature` + `FeatureValue` |
| categories[] | Категория | `product_cat` (термины) | раздел инфоблока | `Category` |
| brand | Бренд | **атрибут `pa_*` / таксономия `product_brand` / CPT** (настройка) | свойство/справочник/раздел | `Manufacturer`/feature |
| country | Страна происх. | **атрибут `pa_*` / таксономия** (настройка) | свойство/справочник | `Feature`/своё |
| collections[] | Коллекция | CPT **или** таксономия (настройка) | инфоблок/раздел/свойство | категория/feature/своё |
| tags[] | Теги | таксономия тегов (вкл/выкл, поиск по имени) | свойство/тэги | tag |
| images_urls/files | Медиа | вложения + featured/галерея (дедуп по контент-ключу) | файлы свойства/анонс-детальная | изображения товара |
| measurement_unit, sizes, areas_per_package, product_shape, … | прочее | через событие `product_imported` (сайтовый слой) | то же | то же |

**Ключевой архитектурный выбор (обязателен как настройка):** для «справочных»
сущностей — коллекция, бренд, страна — целевое хранилище в разных проектах разное
(отдельная сущность-страница / фасет-термин / товарный атрибут). Модуль обязан
давать выбор **«тип объекта (атрибут / таксономия / CPT) + конкретная цель»** по
каждой такой сущности (§7), а не зашивать одно решение. Характеристики
дополнительно поддерживают **ручной маппинг** «характеристика OneCatalog →
существующий атрибут платформы» по стабильному `specification_id` (§5.4).

## 4. Архитектура (слои)

Эталон делит логику на независимые модули-классы. На любой платформе должны быть
эти **слои** (имена — из эталона):

```
Plugin            — сборка, жизненный цикл, миграции
  ├─ Api          — клиент Wiki API: base URL, token, lang, GET, обёртка ответа
  ├─ Units        — конвертер единиц (вес/размеры) источник → единицы платформы
  ├─ Media        — выбор размера, скачивание, обложка/галерея, трекинг качества, дедуп
  ├─ Taxonomies   — атрибуты/характеристики (поиск по имени), термины, категории
  ├─ CollectionImporter — коллекция → post type / таксономия + поля
  ├─ BrandImporter      — бренд → атрибут / таксономия / CPT
  ├─ CountryImporter    — страна → атрибут / таксономия
  ├─ ProductImporter    — товар → объект платформы + все зависимости (оркестратор)
  ├─ Queue        — фоновая очередь, порции, лог, статус
  ├─ Rest         — контроллер приёма идентификаторов + статус
  ├─ Settings     — хранение/валидация настроек, UI настроек
  └─ AdminUI      — кнопка выбора на странице товаров, поле внешнего ключа
assets/
  ├─ picker-loader  — встраивание виджета (iframe + postMessage)
  └─ admin-import   — постановка в очередь + поллинг прогресса
```

Зависимости направлены «вниз»: `ProductImporter` оркестрирует `Api/Media/
Taxonomies/CollectionImporter`; `Queue` вызывает `ProductImporter`; `Rest`
вызывает `Queue`. UI-слои зависят только от `Rest`/`Settings`. Циклов нет.

Принцип переноса: на ЛЮБОЙ платформе должны существовать аналоги этих слоёв,
даже если язык/фреймворк другой (PHP-классы, Bitrix-модуль с классами, PrestaShop
Module + Services).

## 5. Обязательная внутренняя логика (инварианты)

### 5.1 Идемпотентность

- Хранить **внешний стабильный ключ** (`public_id`) в отдельном поле объекта.
- Перед импортом: `find_by_external_id(public_id)` → если найден, **обновить**,
  иначе создать. Никогда не использовать для поиска нестабильные поля (название/slug).
- Справочники (категории, термины характеристик, бренды, страны, коллекции) тоже
  идемпотентны — «найти, иначе создать».
- ⚠️ **Искать справочник по ЧЕЛОВЕКОЧИТАЕМОМУ имени (label), а не по вычисленному
  слагу.** Разные схемы транслитерации (плагины Cyr-to-Lat, локаль CMS, ручное
  создание) дают РАЗНЫЕ слаги для одного имени → поиск по слагу плодит дубли
  (эталон: «Толщина» → `tolschina` нашего транслита vs `tolshhina` от Cyr-to-Lat).
  Матчить по имени без учёта регистра; слаг — только для создания нового.
- Легаси-миграция: при смене схемы ключа поддержать старый способ поиска как
  фолбэк (эталон: раньше идентификатор лежал в SKU → теперь в отдельной мете,
  старые товары находятся по SKU и мигрируют).

### 5.2 Политика артикулов (SKU)

- **Внешний ID (`public_id`) ≠ артикул производителя (`article`).** Это разные
  поля и разный смысл.
- Родное поле «Артикул/SKU» платформы заполнять ТОЛЬКО артикулом производителя
  (`article`), если он пришёл; иначе оставлять пустым.
- `public_id` хранить в отдельном поле (редактируемом в админке), по нему —
  идемпотентность.
- При коллизии артикула с другим товаром — не перезаписывать (SKU уникален).

### 5.3 Трекинг качества изображений (обязателен)

- При загрузке фиксировать выбранный размер (`min|middle|max`) рядом с файлом.
- На повторном импорте: если доступен размер **лучше** записанного (например,
  появился токен) — перекачать и заменить, старый файл удалить; если не лучше —
  не трогать.
- Галерея идемпотентна **по имени файла** (подписи в URL меняются между
  запросами — по URL сравнивать нельзя).
- Без этого инварианта повторные импорты либо дублируют картинки, либо навсегда
  застревают в низком качестве.
- **Дедупликация медиа по контент-ключу.** Один файл встречается у многих товаров
  (и как обложка, и как логотип бренда/коллекции) — чтобы не качать повторно:
  извлечь из источника **стабильный контент-ключ** (эталон: в base64-префиксе
  `media_files` зашит `path` — неизменен между запросами и сменой токена;
  ключ = `hash(path # size)`), сохранить его на вложении и перед загрузкой искать
  существующее вложение по ключу — нашли переиспользуем (без скачивания), нет —
  качаем и записываем ключ. ⚠️ Общие (дедуплицированные) файлы НЕ удалять при
  замене качества — это сломает другие товары, ссылающиеся на тот же файл.

### 5.4 Порядок и зависимости

Внутри импорта одного товара строгий порядок:

```
payload → апдейт базовых полей (название, статус[только при создании], габариты)
          (цена НЕ задаётся по умолчанию — её в источнике нет, §5.6)
        → категории (создать/найти, привязать)
        → характеристики: резолв атрибута (ручной маппинг по specification_id ИЛИ
          по имени/создать); boolean → значение для true/false; привязать термины
        → сохранить объект (получить ID)
        → внешний ключ (public_id) в отдельное поле
        → бренд   (атрибут / таксономия / CPT по настройке)
        → страна  (атрибут / таксономия по настройке)
        → теги    (если вкл → таксономия тегов, поиск по имени)
        → обложка + галерея (трекинг качества + дедуп по контент-ключу)
        → коллекция (создать объект цели, привязать, сохранить поля)
        → событие «товар импортирован» (сайтовый слой дозаполняет остальные поля)
```

**Ручной маппинг характеристик (строгий режим).** Опционально вебмастер
сопоставляет «характеристика OneCatalog (`specification_id`) → существующий атрибут
платформы». При ВКЛ — импортируются ТОЛЬКО сопоставленные характеристики (прочие
пропускаются); ключ привязки — стабильный `specification_id`, а не переводимое
название. При ВЫКЛ — поведение «по имени»: найти атрибут по названию, иначе создать.
Для `boolean`-характеристик в маппинге задаются значения терма для `true` и `false`.

### 5.5 Устойчивость и логирование

- Сетевые/HTTP-ошибки одного запроса → `null`/skip, не исключение наверх.
- Один битый идентификатор в пакете помечается ошибкой и **не прерывает** пакет.
- Best-effort обогащение (например, доп. данные коллекции): при неудаче метку
  «обогащено» НЕ ставить, чтобы повторить позже.
- Лог последних N результатов (статус/имя/ошибка) для отображения прогресса.

### 5.6 Не выдумывать данные; нормализация значений

- **Чего нет в источнике — не синтезировать.** Если в API нет цены — НЕ
  генерировать «правдоподобную» (эталон: была формульная цена `f(id)` — убрана как
  вводящая в заблуждение). По умолчанию такое поле не задаётся; для подстановки из
  своего источника — фильтр/событие.
- **Поля «только при создании» не перезаписывать при апдейте.** Статус, цена и
  т.п. задаются новому объекту, но при переимпорте НЕ затираются — иначе теряются
  ручные правки вебмастера. (Идемпотентность ≠ «затереть всё заново».)
- **boolean → false по умолчанию.** Пустое / `null` / отсутствующее булево значение
  источника трактуется как `false` (берётся «значение для false»).
- **Конверсия единиц измерения (обязательна).** OneCatalog отдаёт величины в
  НАИМЕНЬШЕЙ единице: вес — граммы, размеры — миллиметры (подпись единицы
  локализована: «мм», «г»). Платформа хранит в СВОИХ единицах
  (WooCommerce: `woocommerce_weight_unit` / `woocommerce_dimension_unit` — кг/см и
  т.п., зависят от страны/настроек). Перед записью **конвертировать источник →
  единицы платформы** (универсальный конвертер: таблицы коэффициентов к базовой
  единице + нормализация локализованных подписей). Без этого 1200 мм превращаются в
  1200 см, а вес в граммах — в килограммы. Единицу источника брать из payload
  (`sizes.*_unit`), иначе мм/г; переопределяемо фильтром.

## 6. Фоновая очередь (обязательна для пакетов)

Зачем: выбранных товаров может быть много; синхронный импорт упрётся в таймаут
веб-запроса и память. **Ничего не должно грузиться всё сразу.**

- Список идентификаторов режется на **порции по «шагу импорта»** (минимум 10).
- Порции ставятся фоновыми задачами; обработчик импортирует элементы порции.
- UI ставит в очередь и **опрашивает статус** (pending/running + лог) до готовности.
- Фолбэк без планировщика: синхронная обработка с тем же отчётом.

| Платформа | Механизм очереди |
|---|---|
| WooCommerce/WP | **Action Scheduler** (встроен в WooCommerce), WP-Cron/async runner |
| Bitrix24 | Агенты, очередь `\Bitrix\Main\…`, или фоновые шаги через `CAgent` |
| PrestaShop | Cron-таск, Symfony Messenger, или AJAX-степпер (как в нативном импорте) |
| Кастом | Любая job-очередь (Redis/DB), worker |

## 7. Настройки (обязательный минимум)

| Настройка | Назначение | По умолчанию |
|---|---|---|
| Токен Wiki API | `X-API-Key` для запросов и виджета | пусто (env приоритетнее) |
| Язык товаров | `lang`; список из API (`en/ru/ar/zh/kk`) | `en` |
| Шаг импорта | размер порции очереди, **минимум 10** | 10 |
| Статус новых товаров | publish/pending/draft/private; **только при создании** | publish |
| Импортировать коллекции | вкл/выкл | вкл |
| Хранить коллекции как + цель | post type \| taxonomy + конкретная цель | post type |
| Импортировать бренд + хранить как | вкл; **атрибут \| таксономия \| CPT** + цель | вкл, таксономия (нативная брендов) |
| Импортировать страну + хранить как | вкл; **атрибут \| таксономия** + цель | выкл, атрибут |
| Импортировать теги | вкл/выкл (в таксономию тегов, поиск по имени) | выкл |
| Ручной маппинг характеристик | вкл/выкл; отдельная страница: характеристика→атрибут, для `boolean` — значения true/false (из термов атрибута) | выкл (иначе — по имени/создать) |

Требования: валидация и кламп значений; приоритет переменной окружения над
полем токена; язык — только из списка поддерживаемых. Для «справочных» сущностей
(коллекция/бренд/страна) — единый паттерн «тип объекта + цель»; UI скрывает
нерелевантные поля цели по выбранному типу.

## 8. Точки расширения (ключевой принцип переносимости)

Универсальное ядро публикует **события/фильтры**, через которые сайт дополняет
логику, НЕ форкая модуль. На WP это `do_action`/`apply_filters`; на других
платформах — события модуля/обсерверы/DI-хуки.

**Фильтры (меняют значение):** базовый URL, токен, аргументы запроса, язык,
список языков, шаг очереди, payload товара перед импортом, SKU, **цена** (по
умолчанию не задаётся), порядок размеров изображений, **дедупликация медиа**,
файлы галереи, **справочник характеристик**, флаги импорта (бренды/страна/теги/
маппинг характеристик) и карта маппинга, origin виджета.

**События (реагируют):**
- `before_import_product(payload, existing_id)`
- `product_imported(product_id, payload, report)` ← **сайт дозаполняет поля здесь**
  (ед. изм., площадь упаковки, вид товара, alt картинок, детали бренда, upsells — всё,
  чего нет в маппинге ядра; в событие передаётся ПОЛНЫЙ payload)
- `collection_imported(object_id, payload, result)`
- `brand_imported(object_id, payload, result)`
- `country_imported(object_id, payload, result)`
- `queue_enqueued(ids, batches, step)`

Эталон: оформительские мета-поля витрины (текстура, единица, рейтинг, привязки) и
прочие неиспользуемые ядром поля payload живут в **отдельном mu-плагине сайта** на
событии `product_imported` — сам импортёр о витрине ничего не знает. Это и есть
граница «универсальное ↔ проектное»: ядро переносит то, что отображается на
**таблицу маппинга** (§3), остальное — забота сайта через события.

## 10. Жизненный цикл и совместимость

- Заголовок/манифест модуля: версия, требования (CMS, движок магазина, версия
  языка), путь переводов, зависимость от движка магазина.
- **Версионирование — SemVer** (`MAJOR.MINOR.PATCH`: ломающее / фича / багфикс);
  вести **CHANGELOG** (Keep a Changelog). Поднимать версию **по вехам, по ходу**:
  каждая законченная веха сразу получает свой номер (фича → MINOR, фикс → PATCH) и
  датированную секцию в CHANGELOG — не копить большой объём под одним номером задним
  числом. Поднимать версию во **всех** местах манифеста; новые строки UI — переводить
  и пересобирать перевод (§9). Бэклог — секция «Не выпущено».
- **Миграция** опций/мета при смене схемы (перенос старых ключей, удаление
  устаревших, флаг «миграция выполнена»).
- Активация не должна падать без движка магазина; функции импорта проверяют его
  наличие.

## 12. Карта переноса: Bitrix24 / PrestaShop (кратко)

**Bitrix24 (1С-Битрикс, инфоблоки):**
- Товар → элемент инфоблока «Каталог»; `public_id` → `XML_ID` или внешнее
  свойство (идемпотентность по `XML_ID`).
- Характеристики → свойства инфоблока типа «список» (значения = `enum`).
- Категории → разделы инфоблока; бренд/страна → свойство-справочник/highload-блок
  ИЛИ свойство-список (выбор «тип объекта + цель» в настройках); теги → свойство.
- Коллекция → инфоблок/раздел/свойство (выбор в настройках модуля).
- Изображения → `DETAIL_PICTURE`/`PREVIEW_PICTURE` + множественное свойство-файл;
  скачивание `CFile::MakeFileArray` из временного файла.
- Очередь → агенты (`CAgent`) или пошаговый AJAX-импорт.
- Настройки → опции модуля (`COption`); локализация → `lang/`.

**PrestaShop (Symfony-модуль):**
- Товар → `Product`; `public_id` → `reference` или своё поле + идемпотентность.
- Характеристики → `Feature`/`FeatureValue` (или `Attribute` для вариаций).
- Категории → `Category`; бренд → `Manufacturer`.
- Коллекция → категория/feature/своя сущность (настройка).
- Изображения → `Image` + `ImageManager`/copy из временного файла.
- Очередь → cron-контроллер или Symfony Messenger; UI — AJAX-степпер.
- Настройки → `Configuration`; i18n — `trans()` + переводы модуля.
- События PrestaShop (`hook`) — аналог §8.

Во всех случаях **внешние контракты §2 и инварианты §5 неизменны** — меняется
только слой «как это лечь в конкретную CMS» (§3, §12).

## 13. Синхронизация цен и остатков (B2B-фид)

Отдельный сценарий поверх импорта каталога: периодически обновлять **цену и остаток**
товаров из **B2B-фида ритейлера**. Внедряется независимо от импорта (§1–§12).

### 13.0 Назначение и граница с импортом

- **Импорт каталога** переносит структуру/контент и **цену НЕ задаёт** (§5.6) — цена
  на совести вебмастера/фида.
- **Синхронизация цен/остатков** — это **живой фид**: она ВЛАДЕЕТ ценой и остатком
  синхронизируемых товаров и **перезаписывает** их при каждом запуске. Это не
  нарушение §5.6 (там про импорт), а отдельная авторитетная подсистема.
- Граница владения: если у товара есть оффер в фиде — ценой/остатком управляет синк;
  у кого оффера нет — синк их не трогает (см. 13.4 «обнуление»).

### 13.1 Внешний контракт B2B API

- **Эндпоинт:** `…/b2b/v1/retailer-share-products/{url_key}/?private_key=…`.
- **Авторизация ОТЛИЧАЕТСЯ от Wiki API:** ключ ритейлера в **пути** (`url_key`) +
  `private_key` в **query** (это НЕ заголовок `X-API-Key`). Хранить как отдельные
  настройки.
- **Пагинация** `?start&limit`; `meta.counts` — ОБЩЕЕ число (как в §2.1).
- **Формат ответа:**
  ```
  data.products.known   = { "<public_id>": [ offer, … ] }   // есть wiki-товар
  data.products.unknown = [ offer, … ]                       // без wiki
  data.warehouses       = { warehouse_id: {name, city, address, …} }
  data.regions          = { region_id: {slug, menutitle} }
  data.suppliers        = { supplier_id: {id, name} }        // справочник поставщиков
  meta: { counts, total }
  ```
- **offer:**
  ```
  name, code (код поставщика), public_id, status (bool),
  supplier_id,                                            // id поставщика (имя — в data.suppliers)
  products_stocks [ { warehouse_id, quantity } ],          // остатки по складам (часто дробные!)
  product_prices  [ { region_id, base_price, promo_price, purchasing_price } ]  // цены по регионам
  ```
  Справочники `regions`/`warehouses`/`suppliers` — верхнеуровневые; в оффере — только id
  (`supplier_id`, `region_id`, `warehouse_id`). `promo_price`/`purchasing_price` могут быть `null`.
  ⚠️ `base_price` — **розничная** цена (как есть на витрину; наценку не добавлять).
  `promo_price` = 0 → акции нет. `purchasing_price` (закупочная) — не на витрину.
  ⚠️ Поле `wiki` оффера содержит URL **с api-key Wiki в query** — не использовать,
  не логировать (утечка ключа).

### 13.2 Доменная модель и маппинг

| B2B-источник | Платформа | Правило |
|---|---|---|
| `known[public_id]` | товар по **тому же** внешнему ключу `public_id` (§5.1) | префетч карты `public_id→товар` |
| `unknown[].code` | товар по **коду поставщика** (отдельное поле кодов на товаре) | сопоставление только если код задан |
| `base_price` (регион по приоритету) | обычная цена | стратегия цены × приоритет регионов |
| `promo_price` (>0 и `< base`) | акционная цена | гард: акция только дешевле обычной |
| `products_stocks[].quantity` (Σ по складам, по всем поставщикам) | остаток (кол-во) | сумма; дробные поддержать |
| `code` + `supplier.id` | поле «коды поставщиков» товара | для будущего сопоставления |

- **Несколько офферов на один `public_id`** (разные поставщики): **остаток —
  сумма** по всем; **цена** — по стратегии (13.3).
- **Регион:** у большинства платформ **одна цена на товар** → берём приоритет
  регионов (первый регион с валидной ценой). Мультирегиональные цены — сайтовый слой
  (13.6), не ядро.

### 13.3 Стратегия цены (настройка)

Остаток всегда суммируется; стратегия управляет ТОЛЬКО ценой:
- **Приоритет поставщиков** — цена от самого приоритетного поставщика с валидной
  ценой (порядок настраивается, напр. drag-and-drop); фолбэк — минимальная.
- **Минимальная** `base_price` среди офферов.
- **Конкретный поставщик** — жёстко один; прочие офферы для цены игнорируются.

### 13.4 Обязательные инварианты

- **Change-detection (scan-and-diff) — КЛЮЧЕВОЕ для нагрузки.** Нельзя при каждом
  запуске перезаписывать все товары — это роняет сайт. Разделить на две фазы:
  1. **СКАН (дёшево):** прочитать фид, **в памяти** сравнить **сигнатуру** итоговых
     значений (цена+скидка+кол-во+статус) с сохранённой (префетч ключей и сигнатур
     одним запросом, **без загрузки объекта товара**). Совпала — пропустить.
  2. **ЗАПИСЬ (тяжело):** только изменившиеся товары — порциями в фоновую очередь;
     сохранение объекта товара вызывать ТОЛЬКО для них.
  Сигнатуру считать от **итоговых записываемых значений** (а не от сырого фида):
  тогда смена настроек (регион/стратегия/единицы) корректно вызывает переобновление,
  а одинаковый фид — нет. Хранить сигнатуру рядом с товаром.
- **Чтение дёшево, запись дозированно.** Никаких поштучных запросов в цикле; фид
  тянуть страницами; запись — фоном порциями, чтобы не блокировать веб-запросы.
- **Дробные остатки.** Количества бывают дробными (напр. м²) — поддержать (или
  округление вниз как опция); сырое значение хранить рядом.
- **Не выдумывать (§5.6).** Нет валидной цены для региона/оффера — **не ставить 0**,
  не трогать цену. Нет доступного оффера (`status=false` или сумма 0) → «нет в наличии».
- **Поля «только при создании» не трогать.** Статус публикации и т.п. — как в §5.6.
- **Устойчивость к изменению состава фида.** Справочники (регионы/поставщики/склады)
  со временем меняются (добавляются/исчезают). Требования:
  - приоритеты/выбор хранить по **стабильным id**, а не по позициям — тогда настройки
    не «слетают» при изменениях; исчезнувший id безвреден (просто не находится);
  - **новые элементы НЕ применять автоматически** (не давать им влиять на цену по
    умолчанию) — помечать как «новое, не настроено» (низший приоритет), пока вебмастер
    их не настроит;
  - вести **два снимка** справочника: «подтверждённый» (то, что настроено) и
    «последний из фида»; их разница = новые/неподтверждённые элементы; подтверждение —
    действие вебмастера (сохранение настроек / «загрузить и проверить»);
  - при появлении новых элементов **уведомлять администратора** (баннер в админке +
    письмо, один раз на новый состав); при **ошибке выгрузки** — письмо (с троттлингом).
    Уведомления — переключаемы.

### 13.5 Ненайденные товары (настройка вариативности)

- **`known`, которых нет в магазине:** либо **дозагрузить штатным импортом каталога**
  (§6 — у них есть `public_id`), либо пропустить.
- **`unknown` (без wiki)** — три режима (выбор вебмастера):
  1. **Пропустить** (рекомендуется по умолчанию) — игнорировать.
  2. **Отстойник (staging) — ручной отбор.** Несопоставленные `unknown` НЕ создаются
     сразу, а складываются в отдельное хранилище (своя таблица: код, payload оффера,
     статус `new|imported|ignored`, first/last seen). Вебмастер на отдельной странице
     просматривает их (с ценой/остатком, посчитанными теми же резолверами §13.3) и
     **выбирает, что импортировать**, что — **игнорировать** (больше не предлагать).
     Импорт из отстойника создаёт товар + фиксирует код → далее обновляется обычным
     синком по коду (статус `imported`). Это «шлюз» против засорения каталога
     случайными позициями поставщика.
  3. **Авто-создание всех** — создать товар и зафиксировать код сразу (на свой риск).
- **Сопоставление по коду** первично во всех режимах: если `unknown` уже сопоставлен
  с товаром по коду поставщика — это обычное обновление цены/остатка (change-detection),
  а не повторное создание/отстойник.
- **Поле кодов поставщиков** на товаре обязательно, если поддерживаем сопоставление
  по коду: редактируемое (вебмастер привязывает коды вручную) + автозаполняемое синком.

### 13.6 Расписание

- **Ручной запуск** + прогресс/лог (как §6).
- **Авто-синхронизация по расписанию** (интервалы + для «ежедневно» выбор времени в
  часовом поясе сайта). Снимать расписание при деактивации модуля.

### 13.7 Точки расширения (для сайтового слоя — раскладка по регионам/складам)

Ядро пишет ОДНУ цену и ОДИН суммарный остаток. Всё остальное — через события/фильтры,
чтобы сайт мог разложить **цены по регионам** и **остатки по складам** в свои поля
(напр. ACF), не форкая модуль:
- фильтр на **сырые данные фида** при чтении;
- фильтры на цену/остаток/офферы перед записью;
- **фильтр сигнатуры change-detection** — ⚠️ обязателен, если сайт раскладывает весь
  payload: иначе изменения в неосновных регионах/складах не вызовут запись и
  сайтовые поля устареют;
- события **до/после записи**, в которые передаются **сырые офферы** + справочники
  `regions`/`warehouses`/`suppliers` (чтобы разложить по регионам/складам/поставщикам).

### 13.8 Чек-лист «синхронизация цен/остатков на новой платформе»

- [ ] Клиент B2B API: `url_key` (путь) + `private_key` (query), пагинация, ошибки → skip.
- [ ] `known` по `public_id` (тот же ключ §5.1); `unknown` по коду поставщика (поле кодов).
- [ ] Цена: стратегия (приоритет/мин/конкретный) × приоритет регионов; `promo` → акция
      (гард `< base`); `base_price` как розничную, без наценки.
- [ ] Остаток: сумма по складам/поставщикам; дробные поддержать; нет оффера → не трогать.
- [ ] **Change-detection:** сигнатура итоговых значений; обновлять ТОЛЬКО изменившееся;
      скан без загрузки объектов, запись порциями в фоне.
- [ ] Ненайденные: known → штатный импорт / skip; unknown → skip / **отстойник
      (ручной отбор)** / авто-создание+код. Отстойник: хранилище + страница отбора
      (импортировать / игнорировать), статусы `new|imported|ignored`.
- [ ] Расписание: ручной + авто (интервал/время), снятие при деактивации.
- [ ] **Изменение состава фида:** приоритеты по стабильным id; новые элементы не
      применять автоматически (пометка «новое»); два снимка (подтверждённый/из фида);
      уведомление администратора при изменении состава и при ошибке выгрузки (переключаемо).
- [ ] **Журналы:** история запусков синка (счётчики/длительность) и журнал очереди
      импорта; ручной прогон, когда фоновый планировщик не срабатывает.
- [ ] Точки расширения: сырые офферы + regions/warehouses/suppliers в события; фильтр сигнатуры.
- [ ] i18n новых строк; владение ценой/остатком задокументировать.
