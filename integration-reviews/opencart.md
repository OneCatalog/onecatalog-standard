# Интеграционный план: OpenCart

Реализация стандарта **OneCatalog Import** (`INTEGRATION-STANDARD.md`) на OpenCart
3.x / 4.x. Эталон — WooCommerce; ниже — перенос инвариантов §2/§5 на реальные
механизмы OpenCart (MVC(L), таблицы `oc_*`, event system, OCMOD/extension, cron).

> Допущения о платформе (зафиксированы, чтобы не выдумывать): целимся в **OpenCart
> 3.0.3.x и 4.0.2.x** одновременно — это две заметно разные кодовые базы (в 4.x
> namespaces, `Registry`, маршрут `extension/<ext>/<type>/<name>`, иной формат
> расширения, изменены имена событий). Поэтому модуль фактически = **два таргета**
> с общим ядром-логикой (см. §9 «Жизненный цикл»). Префикс таблиц по умолчанию
> `oc_`; в коде только через `DB_PREFIX`. Движок каталога в OpenCart встроенный
> (отдельного «движка магазина», как WooCommerce поверх WP, нет) — проверка §10
> «активация без движка магазина» вырождается в проверку версии OpenCart.

## Маппинг сущностей и реализация

Сводная таблица соответствий (§3 стандарта → реальные таблицы/модели OpenCart).
Все имена таблиц — без префикса; в коде `DB_PREFIX`.

| OneCatalog | Абстракция | OpenCart: хранилище | Модель/механизм |
|---|---|---|---|
| product | Товар | `product` + `product_description` (по `language_id`) + `product_to_store` + `product_to_layout` | `catalog/product` admin-model (3.x) / `Cart\...`/admin model (4.x) |
| public_id | Внешний ключ | **своя таблица** `oc1c_product_map (product_id, public_id, source, quality_json, created, updated)` + дубль в `product.sku`-фолбэк для легаси | прямой SQL в своей model |
| article | Артикул произв. | `product.model` (обязательное поле!) + опц. `product.sku` | см. §«Артикул» ниже |
| options[] text | Характеристика | `attribute` + `attribute_description` (по языку) + `attribute_group`/`attribute_group_description` + `product_attribute (product_id, attribute_id, language_id, text)` | характеристика хранится **текстом по языку**, не «термин» |
| options[] boolean | Характеристика-флаг | то же `product_attribute.text` = строка «Да»/«Нет» (значения true/false из маппинга) | нет нативного типа boolean |
| options[] numeric | Характеристика-число | то же `product_attribute.text` = число + (опц.) единица | текстом |
| categories[] | Категория | `category` + `category_description` + `category_path` + `product_to_category` | дерево; нужен пересчёт `category_path` |
| brand | Бренд | `manufacturer` (+ `manufacturer_to_store`); опц. как атрибут/категория (настройка) | нет `*_description` — бренд НЕ мультиязычен (см. блокер B1) |
| country | Страна происх. | **атрибут** `attribute` (рекоменд.) ИЛИ категория-ветка (настройка). НЕ таблица `country` (это страны доставки!) | см. блокер B2 |
| collections[] | Коллекция | **категория-ветка** «Коллекции/{Бренд}» (рекоменд.) ИЛИ атрибут ИЛИ кастом-таблица (настройка) | нет CPT/таксономий (блокер B3) |
| tags[] | Теги | `product_description.tag` (CSV-строка по языку) + опц. `product.meta_keyword` | нет таблицы тегов; «поиск по имени» вырождается |
| images_urls (обложка) | Медиа-обложка | `product.image` (путь относительно `image/catalog/...`) | файл в `DIR_IMAGE` |
| files[] (галерея) | Медиа-галерея | `product_image (product_id, image, sort_order)` | дедуп по имени файла |
| measurement_unit | Ед. измерения | **нет аналога**; через событие + опц. атрибут | сайтовый слой |
| sizes.weight/+unit | Вес | `product.weight` + `product.weight_class_id` | конвертер → `weight_class` (§«Единицы») |
| sizes.length/width/height | Габариты | `product.length/width/height` + `product.length_class_id` | конвертер → `length_class` |
| areas_per_package, product_shape, upsells, alt картинок… | прочее | событие `oc1c.product.imported` (сайтовый слой) | граница «универсальное↔проектное» |

### Слои (перенос §4 на OpenCart)

OpenCart MVC(L) не даёт класс-слоёв «из коробки», но library-классы в
`system/library/` (3.x) / autoload-namespace (4.x) это позволяют. Раскладка:

```
admin/controller/extension/module/onecatalog_import.php   — настройки + кнопка + REST-приём (Rest+AdminUI+Settings)
admin/model/extension/module/onecatalog_import.php        — все find-or-create + SQL карты соответствий
admin/view/template/extension/module/onecatalog_import.twig
admin/language/{en-gb,ru-ru}/extension/module/onecatalog_import.php
system/library/onecatalog/Api.php        — клиент Wiki API (base URL/token/lang/GET/обёртка)
system/library/onecatalog/Units.php      — конвертер источник→weight_class/length_class
system/library/onecatalog/Media.php      — скачивание/MIME/обложка-галерея/качество/дедуп
system/library/onecatalog/Taxonomies.php — attribute/attribute_group/category find-or-create по ИМЕНИ
system/library/onecatalog/CollectionImporter.php
system/library/onecatalog/BrandImporter.php
system/library/onecatalog/CountryImporter.php
system/library/onecatalog/ProductImporter.php  — оркестратор (вызывает остальные через Registry)
system/library/onecatalog/Queue.php      — порции + cron-обработчик + лог в своей таблице
catalog/controller/extension/module/onecatalog_cron.php   — cron-точка обработки порций
install.xml (OCMOD) ИЛИ extension-install (4.x) — внедрение кнопки на страницу товаров, схема таблиц
```

`ProductImporter` получает `Registry` (доступ к `db`, `config`, `load->model`,
`event`) и оркестрирует остальные library-классы — прямой аналог DI эталона.
Циклов нет: `Queue` → `ProductImporter` → `Api/Media/Taxonomies/*Importer`.

### Идемпотентность (§5.1) — критично

OpenCart **не имеет** аналога WP-меты на произвольном объекте. Решение —
**своя таблица** `oc1c_product_map`:

```sql
CREATE TABLE `oc1c_product_map` (
  `product_id` INT NOT NULL,
  `public_id`  VARCHAR(64) NOT NULL,
  `source`     VARCHAR(32) NOT NULL DEFAULT 'onecatalog',
  `quality`    TEXT NULL,          -- JSON: {image_path: 'max', ...} трекинг качества (§5.3)
  `date_added` DATETIME, `date_modified` DATETIME,
  PRIMARY KEY (`public_id`,`source`),
  UNIQUE KEY (`product_id`,`source`)
);
```

- `find_by_external_id(public_id)` = `SELECT product_id FROM oc1c_product_map WHERE public_id=? AND source=?`.
- public_id показывать/редактировать в админке — добавить **вкладку/поле** на
  карточку товара через OCMOD-вставку в `product_form.twig` + сохранять в свою
  таблицу через event `admin/model/catalog/product/editProduct/after` (см. §«События»).
- **Справочники find-or-create по ИМЕНИ (label), не по слагу** (§5.1):
  для атрибутов/категорий/брендов матчить по `*_description.name` (или
  `manufacturer.name`) `LOWER(name)=LOWER(?)`. У OpenCart нет автогенерации slug
  у этих сущностей (slug = только SEO-keyword в `seo_url`), так что риск «разных
  транслитов» ниже, но матч по имени всё равно обязателен (ручные дубли).
- Легаси-фолбэк: если public_id раньше клали в `product.sku` — при отсутствии в
  карте искать `WHERE sku=?`, найдя — записать в карту и продолжить (миграция).

### Артикул / SKU (§5.2)

Конфликт с моделью OpenCart: поле **`product.model` ОБЯЗАТЕЛЬНО** (NOT NULL,
валидируется в админ-форме), а `product.sku` — необязательно и не уникально на
уровне БД.

- `article` (артикул производителя) → писать в `product.model` если пришёл,
  иначе в `product.model` положить **fallback** (нельзя оставить пустым — форма
  не сохранит; кандидат: сам `public_id` как технический model, но это нарушает
  §5.2 «public_id отдельно»). Рекомендация: `product.model` = `article` ?: `public_id`
  (технический), `product.sku` = `article` ?: '' , public_id всегда в своей карте.
  → **Это компромисс, см. блокер B4.**
- `product.sku` НЕ использовать как ключ идемпотентности (нестабилен, не уникален).
- Коллизия артикула: SKU не уникален в OC, перезапись допустима, но по стандарту
  «не перезаписывать» — проверять занятость `model`/`sku` другим product_id и
  при коллизии не трогать.

### Характеристики (§5.4) — text по языкам, НЕ термины

Ключевое отличие от Woo: в OpenCart характеристика = `product_attribute.text`
(строка, по `language_id`), **значения не нормализуются в термины/enum**. Поэтому:

- «Поиск термина по имени» не нужен — нужен **поиск `attribute` (определения)
  по имени** в `attribute_description`, иначе создать (`attribute` +
  `attribute_description` на ВСЕ языки магазина + привязка к `attribute_group`).
- Значение пишем как текст: `INSERT product_attribute (product_id, attribute_id,
  language_id, text)`. Для импорта в один язык — заполнять только текущий
  `lang`-язык; для остальных языков магазина — пустая строка или дубль (настройка).
- `attribute_group` обязателен (attribute без группы не отображается на витрине) —
  завести группу по умолчанию «Характеристики» (find-or-create по имени), либо
  маппить `specification` → group по `product_shape`/категории (расширение).
- **boolean** (§2.2/§5.6): `specification_option_name=null`, значение в
  `bool_option`. Маппинг задаёт строки для true/false (по умолч. «Да»/«Нет»,
  локализуемо); отсутствие/null/false → строка «Нет».
- **numeric**: `numeric_option` + (опц.) единица из `specification_label` → текст.
- **Ручной маппинг (строгий режим, §5.4):** таблица `oc1c_spec_map
  (specification_id, attribute_id, true_text, false_text)`. При ВКЛ импортировать
  ТОЛЬКО сопоставленные `specification_id`; ключ — стабильный `specification_id`,
  не переводимое название. UI маппинга — отдельная вкладка настроек (список
  спек из `/specifications/` ↔ select существующих `attribute`).
- **Альтернатива «option вместо attribute»**: OC `option`/`product_option`
  предназначены для вариаций влияющих на цену/склад/выбор покупателем — для
  справочных характеристик это неверно (плодит варианты, ломает корзину). Поэтому
  **только `attribute`**, `option` НЕ использовать (зафиксировано).

### Категории (§5.4)

- find-or-create по имени в `category_description.name` (по языку), учёт
  иерархии если категории придут деревом (сейчас `categories[]` плоский — все
  как top-level либо под общим родителем «Каталог», настройка).
- После вставки/перепривязки — **пересчитать `category_path`** (OpenCart хранит
  материализованный путь предков) и обновить `product_to_category`. Это легко
  забыть → категория не покажется в дереве. Использовать admin-model
  `catalog/category` (она пересчитывает path), не голый INSERT.

### Бренд / Страна / Коллекция — «тип объекта + цель» (§3/§7)

OpenCart беднее Woo по «справочным» сущностям. Реализация настройки
«тип объекта + цель» сводится к набору {`manufacturer` | `attribute` | `category`}:

- **Бренд** по умолчанию → `manufacturer` (find-or-create по `manufacturer.name`),
  `product.manufacturer_id`. Альтернативы: атрибут или категория. ⚠️ `manufacturer`
  **не мультиязычен** — description бренда в один глобальный текст (блокер B1).
- **Страна** по умолчанию → **атрибут** «Страна происхождения» (значение текстом).
  НЕ путать с таблицей `country` (страны для адресов доставки — туда писать нельзя).
  Альтернатива: категория-ветка. (блокер B2)
- **Коллекция** по умолчанию → **категория-ветка** (родитель «Коллекции», ребёнок =
  бренд, лист = коллекция) — даёт навигацию/фасет «из коробки». Альтернативы:
  атрибут (теряет ссылки `link_3d`/`link_official_site`) или **кастом-таблица**
  `oc1c_collection` (полный контроль, но нужен свой контроллер витрины). Поля
  коллекции (`link_3d`, `link_official_site`, изображения) при «категория»-режиме
  складывать в `category_description.description` (HTML) или доп. SEO-поля. (блокер B3)

### Теги (§3)

OpenCart **не имеет** таблицы тегов и таксономии тегов. Есть лишь
`product_description.tag` — **CSV-строка по языку** (используется только во
встроенном поиске). Поэтому:

- «Импорт тегов в таксономию тегов, поиск по имени» → деградирует до: собрать
  имена тегов в строку, записать в `product_description.tag` (CSV). Поиск/дедуп
  по имени внутри строки тривиален, но это не сущность (нет страниц тегов,
  фильтрации). Опц. дублировать в `product.meta_keyword`. (см. блокер B5)

### Медиа (§2.3/§5.3)

- URL без расширения → как и в эталоне: **скачивать вручную** (`curl`/stream),
  определять MIME по содержимому (`finfo_buffer`), писать файл в
  `DIR_IMAGE.'catalog/onecatalog/<хеш>.<ext>'`, в БД хранить путь относительно
  `image/` (OpenCart так хранит в `product.image`/`product_image.image`).
- Размер `min|middle|max`: выбирать `max` при наличии токена, иначе `min`.
- **Трекинг качества**: записанный размер хранить в `oc1c_product_map.quality`
  (JSON по имени файла). На повторе: если доступен лучший размер — перекачать,
  заменить файл, старый удалить (если не дедуплицирован).
- **Галерея идемпотентна по имени файла** (`product_image`), не по URL (подписи
  в URL меняются).
- **Дедуп по контент-ключу**: вынуть `path` из base64-префикса `media_files`,
  ключ = `hash(path # size)`. Хранить в своей таблице
  `oc1c_media (content_key, image_path)`; перед загрузкой искать по ключу —
  нашли переиспользовать путь (без скачивания), нет — качать и записать.
  ⚠️ Общие файлы НЕ удалять при апгрейде качества (сломает другие товары) —
  считать ссылки/`source` перед удалением.
- ⚠️ В OpenCart нет «вложений»-объектов (как WP attachments) — медиа это просто
  файлы + строки-пути. Reuse реализуется только своей таблицей `oc1c_media`.
  Нет авто-генерации превью при импорте — OC генерит ресайзы лениво при выводе
  (`image->resize`), так что доп. действий не нужно, но и контроля размеров меньше.

### Единицы (§5.6) — конвертер с учётом weight_class/length_class

OpenCart хранит вес/габариты в **настраиваемых классах** с таблицами
коэффициентов: `weight_class`/`weight_class_description` (`unit`,`value`) и
`length_class`/`length_class_description` (`value`). Поле `*.value` = множитель
к **базовому** классу (у базового value=1). Конвертер:

1. Источник OneCatalog: вес в **граммах**, размеры в **миллиметрах** (подпись
   локализована: «мм»/«г» — нормализовать подпись из `sizes.*_unit`, иначе мм/г).
2. Целевой класс магазина: настройка модуля «целевой `weight_class_id` /
   `length_class_id`» (по умолчанию — базовый класс магазина).
3. Конверсия: `value_target = value_source_in_base * (1 / target_class.value)`,
   где `value_source_in_base` = граммы→базовая_ед_веса, мм→базовая_ед_длины через
   собственную таблицу коэффициентов (г, кг, мм, см, м, дюйм…). Т.е. **два этапа**:
   источник→СИ-базовая→класс магазина. Нельзя слепо записать число — иначе
   1200 мм станут 1200 см.
4. Писать `product.weight`, `product.weight_class_id`, `product.length/width/height`,
   `product.length_class_id`.

> ⚠️ Тонкость: классы единиц в OpenCart **создаёт администратор** и их набор
> произволен (может не быть «мм» или «г»). Конвертер не должен полагаться на
> наличие конкретного класса — он работает через value-множители к базовому
> классу магазина, а собственную таблицу г/мм держит независимо. Если нужного
> целевого класса нет — настройка указывает, в какой существующий класс писать.

### Идемпотентность справочников и порядок (§5.4)

Порядок внутри импорта одного товара — как в стандарте: базовые поля
(статус только при создании) → категории → характеристики → save (получить
`product_id`) → запись `public_id` в карту → бренд → страна → теги → обложка+галерея
→ коллекция → событие `oc1c.product.imported`. Save товара — через admin-model
`catalog/product` (она консистентно пишет во все связанные таблицы:
`product`, `product_description`, `product_to_store`, `product_to_category`,
`product_attribute`, `product_image`, `seo_url`), плюс свои `oc1c_*`-таблицы
прямым SQL.

## Фоновая очередь, медиа, настройки, i18n, жизненный цикл

### Фоновая очередь (§6) — у OpenCart нет Action Scheduler

OpenCart НЕ имеет встроенной очереди задач. Механизмы:

- **OpenCart Cron** (есть в 3.0.3+/4.x: таблица `cron`, контроллер
  `catalog/controller/.../cron`, запуск по URL/системному cron) — основной путь.
  Свои таблицы:
  ```sql
  oc1c_queue_batch (batch_id PK, status[pending|running|done|error], created, started, finished)
  oc1c_queue_item  (item_id PK, batch_id, public_id, status, message, processed_at)
  ```
- REST-приём (контроллер админки) режет список public_id на **порции по шагу
  (минимум 10)**, пишет в `oc1c_queue_*`, ставит cron-задачу.
- Обработчик (cron-контроллер) берёт одну порцию pending → импортирует элементы →
  пишет статус/лог. Идемпотентность гарантирует безопасный повтор порции.
- UI **опрашивает статус** AJAX-ом (контроллер `getStatus` читает `oc1c_queue_*`).
- **Фолбэк без системного cron**: AJAX-степпер — фронт сам дёргает «обработать
  следующую порцию» в цикле (как нативный импорт PrestaShop). Это и есть
  «синхронный фолбэк с тем же отчётом». ⚠️ OpenCart-cron сам по себе не
  «фоновый» демон — он срабатывает по внешнему триггеру (системный crontab/визит),
  поэтому AJAX-степпер часто оказывается основным, а не фолбэком (блокер B6).

### Настройки (§7) — таблица `setting`

Хранение через таблицу `setting` (key/value по `store_id`, JSON для массивов),
group = `module_onecatalog_import`. Доступ — `$this->config->get(...)` /
admin-model `setting/setting`. Минимум настроек из §7:

| Настройка | Ключ | По умолчанию |
|---|---|---|
| Токен Wiki API | `..._token` (env `ONECATALOG_TOKEN` приоритетнее) | пусто |
| Язык товаров | `..._lang` (из `en/ru/ar/zh/kk`) | `en` |
| Шаг импорта | `..._step` (кламп ≥10) | 10 |
| Статус новых | `..._new_status` (0/1 → `product.status`; OC имеет только enabled/disabled!) | 1 |
| Импортировать коллекции + тип+цель | `..._collections`, `..._collection_target_type`, `..._collection_target_id` | вкл, category |
| Бренд + тип+цель | `..._brand*` | вкл, manufacturer |
| Страна + тип+цель | `..._country*` | выкл, attribute |
| Теги вкл/выкл | `..._tags` | выкл |
| Маппинг характеристик вкл/выкл | `..._spec_map_enabled` | выкл |
| Целевой weight_class/length_class | `..._weight_class_id`, `..._length_class_id` | базовые классы магазина |

⚠️ **Статус товара в OpenCart — только bool** (`product.status` enabled/disabled).
Маппинг §7 «publish/pending/draft/private» вырождается до enabled/disabled — нет
аналогов pending/private (блокер B7). Валидация/кламп — в контроллере перед save.
Приоритет env-токена — читать `getenv('ONECATALOG_TOKEN')` в `Api`.

### i18n (§9)

- Исходные строки — **EN** в `admin/language/en-gb/.../onecatalog_import.php`
  (массив `$_[...]`), перевод RU в `admin/language/ru-ru/...`. Это нативный
  механизм OpenCart (вместо POT/PO/MO Woo — соответствие в стандарте §9 «родной
  механизм платформы»).
- Каталог-сторона (cron-логи, значения «Да/Нет» для boolean) — в
  `catalog/language/...`.
- Строки фронтовых JS (admin-import.js, picker) — **отдавать из контроллера**
  локализованными (через `$data` в Twig / `data-*`), не хардкодить в JS (§9).
- ⚠️ В OpenCart язык в БД = `language_id` (число), а `lang` API = код
  (`ru`/`en`). Нужна **карта** код-API ↔ `language_id` магазина (настройка/
  автоопределение по `language.code`). Если в магазине нет языка из API —
  писать в дефолтный язык магазина (блокер B8).

### События / точки расширения (§8) — слабая модель OpenCart

OpenCart event system (`event` таблица + триггеры `model.*`/`controller.*`/
`view.*`) — есть, но публиковать **произвольные доменные события**
(`product_imported` с кастомным payload) штатно нельзя так же гибко, как WP
`do_action`. Решение:

- Использовать `$this->event->trigger('oc1c.product.imported', [&$args])` —
  OpenCart 3.x/4.x позволяет тригерить **кастомные имена событий**, на которые
  сторонние расширения подписываются через таблицу `event`. Это рабочий аналог
  `do_action`, хотя и менее известный/документированный.
- Аналог `apply_filters` (изменение значения) — передавать аргумент **по ссылке**
  в массиве (`['payload' => &$payload]`); подписчик меняет на месте. OpenCart
  events это поддерживают (массив аргументов по ссылке).
- Минимальный набор (имена в неймспейсе `oc1c.*`): `oc1c.before_import_product`,
  `oc1c.product.imported` (полный payload + product_id + report — сайт дозаполняет
  ед.изм., areas_per_package, product_shape, alt картинок, upsells),
  `oc1c.collection.imported`, `oc1c.brand.imported`, `oc1c.country.imported`,
  `oc1c.queue.enqueued`. Фильтры (по ссылке): base URL, токен, args, lang, step,
  payload, sku, price (по умолч. не задаётся), порядок размеров картинок, дедуп
  медиа, файлы галереи, справочник характеристик, флаги/карта маппинга, origin
  виджета.
- Оформительские поля витрины (как mu-плагин в эталоне) → отдельное
  **подписчик-расширение** на `oc1c.product.imported`, регистрируется в `event`.
  Ядро о витрине не знает. ⚠️ Но: чтобы подписаться, расширению нужны строки в
  `event`-таблице — это менее эргономично, чем `add_action` в коде (блокер B9).

### Виджет-пикер (§2.4)

- Кнопку «Импорт из OneCatalog» вставить на admin-страницу товаров
  (`catalog/product`) через **OCMOD** (`<file path=".../product_list.twig">` +
  `<add>`) или через event `view` (4.x) — модификация шаблона списка. В 4.x
  допустимо через extension-механизм.
- По клику — `<iframe>` на `https://tools.onecatalog.net/picker.html?token=…&parentOrigin=…`,
  sandbox `allow-scripts allow-forms allow-same-origin`, origin задавать явно,
  проверять `event.origin`.
- `postMessage` → `{type:'ONECATALOG_SELECTED', productPublicIds}` →
  импорт **по publicIds** (не числовым id). Альтернатива пикеру — textarea со
  списком public_id (контракт §2.4 «можно по списку»).
- ⚠️ OpenCart admin часто за HTTP Basic / в подпапке / с CSP — origin/iframe
  настройки и `user_token` в URL учитывать (CSRF в OC = `user_token` в query,
  не nonce; защита REST-приёма — проверка `user_token` + прав группы).

### Установка / обновление (§10)

- **3.x**: дистрибутив = OCMOD-zip (`install.xml` + файлы), ставится через
  «Модификаторы»/Marketplace. Схему таблиц `oc1c_*` создавать в
  `install()`-методе контроллера (срабатывает при установке модуля в
  «Расширения»). OCMOD внедряет кнопку в шаблон.
- **4.x**: формат расширения иной (`install.json`, иные пути
  `extension/<name>/...`), OCMOD-движок переработан. Фактически нужна **отдельная
  сборка** под 4.x (общая логика library-классов, разные обвязки) — см. блокер B10.
- Версионирование SemVer + CHANGELOG (Keep a Changelog), как в эталоне; версия в
  манифесте(ах) расширения; новые UI-строки переводить в `ru-ru` перед сборкой.
- **Миграция** опций/схемы: версия схемы в `setting` (`..._schema_version`),
  `install`/`upgrade` сравнивает и докатывает (ALTER, перенос легаси public_id
  из `product.sku` в `oc1c_product_map`).

### Права (§«права»)

- Доступ к контроллеру модуля — стандартная проверка OpenCart:
  `$this->user->hasPermission('modify', 'extension/module/onecatalog_import')` +
  `access`. Добавить право группе при установке.
- REST-приём идентификаторов — тот же контроллер админки (под `user_token` +
  permission), не публичный. Cron-контроллер каталога — защитить токеном cron
  (`cron` таблица/секрет в URL), т.к. он в публичном `catalog/`.

## 🚫 Сложно / невозможно на OpenCart

**B1. Бренд не мультиязычен.** Таблица `manufacturer` не имеет
`manufacturer_description` — есть только глобальные `name`/`image`. Описание
бренда, `foundation_year`, `website`, `country`, локализованное имя — некуда
положить штатно. → **Обход:** базовое имя в `manufacturer.name` (одно на все
языки); богатые поля бренда — в кастом-таблицу `oc1c_brand_meta` и выводить через
своё расширение/событие, либо отдать целиком сайтовому слою через
`oc1c.brand.imported`.

**B2. Нет сущности «страна происхождения товара».** Таблица `country` — это
страны доставки/адресов, писать туда нельзя. Нет нативного «country of origin».
→ **Обход:** атрибут «Страна происхождения» (текст по языку) или категория-ветка
«Страны». Фасетная фильтрация по стране — только если магазин/тема умеет
фильтр по атрибутам (нативно слабо).

**B3. Нет CPT / таксономий — коллекция негде «жить» нативно.** В Woo коллекция =
CPT или таксономия с собственными страницами/архивами. В OpenCart выбор беднее:
категория (засоряет дерево, но даёт страницу/навигацию), атрибут (нет страницы,
теряет `link_3d`/`link_official_site`), либо **кастом-таблица + свой
контроллер/маршрут/SEO** (чисто, но это фактически новый раздел витрины —
много работы). → **Обход:** дефолт «категория-ветка», богатые поля коллекции в
`category_description.description`; для полноценных страниц коллекций — отдельный
кастом-модуль (вне ядра импорта, через событие).

**B4. `product.model` обязателен, `sku` не уникален.** Модель OpenCart жёстко
требует непустой `model` и не даёт уникального бизнес-ключа на уровне схемы под
`public_id`. → **Обход:** своя таблица-карта под public_id (B-решение в маппинге);
`model` заполнять `article` или техническим `public_id`-производным. Уникальность
public_id гарантирует только своя таблица (`UNIQUE`), не схема OC.

**B5. Нет таксономии тегов.** Теги в OC — лишь CSV-строка
`product_description.tag` во встроенном поиске; нет страниц тегов, нет
фильтрации, нет «сущности тега» для дедупа/связей. → **Обход:** писать имена в
CSV-строку (+ meta_keyword); полноценные теги = кастом-таблица + контроллер
(вне ядра).

**B6. Нет фоновой очереди из коробки.** Нет аналога Action Scheduler/демона.
OpenCart-cron срабатывает по внешнему триггеру (системный crontab или визит на
cron-URL), не сам по себе. → **Обход:** свои таблицы очереди + AJAX-степпер как
ОСНОВНОЙ механизм (фронт держит прогресс, дёргает порции), системный cron — как
опциональный «настоящий фон» если хостинг позволяет. «Фоновость» в общем случае
условна — честно отразить в UI («не закрывайте вкладку» при степпере).

**B7. Статус товара только enabled/disabled.** Нет pending/draft/private из §7.
→ **Обход:** маппинг статуса вырождается в bool (новый товар enabled или
disabled). Если нужен «черновик» — только disabled (скрыт), без промежуточных
состояний.

**B8. Язык как `language_id` (число), не код.** API оперирует кодами
(`ru/en/…`), OC — числовыми `language_id` и отдельными `*_description`-строками
на каждый язык. → **Обход:** карта код↔`language_id` (авто по `language.code` +
настройка); при отсутствии языка из API в магазине — писать в дефолтный язык.
Полноценная мультиязычность товара требует N запросов к API (по языку) — см.
вопрос 6.

**B9. Событийная модель слабее `do_action`/`add_action`.** Кастомные доменные
события тригерятся (`$this->event->trigger('oc1c.*')`), но подписка идёт через
строки в таблице `event` (надо ставить расширение-подписчик), а не лаконичным
`add_action` в коде темы. Передача «фильтров» — только через аргумент по ссылке.
→ **Обход:** документировать имена событий и регистрировать подписчиков
расширениями; это работает, но эргономика и открытость ниже эталона. Сайтовый
слой = отдельное расширение-подписчик (аналог mu-плагина), не «пара строк».

**B10. Две несовместимые платформы 3.x vs 4.x.** Namespaces, `Registry`,
маршрутизация `extension/...`, формат расширения (`install.json`), переработанный
OCMOD/event-движок, изменённые имена системных событий — 3.x и 4.x фактически
разные таргеты. → **Обход:** общее ядро в platform-agnostic library-классах
(`system/library/onecatalog/*` с минимумом зависимостей от версии), две тонкие
обвязки (controller/model/install) и две сборки. Это удваивает работу обвязки и
тестирования (отразить в оценке).

**B11. Нет медиа-объектов (вложений).** В OC нет сущности «attachment» с метой —
только файлы и строки-пути. Reuse/дедуп/трекинг качества (§5.3) нельзя повесить
на объект медиа. → **Обход:** всё через свои таблицы `oc1c_media`
(content_key→path, счётчик ссылок) и `oc1c_product_map.quality`. Без них §5.3
невыполним. Удаление общих файлов — только после проверки счётчика ссылок.

**B12. Save товара завязан на admin-model и валидацию формы.** Чистый
программный импорт лучше делать через admin `model/catalog/product`
(`addProduct`/`editProduct`) ради консистентности связанных таблиц и
`category_path`/`seo_url`, но эти методы тянут ожидания формата формы (массивы
`product_description[lang_id][...]`, `product_category[]`, `product_attribute[]`).
→ **Обход:** собирать data-массив в формате формы и звать admin-model; либо
писать таблицы напрямую, но тогда вручную поддерживать `category_path`, `seo_url`,
`product_to_store`. Рекоменд. — через admin-model (меньше дыр), приняв связанность
с форматом формы.
