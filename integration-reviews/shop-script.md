# Интеграционный план: Shop-Script (Webasyst)

> Источник требований: `backend/web/app/plugins/onecatalog-import/INTEGRATION-STANDARD.md`
> (эталон — WooCommerce). Здесь — перенос на **Shop-Script (приложение `shop` фреймворка
> Webasyst)**. Внешние контракты §2 и инварианты §5 стандарта неизменны; меняется только
> слой «как лечь в CMS».
>
> Платформенные факты помечены `(проверено)` со ссылкой на developers.webasyst.com либо
> `(не уверен, проверить)` — там, где документация неоднозначна и нужна сверка с кодом
> конкретной версии Shop-Script на проекте.

## Архитектурное решение: плагин приложения `shop`, не отдельное приложение

Модуль строим как **плагин Shop-Script** в `wa-apps/shop/plugins/onecatalogimport/`
(а НЕ как отдельное приложение Webasyst). Причины: нужен доступ к моделям товара,
категорий, фич, типов; плагин получает события `shop`-приложения и нативный UI-вход
(кнопка/блок в бэкенде магазина). Зависимость объявляется в манифесте плагина: требуется
приложение `shop` определённой версии — без него плагин не активируется (§10 стандарта).

Структура (имена слоёв из §4 стандарта → файлы Webasyst):

```
wa-apps/shop/plugins/onecatalogimport/
  lib/
    config/
      plugin.php            — манифест: name, version, vendor, app shop, handlers, settings
      settings.php          — описание полей настроек (§7)
      db.php                — свои таблицы (карта public_id, трекинг качества медиа, очередь)
    classes/
      onecatalogimportApi.class.php          — Api (base URL, X-API-Key, lang, GET, обёртка)
      onecatalogimportUnits.class.php        — Units (конвертер г/мм → ед. магазина)
      onecatalogimportMedia.class.php        — Media (скачивание, MIME, качество, дедуп)
      onecatalogimportTaxonomies.class.php   — фичи/категории/термины (find-or-create по имени)
      onecatalogimportCollectionImporter.class.php
      onecatalogimportBrandImporter.class.php
      onecatalogimportCountryImporter.class.php
      onecatalogimportProductImporter.class.php  — оркестратор импорта одного товара
      onecatalogimportQueue.class.php        — порции, лог, статус
    cli/
      shopOnecatalogimportPluginImport.cli.php   — фоновый обработчик очереди (waCliController)
    actions/  (или controllers/)
      …Import…Controller.php  — приём списка public_id + статус (AJAX/JSON)
      …Settings…Action.php    — UI настроек
    handlers/                — обработчики событий shop (кнопка на странице товаров и т.п.)
  js/   picker-loader.js, admin-import.js
  css/  …
  img/  icon
  locale/  onecatalogimport.po / .mo  (gettext, §9)
```

Зависимости направлены «вниз» (ProductImporter → Api/Media/Taxonomies/*Importer; Queue →
ProductImporter; Controller → Queue), как в §4. Циклов нет.

## Маппинг сущностей и реализация

Таблица соответствий (§3 стандарта) под модель Shop-Script:

| OneCatalog | Абстракция | Shop-Script | Как |
|---|---|---|---|
| product | Товар | `shop_product` / `shopProduct` | `shopProduct` (save) или `shop.product.add/update` API |
| public_id | Внешний идемпотентный ключ | **своя таблица карты** (`*_public_id_map`) ИЛИ фича-`varchar` `_oc_public_id` | см. «Идемпотентность» ниже |
| article | Артикул производителя | `sku` товара/варианта | только из `article`; пусто если нет |
| options[] text/select | Характеристика | **feature** (`shop_feature` + `shop_feature_values`), тип `varchar`/select | find-or-create по имени, термин-значение |
| options[] boolean | Характеристика-флаг | feature тип `boolean` | значение true/false по правилу маппинга |
| options[] numeric | Характеристика-число | feature тип `double`/`dimension` | numeric_option, ед. — если задана |
| categories[] | Категория | `shop_category` (дерево) | `shopCategoryModel`, find-or-create |
| brand | Бренд | feature `varchar` / категория / тип товара | **настройка «тип объекта + цель»** |
| country | Страна происх. | feature `varchar` | настройка; по умолч. feature |
| collections[] | Коллекция | **`shop_set` (подборка)** / категория / feature | настройка «тип объекта + цель» |
| tags[] | Теги | `shop_tag` (`shopTagModel`) | вкл/выкл, поиск по имени |
| files/images_urls | Медиа | `shop_product_images` (`shopImage`) | ручное скачивание + MIME + дедуп |
| measurement_unit, sizes, areas_per_package, product_shape, alt картинок, детали бренда, upsells | прочее | НЕ ядром | событие `product_imported` → mu-слой сайта |

### Товар (`shop_product`)

- Создание/обновление через класс `shopProduct` (`$p = new shopProduct($id|null); $p->save();`)
  ИЛИ через API-методы `shop.product.add` / `shop.product.update` (проверено — есть в API
  Explorer). Класс предпочтительнее: даёт доступ к save-флоу без HTTP-обёртки.
- **Тип товара (`type_id`) обязателен и решающ.** В Shop-Script фичи привязаны к
  **типу товара**: фича видна/заполняема у товара только если она назначена его типу
  (проверено: «Product types in Shop-Script» — фичи группируются по типам). Это
  принципиальное отличие от WooCommerce, где глобальный атрибут `pa_*` цепляется к любому
  товару. **Решение:** настройка «тип товара по умолчанию для импорта» (выбор из существующих
  или авто-создание типа «OneCatalog»); все импортируемые фичи/бренд/страна назначаются
  этому типу при создании. Без этого фичи сохранятся в БД, но не покажутся на карточке.
- Базовые поля при создании: `name` ← `menutitle`, `status` ← настройка статуса
  (1 publish / 0 draft) — **только при создании** (§5.6), при апдейте не трогаем.
  `description` ← `description_text` (часто null — не перезаписывать пустым).
- **Цена НЕ задаётся** (§5.6) — в источнике её нет. Поле фильтра/события для подстановки
  из своего прайса. У Shop-Script цена — обязательное поле товара, но `0.00` допустимо;
  товар с ценой 0 в большинстве тем не покупается (нужна валидация витрины) —
  это нормально для каталога-визитки.

### Идемпотентность и `public_id` (§5.1, §5.2)

- **Рекомендую отдельную таблицу карты** `onecatalogimport_public_id_map(public_id PK,
  product_id, quality_…)` вместо хранения в SKU. Причины: (1) §5.2 запрещает класть
  public_id в SKU; (2) у Shop-Script нет универсального «поля произвольной меты товара»
  как `wp_postmeta` — единственные «свободные» хранилища значений на товаре это **фичи**
  и **сам товар**. Таблица карты даёт быстрый `find_by_external_id(public_id)` и не
  засоряет фичи.
- Альтернатива/дубль для админ-редактируемости: фича `varchar` `_oc_public_id`, чтобы
  вебмастер видел/правил ключ в карточке. Стандарт требует «отдельное **редактируемое**
  в админке поле» (§5.2) — фича это обеспечивает; таблица карты — нет. **Решение:**
  держать обе — фича для UI/правки, таблица для скорости и легаси-фолбэка; источником
  истины при поиске считать таблицу, фолбэк — фича, затем (легаси) SKU.
- `sku` товара = `article` (артикул производителя), пусто если нет; коллизия SKU с другим
  товаром → не перезаписывать (SKU уникален в Shop-Script).
- Справочники (категории, фичи, термины, бренд, страна, коллекции, теги) — find-or-create,
  **матч по человекочитаемому имени (label) без учёта регистра**, не по слагу (§5.1):
  Shop-Script сам генерирует `url`/slug из имени по своей транслитерации — поиск по слагу
  плодил бы дубли.

### Характеристики → features (§5.4)

- `options[].specification_type`:
  - `text` → feature `varchar` (или select-набор значений); значение = `specification_option_name`,
    термин find-or-create в `shop_feature_values` по имени.
  - `boolean` → feature `boolean`; значение из `bool_option`; `null/отсутствует/false → false`
    (§5.6). В ручном маппинге задаются «значения для true/false».
  - `numeric` → feature `double` либо `dimension` (если есть ед. изм.); значение `numeric_option`.
- Тип фич Shop-Script (проверено): `varchar | double | text | boolean | color | dimension`.
  `dimension` = число + единица (length/weight/area/…), что удобно для numeric с единицей.
- **Ручной маппинг (строгий режим, §5.4):** ключ привязки — стабильный `specification_id`
  (не переводимое название). При ВКЛ импортируются только сопоставленные фичи. Карту хранить
  в настройках плагина (`specification_id → feature.id`).
- **Поиск фичи по имени, а не по `code`/слагу** (§5.1): код фичи Shop-Script генерируется
  транслитом; матчим по `name`.
- Назначить используемые фичи **типу товара** (см. выше), иначе не отобразятся.

### Варианты (skus) vs характеристики — важное различие

- В стандарте `options[]` — это **характеристики** (features), НЕ товарные вариации.
  В Shop-Script отдельно есть **SKU/варианты** (`shop_product_skus`) и «селектируемые в
  вариантах» фичи (флаг `selectable`). **Импорт OneCatalog НЕ порождает вариаций** — у
  товара один SKU (= `article`), все `options[]` идут как обычные (неселектируемые) фичи.
  Не путать: не включать `selectable`/не разворачивать комбинации, иначе Shop-Script
  «размножит» SKU по комбинациям значений (проверено: add/update генерирует SKU по
  комбинациям selectable-фич — нам это не нужно).

### Категории (§5.4)

- `shop_category` — дерево (nested sets). `categories[]` OneCatalog приходит **плоским
  списком без иерархии** в payload товара → импортируем как плоские категории верхнего
  уровня (или под общий корень «OneCatalog» — настройка). Иерархию из `/categories/`
  справочника при необходимости достроить отдельно (best-effort, не блокер).
- find-or-create по имени; привязка товар↔категория через модель категорий.
- ⚠️ Shop-Script различает **«категории»** (витринная навигация) и **«типы»**. Бренд/страна/
  коллекция могут лечь и в категории — это часть настройки «тип объекта + цель».

### Бренд / страна / коллекция — «тип объекта + цель» (§3, §7)

Ключевой инвариант стандарта: **не зашивать одно хранилище**. У Shop-Script НЕТ нативной
сущности «бренд» и «страна» (в отличие от WooCommerce `product_brand` или PrestaShop
`Manufacturer`) — это надо честно отразить.

- **Бренд:** варианты цели — feature `varchar` `brand` (по умолчанию) | категория | тип
  товара. По умолчанию рекомендую **feature**, т.к. бренд в Shop-Script нативно обычно
  делают именно фичей.
- **Страна:** feature `varchar` (по умолчанию, как в стандарте) | категория. По умолчанию
  выкл.
- **Коллекция:** **`shop_set` (подборка/«наборы»)** — наиболее близкий нативный аналог
  «коллекции/витринного списка» в Shop-Script (`shopSetModel`) `(не уверен — проверить, что
  set допускает программную привязку товаров и хранение доп.полей link_3d/official_site)`.
  Альтернативы цели: категория | feature. Доп. поля коллекции (link_3d, official_site,
  images) ядро НЕ обязано хранить нативно — отдать в событие `collection_imported` для
  mu-слоя сайта.

UI настроек по каждой справочной сущности — единый паттерн «тип объекта (feature/категория/
тип/set) + конкретная цель»; нерелевантные поля скрываются JS по выбранному типу (§7).

### Теги (§5.4)

- `shop_tag` / `shopTagModel` — нативные теги товара. Вкл/выкл настройкой; поиск/привязка
  по имени.

## Фоновая очередь, медиа, настройки, i18n, жизненный цикл

### Фоновая очередь (§6)

- **Механизм:** собственная CLI-команда плагина —
  `wa-apps/shop/plugins/onecatalogimport/lib/cli/shopOnecatalogimportPluginImport.cli.php`,
  класс расширяет `waCliController`, логика в `execute()` (проверено: «Scheduled actions» —
  плагин может объявлять CLI-методы; путь `…[app][Plugin][Action].cli.php`).
- **Запуск:** cron-строкой через `cli.php`:
  `*/5 * * * * /usr/bin/php /path/wa-apps/.../cli.php shop onecatalogimportPluginImport`
  `(не уверен в точном синтаксисе вызова плагинной CLI-команды из cli.php — проверить:
  возможно регистрация через секцию cron в plugin.php/config, формат "appId pluginAction")`.
- **Модель работы:** контроллер кладёт список `public_id` в свою таблицу-очередь, режет на
  порции по «шагу импорта» (минимум 10, §6); CLI-воркер берёт следующую порцию, импортирует,
  пишет лог последних N результатов и статус (pending/running/done) в таблицу; UI **опрашивает
  статус** (AJAX-поллинг `admin-import.js`).
- **Фолбэк без планировщика (важно — у многих shared-хостингов Webasyst cron не настроен):**
  - (а) **AJAX-степпер**: фронт сам дёргает контроллер «обработай следующую порцию» циклом
    до готовности — синхронно, но не упирается в один таймаут (каждый запрос = одна порция).
    Это основной фолбэк, рекомендую сделать его дефолтным, а cron — опциональным ускорением.
  - (б) Полностью синхронная обработка всего пакета с тем же отчётом — для малых выборок.
- Webasyst НЕ имеет встроенного «Action Scheduler»-аналога с гарантированным фоновым
  раннером как WooCommerce → **на cron полагаться нельзя как на данность**; AJAX-степпер
  обязателен (см. блокеры).

### Медиа (§2.3, §5.3)

- URL `…/media_files/<base64hash>` **без расширения** → стандартные загрузчики могут
  отвергнуть. **Скачивать вручную** (`waNet`/curl), писать во временный файл,
  **MIME определять по содержимому** (`finfo`/`mime_content_type`), затем добавлять как
  изображение товара (`shopImage` / `shopProductImagesModel`).
- Три размера `min|middle|max`; с токеном — `max`. **Трекинг качества** (§5.3): рядом с
  файлом хранить выбранный размер (в своей таблице, привязка к image_id); на переимпорте
  если доступен лучший — перекачать и заменить, иначе не трогать.
- Галерея идемпотентна **по имени файла** (подписи в URL меняются — по URL не сравнивать).
- **Дедуп по контент-ключу** (§5.3): из base64-префикса `media_files` извлечь стабильный
  `path`, ключ = `hash(path # size)`, хранить на вложении (своя таблица), перед загрузкой
  искать существующее — нашли → переиспользовать без скачивания. ⚠️ Общие
  (дедуплицированные) файлы НЕ удалять при замене качества.
- ⚠️ Shop-Script хранит производные размеры изображений своим механизмом (thumbs). Наш
  «размер источника» (min/middle/max) — это качество **оригинала**, который мы скармливаем;
  не путать с thumbnail-размерами Shop-Script. Сохраняем максимально доступный оригинал,
  ресайз витрины делает сам магазин.

### Настройки (§7)

- Хранение — нативный механизм плагина: `settings.php` (описание полей) +
  `getSettings()/saveSettings()` (`waAppSettingsModel`) — проверено.
- Поля (минимум §7): токен API (приоритет ENV/переменной над полем); язык товаров
  (`en/ru/ar/zh/kk`, валидация по списку из API); шаг импорта (кламп ≥10); статус новых
  (publish/draft — только при создании); тип товара по умолчанию; импорт коллекций +
  «тип+цель»; импорт бренда + «тип+цель»; импорт страны + «тип+цель»; импорт тегов вкл/выкл;
  ручной маппинг характеристик вкл/выкл + страница карты `specification_id→feature` (+ true/false
  для boolean). Валидация и кламп обязательны.

### Точки расширения — события (§8)

- Webasyst даёт `wa_event` + классы-обработчики `waEventHandler`
  (`…/lib/handlers/shop.<event>.handler.php`) — проверено. Это аналог WP `do_action`.
- ⚠️ **Нюанс:** в Webasyst события обычно публикуют **приложения**, а обработчики ставят
  плагины. Наш модуль — плагин, и ему нужно **самому ПУБЛИКОВАТЬ** свои события
  (`before_import_product`, `product_imported`, `collection_imported`, …), чтобы mu-слой
  сайта дозаполнял поля. Плагин может вызвать `wa()->event(...)` от своего имени `(не уверен,
  что плагин может объявлять собственное пространство событий, на которое подпишутся другие —
  проверить; запасной вариант — публиковать событие в неймспейсе shop с префиксом плагина,
  либо отдать DI-хук/колбэк)`.
- **Фильтры** стандарта (меняют значение: base URL, токен, аргументы запроса, язык, шаг,
  payload перед импортом, SKU, цена, порядок размеров, дедуп медиа, файлы галереи, справочник
  характеристик, флаги импорта и карта маппинга, origin виджета) — в Webasyst нет нативных
  «filter»-хуков, возвращающих значение, как `apply_filters`. **Обходной путь:** реализовать
  их через события с **изменяемым параметром по ссылке** (обработчик правит элемент массива
  `$params`) — это рабочая практика Webasyst для «фильтров». Документировать перечень в README.
- Граница «универсальное ↔ проектное»: всё неотображаемое на таблицу маппинга (ед. изм.,
  area_per_package, product_shape, alt картинок, детали бренда, upsells) ядро НЕ пишет —
  забирает mu-слой сайта на `product_imported` (передаём ПОЛНЫЙ payload).

### Пикер выбора товаров (§2.4)

- Кнопка в бэкенде магазина (через хук/событие отрисовки страницы товаров shop) → iframe
  `https://tools.onecatalog.net/picker.html?token=…&parentOrigin=…`; связь `postMessage`;
  импорт по `productPublicIds` (стабильные ключи), не по числовым id; проверять `event.origin`;
  origin виджета задавать явно. JS — `js/picker-loader.js`, `js/admin-import.js`.
- ⚠️ Точка встраивания кнопки на странице товаров Shop-Script зависит от наличия подходящего
  backend-хука; если нативного хука на тулбар списка товаров нет — fallback: отдельная
  страница плагина в бэкенде «Импорт из OneCatalog» с пикером и полем «список public_id».

### i18n (§9)

- Нативный gettext Webasyst: исходные строки **английские**, единый text-domain плагина,
  `locale/onecatalogimport.po/.mo`; строки фронтовых скриптов отдавать из бэкенда
  локализованными (не хардкодить в JS). Соответствует памяти проекта (ru_RU .po/.mo,
  пересборка .mo при новых строках).

### Жизненный цикл (§10)

- Манифест `plugin.php`: name, version (SemVer, поднимать перед каждой сборкой),
  vendor, требование приложения `shop` и его версии, путь locale.
- Установка/удаление: `db.php` (свои таблицы карты/очереди/трекинга создаются и удаляются
  автоматически по db.php — проверено). Если нужны действия сложнее CREATE TABLE
  (миграции опций, создание типа товара «OneCatalog», создание дефолтных фич) — install
  через `lib/config/install.php` / соответствующий хук плагина `(не уверен в точном имени
  install-скрипта для плагина shop — проверить; у приложений это install.php, у плагинов
  механизм аналогичный)`.
- Миграция схемы ключа (§5.1): легаси-фолбэк поиска (таблица → фича → SKU), флаг «миграция
  выполнена».
- Активация не должна падать без приложения `shop`; функции импорта проверяют его наличие
  (manifest-зависимость это уже частично гарантирует).

### Права (permissions)

- Импорт — административное действие. Доступ к контроллеру импорта/настроек ограничить
  правами на приложение `shop` (бэкенд-доступ + право на товары). Контроллеры —
  CSRF-защита нативным механизмом Webasyst (`waRequest`/токен формы), аналог WP nonce.

## ❓ Вопросы и сомнения (ответить ДО старта)

1. **Тип товара для импорта.** Фичи в Shop-Script привязаны к **типу товара** — какой тип
   присваивать импортируемым товарам? Создавать выделенный тип «OneCatalog» автоматически,
   дать выбор существующего в настройках, или и то и другое? От этого зависит, увидит ли
   вебмастер импортированные характеристики на карточке вообще.

2. **Хранилище `public_id`.** Согласны на гибрид «таблица карты (источник истины) + фича
   `_oc_public_id` (редактируемость в админке) + легаси-фолбэк по SKU»? Или достаточно
   одного механизма? Стандарт требует «отдельное **редактируемое** поле» — только таблица
   этого не даёт.

3. **Коллекция = `shop_set`?** Это правда наиболее близкий нативный аналог? Нужно
   подтвердить, что подборки (`shopSetModel`) допускают (а) программную привязку набора
   товаров, (б) идемпотентный find-or-create, (в) хранение наших ключей. Если нет —
   дефолт «коллекция → категория» или «→ feature». Какой дефолт хотим?

4. **Цена = 0.** Цена в источнике отсутствует, у Shop-Script поле цены обязательно.
   Допустим товар с ценой `0.00` (каталог-визитка), или нужен «статус по запросу»/скрытие
   из продажи/подстановка из внешнего прайса через фильтр? Поведение тем витрины при цене 0
   надо проверить.

5. **Cron доступен?** На целевом хостинге настроен системный cron для `cli.php`? Если нет —
   подтверждаем, что **AJAX-степпер — основной** режим, а cron — опциональное ускорение.
   (Я закладываю степпер как дефолт.)

6. **Точка входа пикера.** Есть ли в текущей теме/версии бэкенда Shop-Script подходящий
   хук, чтобы повесить кнопку «Импорт из OneCatalog» прямо на тулбар списка товаров? Если
   нет — устраивает отдельная страница плагина в бэкенде?

7. **Публикация событий из плагина.** Нужно подтвердить, что плагин может ПУБЛИКОВАТЬ
   собственные события (`product_imported` и пр.), на которые подпишется mu-слой сайта, и в
   каком неймспейсе. От этого зависит, как реализовать «сайтовый слой» (§8).

8. **«Фильтры» (возврат значения).** Webasyst-события возвращают значение хуже, чем WP
   `apply_filters`. Принимаем реализацию фильтров через «событие с изменяемым параметром по
   ссылке»? Перечень фильтров из §8 фиксируем как контракт?

9. **Иерархия категорий.** `categories[]` в payload товара — плоский, без родителей.
   Импортировать плоско (под общий корень?) или дочитывать дерево из справочника
   `/categories/`? Какой корень/родитель по умолчанию?

10. **Мультиязычность.** Импортируем товары на ОДНОМ языке (настройка `lang`) или нужен
    мультиязычный импорт? Shop-Script хранит контент товара в одной локали магазина по
    умолчанию (перевод полей — отдельная история). Если нужен мультиязык — это сильно
    расширяет задачу (см. блокеры).

11. **Единицы магазина.** В какие единицы конвертировать вес/размеры (источник: граммы/мм)?
    Где в Shop-Script берётся «единица веса/размера магазина» — глобальная настройка
    магазина, поле типа `dimension` фичи, или ничего (тогда писать в фичи `dimension` с
    явной единицей)? Нужно решить целевое поле для `sizes` (вес/габариты).

12. **Лимиты API источника.** Какой rate-limit у Wiki API и какой реальный `limit` отдают
    справочники (`/specifications/` и пр.)? Стандарт велит брать одним запросом с высоким
    `limit` и сверяться с `counts` — подтвердить максимум.

13. **Артикул-варианты.** У OneCatalog один `article` на товар → один SKU. Подтверждаем, что
    вариаций (selectable-фич, разворачиваемых в комбинации SKU) НЕ создаём?

14. **Объём выборки.** Сколько товаров планируется импортировать единоразово (десятки/тысячи)?
    От этого — размер порции, нужность cron vs степпер, требования к памяти.

15. **Версия Shop-Script на проекте.** Точная версия приложения `shop` — чтобы сверить
    наличие `shop_set`, сигнатуры моделей, состав событий бэкенда и формат CLI-cron.

## 🚫 Сложно / невозможно на Shop-Script

1. **Нет произвольной «меты товара» (как `wp_postmeta`).** В WooCommerce public_id, размер
   качества картинки, контент-ключ дедупа кладутся в мету одной строкой. В Shop-Script
   свободных скаляров на товаре нет — есть только фичи (с типизацией и привязкой к типу) и
   столбцы таблицы товара. **Обход:** свои таблицы плагина (`db.php`) для карты public_id,
   трекинга качества медиа и контент-ключей дедупа. Это работает, но добавляет схему БД,
   которую WooCommerce-эталон не имеет.

2. **Фичи привязаны к ТИПУ товара (не глобальны).** Глобальный WooCommerce-атрибут `pa_*`
   цепляется к любому товару; фича Shop-Script отображается только если назначена типу
   товара. **Обход:** при импорте назначать каждую используемую фичу типу импортируемого
   товара (и заранее иметь/создать выделенный тип). Иначе характеристики «есть в БД, но
   невидимы» — классическая ловушка.

3. **Нет нативных сущностей «бренд» и «страна».** WooCommerce имеет `product_brand`,
   PrestaShop — `Manufacturer`. В Shop-Script их нет как первого класса. **Обход:** настройка
   «тип объекта + цель» (feature/категория/тип) — что стандарт и так требует; дефолт — feature.
   Минус: бренд как фича не даёт нативной бренд-страницы/логотипа — детали бренда (логотип,
   год основания, сайт) уходят в mu-слой через `brand_imported`.

4. **«Коллекция» не имеет точного аналога.** `shop_set` (подборки) — ближайшее, но это
   витринный инструмент, не контентная сущность с произвольными полями (link_3d,
   official_site, галерея). **Обход:** базовая привязка через set/категорию; доп.поля
   коллекции — в событие `collection_imported` для сайтового слоя. Идемпотентность подборок
   под вопросом — `(проверить)`.

5. **Нет встроенного гарантированного фонового раннера (аналога Action Scheduler).**
   WooCommerce-эталон опирается на Action Scheduler — самозапускающуюся очередь. У Webasyst
   фон = системный cron на `cli.php`, которого на shared-хостинге может не быть. **Обход:**
   AJAX-степпер как основной режим (фронт пошагово дёргает контроллер), cron — опциональное
   ускорение, синхронный режим — для малых партий. Стандарт это разрешает (§6 «фолбэк без
   планировщика»), но «настоящего» фона из коробки нет.

6. **«Фильтры» с возвратом значения слабее, чем `apply_filters`.** §8 опирается на
   WP-фильтры, изменяющие значение. У Webasyst события для «реакции», а не для «возврата
   модифицированного значения». **Обход:** события с изменяемым параметром по ссылке
   (`$params['value']` правится обработчиком). Работает, но менее идиоматично и требует
   дисциплины; перечень «фильтров» надо явно задокументировать.

7. **Публикация собственных событий плагином — неочевидна.** В Webasyst события обычно
   эмитят приложения, плагины — слушают. Нам нужно наоборот: плагин эмитит `product_imported`
   для mu-слоя. **Обход/риск:** вызвать `wa()->event()` из плагина с префиксным неймспейсом;
   `(не уверен, что сторонний код легко подпишется на событие, эмитируемое плагином —
   проверить на версии проекта)`. Если механизм слабый — отдать DI-колбэк/конфиг-хук вместо
   события.

8. **Мультиязычный импорт.** WooCommerce-эталон импортит на одном языке; OneCatalog отдаёт
   `lang`-зависимый payload. В Shop-Script перевод полей товара/фич — отдельная подсистема
   (локали контента), и «один товар на N языков» из одного payload-запроса не ложится
   тривиально (надо N запросов с разным `lang` и запись в локали). **Обход:** v1 — один
   язык (настройка `lang`); мультиязык — отдельная фича позже (не в MVP). Помечаю как
   осознанное сокращение объёма.

9. **iframe-пикер и origin/sandbox.** Технически реализуемо (как в эталоне), но точка
   встраивания кнопки в бэкенд Shop-Script зависит от наличия хука на список товаров.
   **Обход:** если хука нет — отдельная страница плагина с пикером (без интеграции в тулбар).
   Не блокер, но UX слабее эталона.

10. **Загрузка изображений без расширения в URL.** Нативные хелперы Shop-Script могут
    опираться на расширение/Content-Type. **Обход:** ручное скачивание + `finfo` по
    содержимому + явный MIME при добавлении `shopImage` (тот же приём, что в эталоне с WP
    `media_sideload_image`). Реализуемо, но в обход «удобного» пути платформы.

*Сводка масштаба:* основная масса работы переносится 1:1 по стандарту; реальные
платформенные трения — (а) фичи-через-тип, (б) отсутствие меты → свои таблицы, (в) фон без
Action Scheduler → степпер, (г) бренд/страна/коллекция без нативных сущностей, (д) фильтры/
эмит-событий слабее WP. Все имеют рабочий обход; ни один не делает стандарт нереализуемым.
