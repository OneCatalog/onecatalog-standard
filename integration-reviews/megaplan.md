# Интеграционный план: Мегаплан

> Перенос стандарта `INTEGRATION-STANDARD.md` (импорт каталога из OneCatalog Wiki API)
> на **Мегаплан (Megaplan)** — российскую CRM (продажи + задачи/проекты/совместная
> работа) с модулем «Товары и услуги», сделками, счетами и REST API (`api/v3`,
> Bearer-токен). Эталон стандарта — плагин WooCommerce.
>
> **Что за поставка.** На Мегаплане нет понятия «плагин внутри платформы» (как у
> WordPress) и нет клиентского cron. Поставка — это **внешний сервис (наш код на
> нашем хостинге)**, который тянет OneCatalog Wiki API и **императивно пушит**
> сущности «Товар/услуга» в Мегаплан через его **REST API v3** под Bearer-токеном
> аккаунта. У Мегаплана **нет декларативного фид-импорта** каталога (нет аналога
> ICML/YML как у RetailCRM/Битрикса для каталога — см. §«Сложно», п.5; **проверить**
> по docs, но штатного «загрузить каталог по URL-фиду» в Мегаплане я не знаю).
> Значит, в отличие от RetailCRM, **вся оркестрация — очередь, идемпотентность,
> дедуп, трекинг качества картинок, rate-limit — на нашей стороне**.
>
> **Принципиальная рамка (читать до всего остального).** Мегаплан — это **CRM для
> продаж/сделок и управления задачами**, а НЕ витринная CMS со своим storefront.
> У него нет фронтенда-магазина, taxonomy-страниц, тем, SEO-страниц
> коллекций/брендов, фасетного поиска для покупателя, страниц «коллекция»/«бренд».
> «Каталог» в Мегаплане — это **справочник «Товары и услуги»**, позиции которого
> подставляются в **счета, сделки, КП** (продажная сущность). Поэтому значительная
> часть инвариантов стандарта, завязанных на витрину (страницы коллекций/брендов,
> фасеты, alt-тексты картинок, SEO, статусы публикации draft/pending), на стороне
> Мегаплана **просто не живут**: их место — внешний сайт-витрина (ceram-online.ru),
> а Мегаплан в этой связке — «бэкенд продаж/счетов». Каталог нужен ему, чтобы
> по этим товарам выставлять счета и вести сделки, а не чтобы их показывать
> покупателю.
>
> **Целевая версия API:** Мегаплан **REST API v3** (`/api/v3/...`, JSON, **проверено**
> — актуальная публичная линейка API Мегаплана; есть и легаси `Bums*`-API v1, но
> для новой интеграции берём v3). Аутентификация — **Bearer-токен** (access-token,
> получается через `/api/v3/auth/access_token` по логину/паролю или выдаётся в
> настройках аккаунта; **проверено** — Мегаплан использует токен в заголовке
> `Authorization: Bearer <token>`). Точные имена эндпоинтов модуля «Товары и услуги»
> и состав полей карточки товара — **(не уверен, проверить по docs Мегаплана,
> раздел «Товары»/`/api/v3/...`)**: я НЕ выдумываю имена `/api/v3/goods` и т.п.

## Маппинг сущностей и реализация

### Таблица соответствий: OneCatalog → абстракция → Мегаплан

| OneCatalog | Абстракция | Мегаплан — механизм | Пометка достоверности |
|---|---|---|---|
| product | Товар | сущность **«Товар/услуга»** модуля «Товары и услуги» | (проверено: модуль есть; имя сущности/эндпоинт **проверить**) |
| public_id (`OC.IND.1`) | Внешний ключ | **доп. поле (custom field)** товара `oc_public_id` — центр идемпотентности | (не уверен в нативном externalId — **проверить**) |
| article (артикул произв.) | Артикул | нативное поле «Артикул» товара, если оно есть | (не уверен, что у товара есть поле «Артикул» — **проверить**) |
| options[] (характеристики) | Свойство товара | **доп. поля товара** (custom fields товара) — НЕ доп.поля сделки! | (не уверен в типах доп.полей товара — **проверить**) |
| categories[] | Категория | **группы/категории товаров** (папки справочника), если есть; иначе доп.поле | (не уверен в нативных группах товаров — **проверить**) |
| brand | Бренд | доп. поле товара `oc_brand` (нет нативной сущности «бренд») | (нет нативного бренда — почти наверняка) |
| country | Страна происх. | доп. поле товара `oc_country` (нет нативного поля «страна товара») | (нет нативного — почти наверняка) |
| collections[] | Коллекция | доп. поле товара `oc_collection` или группа товаров | (нет нативной коллекции — почти наверняка) |
| tags[] | Теги | доп. поле товара `oc_tags` (нет нативных тегов у товара) | (не уверен про теги товара — **проверить**) |
| images_urls / files | Медиа | **вложение/картинка товара** (если поддержана) — загрузка файлом | (не уверен в картинках у товара — **проверить**, критично) |
| sizes.weight | Вес | доп. поле товара (нет нативного веса в продажном товаре) | (не уверен — **проверить**) |
| sizes.length/width/height | Габариты | доп. поля товара | (не уверен — **проверить**) |
| measurement_unit (м²/шт) | Ед. измерения | нативная **единица измерения** товара (справочник единиц) | (проверено концептуально: у позиции счёта есть единица; имя — **проверить**) |
| **цена (нет в источнике)** | Цена | **поле «Цена» товара — в Мегаплане фактически нужна** (товар для счетов) | (проверено: товар для счёта подразумевает цену; обязательность — **проверить**) |
| measurement_unit детали, areas_per_package, product_shape, alt картинок, upsells | прочее | НЕ ложится на каталог Мегаплана → витрина/внешний сайт | — |

**Ключевой вывод по таблице.** В Мегаплане почти НЕТ нативных «справочных»
сущностей стандарта (бренд / страна / коллекция / теги). В отличие от WooCommerce
(таксономии/атрибуты) и даже RetailCRM (`<vendor>`, properties), здесь почти всё
описательное уходит в **доп. поля (custom fields) товара**. Это центральная
особенность переноса: модель стандарта «справочник = объект с собственной
страницей» (§3) на Мегаплане схлопывается в «значение доп. поля у товара». Выбор
из §3/§7 «тип объекта (атрибут/таксономия/CPT) + цель» здесь вырождается:
реально доступны только варианты **«доп. поле товара»** и (если есть) **«группа
товаров»**. CPT/таксономий-страниц нет как класса.

### product → «Товар/услуга» (§3, §5.1)

- Целевая сущность — позиция справочника **«Товары и услуги»** Мегаплана
  (**проверено**: такой модуль в Мегаплане есть, позиции подставляются в счета и
  сделки). У OneCatalog **нет вариаций** (`options[]` — это характеристики-фасеты,
  не SKU-варианты), поэтому маппим **один товар OneCatalog → одна позиция
  справочника**. Никаких «торговых предложений»/вариаций заводить не надо.
- Точное имя REST-сущности и эндпоинты CRUD товара — **(не уверен, проверить по
  docs)**. Я НЕ называю `/api/v3/goods`/`/api/v3/products` наугад. До кода —
  обязательная **разведка API** (шаг 0 стандарта): получить payload одного товара,
  список полей, формат создания/обновления, формат доп. полей.
- В Мегаплане товар может быть **«товар» или «услуга»** (флаг типа); каталог
  OneCatalog — физические товары (плитка/керамика) → тип «товар». Услуги не из
  этого источника.

### public_id → externalId / доп. поле (центральный вопрос идемпотентности, §5.1)

Это **самый важный вопрос всей интеграции** и главное отличие от платформ с
нативным внешним ключом.

- Стандарт (§5.1) требует хранить стабильный внешний ключ `public_id` (`OC.IND.1`)
  в **отдельном поле** и матчить по нему: `find_by_external_id` → обновить, иначе
  создать. Поиск по slug/названию запрещён (нестабильны).
- **Вопрос: есть ли у товара Мегаплана нативный `externalId` / внешний код?**
  - В ряде сущностей Мегаплана (особенно через интеграции 1С/обмен) встречается
    понятие внешнего идентификатора, но **есть ли он именно у сущности «Товар»
    в API v3 — (не уверен, проверить)**. Я НЕ утверждаю, что поле `externalId`
    у товара существует.
  - **План по умолчанию (безопасный):** завести **доп. поле товара**
    `oc_public_id` (тип «строка»), писать туда `public_id`, и матчить по нему.
    Это полностью соответствует §5.1 «хранить внешний ключ в отдельном поле».
- **Как искать товар по доп. полю перед импортом?** Критично: нужен способ
  серверного поиска/фильтра товаров по значению доп. поля через API v3
  (`filter`/`search` по custom field). **(не уверен, поддерживает ли API v3 фильтр
  по произвольному доп. полю — проверить.)** Если **не поддерживает** — это
  серьёзная проблема идемпотентности (см. §«Сложно», п.3): тогда нам придётся
  **держать у себя индекс** `public_id → megaplan_id` (в БД нашего сервиса) и по
  нему решать create/update. Это рабочий обход, но он переносит ответственность за
  идемпотентность целиком на наш сервис (и ломается, если кто-то правит каталог в
  Мегаплане в обход нас или мы теряем индекс).
- **Если нативный externalId у товара ЕСТЬ** — используем его (предпочтительно):
  тогда матчинг и уникальность гарантирует платформа, локальный индекс не нужен.
  Это первый вопрос к docs/заказчику.
- Легаси-фолбэк (§5.1): первый запуск — каталог пустой, миграции нет. При смене
  схемы ключа позже — фолбэк через старый признак (если использовали артикул как
  ключ — мигрировать в доп. поле).

### article → артикул товара (§5.2)

- `public_id` ≠ `article`. Жёстко соблюдаем §5.2: **нативное поле «Артикул»**
  товара заполняем ТОЛЬКО артикулом производителя (`article`), если он пришёл;
  иначе оставляем пустым (не синтезируем). `public_id` — отдельно, в доп. поле
  (см. выше), НЕ в артикул.
- **Есть ли у товара Мегаплана нативное поле «Артикул»?** — **(не уверен,
  проверить)**. Если нет — `article` тоже уходит в доп. поле `oc_article`. Если
  есть — туда.
- Артикул не обязан быть уникален; чужой не перезаписываем (наш импортёр идёт по
  одному товару).

### options[] → доп. поля ТОВАРА, НЕ доп. поля сделки! (§5.4)

Это **второй критичный пункт переноса** (наряду с идемпотентностью).

- Характеристики OneCatalog (`options[]`) — это **свойства товара**. В Мегаплане
  их нативного аналога «атрибут+термины» (как таксономия Woo / property RetailCRM)
  **скорее всего нет** → выражаем как **доп. поля (custom fields) самого товара**.
- 🚫 **Категорически не путать с доп. полями СДЕЛКИ/счёта.** У Мегаплана доп. поля
  есть у разных сущностей: у **сделки**, у **задачи**, у **контакта/компании**, и
  (предположительно) у **товара**. Характеристики плитки («Толщина», «Размер»,
  «Поверхность») — это атрибуты **товара**, и они обязаны жить в доп. полях
  **товара**, а НЕ в доп. полях сделки. Сложить характеристики товара в поля
  сделки — классическая и дорогая ошибка (прямой аналог «properties vs customFields»
  в RetailCRM). См. §«Сложно», п.1.
- **Поддерживает ли товар Мегаплана доп. поля вообще?** — **(не уверен, проверить,
  критично)**. Если у сущности «Товар» в API v3 кастомных полей **нет** (а они есть
  только у сделок/задач) — это ломает почти весь маппинг характеристик/бренда/
  страны и придётся либо складывать всё в **описание товара** (текст, неструктурно),
  либо хранить характеристики на стороне сделки (плохо, см. выше), либо признать,
  что структурные характеристики в Мегаплан не переносятся. Это **вопрос №1 к
  технической документации**.
- Резолв характеристики (если доп. поля товара есть): для каждой `options[]` —
  доп. поле с именем `specification_label`; значение — `specification_option_name`.
  Идемпотентность поля — по стабильному ключу (`specification_id`), не по
  переводимому названию (§5.1): имя доп. поля в Мегаплане кодируем как
  `oc_spec_<specification_id>` (или ведём карту «specification_id → id доп. поля»).
- **boolean-характеристики (§2.2, §5.6):** `specification_option_name = null`,
  значение в `bool_option`. Если в Мегаплане доступен тип доп. поля «флажок/
  булево» — пишем напрямую; **по умолчанию false** (null/false/нет → false, §5.6).
  Если булева типа нет — текстовое «Да»/«Нет» (подписи из настройки маппинга).
- **numeric-характеристики:** `numeric_option` → числовое доп. поле (если тип
  «число» есть) либо строка. Тип доп. полей товара Мегаплана — **проверить**.
- **Ручной маппинг (строгий режим, §5.4):** при ВКЛ создаём/заполняем только доп.
  поля для сопоставленных `specification_id`; ключ привязки — `specification_id`.
  При ВЫКЛ — все характеристики «по имени/создать доп. поле».
- ⚠️ Доп. поля в Мегаплане, как правило, **создаются заранее в настройках
  аккаунта** (конфигурация полей), а через API заполняются значения. **(не уверен,
  можно ли СОЗДАВАТЬ доп. поле через API v3 или только заполнять заранее
  заведённое — проверить.)** Если создавать поля через API нельзя — нужен этап
  «преднастройка полей в аккаунте» (вручную/полуавтоматически) до импорта, и режим
  «создать атрибут на лету» из стандарта здесь не работает → только ручной маппинг
  на заранее заведённые поля. Это меняет UX настройки (см. §Вопросы, п.4).

### categories[] → группы товаров / доп. поле (§5.4)

- В справочнике «Товары и услуги» Мегаплана **могут быть группы/папки** товаров
  (категории каталога) — **(не уверен, есть ли иерархические группы и доступны ли
  через API — проверить)**.
- Если группы есть: `categories[]` → группы товаров, идемпотентность группы по
  её внешнему ключу (id категории OneCatalog) или по имени (label, §5.1 — матчить
  по имени, не по слагу/транслиту). Привязываем товар к группе.
- **Иерархия:** payload `categories[]` товара плоский (id, menutitle, slug) **без
  явного parent** (§2.2). Для дерева — отдельный вызов `/categories/` Wiki API и
  его поле родителя (шаг 0 разведки). Если родитель не приходит — кладём плоско.
  Поддерживает ли группы товаров Мегаплана вложенность — **проверить**.
- Если групп товаров в Мегаплане нет/недоступны через API — категории уходят в
  **доп. поле** `oc_category` (множественное/строка), без навигационной структуры.
  Это деградация (нет дерева каталога), но не блокер для счетов.

### brand / country / collections / tags → доп. поля (нет нативных сущностей) (§3, §7)

В Мегаплане **нет нативных сущностей «бренд», «страна происхождения», «коллекция»,
«теги товара»** (почти наверняка — это витринные/каталожные понятия, чуждые CRM
продаж). Поэтому все четыре уходят в **доп. поля товара**:

- **brand** → доп. поле `oc_brand` (значение `brand.menutitle`). Доп. поля бренда
  (страна, год основания, сайт, лого из `brand.images_*`) в Мегаплан **не ложатся**
  (нет «карточки бренда») → это витрина/внешний сайт. Альтернатива из §3/§7
  «тип объекта + цель» здесь почти вырождена: вариант «группа товаров для бренда»
  возможен, но обычно бренд логичнее как доп. поле (фильтр/выгрузка), а группами
  занять категории.
- **country** → доп. поле `oc_country` (`country.menutitle`). Дефолт — выключено
  (как §7). Витринной «страницы страны» нет.
- **collections** → доп. поле `oc_collection` (`menutitle`) **или** группа товаров
  «Коллекции/<имя>» (если нужна группировка внутри Мегаплана). Доп. поля коллекции
  (`link_3d`, `link_official_site`, лого) — в Мегаплан не ложатся → витрина.
  Поля берём из эмбеда `collections[]` товара (стандарт §2.1: `/collections/{slug}/`
  отдаёт 500 — НЕ дёргать).
- **tags** → доп. поле `oc_tags` (множественное, `title`). Вкл/выкл, дефолт выкл
  (§7). Семантика «фасет-тег витрины» в Мегаплане не воспроизводится.
- ⚠️ Все четыре зависят от ответа «**есть ли вообще доп. поля у товара**» (см.
  options[] выше). Если нет — переезжают в текст «Описание» товара (неструктурно)
  или не переносятся.

### media → картинки товара (есть ли вообще? как?) (§2.3, §5.3)

Это **третий критичный неизвестный**.

- **Поддерживает ли «Товар/услуга» Мегаплана изображения?** — **(не уверен,
  проверить, критично)**. У товара в CRM-справочнике картинка не всегда есть (товар
  для счёта — это строка прайса, а не карточка с галереей). Возможны варианты:
  (а) у товара есть поле-картинка/вложения; (б) картинок у товара нет вовсе.
- **Если картинки поддерживаются:**
  - Загрузка — **только файлом через API** (multipart/attachment-эндпоинт
    Мегаплана; имя — **проверить**). По URL Мегаплан картинку, скорее всего, сам
    НЕ скачает (нет фид-импорта) → **наш сервис качает файл из OneCatalog с
    токеном и заливает в Мегаплан как вложение**. Это полностью на нас.
  - **URL OneCatalog без расширения** (§2.3) → качаем вручную, **определяем MIME по
    содержимому**, заливаем с корректным именем/типом. Полностью совпадает с
    предупреждением стандарта про `media_sideload`.
  - **Качество/токен (§2.3):** наш сервис держит токен, берёт `max`; без токена —
    только `min`. Выбор размера — у нас.
  - **Трекинг качества + дедуп по контент-ключу (§5.3):** целиком в нашем сервисе.
    Контент-ключ `hash(path#size)` из base64-префикса `media_files` (§5.3) храним
    в нашем индексе и в Мегаплане (если есть куда — доп. поле/имя файла). Перед
    заливкой ищем по ключу: уже залит → переиспользуем `attachment_id`, не качаем.
    Галерея идемпотентна **по имени файла** (§5.3), не по URL.
- **Если картинок у товара нет** — медиа в Мегаплан не переносятся; обложку/галерею
  показывает витрина (внешний сайт). Это нормально для CRM продаж: для счёта
  картинка не нужна. Честно фиксируем как «не реализуется на Мегаплане».

### sizes/weight → доп. поля; единицы измерения (§5.6) — Units

- OneCatalog отдаёт **вес в граммах, размеры в миллиметрах** (подписи локализованы;
  ед. из `sizes.*_unit`, иначе мм/г) — §5.6. Конвертер `Units` (§4) обязателен:
  нормализация подписи → коэффициент → единица назначения.
- **Вес/габариты:** у продажного товара Мегаплана нативных полей веса/габаритов,
  скорее всего, **нет** (это логистика, не продажа) — **(не уверен, проверить)**.
  → вес/length/width/height уходят в **доп. поля** товара (числовые), без участия
  в каких-либо расчётах. Конвертер всё равно нужен, чтобы значения были в
  понятных единицах (мм/см, г/кг — по настройке).
- **measurement_unit (м²/шт):** это **продажная единица** товара. У позиции
  счёта/товара Мегаплана единица измерения **есть** (концептуально — **проверено**;
  точный механизм/справочник единиц — **проверить**). Маппим
  `measurement_unit.slug/name` на единицу Мегаплана (find-or-create в справочнике
  единиц, если он редактируемый, иначе на ближайшую существующую).
  ⚠️ **Не путать продажную единицу (м²/шт) с единицей габаритов (мм)** — §5.6 прямо
  предупреждает. Это два разных поля/смысла.

### Цена (§5.6) — но товар Мегаплана для счетов почти наверняка требует цену

В источнике **цены нет** (§5.6: не синтезировать). Но **Мегаплан — CRM продаж**, и
позиция «Товара/услуги» нужна именно чтобы подставляться в **счета/сделки**, где
цена — основа документа.

- **Цена у товара Мегаплана — фактически обязательна/ожидаема** (товар без цены в
  счёте бессмысленен). Обязательность поля на уровне API — **(проверить)**, но по
  природе платформы цена нужна. Это прямой конфликт §5.6 «не выдумывать» с природой
  Мегаплана (см. §«Сложно», п.2). Варианты:
  1. Цена `0` — формально, но «нулевые» товары текут в счета (плохо, искажает
     суммы/аналитику продаж).
  2. Цену подставляет **наш сервис из СВОЕГО источника** (реальный прайс заказчика) —
     нарушает §5.6, делаем ТОЛЬКО если у заказчика есть настоящий прайс-лист.
  3. Цена приходит в Мегаплан **из другой системы** (1С/сайт), а наш импорт несёт
     только описательную часть товара — но тогда надо не затирать цену при
     переимпорте (§5.6 «поля только при создании»): цену задаём при создании, при
     апдейте **не трогаем** (ручную/1С-цену сохраняем).
- **Решение — вопрос к заказчику (§Вопросы, п.5–6):** нужен ли каталог в Мегаплане
  для **реальных счетов/сделок** (тогда цена обязательна + источник цены) или
  справочно/для маркетинга (тогда цена может быть 0/пустой, либо Мегаплан вообще не
  целевая система для этих товаров).
- Стандартный механизм: цена — поле «только при создании» (§5.6); по умолчанию не
  задаётся (0/пусто); подстановка из прайса — через настройку/событие нашего
  сервиса, не зашита.

## Фоновая очередь, медиа, настройки, i18n, жизненный цикл

### Как лить каталог: только API-пуш, своя очередь (§6)

- **Нет фид-импорта.** В отличие от RetailCRM (ICML) и Битрикса (YML/CommerceML),
  у Мегаплана я **не знаю** штатного «загрузить каталог по URL-фиду» —
  **(проверить, но скорее нет)**. Значит, единственный путь — **императивный
  REST-пуш**: наш сервис сам создаёт/обновляет товары через API v3. → **вся
  фоновая очередь стандарта (§6) реализуется в НАШЕМ сервисе.**
- **Оркестрация в нашем сервисе (аналог слоёв §4):**
  - `Api` — клиент Wiki API (base URL, `X-API-Key`, `lang`, GET, обёртка ответа,
    ошибки → null) **и** клиент Мегаплан API v3 (Bearer-токен, CRUD товара,
    доп. поля, единицы, вложения).
  - `Units` — конвертер г/мм → единицы назначения.
  - `Media` — скачивание из OneCatalog (MIME по содержимому), заливка вложением в
    Мегаплан, трекинг качества, дедуп по контент-ключу — **всё у нас**.
  - `Taxonomies`/маппинг — резолв доп. полей/групп товаров, единиц.
  - `ProductImporter` — оркестратор импорта одного товара по порядку §5.4.
  - `Queue` — **наша job-очередь** (Redis/БД + worker): список public_id режется на
    порции **≥10** (§6), порции — фоновые задачи, статус/лог, синхронный фолбэк.
  - `Rest` — наш endpoint приёма списка public_id + статус (если нужен UI).
  - `Settings` — наши настройки (БД/конфиг сервиса).
- **rate-limit Мегаплан API.** У Мегаплана **есть ограничение частоты запросов**
  к API — **(не уверен в конкретном лимите/окне — проверить по docs)**. Поскольку
  каждый товар = несколько вызовов (поиск по ключу, create/update, доп. поля,
  заливка картинок), порции и паузы между запросами **обязательны**, иначе 429/
  блокировка. Это усиливает требование §6 (очередь) — без неё импорт сотни товаров
  упрётся в лимит.
- **rate-limit / пагинация OneCatalog (§2.1):** справочники (`/specifications/`
  и т.п.) брать **одним запросом с высоким `limit`** (иначе дефолтный лимит молча
  обрежет); сверяться с `meta.counts`. Это наша забота при разведке/импорте.

### Токен и настройки (§7)

- **Аутентификация Мегаплан:** Bearer access-token (**проверено**: заголовок
  `Authorization: Bearer <token>`). Токен получаем через эндпоинт авторизации
  (`/api/v3/auth/access_token` по логину/паролю/одноразовому — **точное имя/схему
  проверить**) либо берём выданный в аккаунте. Храним и **обновляем** токен в нашем
  сервисе; учесть истечение/refresh (**проверить TTL и refresh-механику**).
- **Наш сервис хранит:** токен OneCatalog (`X-API-Key`), язык (`lang`,
  `ru/en/ar/zh/kk`, дефолт под ceram — `ru`), шаг очереди (≥10), статус новых
  (на Мегаплане почти неприменим — у товара нет 4 витринных статусов; есть, может
  быть, «активен/архив» — **проверить**), флаги (бренд/страна/коллекции/теги),
  карту ручного маппинга характеристик → доп. поля, правила цены, креды Мегаплана
  (URL аккаунта + токен).
- Приоритет переменной окружения над полем токена (§7) — соблюдаем в нашем сервисе.
- Язык — только из списка поддерживаемых API (§7).

### Мультиязычность (§9)

- Wiki API отдаёт контент по `lang`. Мегаплан **не мультиязычная витрина** — каталог
  ведётся в одной языковой проекции. → импортируем на **одном языке** (для ceram —
  `ru`). Несколько языков = несколько прогонов/наборов полей — отдельная задача,
  на старте не нужно.
- i18n самого **нашего сервиса** (строки UI настроек, если UI есть): по стандарту
  EN-исходник + переводы (§9). Механизм POT/PO/MO — это WordPress; у нас — механизм
  нашего стека (i18n-библиотека). Строки фронта — из бэкенда локализованными, не
  хардкод в JS (§9). Внутри Мегаплана своего UI у нас нет (нет плагинов платформы).

### Жизненный цикл интеграции (§10)

- **Поставка — приватный сервис «только для ceram»** на Bearer-токене конкретного
  аккаунта Мегаплана (не публичное приложение маркетплейса). Это, скорее всего, наш
  случай: один аккаунт, не тиражируемый продукт. → не нужны manifest приложения,
  OAuth-установка, листинг в маркетплейсе.
  - **Если бы понадобилось публичное приложение** (другим клиентам) — у Мегаплана
    есть механизм приложений/интеграций; схема регистрации и OAuth — **(проверить
    по docs «Приложения/Интеграции Мегаплана»)**. Для ceram это избыточно.
- **SemVer + CHANGELOG (§10, и по практике проекта):** ведём для **нашего сервиса**
  (а не «плагина Мегаплана», которого как файла нет). Версия в манифесте сервиса +
  CHANGELOG (Keep a Changelog), бэклог = «Не выпущено», поднимать версию перед
  релизом. Новые строки UI — переводить (§9).
- **Деградация без падений (§5.5, §10):** один битый товар/недоступный эндпоинт не
  валит пакет — `null`/skip, лог, продолжаем. При недоступности Мегаплана/OneCatalog
  — повтор по расписанию нашего worker'а; best-effort обогащение без ложной метки
  «готово».
- **Cron — наш.** Поскольку у клиента (Мегаплан SaaS) **нет управляемого
  серверного cron** (§«Сложно», п.6) — расписание импорта живёт в нашем сервисе
  (наш worker/cron), не в Мегаплане.

## 🚫 Сложно / невозможно на Мегаплане

1. **Доп. поля ТОВАРА ≠ доп. поля СДЕЛКИ/счёта.** В стандарте «характеристика» —
   единая абстракция (атрибут+термины). В Мегаплане доп. поля есть у разных
   сущностей (сделка, задача, контакт, и — предположительно — товар). Характеристики
   товара (Толщина/Размер/Поверхность) обязаны идти в доп. поля **товара**, НЕ в
   поля сделки. *Обход:* жёсткое правило в маппинге + ревью; убедиться, что у товара
   доп. поля вообще есть (Вопрос 3). Самая частая и дорогая ошибка (полный аналог
   «properties vs customFields» в RetailCRM).

2. **Цены нет в источнике, но товар Мегаплана для счетов её требует.** §5.6 «не
   синтезировать», но Мегаплан — CRM **продаж**: позиция нужна для счёта, где цена —
   основа. *Обход:* источник цены (1С/прайс заказчика), либо цена 0 (с риском «нулей»
   в счетах/аналитике), либо признать, что для реальных счетов эти товары Мегаплан
   не обрабатывает (только справочник). Цена — «только при создании», не затирать
   при апдейте. Прямой конфликт инварианта с природой платформы — решать с
   заказчиком (Вопросы 7–8).

3. **Идемпотентность без нативного externalId — на нашем индексе.** Если у товара
   нет нативного внешнего ключа и/или API не умеет фильтровать по доп. полю — мы НЕ
   можем дешево «найти по public_id» на стороне Мегаплана. *Обход:* держать
   локальный индекс `public_id → megaplan_id` в БД нашего сервиса и по нему решать
   create/update. Минусы: индекс — единственный источник правды о соответствии;
   ломается при ручных правках каталога в обход нас или потере индекса; нужна
   процедура «пересборки индекса» (вычитать все товары, прочитать доп. поле
   public_id). Менее надёжно, чем нативный externalId (WP-мета/RetailCRM externalId).

4. **Почти нет нативных «справочных» сущностей (бренд/страна/коллекция/теги).**
   Стандарт (§3/§7) даёт выбор «атрибут / таксономия / CPT + цель». В Мегаплане
   реально доступны лишь «доп. поле товара» и (если есть) «группа товаров». CPT/
   таксономий-страниц нет как класса. *Обход:* всё описательное — в доп. поля
   товара; категории/коллекции — опционально в группы товаров. Витринные «страницы
   бренда/коллекции/страны» не реализуются (это внешний сайт).

5. **Нет фид-импорта каталога — только императивный API-пуш.** В отличие от
   RetailCRM (ICML) и Битрикса (YML/CommerceML), штатной «загрузки каталога по
   URL-фиду» в Мегаплане нет (**проверить, но почти наверняка**). Значит, очередь,
   идемпотентность, дедуп, трекинг качества картинок, пагинация OneCatalog,
   rate-limit — **всё в нашем сервисе** (своя job-очередь + worker, порции ≥10,
   статус/лог, синхронный фолбэк). Это БОЛЬШЕ нашего кода, чем в декларативных
   платформах: там платформа берёт идемпотентность/расписание/медиа на себя — здесь
   всё на нас.

6. **Нет управляемого клиентом cron внутри Мегаплана (SaaS).** §6 (очередь/шаг/
   статус/прогресс) — наш код; «по расписанию» — наш worker/cron, не Мегаплан.
   *Обход:* расписание и оркестрация целиком в нашем сервисе; в Мегаплане видны
   лишь результаты (созданные/обновлённые товары), процессом импорта мы управляем у
   себя.

7. **Нет витрины: страниц коллекций/брендов, фасетов, SEO, alt-текстов.** Инварианты
   стандарта про «коллекция/бренд как страница с полями, alt картинок, фасеты, SEO,
   статусы публикации» в Мегаплане **не живут** — это функции витрины-сайта, не CRM
   продаж. *Обход:* в Мегаплане — только значения доп. полей (для фильтра/выгрузки/
   подстановки в счёт); вся витринная часть — на внешнем сайте (ceram-online.ru).
   Честно: бОльшая часть «витринного» слоя стандарта на Мегаплане не реализуется и
   не должна.

8. **Картинки товара под вопросом и грузятся только файлом.** Если изображения у
   товара поддерживаются — Мегаплан не скачает URL сам (нет фид-импорта) → наш
   сервис качает из OneCatalog (URL без расширения → MIME по содержимому, токен для
   `max`) и заливает вложением; трекинг качества/дедуп (§5.3) — целиком у нас. Если
   изображения у товара НЕ поддерживаются — медиа в Мегаплан не переносятся вовсе
   (только витрина). *Обход:* при отсутствии поддержки — фиксируем как
   нереализуемое; галерею показывает внешний сайт.

9. **Создание доп. полей на лету может быть невозможно.** Если доп. поля заводятся
   только в настройках аккаунта (не через API) — режим стандарта «найти атрибут по
   имени, иначе создать» (§5.4) не работает. *Обход:* этап преднастройки полей в
   аккаунте + строгий ручной маппинг «характеристика → существующее доп. поле» по
   `specification_id`. Это сдвигает работу на конфигурацию аккаунта до импорта.

10. **«Поля только при создании» при императивном апдейте.** §5.6 (не затирать
    ручные правки вебмастера при переимпорте: статус, цена) реализуемо, но требует
    дисциплины: при `update` мы шлём ТОЛЬКО управляемые нами поля и НЕ шлём
    цену/статус (или шлём их лишь при `create`). *Обход:* в `ProductImporter`
    раздельные наборы полей для create и update; цену/статус — в «create-only».
    На стороне Мегаплана нет «декларативного слепка», так что контроль за
    незатиранием — наш (это даже проще, чем на ICML RetailCRM).

## Резюме для исполнителя

- **Основной путь — внешний сервис с императивным API-пушем** в Мегаплан **REST
  API v3** (Bearer-токен, **проверено**). Фид-импорта каталога у Мегаплана нет
  (**проверить, но почти наверняка**) → **вся оркестрация на нашей стороне**:
  своя job-очередь (§6, порции ≥10, статус/лог), идемпотентность, дедуп, трекинг
  качества картинок, пагинация OneCatalog, паузы под rate-limit. Целевая сущность —
  справочник **«Товары и услуги»** (имена эндпоинтов/полей **проверить**, не
  выдуманы).
- **Маппинг:** `public_id` → доп. поле `oc_public_id` (если нет нативного
  externalId — а это под вопросом); `article` → артикул/доп. поле; характеристики
  `options[]` → **доп. поля ТОВАРА** (НЕ сделки!); категории → группы товаров или
  доп. поле; бренд/страна/коллекция/теги → **доп. поля** (нативных сущностей нет);
  единица измерения → нативная продажная единица; вес/габариты → доп. поля через
  конвертер г/мм; цена — в источнике нет, но Мегаплану для счетов нужна.
- **Главная рамка:** Мегаплан — **CRM продаж/сделок**, не витрина. Витринные
  инварианты стандарта (страницы коллекций/брендов, фасеты, SEO, alt, статусы
  публикации) на Мегаплане **не живут** — это внешний сайт ceram-online.ru;
  Мегаплан — «бэкенд счетов/сделок».
- **Самые острые вопросы:** (1) есть ли у товара доп. поля ВООБЩЕ — иначе рушится
  маппинг характеристик; (2) идемпотентность — нативный externalId у товара или
  доп. поле + можно ли фильтровать по нему через API (иначе наш локальный индекс);
  (3) ЦЕНА — её нет в источнике, а Мегаплану для счетов она нужна (источник цены?);
  (4) картинки у товара — поддержаны ли и как грузить.
- **Главные «невозможно/сложно»:** (1) доп. поля товара ≠ доп. поля сделки — не
  перепутать; (2) цена обязательна для счетов, но в источнике её нет; (3)
  идемпотентность без нативного externalId держится на нашем локальном индексе
  (хрупко); (4) нет фид-импорта и клиентского cron — вся очередь/идемпотентность/
  дедуп/медиа на нас (больше кода, чем у декларативных платформ); (5) нет витрины —
  страницы коллекций/брендов/фасеты/SEO отсутствуют как класс.
