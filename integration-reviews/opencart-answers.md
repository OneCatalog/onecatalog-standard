# Ответы OpenCart-разработчику — OneCatalog Import

Ответы на [opencart.md](opencart.md) (20 вопросов + 12 блокеров). Маркировка:
🟢 — авторитетно (проверено на живом Wiki API / решение эталона WooCommerce `onecatalog-import`
v1.7.0); 🟡 — продуктовое решение/зависит от магазина, рекомендация + что подтвердить.

> ⚠️ Эталонная витрина `ceram-online` — headless WordPress+Nuxt, не OpenCart. Вопросы про «этот
> OpenCart-магазин» (версия, cron, мультимагазин, CSP) — про новое развёртывание; по ним рекомендация
> + «подтвердить». API/логику стандарта отвечаю точно.

> 👍 **Хорошая новость для OpenCart:** в отличие от Magento/PrestaShop/Drupal, у вас **нет блокеров
> «цена обязательна» и «характеристика обязана быть вариацией»** — `product.price` спокойно живёт с
> `0.0000` (каталожный режим), а характеристики честно ложатся на `attribute`. Это упрощает порт.

## Ответы на вопросы

**1. Целевые версии (3.x / 4.x / обе).** 🟡 **Не стройте обе спекулятивно** — это удвоение обвязки и
тестов (ваш же блокер B10). Целиться в ту версию, на которой реально работает магазин-заказчик. Общую
логику держать в version-agnostic library-классах, но обвязку (controller/model/install) делать под
одну версию; вторую — только если есть конкретный второй магазин. **Подтвердить версию у владельца.**

**2. Поле под `public_id` (своя таблица `oc1c_product_map`).** 🟢 **Да, правильно.** Точный аналог
эталона: ключ идемпотентности — в выделенном хранилище, НЕ в `model`/`sku`. `public_id ≠ model ≠ sku`.
`UNIQUE(public_id, source)` — то, что нужно.

**3. `product.model` при пустом `article`.** 🟢 Эталон: SKU=`article`, пусто если нет (в WC можно). У
OpenCart `model` обязателен → фолбэк. Рекомендация: **`model = article ?: public_id`**, `sku = article ?: ''`,
а `public_id` ВСЕГДА в карте (по нему идемпотентность, не по model). Префикс `OC-` добавляйте, только
если `model` показывается покупателю и «OC.IND.1» смущает — иначе не нужно. 🟡 Подтвердить, видим ли
`model` на витрине.

**4. Коллекция — дефолтное хранилище.** 🟡 Эталонный дефолт коллекции — CPT (в WC); на OpenCart CPT
нет. **Дефолт — категория-ветка** (навигация и страница «из коробки», богатые поля `link_3d`/
`link_official_site` → в `category_description.description`). Кастом-таблица + свой контроллер витрины —
только если нужны полноценные страницы коллекций (отдельная задача, вне ядра). Эталон недавно добавил и
режим «атрибут» для коллекций — на OpenCart это attribute (но теряет ссылки). Итог: category (дефолт) /
attribute / custom-table.

**5. Страна — атрибут, не таблица `country`.** 🟢 **Подтверждаю: НЕ писать в `country`** (это страны
адресов/доставки). Страна происхождения → **атрибут** «Страна происхождения» (текст по языку),
find-or-create по `menutitle` («Индия»/«Испания»). 🟢 Примечание: в API `country_code` — это ISO
**alpha-3** (`IND`,`ESP`,…), но при подходе «атрибут по имени» код не нужен — берём `menutitle`.
Фасетная фильтрация по стране — только если тема умеет фильтр по атрибутам.

**6. Мультиязычность характеристик (один язык vs N вызовов).** 🟡 Эталон импортирует **один язык за
прогон**. Фаза 1 — один язык: писать `product_attribute.text` только для языка из настройки `lang`, для
прочих языков магазина — пусто или дубль (настройка). N запросов на язык — фаза 2. **Подтвердить
число языков магазина.**

**7. Бренд не мультиязычен.** 🟢 Принять одноязычное имя бренда (`manufacturer.name`). Это НЕ потеря:
эталон в `BrandImporter` хранит только имя + лого + OC-id, а `description_text/foundation_year/website/
country` бренда **не сохраняет даже в WC** — это «прочий payload» → через событие `oc1c.brand.imported`
или кастом-таблицу `oc1c_brand_meta`, если реально нужны.

**8. Карта `lang` API ↔ `language_id`.** 🟡 Авто-сопоставление по `language.code` (в OpenCart код вида
`en-gb`/`ru-ru`; API даёт `en`/`ru`) — мапим по префиксу + ручное переопределение в настройках. Языка
из API нет в магазине → писать в язык магазина по умолчанию. Разумно.

**9. Статус товара.** 🟢 OpenCart знает только `status` 0/1. Настройка «статус новых» → enabled(1)/
disabled(0), **только при создании**, при апдейте не трогать (эталон так и делает). `pending/draft/
private` аналога нет → «не опубликован» = disabled. 🟢 Опционально можно уважать `status_publish` из
API (значение `"publish"` приходит), но эталонный путь — своя настройка.

**10. Категории: плоско или дерево.** 🟢 **Дерево — иерархия в API ЕСТЬ.** Факт API: у товара
`categories[]` плоский (`{id, menutitle, slug}`), но справочник **`/categories/` отдаёт
`parent_category: {id, slug, menutitle}`** (или `null` = корень). Пример: «Керамогранит»(3) → «Плитка»(1)
→ корень. Тяните `/categories/?limit=1000`, стройте дерево, привязывайте по полной цепочке и
**пересчитывайте `category_path`** (через admin-model `catalog/category`, не голым INSERT). Эталон WC
строит плоско — OpenCart может **лучше**.

**11. Теги: деградация в CSV приемлема?** 🟡 Эталон кладёт теги в нативную таксономию `product_tag`. У
OpenCart её нет → `product_description.tag` (CSV) — прагматичная деградация для поиска (+ опц.
`meta_keyword`). Страниц тегов/фильтрации не будет (это кастом, вне ядра). Теги по умолчанию ВЫКЛ —
так что на старте это не критично. **Подтвердить, нужны ли страницы тегов.**

**12. Целевые `weight_class`/`length_class`.** 🟡 **Писать в существующие базовые классы магазина (по
умолчанию кг/см) с конверсией — НЕ создавать новые мм/г классы.** Ваш двухэтапный конвертер
(источник г/мм → СИ-база → класс магазина через `value`-множители) — верный. Настройка «целевой
`weight_class_id`/`length_class_id`» с дефолтом = базовые классы. Набор классов произволен — конвертер
не должен зависеть от наличия конкретного класса.

**13. Лимиты Wiki API.** 🟢 Жёсткого rate-limit на `X-API-Key` в тестах не наблюдалось. Справочники
(`/specifications/`, `/categories/`) брать **одним запросом `limit=1000`** (проверено; `meta.counts` =
общее число, дефолтный лимит молча обрезает). Товары — по одному `/products/{public_id}/`, таймаут
~40с, ошибка → skip (§5.5). 🟡 Реальный объём уточнить (шаг порции для OpenCart — 10–20, частота
cron-порций — без агрессии).

**14. Cron-режим.** 🟡 Зависит от хостинга. **AJAX-степпер — основной** (работает без системного cron,
но нужна открытая вкладка). Системный crontab → настоящий фон. Эталонный Action Scheduler тоже зависит
от cron/визитов — не минус OpenCart. **Подтвердить наличие системного cron.**

**15. Картинки: путь и чистка.** 🟡 `image/catalog/onecatalog/...` — ок. Дедуп-таблица `oc1c_media`
(content_key→path) + **счётчик ссылок перед удалением** — обязательно (в OC нет вложений-объектов).
Это аналог эталонного `_onecatalog_file_key` + правила «общие файлы не удалять».

**16. Пикер-iframe в admin.** 🟡 OpenCart admin часто за Basic-auth / в подпапке / с CSP; CSRF в OC =
`user_token` в query. **Фолбэк (всегда работает) — textarea со списком public_id** (стандарт §2.4
допускает). Рекомендую сначала ID-list, пикер — опционально при совместимом окружении.

**17. Один store или мульти-store.** 🟡 Дефолт — один store (`product_to_store` = текущий). Мульти-store
— настройка (в какие store писать). **Подтвердить.**

**18. `attribute_group` для характеристик.** 🟡 **Одна группа «Характеристики» для всех** (проще;
эталон вообще не группирует — глобальные атрибуты WC без групп). Группировка по `product_shape`/
категории — улучшение на потом.

**19. `option` vs `attribute`.** 🟢 **ПОДТВЕРЖДАЮ КАТЕГОРИЧЕСКИ: характеристики только через `attribute`,
`option` НЕ использовать.** OneCatalog `options[]` — это фасетные характеристики (цвет/формат/
поверхность), а НЕ покупательские вариации; вариаций у товара нет вообще. `option`/`product_option` в
OpenCart влияют на цену/склад/выбор в корзине — использование их под характеристики наплодит фантомные
варианты и сломает корзину. Это **самый важный пункт корректности** для OpenCart.

**20. Цена = 0.** 🟢 Эталон цену не задаёт (в WC товар без цены валиден). У OpenCart `product.price`
по умолчанию `0.0000` и не обязан быть непустым → **ставьте `price = 0` при создании, не трогайте при
апдейте** — каталожный режим работает нормально (это НЕ блокер, в отличие от Magento/PrestaShop). Реальную
цену даёт сайт через событие. Если нужно «цена по запросу»/скрыть — у OpenCart есть опции «показывать цену
только авторизованным»/тема; `0` оставлять безопасно. 🟡 Подтвердить политику отображения.

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id`** (своя карта), `model = article ?: public_id`, `sku = article`. Искать
   ВСЕГДА по public_id, не по model/sku.
2. **Справочники — поиск по имени** (`LOWER(name)=LOWER(?)`), не по слагу. В OpenCart авто-слагов у
   `attribute`/`category`/`manufacturer` нет (slug только в `seo_url`), поэтому риск «разных транслитов»
   ниже эталонного — но матч по имени всё равно обязателен (ручные дубли). Стабильный ключ маппинга —
   `specification_id`/category id.
3. **Характеристики — только `attribute`, не `option`** (Q19) — критично.
4. **Ручной маппинг по `specification_id`, строгий режим**; список спек из `/specifications/`
   (`limit=1000`). boolean: текст true/false из маппинга (дефолт «Да»/«Нет»).
5. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие
   = false**. (Факт API: в этом инстансе `numeric` не используется — все характеристики text/boolean.)
6. **Медиа:** дедуп по `sha1(path#size)`; общие файлы НЕ удалять (счётчик ссылок); галерея по имени
   файла, не по URL. Скачивать вручную + MIME по содержимому (URL без расширения).
7. **Единицы:** конвертер обязателен (источник г/мм, локализованные подписи «мм»/«г», `weight_unit`
   бывает `null`) → целевые `weight_class`/`length_class`. `measurement_unit` (м²/шт) — продажная, не
   габарит → событие/атрибут.
8. **Цена `0` при создании, не перезатирать**; статус — только при создании.
9. **Категории — дерево** из `/categories/` (`parent_category`), пересчёт `category_path`.
   **`/collections/{slug}/` отдаёт 500 — не дёргать**, всё в эмбеде `collections[]` товара.
10. **Прочий payload** (`measurement_unit`, `areas_per_package`, `product_shape`, alt/описание картинок,
    детали бренда, `upsells/cross_sells`) — не в ядро, а в событие `oc1c.product.imported` (полный payload).
11. **SemVer + CHANGELOG** (версия в манифесте расширения), новые UI-строки — в `en-gb` + `ru-ru` перед
    сборкой.
