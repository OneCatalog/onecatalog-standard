## Главное про данные (меняет Q3 и Q4)

🟢 **Вариаций товара НЕТ вообще** — один `public_id` = один готовый товар (характеристики — фасеты, не
варианты). Это снимает любые сомнения про «варианты».

🟢 **Характеристики многозначны** — у товара «Цвет» = 2 значения, «Помещение» = **6**. Справочник
`/specifications/` отдаёт флаг **`multitude`** (true = многозначная; boolean-характеристики `false`).
В UMI это ложится на **множественное поле-`relation`** к справочнику (несколько объектов-значений).
Используйте `multitude`, чтобы выбрать кардинальность поля.

🟢 **`numeric` в этом инстансе не используется** (все 13 характеристик text/boolean).

🟢 **Иерархия категорий ЕСТЬ** — у товара `categories[]` плоский, но `/categories/` отдаёт
`parent_category` (Q4).

## Ответы на вопросы

**1. Тип объекта товара (существующий vs свой).** 🟡 **Рекомендую импортировать в существующий тип
«Товар»** магазина клиента (+ добавить служебные поля `oc_public_id`/`article`/поля-связи характеристик),
чтобы товары видела штатная витрина/`emarket`. Свой тип-наследник — изолированно, но товары «не увидит»
магазин. Эталон тоже пишет в нативный тип товара платформы. **Подтвердить наличие типа товара у клиента.**

**2. Цена и `emarket`.** 🟢 В источнике цены НЕТ → **ядро её не пишет** (§5.6). Если нужны продажи —
цену подставляет сайт через событие `oc_product_price` (как поле «только при создании»). Если «каталог-
витрина без продаж» — `emarket` не обязателен, товар = объект каталога. 🟡 Подтвердить: нужны ли продажи
(→ кто/как ставит цену) или только каталог. Активацию делать не падающей без `emarket`.

**3. Характеристики: стратегия А (справочник на характеристику) или Б (универсальный склад).** 🟢
**Рекомендация — гибрид, как на Magento (главная рекомендация там — ручной маппинг по умолчанию):**
- **Ручной маппинг ВКЛ (рекомендуемый дефолт на UMI):** справочники/поля под характеристики
  **предсоздаются заранее на admin-странице маппинга** (вне очереди!), а очередь только **привязывает
  значения**. Это стратегия А (фасетная фильтрация хорошая), но **без динамической правки схемы в фоне**
  — снимает ваш блокер 1.
- **Ручной маппинг ВЫКЛ (авто):** чтобы не менять схему в очереди — либо стратегия Б (универсальный
  склад), либо **пред-сканировать `/specifications/` и создать справочники один раз ДО прогона** (как на
  Magento).
Многозначные (`multitude`) → множественное поле-`relation`. Никогда не создавать тип/поле «по ходу
каждого товара». Это самый важный архитектурный совет для UMI.

**4. Множественные категории и иерархия.** 🟢 **Иерархия ЕСТЬ:** `/categories/` отдаёт
`parent_category` (или `null` = корень). Пример: «Керамогранит»(3) → «Плитка»(1) → корень. Тяните
`/categories/?limit=1000`, стройте дерево разделов (`addElement` с правильным родителем), find-or-create
по имени в пределах родителя. 🟡 Товар в нескольких разделах: основной раздел = родитель страницы,
дополнительные — штатным механизмом «товар в нескольких разделах» модуля `catalog` (имя сверить на
инстансе) или полем-связью «доп. категории». Эталон строит плоско — UMI может **лучше**.

**5. Версия UMI / API-стиль.** 🟡 Зависит от сборки клиента (`selector` vs `umiSelection`; сигнатуры
`addField`/`setGuideId`; `commit()`/переиндексация). 🟢 Согласен с «шагом 0 — разведка схемы на
инстансе» и инкапсуляцией работы со схемой в install/update + слой `Taxonomies`-аналога. **Подтвердить
версию/сборку и доступ к коду модулей.**

**6. Фон: cron или AJAX-степпер.** 🟡 **AJAX-степпер — основной** (детерминирован, не зависит от
расписания хостинга); `cron.php` + слушатель события `cron` — вторичный авто-добор, если cron реально
запускается. Эталонный Action Scheduler тоже cron/визит-зависим — не минус UMI. **Подтвердить, запускается
ли `cron.php` регулярно, и объём импорта** (шаг порции 10–20).

**7. Iframe-пикер / CSP.** 🟡 Не гарантировано, что админка UMI допустит сторонний iframe + `postMessage`
(CSP/`X-Frame-Options`). **Фолбэк (всегда работает) — импорт по списку `public_id`** (стандарт §2.4
допускает). Рекомендую сначала ID-list, пикер — опционально. `event.origin` явно.

**8. Медиа-хранилище.** 🟡 Обложка → поле `img_file`; галерея → множественное `file`/`relation`-поле
или модуль «Фотогалереи» (на ваш выбор/витрину). 🟢 Трекинг качества и контент-ключ — **собственная
таблица модуля `oc_media_index`** (у файлов UMI нет удобной произвольной меты, в отличие от WP-вложений).
Это прямой аналог эталонной меты `_onecatalog_file_key`. Дедуп по `sha1(path#size)`; общие файлы не
удалять при апгрейде.

**9. Мультиязычность.** 🟡 API отдаёт **один язык за запрос**. **Фаза 1 — один язык** (основной).
Мультиязык (UMI хранит языковые версии связанными страницами) = N проходов с разным `lang` + аккуратная
привязка версий — отдельная итерация (общий пробел стандарта, его отметили все агенты). Эталон тоже
один язык за прогон. **Подтвердить, нужен ли мультиязычный каталог.**

**10. Лимиты API / пагинация.** 🟢 Справочники (`/specifications/`, `/categories/`) — **одним запросом
`limit=1000`** (проверено; `meta.counts` = общее число, дефолт молча обрезает). Жёсткого rate-limit на
`X-API-Key` в тестах не наблюдалось. Товары — по одному `/products/{public_id}/`; базовый URL API
переопределяем (настройка). Ошибка → skip (§5.5).

**11. Артикул.** 🟢 `article` → штатное поле «Артикул» типа товара, если есть, иначе своё поле `article`
(только при непустом `article`, не синтезировать). Уникальность артикула на стороне UMI не требуется —
при коллизии не перезаписывать чужой, максимум лог. `public_id` — отдельно (не в артикул). 🟡
Подтвердить наличие штатного поля артикула.

**12. Права/роли.** 🟡 Отдельное право модуля `onecatalog/import` + CSRF-токен админки на каждый POST/
AJAX (аналог nonce эталона). Запускает — роль с этим правом. Стандартно.

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id`** (отдельное поле `oc_public_id`), `article` отдельно. Искать
   ВСЕГДА по `oc_public_id`, не по slug/name.
2. **Справочники — поиск по имени (label), не по слагу/`alt-name`** — критично: UMI генерит alt-name
   своей транслитерацией → разные слаги для одного имени → дубли (в эталоне был баг «Толщина»
   tolschina/tolshhina). Объект справочника по `name`; раздел по имени в пределах родителя. Стабильный
   ключ маппинга — `specification_id`/category id.
3. **Характеристики:** предсоздание справочников/полей вне очереди (Q3); многозначные → множественное
   `relation`; ручной маппинг по `specification_id`, строгий режим.
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие =
   false**; объекты-значения true/false из маппинга. (`multitude=false` у boolean.)
5. **Медиа:** дедуп по `sha1(path#size)` (своя `oc_media_index`); общие файлы не удалять; галерея по
   имени файла, не по URL; скачивать вручную + MIME по содержимому (URL без расширения).
6. **Не выдумывать цену** (UMI это позволяет); статус публикации — только при создании, не перезатирать.
7. **Единицы:** конвертер г/мм → целевые единицы (настройка модуля `oc_weight_unit`/`oc_dimension_unit`);
   `measurement_unit` (м²) — продажная единица, не габарит → событие.
8. **Категории — дерево** из `/categories/` (`parent_category`). **`/collections/{slug}/` 500 — не
   дёргать**, всё в эмбеде `collections[]` товара. **`/specifications/`, `/categories/` — `limit=1000`.**
9. **Прочий payload** (`measurement_unit`, `areas_per_package`, `product_shape`, alt/описание картинок,
   детали бренда, `upsells/cross_sells`) — через событие `oc_product_imported` (полный payload).
10. **SemVer + CHANGELOG**, версию поднимать перед сборкой; новые UI-строки — в языковые файлы модуля.
