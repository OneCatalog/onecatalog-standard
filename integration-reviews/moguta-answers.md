## Главное про данные (уточняет характеристики и категории)

🟢 **Вариаций товара НЕТ вообще** — один `public_id` = один готовый товар; `options[]` — фасеты, не
варианты. `color`/`size` нативные типы Moguta под них использовать не нужно.

🟢 **Характеристики многозначны** — у товара «Цвет» = 2 значения, «Помещение» = **6**. Справочник
`/specifications/` отдаёт флаг **`multitude`** (true = многозначная; boolean-характеристики `false`).
→ многозначные кладите в характеристику типа **«Набор»** с несколькими выбранными значениями. Используйте
`multitude`, чтобы понять, одно значение или несколько.

🟢 **`numeric` в этом инстансе не используется** (все 13 характеристик text/boolean).

🟢 **Иерархия категорий ЕСТЬ** — у товара `categories[]` плоский, но `/categories/` отдаёт
`parent_category` (Q ниже).

## Ответы на вопросы

**1–3. Бренд / страна / коллекция как сущности.** 🔎 Нативность — домен Moguta (вы верно пометили
низкую уверенность, нужен осмотр БД/админки). 🟡 Рекомендованные дефолты (паттерн «тип объекта + цель»):
- **Бренд** — характеристика «Набор» (фасет) ИЛИ категория-ветка. В WC бренд = нативная таксономия; на
  Moguta нативной сущности, видимо, нет → характеристика/категория. Доп. поля бренда эталон **не хранит
  даже в WC** (только имя+лого+id) → через событие `oc_brand_imported`.
- **Страна** — характеристика «Набор» «Страна производства» (НЕ путать со странами доставки). 🟢
  `country_code` в API — ISO **alpha-3** (`IND`,`ESP`), но при «характеристике по имени» (`menutitle`)
  код не нужен.
- **Коллекция** — **категория-ветка** (даёт страницу-листинг, для плитки обычно нужно) ИЛИ характеристика
  ИЛИ кастом-таблица. Поля коллекции (`link_3d`/`link_official_site`/лого) — через событие.

**4. Габариты товара.** 🔎 Наличие полей length/width/height у `mg_product` — проверить (в сигнатуре
`addProduct` их нет). 🟡 Если нет — писать в **характеристики-«Строка»** «Длина/Ширина/Высота» (с
конверсией мм→целевая единица), чтобы попадали в фильтры; либо целиком в событие. Эталон пишет их в
нативные поля WC — на Moguta нативного, похоже, нет. Для плитки 600×1200 размер важен → рекомендую
характеристики.

**5. Единицы веса магазина.** 🟢 Источник: вес — **граммы**, размеры — **миллиметры** (подписи
локализованы «мм»/«г», `weight_unit` бывает `null`). 🔎 В каких единицах `mg_product.weight` (кг?) —
проверить. 🟡 Конвертер (таблицы коэффициентов к базе + нормализация подписи) → целевую единицу из
настройки. Коэффициенты (эталонные): вес к г — `g=1, kg=1000, oz=28.3495, lb=453.592`; длина к мм —
`mm=1, cm=10, m=1000, in=25.4`. Без конвертера 1200 мм станут 1200 см.

**6. Мультиязычность.** 🟡 API отдаёт **один язык за запрос**. **Фаза 1 — один язык** (`ru` для
российского магазина). Мультиязык (если Moguta его поддерживает — 🔎 проверить) = N проходов с разным
`lang` — отдельная итерация (общий пробел стандарта). Эталон тоже один язык за прогон.

**7. i18n-механизм.** 🔎 Формат языковых файлов плагина Moguta — проверить. 🟢 Принцип эталона: исходные
строки EN в коде + перевод RU родным механизмом платформы (не WP-овский `.po/.mo`). Память проекта: UI
по-русски. JS-строки передавать из PHP локализованными.

**8. Загрузка/хранение изображений.** 🔎 Точный служебный метод сохранения картинки товара/галереи —
проверить в исходниках (`Models_Product` упоминает копирование картинок; есть `mg_product_image`?). 🟢
Принцип эталона: URL без расширения → скачивать вручную + **MIME по содержимому** (`finfo`) → сохранять с
правильным расширением → прописать в `image_url`/галерею. Не писать файлы «мимо ядра», если есть
служебный метод.

**9. Планировщик/фон.** 🟢 Согласен: в ядре очереди/cron нет; путь — **AJAX-степпер** (как штатный
пошаговый CSV/YML-импорт) + опц. хостинговый cron-endpoint (как у платного YML-плагина). Эталонный Action
Scheduler тоже cron/визит-зависим. Состояние очереди — в своей таблице/`setOption`.

**10. Хук установки плагина.** 🔎 Не подтверждён. 🟢 Согласен с обходом: **ленивое
`CREATE TABLE IF NOT EXISTS`** при загрузке + версия схемы в `setOption('oc_schema_ver')`; миграции по
флагу. Деинсталляция таблиц — вручную/скриптом.

**11. CSRF/права админ-AJAX.** 🔎 Штатный CSRF-токен и проверка роли в `Pactioner` — уточнить. 🟢
Принцип: AJAX-приём только для авторизованного администратора + CSRF-токен (штатный Moguta или свой в
опции) на каждый POST — аналог nonce эталона.

**12. Привязка характеристик к категориям.** 🟢 Подтверждаю важность: по доке Moguta характеристика без
привязки к разделу **не видна** в товаре → **импортёр ОБЯЗАН после find-or-create характеристики ещё и
привязать её к категориям импортируемого товара**. Это дополнительный шаг, которого нет в WC-эталоне (там
глобальные атрибуты видны всегда). 🔎 Метод привязки — уточнить. Заложите это в `Taxonomies`-слой.

**13. `CATALOG_PROP`.** 🔎 Нужно ли `CATALOG_PROP=0` ради чтения `characteristics[id]` — для ЗАПИСИ через
`userProperty`, по вашему же анализу, не требуется. Проверить только если читаете характеристики обратно
для сверки.

**14. Уникальность `code`.** 🟢 Не гарантируется → **перед записью `article` в `code` проверять коллизию,
не перезаписывать чужой** (эталон так же: SKU=`article`, при коллизии не трогать). `public_id` — отдельно
(в своей карте), не в `code`.

**15. `Controllers_Api` vs `Pactioner`.** 🔎 Что меньше «дерётся» с кэшем/хуками — оценить на инстансе.
🟢 Согласен: свой `Pactioner` для приёма списка public_id + оркестрации (штатный API «не расширяемый
хуками»); `Controllers_Api::importProduct` — опционально как низкоуровневый upsert внутри.

**16. Лимиты Wiki API / справочник.** 🟢 Справочник `/specifications/` (и `/categories/`) — **одним
запросом `limit=1000`** (проверено; `meta.counts` = общее число, дефолт молча обрезает). 🟢
`/collections/{slug}/` **отдаёт 500 — не дёргать**, все поля коллекции в эмбеде `collections[]` товара.
Жёсткого rate-limit на токен в тестах не наблюдалось; товары — по одному, ошибка → skip (§5.5).

**17. Кэш каталога.** 🔎 Есть ли служебный сброс кэша каталога/категорий — проверить. 🟡 После
массового импорта рекомендую инвалидировать кэш каталога, иначе товары могут не появиться на витрине без
ручного «Сбросить кэш». ⚠️ Также: `getProduct(..., $disableCashe=true)` при повторном чтении в одном
прогоне (вы это верно отметили).

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id`** (своя `mg_oc_product_map`, не в `code`), `code = article` (при
   непустом, без коллизии). Искать ВСЕГДА по `public_id`.
2. **Справочники — поиск по имени (label), не по url/слагу** — критично: Moguta генерит `url`
   транслитом → разные слаги для одного имени → дубли (в эталоне был баг «Толщина» tolschina/tolshhina).
   Категория/значение характеристики/бренд/страна/коллекция — по имени без учёта регистра; стабильный
   ключ маппинга — `specification_id`/category id.
3. **Характеристики:** «Набор» (фасет); многозначные (`multitude`) → несколько значений; ручной маппинг
   по `specification_id`, строгий режим; **обязательно привязывать характеристику к категориям товара**
   (Q12).
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие =
   false**; значения «Да»/«Нет» из маппинга.
5. **Медиа:** дедуп по `sha1(path#size)` (реестр в `quality_json`/своей таблице); общие файлы не удалять
   при апгрейде; галерея по имени файла, не по URL; скачивать вручную + MIME по содержимому.
6. **Не выдумывать цену** (`mg_product.price` оставлять 0, не перезатирать при апдейте); `activity`/статус
   — только при создании.
7. **Единицы:** конвертер г→кг / мм→единица магазина; `measurement_unit` (м²/шт) → `mg_product.unit` (это
   продажная единица — у Moguta для неё есть нативное поле, удобно!), НЕ путать с габаритами.
8. **Категории — дерево** из `/categories/` (`parent_category`): первую категорию → `cat_id`, остальные →
   `inside_cat`, иерархию строить по `parent`. **`/collections/{slug}/` 500 — не дёргать.**
   **`/specifications/`, `/categories/` — `limit=1000`.**
9. **Прочий payload** (`areas_per_package`, `product_shape`, alt/описание картинок, детали бренда,
   `upsells/cross_sells`) — через событие `oc_product_imported` (полный payload).
10. **SemVer + CHANGELOG**, версию поднимать перед сборкой; новые UI-строки — переводить.

> 💡 Приятный нюанс Moguta: для `measurement_unit` (м²/шт) есть нативное поле `mg_product.unit` — это
> лучше, чем у большинства CMS, где продажную единицу пришлось класть в событие. Пишите туда `name`
> единицы из payload.

## Что подтвердить / разведать на инстансе (шаг 0)

- CSRF-токен админ-AJAX и проверка роли в `Pactioner` — Q11.
