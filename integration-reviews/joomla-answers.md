## Главное про данные (уточняет Q1 и категории)

🟢 **Вариаций товара НЕТ вообще** — один `public_id` = один готовый товар. Это окончательно закрывает
вопрос «characteristics vs custom fields»: HikaShop `characteristic`/`variant` (механизм вариантов) **не
нужен в принципе** — у товара нет вариантов. Характеристики → **custom fields `item`** (см. Q1).

🟢 **Характеристики многозначны** — у товара «Цвет» = 2 значения, «Помещение» = **6**. Справочник
`/specifications/` отдаёт флаг **`multitude`** (true = многозначная; boolean-характеристики `false`).
В HikaShop это ложится на **multi-select custom field** (тип `checkbox`/`multipledropdown`), который
держит несколько значений. Используйте `multitude`, чтобы выбрать тип поля (single vs multi).

🟢 **`numeric` в этом инстансе не используется** (все 13 характеристик text/boolean).

## Ответы на вопросы

**1. Куда класть характеристики `options[]`.** 🟢 **Custom fields `item` (`hikashop_field`), НЕ
characteristics/variants** — подтверждаю однозначно. characteristics в HikaShop порождают **варианты**
(комбинации цены/склада/кода); у нашего товара вариантов нет → они только наплодят лишние строки и
сломают чекаут. Это тот же класс ошибки, что `option`-vs-`attribute` в OpenCart и `Feature`-vs-`Attribute`
в PrestaShop. 🟡 Фасетная фильтрация: HikaShop умеет фильтры по **custom fields** (модуль фильтров /
`filter`), так что фасетка достижима — но включить надо отдельно. Многозначные (`multitude`) →
multi-select поле (Q «главное про данные»). Ручной маппинг по `specification_id`, строгий режим.

**2. `public_id` — своя таблица или custom field.** 🟢 **Своя JTable `#__onecatalog_product_link`
(public_id UNIQUE → product_id)** — правильно, точный аналог эталона (ключ в выделенном хранилище, не в
`product_code`/UI-поле). public_id ≠ product_code. Поиск/уникальность на уровне БД.

**3. Бренд: manufacturer достаточно?** 🟢 Дефолт **manufacturer** (`hikashop_category`
`category_type='manufacturer'` + `product_manufacturer_id`) — ок, нативно. Расширенные поля бренда
(`description_text`, `foundation_year`, `website`, `country`, лого) эталон **не сохраняет даже в WC**
(только имя+лого+id) → это «прочий payload» → через событие `onOnecatalogBrandImported`/поля
manufacturer, если реально нужны лендинги брендов. 🟡 Подтвердить, нужны ли страницы брендов с
произвольными полями (→ своя сущность).

**4. Коллекции: своя JTable / категория / custom field.** 🟡 Эталонный дефолт коллекции — отдельная
сущность (CPT в WC). В Joomla CPT нет. **Дефолт — своя JTable `#__onecatalog_collection`** (есть поля
`link_3d`, `link_official_site`, обложка, которых нет у категории). Альтернативы: `hikashop_category`-
ветка / custom field. Эталон недавно добавил и режим «атрибут» — на HikaShop это custom field. Итог:
JTable (дефолт) / category / custom field. 🟡 Подтвердить, есть ли понятие «коллекция» на витрине
(отдельные страницы → JTable).

**5. Теги: Joomla `#__tags` или custom field.** 🟡 Верно — HikaShop-товар не нативный Joomla content
item, `com_tags` к нему «из коробки» не привяжется. Эталон кладёт теги в нативную таксономию WC; у
Joomla+HikaShop прямого аналога для товара нет. **Рекомендация — custom field-тег** (multi-select),
проще и работает в фильтрах HikaShop; свой контекст в `#__contentitem_tag_map` — если нужен именно UI
Joomla-тегов (сложнее). Теги по умолчанию ВЫКЛ — на старте не критично.

**6. CSP/iframe в админке.** 🟡 Joomla-админка (особенно с Admin Tools/RSFirewall + строгий CSP) может
блокировать внешний iframe. **Фолбэк (всегда работает) — импорт по списку `public_id`** (стандарт §2.4
допускает). Рекомендую сначала ID-list, пикер — опционально при совместимом CSP. `event.origin` явно.

**7. Мультиязычность каталога.** 🟡 **Самый тяжёлый конфликт — согласен.** Фаза 1 — **один язык**
(API отдаёт один `lang` за запрос; пишем в основной язык). Мультиязык: N проходов с разным `?lang=` +
запись переводов через **Falang** (зависимость от стороннего расширения; Joomla-Associations к
HikaShop-товарам неприменимы) — отдельная крупная фаза. Эталон тоже импортирует один язык за прогон
(мультиязык — общий пробел стандарта, его отметили все агенты). **Подтвердить:** нужен ли мультиязычный
каталог; если да — Falang или дубли товаров.

**8. Фоновая очередь / CLI-cron.** 🟡 Зависит от хостинга. **AJAX-степпер — основной** (Joomla Scheduler
4.1+ в lazy-режиме ненадёжен для пакетов — верно); CLI-cron `scheduler:run` → можно задействовать
Scheduler task для авто-импорта. Эталонный Action Scheduler тоже cron/визит-зависим. **Подтвердить
наличие CLI-cron.**

**9. Цена.** 🟢 В источнике цены НЕТ. **HikaShop-товар без цены валиден** (цена — отдельная строка
`hikashop_price`; нет строки → товар в каталожном режиме, «free»/скрыто по настройке). → **не создавать
ценовую запись при создании, не трогать при апдейте**; реальную цену даёт сайт через событие (аналог
`filterOneCatalogProductPrice`). Это НЕ блокер (как у OpenCart/PrestaShop, в отличие от
Magento/BigCommerce, где цена обязательна). 🟡 Подтвердить настройки отображения «нулёвок».

**10. Версии Joomla/HikaShop.** 🟡 Зафиксировать минимальные (Joomla 4.4+/5.x; HikaShop 4.x — минорную).
🟢 Согласен с «шагом 0 — разведка схемы на инстансе» для имён полей (`product_dimensions` формат,
`class.file->updateFiles`) — это правильно (не кодировать жёстко). **Подтвердить версии.**

**11. Наличие (`in_stock`).** 🟢 Факт API: есть `in_stock` (булев), **количества (qty) в источнике
НЕТ**. Рекомендация: при `in_stock=true` — `product_quantity = -1` (неограниченно/в наличии), при
`false` — 0; только индикатор, реальные остатки ведёт магазин. Эталон ставит наличие при импорте.
🟡 Подтвердить, нужно ли вообще трогать склад или только индикатор.

**12. Единицы HikaShop.** 🟢 OneCatalog: вес — **граммы**, размеры — **миллиметры** (подписи
локализованы «мм»/«г», `weight_unit` бывает `null`). Целевые `product_weight_unit`/`product_dimension_unit`
брать из настроек магазина в рантайме; конвертер (таблицы коэффициентов + нормализация подписи) →
целевую единицу. Коэффициенты (эталонные): вес к г — `g=1, kg=1000, oz=28.3495, lb=453.592`; длина к
мм — `mm=1, cm=10, m=1000, in=25.4`. `measurement_unit` (м²/шт) — продажная единица, НЕ габарит →
событие/custom field. 🟡 Подтвердить, что целевые единицы в магазине заданы.

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id`** (своя `#__onecatalog_product_link`), `product_code = article` (при
   непустом, без коллизии). Искать ВСЕГДА по public_id, не по product_code/name.
2. **Справочники — поиск по имени (label), не по alias/slug** — критично: HikaShop/Joomla/Falang
   транслитерируют по-разному → разные alias для одного имени → дубли (в эталоне был баг «Толщина»
   tolschina/tolshhina). Категория по `category_name` в пределах `category_parent_id`; manufacturer по
   `category_name`; значение custom field по тексту. Стабильный ключ маппинга — `specification_id`/
   category id.
3. **Характеристики — custom fields `item`, не characteristics/variants** (Q1); многозначные
   (`multitude`) → multi-select поле; ручной маппинг по `specification_id`, строгий режим.
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие =
   false**; термы true/false из маппинга. (`multitude=false` у boolean.)
5. **Медиа:** дедуп по `sha1(path#size)` (реестр `#__onecatalog_media_key`); общие файлы НЕ удалять при
   апгрейде; галерея по имени файла, не по URL; скачивать вручную + MIME по содержимому (URL без
   расширения).
6. **Цена не задаётся** (HikaShop это позволяет); статус — только при создании, не перезатирать.
7. **Единицы:** конвертер г/мм → единицы HikaShop; `measurement_unit` (м²) — продажная, не габарит → событие.
8. **Категории — дерево** из `/categories/` (`parent_category` ЕСТЬ в справочнике, хотя у товара
   `categories[]` плоский) — стройте реальную иерархию (`category_parent_id`), а не всё под одним корнем.
   **`/collections/{slug}/` 500 — не дёргать**, всё в эмбеде `collections[]` товара.
   **`/specifications/`, `/categories/` — одним запросом `limit=1000`.**
9. **Прочий payload** (`measurement_unit`, `areas_per_package`, `product_shape`, alt/описание картинок,
   детали бренда, `upsells/cross_sells`) — через событие `onOnecatalogProductImported` (полный payload).
10. **SemVer + CHANGELOG**, версию поднимать во всех местах манифеста перед сборкой; новые UI-строки — в
    оба `.ini`.
