## Ответы на вопросы

### 1. Целевой инфоблок: создавать модулем или брать существующий? 🟡
**Решение: создавать СВОЙ инфоблок (`ONECATALOG_CATALOG`) по умолчанию, но `CATALOG_IBLOCK_ID`
оставить настройкой** для импорта в существующий каталог. Свой инфоблок = чистая схема свойств, нет
конфликта кодов; импорт в действующий — для сайтов, где каталог уже есть. Эталон тоже создаёт свои
сущности (CPT/таксономии), а не лезет в чужие. Для ceram это неактуально (там WP), так что решает
владелец конкретного Bitrix-сайта. **Дефолт — свой инфоблок.**

### 2. Простой каталог или торговые предложения (SKU)? 🟢
**Простой каталог, ТП НЕ нужны.** OneCatalog не имеет вариаций товара: один `public_id` = один
готовый товар; `options[]` — это характеристики-фасеты (цвет, формат, поверхность…), а не
SKU/вариации. Эталон импортирует один `WC_Product_Simple` без вариаций. Если витрина сайта жёстко
построена на ТП-компонентах — заверните товар как один оффер, но логически вариаций нет.

### 3. public_id → `XML_ID`: конфликт с обменом 1С? 🟢 рекомендация
**Не кладите public_id в `XML_ID`. Используйте отдельное строковое свойство `OC_PUBLIC_ID`.**
Причина: `XML_ID` — рабочее поле штатного обмена `1С ↔ Битрикс` (CommerceML), он перезапишет его
GUID-ами и сломает идемпотентность. Эталон делает ровно так: идентификатор лежит в **отдельной мете**
`_onecatalog_public_id`, а НЕ в slug/ID поста (по нему и find-before-write). Так вы разом снимаете и
вопрос 3, и вопрос 4. **Дефолт — отдельное свойство, не `XML_ID`** (даже если обмена 1С нет — это
дешёвая страховка и единообразие с эталоном).

### 4. Уникальность `XML_ID` / гонки. 🟢
Снимается решением вопроса 3 (ключ в своём свойстве). Но даже по сути: в эталоне WP-мета **тоже не
уникальна на уровне БД** — уникальность обеспечивает САМ модуль (always find-by-key перед записью).
Это не слабость Битрикса, а общий паттерн. От гонок защищает **очередь**: одна и та же `public_id`
не должна обрабатываться параллельно — дедуп в очереди (find-or-skip по статусу) + последовательная
обработка порций. Жёсткий БД-constraint не нужен. Достаточно: find-by-`OC_PUBLIC_ID` перед Add +
дедуп очереди.

### 5. Тип свойства для характеристик. 🟢
**Делайте тип по `specification_type` — это правильнее «всё в список».**
- 🟢 Факт API: в текущем инстансе все 13 характеристик — `text` или `boolean`, **`numeric` не
  используется ни одним товаром** (тип в спецификации возможен, но пуст). Так что numeric — задел на
  будущее, не блокер.
- Рекомендация: `text` → свойство-список (L); `boolean` → список с двумя enum (значения true/false из
  маппинга, дефолт «Да»/«Нет»); `numeric` → свойство «число» (N) с `XML_ID='oc_spec_<id>'`. Эталон
  для boolean именно так и делает (настраиваемые термы true/false), а numeric не спецобрабатывает
  (положил бы как текст-терм) — ваш подход с типом N лучше.
- HL-блоки-справочники — опционально (нужен модуль `highloadblock`, см. в.10). **Дефолт — plain
  enum-списки** (работают на всех редакциях).
- 🟢 КРИТИЧНО (урок эталона): enum-значения и любые справочники искать **по человекочитаемому VALUE
  (label), без учёта регистра, а НЕ по `CODE`/слагу**. Разные транслитерации дают разные слаги →
  дубли (в эталоне был реальный баг: «Толщина» → `tolschina` vs `tolshhina` от плагина Cyr-to-Lat).
  Стабильный ключ для маппинга — `specification_id` / `specification_option_id`; слаг — только при
  создании нового.

### 6. Иерархия категорий. 🟢
**Иерархия ЕСТЬ — стройте дерево.** Факт API:
- У товара `categories[]` — **плоский** (`{id, menutitle, slug}`, без родителя).
- Но справочник **`/categories/` отдаёт `parent_category: {id, slug, menutitle}`** (или `null`).
  Пример: id=3 «Керамогранит» → `parent_category` id=1 «Плитка»; id=1 «Плитка» → `null` (корень).
- Алгоритм: один раз тянем `/categories/?limit=1000` (high limit — см. в.12), строим карту
  `id → parent`, и при привязке товара резолвим/создаём ВСЮ цепочку до корня (`IBLOCK_SECTION_ID`),
  а не плоско. По стабильному ключу раздела `XML_ID = id категории` (фолбэк — по `NAME`/label).
- 🟢 Эталон (WC) сейчас использует только `menutitle` категории товара плоско — **вы можете сделать
  лучше**, задействовав `/categories/` для дерева. Для плитки это важно (Плитка → Керамогранит →…).

### 7. Единицы габаритов товара. 🟢
**Да: физический вес → `CCatalogProduct.WEIGHT` (граммы), размеры → `WIDTH/LENGTH/HEIGHT` (мм)**, НЕ в
произвольные свойства. Маппинг полей: OneCatalog `sizes.length`→LENGTH, `width`→WIDTH, `height`→HEIGHT
(пример: 1200×600×85 мм плитка). Внутренние единицы Битрикса (г/мм) совпадают с OneCatalog → конверсия
почти тождественна, но **конвертер всё равно обязателен**: нормализовать локализованную подпись
(`length_unit` приходит как объект `{id:"мм", label:"мм"}`, вес/`weight_unit` бывают `null`) и
подстраховаться, если источник отдаст кг/см.
- 🟡 `measurement_unit` товара (факт API: `{id:1, slug:"m2", name:"м2"}`) — это **продажная** единица
  (м²/шт/компл), НЕ физический габарит. Не путать с `WEIGHT/WIDTH`. По стандарту это «прочее» →
  сайтовый слой (событие `OnAfterProductImported`) или своё свойство-список «единица». В ядро габаритов
  — не пишем.

### 8. Уникальность артикула. 🟢
**Не блокировать импорт при коллизии; уникальность не требуется.** Пишите `article` в свойство
`ARTICLE` как есть; если `article` пуст — свойство не заполняем (не синтезировать). Эталон ставит SKU =
`article` и при коллизии с другим товаром просто НЕ перезаписывает (SKU в WC уникален технически), но
это особенность WC, а не требование стандарта. В Битриксе артикул — обычное свойство, уникальность не
обязательна. Максимум — лог при коллизии. Блокировать запись не нужно.

### 9. Статусы товара. 🟢 + 🟡
- 🟢 Факт API: товар отдаёт `status_publish: "publish"`. Эталон его **игнорирует** и использует свою
  настройку «статус новых», применяя её **только при создании** (при переимпорте статус НЕ трогает,
  чтобы не затирать ручные правки).
- 🟡 Рекомендация для Битрикса: настройка «статус новых» → `ACTIVE='Y'|'N'` (+ опц. `ACTIVE_FROM`),
  **только при создании**, при апдейте — не трогать. `pending/draft/private` нативного аналога не
  имеют — не городите статус-свойство без явной потребности; `Y/N` достаточно. Если хотите — можно
  опционально уважать `status_publish` из API (мапить `publish→Y`, прочее→`N`), но эталонный путь —
  своя настройка.

### 10. HL-блоки: редакция и права. 🟡
Зависит от редакции/версии конкретного сайта (модуль `highloadblock`). **Сделайте HL опциональным, а
дефолтом — свойства-списки (L), которые работают везде.** Если HL доступен и выбран — создаёте HL-блок
при установке (с проверкой наличия модуля). Это снимает зависимость от редакции. **Рекомендую изменить
ваш дефолт бренда с HL на список (L)** — иначе установка падает там, где нет `highloadblock`. HL — как
opt-in «продвинутый справочник с доп. полями».

### 11. Очередь: cron настроен? 🟡
Зависит от хостинга — подтвердить у владельца. **AJAX-степпер — основной механизм** (работает без
cron, но нужна открытая вкладка админа). Агенты-на-cron — фоновый фолбэк, ТОЛЬКО если системный cron
есть. Важно: эталонный Action Scheduler в WP **тоже** полагается на WP-Cron/async (ненадёжен без
реального cron) — это не минус Битрикса. Поэтому: степпер как primary; «фон при закрытой вкладке»
честно документировать как «требует системного cron».

### 12. Лимиты API и пагинация. 🟢
- Списки/справочники пагинируются `?start&limit`; `meta.counts` = ОБЩЕЕ число; **дефолтный лимит
  инстанса молча обрезает** → справочники (`/specifications/`, `/categories/`, `/measurement-units/`)
  брать **одним запросом с `limit=1000`** (проверено — работает).
- Жёсткого rate-limit на `X-API-Key` в тестах не наблюдалось (высокий объём отрабатывал). Но ведите
  себя вежливо: товары тянуть по одному `/products/{public_id}/`, таймаут ~40с, ошибки → skip (§5.5).
- 🟡 Реальный объём (десятки/тысячи) — уточнить у владельца; влияет на размер порции (10–50) и тайминги.
  Для плиточного каталога обычно сотни–тысячи позиций.

### 13. Пикер (iframe) в админке Битрикса. 🟡
Зависит от CSP/`X-Frame-Options` инстанса. **Реализуйте СНАЧАЛА путь «импорт по вставленному списку
`public_id`»** (стандарт §2.4 это явно допускает — он всегда работает), а iframe-пикер — как
опциональное улучшение там, где CSP позволяет. Для плиточного кейса пикер — nice-to-have, не
обязателен на первом этапе. `postMessage` + проверка `event.origin` сделать, но не закладываться на
гарантированное встраивание.

### 14. Язык и мультиязычность. 🟡
API отдаёт **один язык за запрос** (`lang`). **Фаза 1 — один язык** (для RU-сайта `LANG=ru`).
Мультиязычный Битрикс (классически «инфоблок на язык») + связка `lang`↔языковая версия + сохранение
идемпотентности по `public_id` между языковыми инфоблоками — отдельная крупная фаза. Эталон тоже
импортирует один язык за раз (мультиязык — общий пробел стандарта, его отметили все агенты).
Подтвердить одноязычность у владельца; если мультиязык нужен — планировать отдельно.

### 15. Версия Битрикса и редакция. 🟡
Уточнить точную версию/редакцию. Рекомендация: **D7-API** (`Bitrix\Main\Web\HttpClient`, ORM, HL) для
современных установок, но операции с каталогом — это по-прежнему legacy-API (`CIBlockElement`,
`CCatalogProduct`, `CIBlockSection`, `CFile`) даже в D7. Смешанный стиль (D7 для HTTP/ORM/настроек +
`CIBlock*` для CRUD каталога) — нормальная практика. Минимальная версия — там, где есть `HttpClient`
и ORM.

### 16. Дедуп медиа: где хранить контент-ключ. 🟢
**Своя таблица `oc_media` — правильно и достаточно** (HL-блок необязателен). Это прямой аналог
эталона: в WC контент-ключ хранится мета-полем вложения `_onecatalog_file_key = sha1(path#size)`; у
Битрикса `CFile` меты нет → своя таблица это и заменяет (`content_key`, `size`, `FILE_ID`, `shared`).
Создание таблицы при установке — приемлемо и ожидаемо.

## Дополнительно — что сделает интеграцию лучше (золото эталона)

Не было в ваших вопросах, но это выстраданные решения эталона — заложите сразу:

1. **Идемпотентность по `public_id`, SKU/артикул отдельно.** `OC_PUBLIC_ID` — ключ; `ARTICLE` =
   `article` (≠ ключ). Никогда не искать по названию/слагу для идемпотентности товара.
2. **Справочники — поиск по ИМЕНИ (label), не по слагу** (защита от дублей при разных транслитах;
   стабильный ключ — id OneCatalog). Касается категорий, enum-значений, брендов, стран, тегов.
3. **Ручной маппинг характеристик** — по стабильному `specification_id` (не по переводимому названию),
   строгий режим: при ВКЛ импортируются только сопоставленные характеристики. Список характеристик —
   из `/specifications/` (`limit=1000`).
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие
   = false**. Термы для true/false задаются в маппинге (дефолт «Да»/«Нет»).
5. **Медиа:** дедуп по `sha1(path#size)` (из base64-префикса URL извлечь `path`); общие/дедуплицированные
   файлы НЕ удалять при апгрейде качества (флаг `shared`/счётчик ссылок). Галерея — по имени файла, не
   по URL (подписи меняются).
6. **Не выдумывать данные:** цены в API нет → не задавать (поля «только при создании» не перезатирать
   при апдейте). Цена/статус — через событие/настройку, только при создании.
7. **Единицы:** конвертер обязателен даже при совпадении г/мм (нормализация локализованной подписи +
   защита от кг/см); `measurement_unit` (м²/шт) — продажная, не физический габарит.
8. **`/collections/{slug}/` отдаёт 500 — не дёргать**; все поля коллекции уже в эмбеде `collections[]`
   товара.
9. **SemVer + CHANGELOG** (бэклог = «Не выпущено»), поднимать версию перед сборкой; новые строки UI — в
   оба `lang/`.
10. **Прочие поля payload** (`measurement_unit`, `areas_per_package`, `product_shape`, alt/описание
    картинок, детали бренда `description_text/foundation_year/website`, `upsells/cross_sells`) — не в
    ядро, а в событие `OnAfterProductImported` (полный payload передавать в событие).

## Что обязательно подтвердить у владельца целевого Bitrix-сайта (перед стартом)

Эти ответы зависят от конкретного развёртывания, не от API:

- Используется ли обмен **1С ↔ Битрикс** (→ ключ `OC_PUBLIC_ID`, не `XML_ID`) — **в.3** (критично).
- **Свой** инфоблок или импорт в существующий каталог — **в.1**.
- Есть ли **системный cron** (→ фон при закрытой вкладке) — **в.11**.
- Сайт **одноязычный** (RU)? — **в.14**.
- **Версия/редакция** Битрикса, доступность `highloadblock` — **в.10, в.15**.
- **CSP** админки разрешает внешний iframe (пикер) — **в.13**.
- Ожидаемый **объём** каталога — **в.12**.
- Витрина на **типовых Bitrix-компонентах** (→ нужна ли `CPrice=0`) или headless — **блокер 12**.
