# Ответы RetailCRM-разработчику — OneCatalog Import

Ответы на [retailcrm.md](retailcrm.md) (18 вопросов + 8 блокеров). Маркировка:
🟢 — авторитетно (проверено на живом Wiki API / решение эталона WooCommerce `onecatalog-import` v1.7.0);
🟡 — зависит от версии RetailCRM/ICML или продуктовое решение; рекомендация + что подтвердить.

> ⚠️ Эталонная витрина `ceram-online` — headless WordPress+Nuxt, не RetailCRM. Вопросы про механику
> платформы (обязательность `<price>` в ICML, единица веса, поддержка габаритов, набор `store/*` методов,
> поведение при пропавших товарах, уникальность externalId на уровне БД, manifest модуля, rate-limit) —
> version-specific; по ним подтверждаю ваш подход «проверить по API/ICML версии» и даю контракт со стороны
> стандарта, эндпоинты/поля RetailCRM не выдумываю. API/факты OneCatalog отвечаю точно.

> 👏 Рамка верная и важная: **RetailCRM — каталог для заказов/маркетинга, не витрина**; витринные
> инварианты стандарта (страницы коллекций/брендов, фасеты, SEO, alt) на ней не живут — это сайт,
> синхронизируемый с CRM. Выбор ICML основным путём правильный. Ниже — факты данных, которые **снимают
> часть рисков** (особенно прокси картинок Q7/блокер 5 и иерархию Q14) + подтверждения.

## Главное про данные

🟢 **Вариаций товара НЕТ вообще** — один `public_id` = один готовый товар → **один product → один offer**
(вы верно это вывели). `options[]` — характеристики-фасеты, не SKU/варианты; selectable/комбинации не
порождать.

🟢 **Иерархия категорий ЕСТЬ — ответ на Q14 и §«categories».** Вы написали «`categories[]` плоский, нужна
разведка `/categories/`». Подтверждаю авторитетно: эмбед `categories[]` в товаре плоский, но справочный
**`/categories/` отдаёт `parent_category`** (id родителя или `null` = корень). → тяните
`/categories/?limit=1000`, стройте `<category id parentId>` дерево (родители раньше детей в фиде),
`externalId` группы = стабильный id категории. Не «плоско под корнем», а полноценное дерево.

🟢 **Характеристики многозначны** — «Цвет» = 2 значения, «Помещение» = **6**. Справочник
`/specifications/` отдаёт флаг **`multitude`** (true = многозначная; boolean `false`). В ICML многозначную
кладите как **несколько `<param name="Помещение">` с разными значениями** (ICML допускает повторяющиеся
param). Используйте `multitude`.

🟢 **`numeric` в этом инстансе не используется** (все 13 характеристик text/boolean) — код пишите
универсально.

🟢 **Цены в источнике НЕТ** (факт API, не упущение); **`sizes.weight` часто `null`** (у OC.IND.1 веса
нет); `country_code` — ISO **alpha-3**. `measurement_unit` (м²/шт) — **продажная** единица, не габарит.

## Ответы на вопросы

**1. ICML или API — основной?** 🟢 **ICML — правильный основной путь.** Совпадает с философией стандарта
«источник истины внешний, локально — проекция» (фид и есть проекция); идемпотентность/дедуп/расписание/
загрузка картинок — на стороне RetailCRM, нам не строить очередь find-or-create. REST API v5 (`store/*`)
— дополнение для точечных операций. 🟡 Точный набор методов записи `store/*` — **проверить по версии**.

**2. Идемпотентность по `externalId`.** 🟢 `public_id` → `externalId` товара/offer — верный ключ
(стабильный бизнес-ключ источника), не slug/имя (§5.1). 🟡 Жёсткость уникальности externalId на уровне БД
RetailCRM vs «по соглашению при импорте» — **проверить**; при ICML коллизий не будет, если наш генератор
не выдаёт дубль externalId (валидация уникальности при генерации фида — наша забота).

**3. Характеристики → свойства каталога, НЕ customFields заказа.** 🟢 **Подтверждаю — критично и верно.**
`options[]` → `<param>`/properties **товара**; `customFields`/`customDictionary` — поля order/customer/
company, это другая подсистема. Не смешивать (ваш блокер 1 — самая дорогая ошибка). `code` свойства =
стабильный `oc_spec_<specification_id>`, `name` = label. Ключ маппинга — `specification_id`, не
переводимый label. 🟡 Типизируются ли params (numeric/boolean) автоматически — **проверить**; если нет,
boolean кладём текстом (см. ниже).

**4. ICML конкретной версии: какие элементы поддержаны.** 🟡 Чисто платформенное — **разведка ICML
обязательна** (обязательность `<price>`, `<weight>`/единица, габариты/`<dimensions>`, множественные
`<picture>`, `<vendor>`, `<param code>`, `parentId`, поведение при пропаже поля/товара). Контракт со
стороны данных даю по каждому пункту ниже (Q5/Q7/Q10/Q16).

**5. ЦЕНА.** 🟢 В источнике цены НЕТ — **не синтезировать** (§5.6). 🟡 Но RetailCRM — система **заказов**,
`<price>` у offer вероятно обязателен (конфликт инварианта с природой платформы — ваш блокер 2, верно).
Решение продуктовое (Q6): (а) реальный прайс-источник у заказчика → подставлять; (б) цена приходит из
другой системы (1С/сайт), наш фид — описательная часть; (в) `<price>0</price>` (риск «нулей» в заказах).
Эталон цену не задаёт (фильтр `onecatalog_product_price`) — тот же принцип: ядро не выдумывает, источник
цены задаёт интегратор/заказчик. **Решать с заказчиком.**

**6. Нужен ли каталог в RetailCRM для ЗАКАЗОВ или справочно?** 🟡 Ключевой стратегический вопрос (верно
вынесли). Если RetailCRM реально обрабатывает заказы по этим товарам — цена/остатки/единицы обязательны
(серьёзная интеграция). Если цель — обогатить карточки витрины (а витрина отдельная) — возможно, RetailCRM
не целевая система для этого каталога. **Определить роль RetailCRM в связке — до архитектуры.**

**7. Прокси картинок.** 🟢 **Хорошая новость — прокси по причине токена/качества НЕ нужен.** Бинарный URL
`…/media_files/<base64hash>` **самодостаточен**: токен зашит в base64-хеш, и файл качается **без заголовка
`X-API-Key`** (в эталоне `Media::sideload` берёт его обычным GET и получает max при наличии токена). Схема:
наш сервис-генератор (с токеном) вызывает `/products/`, получает `images_urls.max` (URL уже кодирует max),
кладёт ЭТОТ URL в `<picture>` — **RetailCRM скачивает его сам и получает max-качество**, токен ему не
нужен. `X-API-Key` нужен только JSON-вызову `/products/`, чтобы он ВЕРНУЛ max-URL. 🟡 Остаётся один риск —
**URL без расширения**: проглотит ли загрузчик RetailCRM URL без `.jpg` и определит ли MIME. Если строго
требует расширение — тогда (и только тогда) лёгкий прокси нашего сервиса, отдающий тот же бинарь по
чистому URL с расширением и корректным `Content-Type`. **Проверить поведение загрузчика на extensionless
URL.** Это сильно упрощает архитектуру против вашего исходного допущения.

**8. Бренд → `<vendor>` или property.** 🟢 **`<vendor>` — дефолт** (нативное поле производителя, лучший
аналог). find-or-create по `brand.menutitle`. Доп. поля бренда (страна/год/сайт/лого `brand.images_urls`)
в каталог RetailCRM не ложатся (нет карточки бренда) → витрина/событие. ОК. Альтернатива `<param "Бренд">`
— настройкой.

**9. Страна/коллекция/теги — нет нативных сущностей.** 🟢 Подтверждаю — `country`/`collections`/`tags` →
`<param>` (свойства), без витринных страниц/фасетов (их в RetailCRM нет — это сайт). `country` (ISO
alpha-3 + локализованное имя), коллекция (поля из эмбеда `collections[]`, `/collections/{slug}/` = 500 не
дёргать), теги (по умолчанию выкл, по `title`). Достаточность свойств для нужд заказчика — **подтвердить**.

**10. Единицы и вес.** 🟢 Источник: **вес в граммах, размеры в миллиметрах** (подпись локализована на
языке `lang`; `sizes.*_unit`, иначе мм/г). **`weight` бывает `null`** — не выдумывать. Конвертер г/мм →
единица RetailCRM **обязателен** (иначе 1200 мм → 1200 см); нормализовать обе раскладки подписей («мм/mm»,
«г/g»). 🟡 Единица веса offer (кг/г), нативная поддержка габаритов, справочник единиц (`unit`) для
`measurement_unit` (м²/шт) — **проверить**; если габаритов нет — в `<param>` (деградация, не блокер).
`measurement_unit` — продажная единица, не путать с единицей габаритов (§5.6).

**11. Мультиязычность.** 🟢🟡 API отдаёт **один язык за запрос**. Для ceram — `ru`, **один язык на
старте**. RetailCRM каталог одноязычный (мультиязык = несколько фидов/инстансов — отдельная фаза). Эталон
тоже один язык за прогон. **Подтвердить достаточность одного языка.**

**12. Rate-limit API v5 (путь B).** 🟡 Лимит/окно на apiKey — **проверить по версии**; для пути B — порции
+ паузы, backoff на 429/503. Для ICML rate-limit RetailCRM не важен (фид тянет он сам). Объём импорта —
**подтвердить** (влияет на размер/частоту фида).

**13. Дефолтный `limit` Wiki API и пагинация.** 🟢 Справочники (`/specifications/`, `/categories/`,
`/countries/`) — **одним запросом `limit=1000`**; дефолтный лимит инстанса молча обрезает → сверяться с
**`meta.counts`** (общее число) и дострелить пагинацией. Жёсткого rate-limit на `X-API-Key` OneCatalog в
тестах не наблюдалось. Товары — по одному `/products/{public_id}/`. Это стратегия генерации фида.

**14. Иерархия категорий.** 🟢 **Дерево СТРОИТЬ** — `/categories/` отдаёт `parent_category` (см. «главное
про данные»). `<category id parentId>`, родители раньше детей.

**15. Тип поставки и ЖЦ.** 🟡 Для ceram — **скорее приватный сервис** (apiKey аккаунта + публичный URL
ICML-фида в настройках магазина), не публичный модуль маркетплейса. Manifest/OAuth/регистрация — только
если тиражируемый продукт. **Подтвердить.** SemVer + CHANGELOG ведём для нашего сервиса.

**16. Поведение при пропавших товарах.** 🟡 Платформенное — **проверить**, деактивирует/удаляет ли
RetailCRM offer, исчезнувший из фида, или оставляет. Со стороны стандарта: нужна осознанная политика
«мягкой деактивации» снятых товаров. Связано с §5.6 «поля только при создании» (см. блокер 8).

**17. Сверка существующего каталога.** 🟡 Реальный риск (как `XML_ID`↔1С в Битриксе). Если в аккаунте уже
есть каталог (1С/сайт) — наши `externalId` могут пересечься. Рекомендация: **неймспейс externalId**
(префикс/схема, чтобы OC-товары не сталкивались с чужими) или опция «не трогать не-OC offer»;
зафиксировать, кто «владеет» какими offer. **Подтвердить наличие другого обмена.**

**18. Пикер OneCatalog (iframe).** 🟢 Для ICML **пикер не нужен** — фид батчевый по всему выбранному
набору; набор `public_id` фиксируется конфигом/файлом (стандарт §2.4 допускает импорт по списку без
виджета). Если UI-выбор всё же нужен — он в нашем сервисе, не в RetailCRM (где iframe маркетплейс-модуля
под CSP). Рекомендую конфиг/список.

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id` → `externalId`** (товар/offer), `article` отдельно в поле артикула
   (только непустой; не синтезировать). Никогда не по slug/имени (§5.1).
2. **Справочники — стабильный ключ, не транслит:** категория — `externalId`=id из OneCatalog; свойство —
   `code=oc_spec_<specification_id>`; бренд — по имени `vendor`. Имя (label) — подпись, не ключ (в эталоне
   был баг дублей «Толщина» из-за матча по транслиту-слагу).
3. **Характеристики → `<param>`/properties каталога** (НЕ customFields заказа!); многозначные
   (`multitude`) → несколько `<param>` с одним name; ручной маппинг по `specification_id`, строгий режим.
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие =
   false**; терм «Да/Нет» из маппинга (локализуемо). (`multitude=false` у boolean.)
5. **Медиа:** `<picture>` по URL — RetailCRM качает сам; URL самодостаточен (токен в хеше) → **прокси по
   токену не нужен**, max доступен напрямую; единственный риск — extensionless URL (тогда лёгкий прокси с
   расширением/MIME). Дедуп `sha1(path#size)` и выбор качества — в нашем сервисе (один URL на контент-
   ключ, всегда лучший доступный размер).
6. **Не выдумывать цену** (источник цены — заказчик/другая система; `<price>0` как крайность); `weight`
   бывает `null`; статус/«поля при создании» — ограничение ICML, см. блокер 8.
7. **Единицы:** конвертер г/мм → единица RetailCRM (обязателен); `measurement_unit` (м²/шт) — продажная
   единица (справочник `unit`/`<param>`), не габарит.
8. **Категории — дерево** из `/categories/` (`parent_category`). **`/collections/{slug}/` = 500 — не
   дёргать**, всё из эмбеда `collections[]`. **Справочники — `limit=1000` + сверка `meta.counts`.**
9. **Прочий payload** (`areas_per_package`, `product_shape`, alt/описание картинок, детали бренда,
   `upsells/cross_sells`, витринные страницы/SEO) — это **слой сайта-витрины**, не RetailCRM (рамка: CRM =
   бэкенд заказов). В нашем сервисе — через событие/хук перед отдачей фида, если витрина их потребляет.
10. **SemVer + CHANGELOG** (Keep a Changelog) для нашего сервиса; версия в манифесте модуля (если
    маркетплейс); деградация §5.5 — битый товар логируем и пропускаем, отдаём прошлый валидный фид.
