## Главное про данные

🟢 **Вариаций товара НЕТ вообще** — один `public_id` = один готовый товар → **одна позиция справочника**
(вы верно вывели). `options[]` — характеристики-фасеты, не SKU/варианты.

🟢 **Иерархия категорий ЕСТЬ** — эмбед `categories[]` в товаре плоский, но справочный **`/categories/`
отдаёт `parent_category`** (id родителя или `null`). → если у Мегаплана есть вложенные группы товаров —
строить дерево (Q10); если групп нет — категории в доп. поле плоско.

🟢 **Характеристики многозначны** — «Цвет» = 2 значения, «Помещение» = **6**. `/specifications/` отдаёт
флаг **`multitude`** (true = многозначная; boolean `false`). → доп. поле под многозначную характеристику
должно быть множественным/мультизначным (Q5).

🟢 **`numeric` в этом инстансе не используется** (все 13 характеристик text/boolean) — код универсально.

🟢 **Цены в источнике НЕТ** (факт API); **`sizes.weight` часто `null`** (у OC.IND.1 веса нет);
`country_code` — ISO **alpha-3**; `measurement_unit` (м²/шт) — **продажная** единица, не габарит;
всего **~13 характеристик** (доп. полей немного).

## Ответы на вопросы

**1. Эндпоинты «Товары и услуги».** 🟡 Платформенное — **шаг 0 обязателен** (payload товара, формат CRUD,
формат доп. полей). Контракт: целевая сущность — позиция справочника «Товары и услуги» (для счетов/
сделок), не сделка. Эталон пишет в нативную товарную сущность платформы — держитесь того же.

**2. Идемпотентность: externalId или доп. поле + фильтр.** 🟡 Главный архитектурный вопрос (верно).
Контракт §5.1: матчить по стабильному `public_id`, не по slug/имени. Рекомендация-приоритет: (1) если у
товара есть нативный externalId/внешний код — использовать его (уникальность/матчинг гарантирует
платформа); (2) иначе доп. поле `oc_public_id` + **серверный фильтр по нему**; (3) если фильтр по доп.
полю API не умеет — **локальный индекс `public_id → megaplan_id`** в БД нашего сервиса (ваш блокер 3,
рабочий обход) + процедура пересборки индекса (вычитать товары, прочитать `oc_public_id`). **Подтвердить
наличие externalId и фильтра по доп. полю.**

**3. Есть ли у товара доп. поля ВООБЩЕ.** 🟡 **Это вопрос №1, согласен** — если доп. поля только у
сделок/задач, рушится маппинг характеристик/бренда/страны. Контракт-фолбэк (по убыванию): доп. поля
товара → описание текстом (неструктурно) → не переносить характеристики (только на витрину). НЕ класть в
сделку (см. блокер 1). **Подтвердить на шаге 0 — критично.**

**4. Создание доп. полей через API vs преднастройка.** 🟡 Платформенное. Если создавать нельзя —
**преднастройка полей в аккаунте + строгий ручной маппинг** на существующие поля по `specification_id`
(режим «создать атрибут на лету» из стандарта не работает). Это нормальный сценарий (так же делаем на
Magento/UMI — справочники предсоздаются вне очереди). **Подтвердить.**

**5. Типы доп. полей.** 🟡 Платформенное. Контракт: нужны string/number/boolean (+множественный для
`multitude`). Если булева типа нет — boolean кладём текстом «Да/Нет» (**пусто/`null`/отсутствие = false**,
§5.6). Если числового нет — numeric строкой. **Подтвердить набор типов.**

**6. Характеристики — строго в доп. поля ТОВАРА, не сделки.** 🟢 **Подтверждаю — критично.** `options[]`
→ доп. поля **товара**; доп. поля сделки/счёта — другая подсистема (прямой аналог «properties vs
customFields» в RetailCRM, блокер 1). Ключ маппинга — `specification_id` (имя поля `oc_spec_<id>` или
карта), не переводимый label.

**7. ЦЕНА.** 🟢 В источнике цены НЕТ — **не синтезировать** (§5.6). 🟡 Но Мегаплан — CRM продаж, цена у
позиции для счёта нужна (конфликт инварианта с природой платформы — блокер 2, верно). Решение продуктовое
(Q8): (а) реальный прайс заказчика → подставлять; (б) цена из 1С/сайта, мы её **не трогаем при апдейте**
(create-only, §5.6); (в) цена `0` (риск «нулей» в счетах/аналитике). Эталон цену не задаёт (фильтр
`onecatalog_product_price`) — тот же принцип. **Решать с заказчиком.**

**8. Роль Мегаплана: счета или справочно?** 🟡 Стратегический вопрос (верно вынесли). Если реальные счета
по этим товарам → цена/единицы/валюта обязательны (полноценная интеграция). Если справочник/маркетинг →
требования мягче, возможно Мегаплан не целевая система для всего каталога. **Определить до архитектуры.**

**9. Картинки у товара.** 🟡 Платформенное (поддержка изображений у товара) — **проверить, критично для
объёма**. Контракт: Мегаплан сам по URL не скачает (нет фид-импорта) → **наш сервис качает из OneCatalog
и заливает вложением**. Полезный факт: бинарный `media_files`-URL **самодостаточен** — токен зашит в
base64-хеш, наш сервис качает его обычным GET **без заголовка `X-API-Key`** (в эталоне `Media::sideload`
так и делает, получает max). URL **без расширения** → MIME по содержимому, имя/тип присваиваем сами. Если
картинок у товара нет — медиа не переносятся (только витрина), это нормально для CRM. **Подтвердить
поддержку + эндпоинт загрузки.**

**10. Группы/категории товаров.** 🟡 Платформенное. Если иерархические группы есть — `categories[]` →
дерево (`/categories/` `parent_category`), find-or-create по имени в пределах родителя; если нет — доп.
поле плоско. **Подтвердить наличие групп и вложенности.**

**11. Единица измерения.** 🟢🟡 `measurement_unit` (м²/шт) — продажная единица, **не путать с единицей
габаритов** (§5.6, вы верно отметили). 🟡 Механизм справочника единиц Мегаплана (редактируемый/
find-or-create) — **подтвердить**.

**12. Вес/габариты.** 🟢 Источник: **вес в граммах, размеры в мм** (подпись локализована; `sizes.*_unit`,
иначе мм/г). **`weight` бывает `null`** — не выдумывать. Конвертер г/мм → единицы назначения **обязателен**
(иначе 1200 мм → 1200 см). 🟡 Нативные поля веса/габаритов у товара vs доп. поля — **подтвердить**.

**13. Rate-limit Мегаплан API.** 🟡 Платформенное — **проверить** лимит/окно. Каждый товар = несколько
вызовов (поиск, create/update, доп. поля, картинки) → порции (≥10) + паузы обязательны. **Подтвердить
лимит и объём каталога.**

**14. Токен: TTL и refresh.** 🟡 Платформенное — **проверить** схему авторизации (TTL, refresh). Наш
сервис хранит и обновляет токен; env приоритетнее поля (§7).

**15. Тип поставки.** 🟡 Для ceram — **приватный сервис** (Bearer-токен аккаунта), не публичное
приложение маркетплейса. **Подтвердить.** SemVer + CHANGELOG ведём для нашего сервиса.

**16. Дефолтный `limit` и rate-limit Wiki API.** 🟢 Справочники (`/specifications/`, `/categories/`,
`/countries/`) — **одним запросом `limit=1000`**; дефолтный лимит инстанса молча обрезает → сверяться с
**`meta.counts`** и дострелить пагинацией. Жёсткого rate-limit на `X-API-Key` OneCatalog в тестах не
наблюдалось. Товары — по одному `/products/{public_id}/`.

**17. Статусы / архивация.** 🟡 Платформенное (есть ли «активен/архив»). Контракт §5.6: статус — **только
при создании**, не перезатирать; «поля только при создании» = цена/статус в create-only наборе (блокер
10). Пропавший из набора товар — осознанная политика деактивации. **Подтвердить модель статусов.**

**18. Существующий каталог.** 🟡 Реальный риск (как `XML_ID`↔1С). Если в аккаунте уже есть товары (1С/
вручную) — наши ключи/артикулы могут пересечься. Рекомендация: неймспейс `oc_public_id`, опция «не трогать
не-OC товары», координация с другими обменами. **Подтвердить наличие другого обмена.**

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id`** (нативный externalId или доп. поле `oc_public_id` + фильтр/локальный
   индекс), `article` отдельно (только непустой; не синтезировать). Никогда не по slug/имени (§5.1).
2. **Справочники/доп. поля — стабильный ключ, не транслит:** характеристика — `code/имя
   oc_spec_<specification_id>`; категория/группа — по имени (в пределах родителя), не по слагу (в эталоне
   был баг дублей «Толщина» из-за матча по транслиту).
3. **Характеристики → доп. поля ТОВАРА** (НЕ сделки!); многозначные (`multitude`) → множественное поле;
   ручной маппинг по `specification_id`, строгий режим; преднастройка полей если API не создаёт.
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие =
   false**; «Да/Нет» из маппинга если нет булева типа. (`multitude=false` у boolean.)
5. **Медиа (если поддержаны):** наш сервис качает (URL самодостаточен, токен в хеше; MIME по содержимому,
   URL без расширения) + заливает вложением; дедуп `sha1(path#size)` и трекинг качества — в нашем индексе;
   галерея по имени файла, не по URL.
6. **Не выдумывать цену** (источник — заказчик/1С; `0` как крайность; create-only, не затирать); `weight`
   бывает `null`; статус — только при создании.
7. **Единицы:** конвертер г/мм → единица назначения (обязателен); `measurement_unit` (м²/шт) — продажная
   единица (нативный справочник), не габарит.
8. **Категории — дерево** из `/categories/` (`parent_category`), если группы поддерживают вложенность.
   **`/collections/{slug}/` = 500 — не дёргать**, всё из эмбеда `collections[]`. **Справочники —
   `limit=1000` + сверка `meta.counts`.**
9. **Прочий payload** (`areas_per_package`, `product_shape`, alt/описание картинок, детали бренда,
   `upsells/cross_sells`, витринные страницы/SEO) — **слой внешнего сайта**, не Мегаплан (рамка: CRM =
   бэкенд счетов). В нашем сервисе — через событие/хук, если витрина потребляет.
10. **SemVer + CHANGELOG** (Keep a Changelog) для нашего сервиса; деградация §5.5 — битый товар логируем и
    пропускаем, не валим пакет.
