# Интеграционный план: МойСклад

> Перенос стандарта `INTEGRATION-STANDARD.md` (импорт каталога из OneCatalog Wiki API)
> на **МойСклад (MoySklad)** — российскую облачную систему складского учёта и
> торговли с JSON API 1.2 (`https://api.moysklad.ru/api/remap/1.2`). Эталон стандарта —
> плагин WooCommerce. Поставка здесь — НЕ плагин внутри платформы, а **внешний
> сервис-синхронизатор**, который тянет OneCatalog Wiki API и пушит сущности в
> МойСклад через JSON API; идемпотентность строится на **`externalCode`**.
>
> **Принципиальная рамка (читать до всего остального).** МойСклад — это
> **складская/торговая/учётная система** (товары, остатки, цены, заказы,
> документы), а НЕ витрина-CMS со своим storefront. У неё нет фронтенда-витрины,
> taxonomy-страниц, тем, Gutenberg, фасетного поиска для покупателя, SEO-страниц
> коллекций/брендов, статусов публикации (publish/draft). Поэтому значительная
> часть инвариантов стандарта, завязанных на витрину (страницы коллекций/брендов,
> фасеты, alt-тексты картинок, SEO, статус публикации), на стороне МойСклада
> **просто не живёт** — их место на внешнем сайте/CMS, который синхронизируется с
> МойСкладом. МойСклад в этой связке — «бэкенд учёта и заказов», витрина — отдельно.
>
> **НО** каталог МойСклада — самый «товарно-родной» из всех CRM/ERP, что мы
> разбирали. У него нативные **country (страна)**, **uom (единицы измерения)**,
> **productfolder (дерево групп товаров = категории)**, **externalCode**
> (готовый внешний ключ идемпотентности на уровне платформы), **attributes**
> (доп. поля метаданных per-entity). Это сильные стороны — где МойСклад объективно
> ЛУЧШЕ Shopify/Woo/Bitrix по «родности» товарной модели (см. раздел «Сложно/
> невозможно», блок про сильные стороны). Ловушка ровно одна и она центральная:
> **«характеристики» (characteristics) в МойСкладе порождают МОДИФИКАЦИИ
> (variants)** — а у OneCatalog вариаций нет. Маппить `options[]` в МойСклад-
> «характеристики» НЕЛЬЗЯ (расплодятся модификации); их место — в **attributes
> (доп. поля товара)**. Это аналог развилки «attribute vs taxonomy» из эталона,
> только злее.

## Что такое целевая поставка

Стандарт описывает «модуль импорта в товарную систему платформы». У МойСклада
плагинов-внутри-платформы нет — расширение делается **снаружи**:

- **Основной путь — внешний сервис на JSON API 1.2 (императивный пуш).** Наш сервис
  (отдельный backend: PHP/Node/Go — что удобно) по списку `public_id` тянет
  OneCatalog Wiki API, маппит и **создаёт/обновляет сущности МойСклада** через
  REST: `POST/PUT entity/product`, `entity/productfolder`, `entity/country` (поиск),
  `entity/uom`, загрузку изображений и т.д. Идемпотентность — по **`externalCode`**
  (см. ниже). Это прямой аналог «императивного» пути; декларативного фид-импорта
  (как ICML у RetailCRM или CommerceML у Bitrix) у МойСклада в общем случае нет —
  загрузка идёт через API (проверено: JSON API 1.2 — основной интеграционный
  канал; есть отдельная загрузка прайс-листов/«импорт» в UI, но это не наш сценарий).

- **Идемпотентность встроена в платформу через `externalCode`** (проверено: у
  большинства сущностей МойСклада — `externalCode`, строковое поле «внешнего кода»,
  явно предназначенное для идентификации из внешних систем; плюс системный `id`
  (UUID, выдаёт МойСклад) и человекочитаемый `code`). `externalCode` уникален в
  пределах типа сущности и аккаунта (**проверить точную гарантию уникальности —
  на уровне платформенного констрейнта или «по соглашению»**, см. вопросы).

**Почему не «фид».** У МойСклада нет штатного механизма «дай URL фида — я сам
скачаю/распарсю/обновлю по расписанию» (как ICML/CommerceML). Значит расписание,
порядок зависимостей, дедуп, трекинг качества картинок, очередь и rate-limit —
**на нашей стороне** (в отличие от RetailCRM, где часть этого делегируется
платформе). Это ближе к Shopify/Woo по объёму нашей логики.

## Маппинг сущностей и реализация

### Таблица соответствий: OneCatalog → абстракция → МойСклад

| OneCatalog | Абстракция | МойСклад |
|---|---|---|
| product | Товар | `entity/product` (нативный товар) |
| public_id | Внешний ключ | **`externalCode`** товара (платформенный внешний код) |
| article | Артикул произв. | `article` (артикул) и/или `code` |
| options[] | Характеристика-фасет | **`attributes` (доп. поля товара)** — НЕ characteristics! |
| categories[] | Категория | **`productfolder` (дерево групп товаров)** — нативное дерево |
| brand | Бренд | `attribute` (нет нативного «бренда») |
| country | Страна происх. | **`entity/country` — НАТИВНАЯ сущность**, ссылка с товара |
| collections[] | Коллекция | `attribute` или отдельный `productfolder` (настройка) |
| tags[] | Теги | `attribute` (строка/справочник) |
| images_urls/files | Медиа | `images` товара (по URL или multipart — проверить) |
| sizes.weight | Вес | **`weight` товара** (нативное числовое поле) |
| sizes.length/width/height | Габариты | `volume` (объём) и/или доп. поля (нет 3 раздельных полей — проверить) |
| measurement_unit | Ед. изм. | **`entity/uom` — НАТИВНАЯ сущность**, ссылка с товара |
| areas_per_package, product_shape, … | прочее | доп. поля `attributes` или не переносится (граница ядра, §2.2) |
| цена | Цена | `salePrices` + `pricetype` — **в источнике цены НЕТ** (§5.6) |

### product → entity/product (§3, §5.1)

Базовая сущность — `entity/product` (проверено: товар — корневая сущность каталога
МойСклада, раздел «Товар»). Поля при создании: `name` (← `menutitle`), `externalCode`
(← `public_id`), `article` (← `article`), `weight`, `volume`, ссылки на
`productFolder`, `country`, `uom`, `attributes[]`, `salePrices[]`, `images`.

Важно: у МойСклада есть **разные типы товарных сущностей**, и выбор типа — это
архитектурное решение (проверено, что эти типы существуют как отдельные сущности):
- **product** — обычный товар. **Наш дефолт для OneCatalog.**
- **variant (модификация)** — порождается «характеристиками»; нам НЕ нужен (см.
  ловушку ниже).
- **bundle (комплект)** — товар-набор; не наш сценарий.
- **service (услуга)** — не наш сценарий.
OneCatalog отдаёт описательные товары без вариаций → всё ложится в **product**.

### public_id → externalCode (центральный вопрос идемпотентности, §5.1)

`public_id` (`OC.IND.1`) → **`externalCode`** товара. Это прямой, «родной» механизм
идемпотентности — сильная сторона МойСклада (в отличие от Woo, где для public_id
заводят отдельную мету):

- Перед импортом: `GET entity/product?filter=externalCode=<public_id>` →
  если найден, `PUT` (обновить), иначе `POST` (создать) (проверено: фильтрация
  выборки по `filter`-параметру — штатный механизм JSON API 1.2; `externalCode`
  фильтруется).
- **НЕ искать по `name`/`code`** — нестабильны (§5.1).
- ⚠️ Артикул (`article`) — отдельное поле, НЕ ключ идемпотентности (см. §5.2).
- Справочники (productfolder, country, uom, attribute-значения) тоже идемпотентны
  «найти-иначе-создать», но у каждого свой механизм (см. ниже).
- **(проверить)** гарантию уникальности `externalCode` на уровне платформы — если
  это не констрейнт, а «по соглашению», нужна защита от гонок на нашей стороне
  (наш сервис — единственный писатель в эти товары).

### article → артикул (§5.2)

`article` OneCatalog (артикул производителя, часто пуст) → `article` товара МойСклада
(проверено: `article` — нативное поле товара). Заполнять ТОЛЬКО если пришёл; иначе
пусто. **Не** класть туда `public_id` (это в `externalCode`). Поле `code`
(человекочитаемый код товара) — отдельное; можно оставить автогенерации МойСклада
или маппить из `article`/`public_id` по согласованию (**уточнить роль `code`** —
он у МойСклада тоже бывает уникальным и используется в документах).

### options[] → attributes (доп. поля), а НЕ characteristics! (§5.4 — КРИТИЧНО)

**Это центральная архитектурная развилка интеграции.** В МойСкладе слово
«характеристика» (`characteristics`) — технический термин: **характеристики
описывают ОСИ вариативности и порождают модификации (variants)** (проверено:
модификация = товар + набор значений характеристик; характеристики — это «Цвет:
красный», «Размер: M», из которых строится матрица модификаций). Если завести
`options[]` OneCatalog как характеристики:
- на каждое сочетание значений МойСклад заведёт **отдельную модификацию-variant**;
- товар «размножится» в десятки SKU-вариантов, которых в OneCatalog НЕТ
  (у OneCatalog `options[]` — описательные фасеты-характеристики, единая карточка
  товара без вариаций).

Поэтому **`options[]` → `attributes` (доп. поля товара)** — метаданные карточки,
не оси вариативности (проверено: `attributes` — пользовательские доп. поля
метаданных, настраиваются per-entity в «Настройки → Дополнительные поля»; у товара
свой набор; значение поля — ссылка на определение поля + значение). Это прямой
аналог развилки «attribute (доп. поле) vs taxonomy (вариация)» из эталона.

Детали реализации:
- Доп. поля МойСклада **типизированы** (проверено, что типы существуют; точный
  перечень — **проверить**): `string`, `long`/`double` (числовые), `boolean`,
  `time`, `file`, и **справочник** (значение — ссылка на элемент справочника /
  на сущность). Это лучше плоских строк.
- Маппинг типов OneCatalog → тип доп. поля:
  - `specification_type=text` → доп. поле типа `string` (или **справочник**, если
    хотим переиспользуемые значения-фасеты — предпочтительно для будущей фильтрации).
  - `specification_type=numeric` → доп. поле числового типа (`numeric_option`).
  - `specification_type=boolean` → доп. поле `boolean` (`bool_option`); пустое/`null`
    → **false** (§5.6).
- **find-or-create определения доп. поля** по имени (`specification_label`),
  не по слагу (§5.1: разные транслиты → дубли). Стабильный ключ привязки —
  `specification_id` (для ручного маппинга, §5.4).
- ⚠️ **Доп. поля надо предварительно ЗАВЕСТИ** (создать metadata доп. поля для
  сущности product), прежде чем присваивать значение товару (**проверить**, можно
  ли создавать определение доп. поля через API: `POST entity/product/metadata/
  attributes` — **проверить точный путь**; если только вручную в UI — это требует
  предварительной настройки аккаунта = ограничение, см. вопросы).
- Boolean-характеристики без текстового значения (`specification_option_name=null`)
  трактуем как `bool_option` → `true`/`false`; правило true/false → значение
  настраивается (§5.4).

### categories[] → productfolder (дерево групп товаров) (§5.4)

`categories[]` → **`entity/productfolder`** (проверено: «Группа товаров» —
нативная иерархическая сущность каталога; товар ссылается на одну группу через
`productFolder`; вложенность — через ссылку на родительскую группу). Это сильная
сторона: нативное дерево категорий, не плоский фасет.

- find-or-create группы по имени (`menutitle`), родитель — по ссылке на
  родительскую группу (если OneCatalog отдаёт иерархию; в payload `categories[]` —
  плоский список без явного parent → **уточнить, есть ли иерархия категорий в
  OneCatalog**; если плоско — заводим плоские группы).
- ⚠️ Товар в МойСкладе принадлежит **одной** группе (`productFolder` — одиночная
  ссылка, не массив — **проверить**). OneCatalog `categories[]` — массив. Развилка:
  (а) брать первую/основную категорию в `productFolder`, остальные — в доп. поле-
  справочник «доп. категории»; (б) согласовать с заказчиком основную категорию.
  Это ограничение (multi-category → single folder), фиксируем в вопросах.

### brand → attribute (нет нативного бренда) (§3, §7)

У МойСклада **нет нативной сущности «бренд/производитель-марка»** (есть
«Контрагент»-поставщик, но это не то). Значит бренд → **доп. поле** товара:
- тип `string` (имя бренда) или **справочник** (предпочтительно: переиспользуемые
  значения, аналог таксономии) — выбор «тип объекта + цель» как в эталоне (§7).
- Доп. поля бренда (страна, год основания, сайт, лого — `brand.*` payload) в
  каталог МойСклада нативно не ложатся; либо отдельные доп. поля, либо игнор
  (это витринные данные → внешний сайт). По умолчанию переносим только имя бренда.

### country → entity/country (НАТИВНАЯ!) (§3) — сильная сторона

`country` OneCatalog → **`entity/country`** (проверено: страна — отдельная
нативная сущность-справочник МойСклада; у товара есть ссылка `country`). Это
объективно ЛУЧШЕ, чем у витринных CMS (где страну пихают в атрибут/таксономию).

- find-or-create по имени страны (`country.menutitle`) и/или коду
  (`country.country_code`). **(проверить)** формат кода страны в МойСкладе —
  alpha-2/alpha-3/числовой ISO; OneCatalog отдаёт `country_code` (**уточнить его
  формат** — см. вопросы). Если не совпадают — матчить по имени (label), код —
  вторично.
- В МойСкладе уже есть **предзаполненный справочник стран** (мировой) — скорее
  всего страну надо **найти существующую**, а не создавать (создание — фолбэк).
  **(проверить)**, есть ли страны из коробки или справочник пуст.

### collections[] → attribute / productfolder (настройка) (§3, §7)

Нет нативной «коллекции». Выбор «тип объекта + цель» (эталон §7):
- **доп. поле-справочник** «Коллекция» (дефолт — как фасет/метаданное), либо
- **отдельная ветка productfolder** «Коллекции/<имя>» (если заказчик хочет
  навигацию по коллекциям в учёте) — но смешивать с деревом категорий рискованно.
Доп. поля коллекции (`link_3d`, `link_official_site`, картинки) — витринные,
по умолчанию не переносим (граница ядра). Дефолт: доп. поле-справочник.

### tags[] → attribute (§7)

Нет нативных тегов. Теги (вкл/выкл, §7) → доп. поле типа `string` (через разделитель)
или **справочник** (предпочтительно). По умолчанию — выкл (как в эталоне).

### media → images товара (§2.3, §5.3)

`images_urls/files` → **`images` товара** (проверено: у товара есть коллекция
изображений; первое часто играет роль обложки/миниатюры). Способ загрузки —
ключевой вопрос:
- **(проверить)** поддерживает ли МойСклад **загрузку изображения по URL** (передать
  `{ "url": "..." }`) или только **multipart/base64** (передать содержимое файла).
  Историческая практика МойСклада — загрузка содержимого (multipart/JSON с base64
  `content`); загрузка «по ссылке» могла появиться позже — **проверить точно**.
- URL OneCatalog `media_files/<base64hash>` — **без расширения**, `max` только с
  токеном (§2.3). Значит наш сервис в любом случае:
  1. сам качает файл с токеном (получаем `max`),
  2. определяет MIME по содержимому (URL без расширения),
  3. отдаёт в МойСклад как multipart/base64 (или по нашему чистому прокси-URL,
     если URL-загрузка поддержана).
  → закладываем **скачивание на нашей стороне** (как в эталоне) — безопаснее.
- **Трекинг качества (§5.3)** и **дедуп по контент-ключу** — целиком на нашей
  стороне: МойСклад этого не делает. Дедуп картинок между товарами в МойСкладе
  затруднён (картинка привязана к товару; общего медиа-хранилища с переиспользованием
  по ключу, как в WP-вложениях, нет — **проверить**). Вероятно, дедуп вырождается:
  каждый товар несёт свои картинки; трекинг качества ведём в нашей БД-проекции
  (запоминаем, какой размер загрузили, чтобы апгрейдить при появлении токена).
- ⚠️ **(проверить)** лимит на число/размер изображений у товара.

### sizes.weight → weight; габариты; единицы (§5.6) — Units

- `sizes.weight` → **`weight` товара** (проверено: `weight` — нативное числовое
  поле). ⚠️ **(проверить)** единицу веса МойСклада — вероятно **килограммы**.
  OneCatalog отдаёт вес в **граммах** (§5.6) → **конвертер г → кг обязателен**
  (иначе 1.2 кг превратятся в 1200 кг).
- Габариты: у МойСклада есть нативный **`volume` (объём)** (проверено) — но
  **нет трёх раздельных полей length/width/height** (**проверить**). OneCatalog
  отдаёт length/width/height в **миллиметрах**. Варианты:
  (а) посчитать объём `l*w*h` → `volume` (в каких единицах объём у МойСклада? —
  **проверить**, вероятно м³ или дм³ → конверсия из мм³ нетривиальна);
  (б) положить три габарита в **доп. поля** (числовые) «Длина/Ширина/Высота, мм»
  как метаданные. **Дефолт — (б)** (сохраняем точные значения), плюс опционально
  `volume`.
- `measurement_unit` → **`entity/uom`** (проверено: единицы измерения — нативная
  сущность-справочник; товар ссылается на `uom`). find-or-create единицы по имени
  (`measurement_unit.name`, напр. «м²», «шт»). В МойСкладе **есть предзаполненный
  справочник единиц** (шт, кг, м, м² и т.п.) — скорее **найти существующую**, чем
  создавать (**проверить** состав справочника из коробки). Это снова сильная
  сторона — нативная единица, не строка-атрибут.
- **Universal Units-конвертер обязателен** (§5.6): таблицы коэффициентов г→кг,
  мм→(см/м/мм) + нормализация локализованных подписей (`sizes.*_unit`: «г», «мм»).

### Цена (§5.6) — в источнике НЕТ, но МойСклад для учёта/заказов её ждёт

В OneCatalog **цены нет** (§5.6 — не синтезировать). Но МойСклад — учётно-торговая
система: товар без цены технически создаётся, **но для заказов/документов цена
обычно нужна** (проверено: цены товара — `salePrices[]`, массив пар
«тип цены (`pricetype`) → значение»; типы цен настраиваются в аккаунте).
- **Дефолт (как эталон): цену НЕ задаём** (`salePrices` пуст или 0). Поле «только
  при создании» (§5.6) — при переимпорте не затираем (вебмастер/1С могли проставить).
- Если у заказчика есть реальный прайс-источник — подставляется через фильтр/событие
  (наш аналог хука), маппится в `salePrices` нужного `pricetype`.
- **Решить роль цены** (см. вопросы): нужна ли вообще цена в МойСкладе под эту
  задачу, или каталог тут справочный, а цены/заказы — в другой системе.

## Фоновая очередь, медиа, настройки, i18n, жизненный цикл

### API-пуш, батч-операции и rate-limit (§6)

- **Канал — JSON API 1.2** (императивный пуш). Декларативного фид-импорта по URL
  у МойСклада нет (в отличие от RetailCRM/Bitrix) → расписание, очередь, порядок
  зависимостей, дедуп, трекинг качества — **на нашей стороне** (полный объём логики
  эталона).
- **Батч / массовые операции (важно для производительности).** JSON API 1.2
  поддерживает **массовое создание/обновление одним запросом** — передать в `POST`
  на коллекцию **массив** объектов (проверено: «массовое создание и обновление»
  через передачу массива сущностей на эндпоинт коллекции — штатный механизм; если
  у объекта есть `id`/`externalCode` — обновление, иначе создание). Это снижает
  число запросов и помогает против rate-limit. ⚠️ **(проверить)** лимит размера
  батча (число сущностей в одном запросе).
- **Rate-limit (есть, критичен).** У МойСклада **жёсткие лимиты запросов**
  (проверено, что лимиты существуют и на превышение возвращается **429**;
  **точные числа — проверить**: исторически ориентир — порядка нескольких запросов
  в секунду на аккаунт и ограничение параллельных соединений, плюс отдельные лимиты
  на тяжёлые операции). Наш сервис обязан:
  - соблюдать паузы/бэкофф, обрабатывать `429` (повтор с задержкой),
  - использовать батч-операции вместо поштучных запросов,
  - вести очередь с шагом ≥10 (§6) и троттлингом.
- **Порядок зависимостей (§5.4)** соблюдаем сами: сначала uom/country/productfolder/
  определения доп. полей (справочники), потом товар, потом картинки.
- Фолбэк без планировщика — синхронная обработка с тем же отчётом (§6).

### Где хранить настройки/токен (§7)

Настройки живут в **нашем сервисе** (своя БД/конфиг), не в МойСкладе:
- **Аутентификация МойСклада** (проверено: два способа) — (а) **Bearer-токен**
  (`Authorization: Bearer <token>`; токен получается логином/паролем или в UI
  аккаунта), либо (б) Basic (login:password, не рекомендуется). Дефолт — **Bearer-
  токен**. Для тиражируемого решения — **vendor-приложение МойСклада** (см. ЖЦ).
- Токен OneCatalog (`X-API-Key`), язык (`ru` дефолт для ceram), шаг очереди (≥10),
  карты маппинга характеристик (`specification_id` → доп. поле), флаги (бренд/
  страна/коллекция/теги), маппинг типов цен (`pricetype`) — всё в нашем конфиге.
- Приоритет env над полем токена (§7); валидация/кламп значений.

### Картинки (§2.3) — скачиваем на своей стороне

См. выше «media». Качаем `max` с токеном OneCatalog, определяем MIME, грузим в
МойСклад (multipart/base64 — **проверить URL-загрузку**). Трекинг качества/дедуп —
в нашей БД-проекции.

### Мультиязычность (§9)

**МойСклад одноязычен** (учётная система; названия товаров/групп — одна строка,
нет встроенной мультиязычности карточки — **проверить**, но почти наверняка так).
Для ceram достаточно `ru` (`lang=ru` в OneCatalog). Несколько языков = несколько
аккаунтов/проекций (отдельная фаза) — на старте один язык. i18n нашего сервиса
(строки админки/логов) — английский исходник + ru-перевод (§9), как в эталоне.

### Жизненный цикл нашего сервиса (§10)

- Версионирование — **SemVer** + CHANGELOG (Keep a Changelog), как требует §10 и
  память проекта (бэклог = «Не выпущено»; версию поднимать перед сборкой).
- **Аутентификация/деплой**:
  - **Приватный токен** (Bearer) — просто, для одного аккаунта ceram. Дефолт на
    старте.
  - **Vendor-приложение МойСклада** (тиражируемое) — если решение пойдёт на много
    аккаунтов: регистрируется в «Вендорском кабинете», ставится из маркета
    приложений, получает доступ через OAuth-подобный флоу (проверено, что механизм
    «приложений вендора» существует; **детали флоу — проверить**). Это серьёзная
    отдельная фаза.
- Миграции схемы нашей проекции (externalCode-маппинги, трекинг качества) — в нашей
  БД. Деградация без падений (§5.5): битый товар/429/недоступный эндпоинт логируется
  и не валит пакет.

## 🚫 Сложно / невозможно на МойСкладе

1. **«Характеристики» ≠ описательные фасеты — порождают модификации (ГЛАВНОЕ).**
   Самая опасная ловушка. Нельзя маппить `options[]` в МойСклад-`characteristics` —
   это оси вариативности, из них строятся модификации-variants, которых в OneCatalog
   нет. **Обход:** `options[]` → **доп. поля товара (`attributes`)**; характеристики
   МойСклада не используем вовсе. Доп. поля надо **преднастроить** (создать
   определения) — возможно вручную в UI, если API создания недоступен (см. вопрос 3).

2. **Нет витрины/storefront — отсутствуют как класс:** страницы коллекций/брендов,
   фасетный поиск покупателя, SEO, alt-тексты картинок, темы, статусы публикации
   (publish/draft/pending). **Обход:** эти инварианты стандарта на МойСкладе не
   живут — их место на внешнем сайте-витрине, синхронизируемом с МойСкладом.
   МойСклад — «бэкенд учёта/заказов».

3. **Нет нативных «бренд / коллекция / теги».** Только category-дерево
   (productfolder), country и uom — нативны; бренд/коллекция/теги — нет.
   **Обход:** доп. поля (`attributes`) типа `string`/справочник; выбор «тип объекта
   + цель» как в эталоне.

4. **Цена обязательна для учёта/заказов, но в источнике её нет (§5.6).** Если
   МойСклад ведёт заказы — товар без цены полуфункционален. **Обход:** цену не
   синтезируем (§5.6); подставляем из реального прайс-источника через фильтр/событие,
   либо цена приходит в МойСклад из 1С/сайта, а наш импорт — описательная часть.

5. **Мульти-категория → одна группа.** Товар привязан к одной `productFolder`
   (массив категорий OneCatalog не помещается). **Обход:** основная категория →
   `productFolder`, остальные → доп. поле-справочник «доп. категории».

6. **Габариты: нет 3 раздельных нативных полей (только `volume`).** OneCatalog даёт
   length/width/height в мм. **Обход:** доп. поля «Длина/Ширина/Высота, мм»
   (точные значения) + опционально расчётный `volume`.

7. **Трекинг качества и дедуп картинок — нет на стороне платформы.** МойСклад не
   дедуплицирует медиа по контент-ключу и не апгрейдит качество. **Обход:** весь
   трекинг качества (§5.3) и дедуп — в нашей БД-проекции; картинки качаем сами
   (URL без расширения + токен `max`) и грузим содержимым; дедуп между товарами,
   вероятно, вырождается (каждый товар несёт свои файлы).

8. **«Поля только при создании» при переимпорте (§5.6).** Платформа сама не различает
   «создать» vs «обновить выбранные поля» — `PUT` затирает переданное. **Обход:**
   логику «не затирать цену/ручные правки» реализуем в нашем сервисе (читаем
   существующий товар, сливаем поля).

9. **Rate-limit (429).** Жёсткие лимиты запросов/параллельности. **Обход:** батч-
   операции (массив в одном запросе), троттлинг/бэкофф, очередь с шагом ≥10,
   обработка 429 с повтором.

10. **Создание определений доп. полей через API — под вопросом.** Если недоступно —
    аккаунт надо преднастроить вручную (определения доп. полей под бренд/страну-
    дубль/коллекцию/теги/характеристики). **Обход:** документировать список доп.
    полей для ручного создания + проверять их наличие перед импортом.

### Где МойСклад объективно ЛУЧШЕ других (сильные стороны — отразить!)

- **`externalCode` — готовый внешний ключ идемпотентности на уровне платформы.**
  Не нужно (как в Woo) заводить кастомную мету `_onecatalog_public_id` — поле уже
  есть и фильтруется. Идемпотентность «из коробки».
- **`country` — НАТИВНАЯ сущность** с предзаполненным мировым справочником. Страна —
  настоящая ссылка, не строка-атрибут (в отличие от Woo/Bitrix/Shopify, где страну
  пихают в фасет).
- **`uom` (единицы измерения) — НАТИВНАЯ сущность** с предзаполненным справочником.
  «м²/шт/кг» — настоящая единица товара, не строка.
- **`productfolder` — нативное иерархическое дерево групп товаров** = категории
  «из коробки», с вложенностью.
- **`weight`/`volume` — нативные числовые поля** товара (вес/объём — родные, не
  доп. поля).
- **Типизированные доп. поля (`attributes`)** — `string`/числовые/`boolean`/
  справочник, per-entity — куда богаче плоских строк-метаполей.
- **Батч-операции** (массив в одном запросе) — эффективная массовая загрузка.

Итог: МойСклад — самый «товарно-родной» бэкенд из CRM/ERP в нашей серии. Слабее он
ровно там, где он не витрина (нет storefront/SEO/фасетов/статусов) и где «характеристики»
значат «модификации».

  Декларативного фид-импорта нет → очередь, порядок зависимостей, дедуп, трекинг
  качества картинок, rate-limit/батч, troттлинг — **на нашей стороне** (объём логики
  как у Woo/Shopify, не как у RetailCRM).
- **Ключ идемпотентности:** `public_id` → **`externalCode`** товара (платформенный
  внешний код — сильная сторона). Категории → **нативный `productfolder` (дерево)**;
  страна → **нативная `country`**; единица → **нативная `uom`**; вес → **`weight`**;
  характеристики `options[]` → **доп. поля (`attributes`), НЕ characteristics**;
  бренд/коллекция/теги → доп. поля (нативных сущностей нет); медиа → `images`
  (качаем сами, грузим содержимым); цена → `salePrices`/`pricetype` (в источнике нет).
- **Чем МойСклад СИЛЬНЕЕ других:** нативные `externalCode` (идемпотентность из
  коробки), `country`, `uom`, `productfolder`-дерево, `weight`/`volume`,
  типизированные доп. поля, батч-операции. Самый «товарно-родной» из CRM/ERP.
- **Чем СЛАБЕЕ:** не витрина (нет storefront/SEO/фасетов/статусов публикации/alt);
  нет нативных бренд/коллекция/теги; «характеристики» значат «модификации»;
  цена ожидается для учёта/заказов, но в источнике её нет.
- **3–4 самых острых вопроса:** (1) характеристики строго в **доп. поля**, не в
  characteristics — иначе расплодятся модификации (критично); (2) **цена** — её нет
  в источнике, а МойСклад для заказов её ждёт (источник цены? роль МойСклада —
  учёт или справочно?); (3) **медиа** — по URL или только multipart/base64
  (грузим сами); (4) **rate-limit/батч** — точные числа и размер батча.
- **3–4 «невозможно/сложно»:** (1) characteristics→variants (главная ловушка,
  обход — доп. поля); (2) нет витрины — коллекции/бренды/фасеты/SEO/статусы как
  класс отсутствуют; (3) цена обязательна для учёта, но в источнике её нет; (4)
  трекинг качества/дедуп картинок и «поля только при создании» — целиком на нашей
  стороне (платформа не помогает), плюс rate-limit (429).
- ⚠️ Метки `(проверено)` опираются на знание JSON API 1.2; всё `(проверить)` —
  сверить по актуальной спецификации `https://api.moysklad.ru/api/remap/1.2`
  (точные имена/пути методов, типы доп. полей, формат кода страны, единицы веса/
  объёма, URL-загрузка картинок, лимиты, размер батча, vendor-флоу) ДО кода.
