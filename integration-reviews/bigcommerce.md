# Интеграционный план: BigCommerce (App)

> Реализация стандарта `INTEGRATION-STANDARD.md` (OneCatalog Wiki API) на платформе **BigCommerce**.
>
> **Принципиальное отличие от эталона (WooCommerce):** на BigCommerce это **НЕ плагин внутри CMS**, а
> **внешнее приложение** (Single-Click App из App Marketplace либо custom/draft App), работающее с магазином
> через **REST Management API V3** (приоритетно) и/или **GraphQL Admin API** по **OAuth 2.0**. У приложения
> **свой сервер, своя БД, своя очередь, свой cron** — у самого магазина серверного cron нет, ФС магазина
> недоступна. «Хуки/события» стандарта §8 живут **внутри приложения** (in-process шина), а наружу магазина —
> только предопределённые **Webhooks** (нет произвольного `do_action`).
>
> **BigCommerce заметно ближе к модели OneCatalog, чем Shopify**, по трём пунктам ядра:
> 1. **Brands — нативная сущность** (`/catalog/brands`), а не строка `vendor`.
> 2. **Categories — нативное дерево** (`parent_id`), а не плоские коллекции.
> 3. **Габариты товара (`width`/`height`/`depth`) и `weight` — в ядре товара**, а не metafields/вариант.
>
> Но BigCommerce **хуже** Woo/Bitrix по части произвольных таксономий (их нет — только Categories + Custom
> Fields/Metafields), и как любой SaaS — ограничен rate limits, отсутствием доступа к ФС и серверному cron.
>
> Целевая версия API: **REST Management V3** + **GraphQL Admin API** (зафиксировать дату/версию в манифесте, §жизненный цикл).

## Маппинг сущностей и реализация

Сводная таблица «OneCatalog → абстракция → BigCommerce», затем детализация по каждой строке.

| OneCatalog | Абстракция | BigCommerce (механизм) |
|---|---|---|
| `product` | Товар | **Product** (`POST/PUT /catalog/products`) |
| `public_id` (`OC.IND.1`) | Внешний ключ (идемпотентность) | **Custom Field** `oc_public_id` на товаре **+ обязательное зеркало в БД приложения** (поиск по Custom Field напрямую API НЕ умеет, см. ниже) |
| `article` | Артикул производителя | `Product.sku` (только если непустой) |
| `options[]` (specifications) | Характеристики | **Custom Fields** (key→value) и/или **Metafields** товара — **НЕ Variant Options, НЕ Modifiers** |
| `categories[]` | Категория | **Categories** — нативное дерево (`/catalog/trees/categories`, `parent_id`) |
| `brand` | Бренд | **Brand** — нативная сущность (`/catalog/brands`), товар → `brand_id` |
| `country` | Страна происхождения | **Custom Field** `oc_country` (или Metafield); в ядре товара поля «страна происхождения» нет |
| `collections[]` | Коллекция | **Category** (доп. ветка дерева) **или** Metafield — выбор в настройках |
| `tags[]` | Теги | **нативных тегов товара НЕТ** → Custom Field `oc_tags` (CSV) / Metafields / search keywords (`search_keywords`) — см. вопрос |
| `images_urls` / `files[]` | Медиа | **Product Images by URL** (`image_url` → BigCommerce сам качает; ФС магазина недоступна) |
| `sizes.weight` + `weight_unit` | Вес | `Product.weight` (нативно; единица — из Store settings, конвертировать) |
| `sizes.{length,width,height}` | Габариты | `Product.width` / `Product.height` / `Product.depth` (**нативно в ядре!**) |
| `measurement_unit` (м²/шт) | Единица измерения | Custom Field / Metafield (нет нативного «unit of measure» на товаре) |
| `product_shape`, `areas_per_package`, alt картинок, детали бренда, upsells/cross_sells | Прочее | Custom Fields / Metafields + событие приложения `product_imported` |

### 1. Товар → Product

- Создание/обновление: REST V3 `POST /catalog/products` / `PUT /catalog/products/{id}`. GraphQL Admin API тоже
  умеет мутации товара, но **массовое создание товаров со всеми вложениями (images, custom fields, variants)
  исторически удобнее и полнее покрыто в REST V3** — рекомендую REST V3 как основной транспорт записи, GraphQL —
  для эффективного чтения/поиска (батч-запросы по metafields).
- Базовые поля стандарта (§5.4):
  - `name` ← `menutitle` (**обязательное, уникальное в рамках магазина** — см. вопросы про коллизии имён);
  - `type` ← `physical` (у нас есть вес/габариты; цифровых товаров OneCatalog не отдаёт);
  - `price` ← **обязателен при создании** (см. §«цена» и вопросы — это блокер!);
  - `weight` ← `sizes.weight` сконвертированный (**обязателен для physical-товара**, см. вопросы);
  - `is_visible` ← маппинг «статуса новых» (`true`/`false`, **только при создании**, §5.6);
  - `description` ← `description_text` (часто null → не трогать при апдейте).
- BigCommerce **требует уникальности `name`** и (если задан) `sku`. Это ужесточает дедуп по сравнению с Shopify.

### 2. `public_id` → Custom Field + идемпотентность (§5.1, §5.2)

Это центральная и **самая болезненная** точка на BigCommerce.

- `public_id` (`OC.IND.1`) — стабильный бизнес-ключ. Хранить на товаре в **Custom Field**:
  `name="oc_public_id"`, `value="OC.IND.1"` (Custom Fields — это пары name/value на товаре; до 200 на товар).
  Альтернатива/дополнение — **Metafield** (`namespace="oc"`, `key="public_id"`, есть `permission_set`,
  виден в API, не на сторефронте по умолчанию). Metafield предпочтительнее как «технический» ключ (его можно
  скрыть от витрины), Custom Field — если значение должно показываться на странице товара.
- ⚠️ **КЛЮЧЕВОЕ ОГРАНИЧЕНИЕ: API НЕ умеет искать товар по значению Custom Field или Metafield.**
  `/catalog/products` фильтруется по `id`, `sku`, `name`, `keyword`, `brand_id`, `categories` и т.п., но
  **нет** `?custom_field=...` или `?metafield=...`. То есть «найти товар по `public_id`» через сам API —
  **невозможно за один запрос**. Поэтому:
  - **БД приложения ОБЯЗАТЕЛЬНА** (в отличие от Shopify, где поиск по metafield в `query` существует и
    БД-зеркало было лишь оптимизацией). Таблица `oc_product_map(store_hash, public_id, bc_product_id, ...)` —
    **единственный** быстрый способ `find_by_external_id`.
  - Custom Field/Metafield на товаре дублирует ключ как «источник истины на стороне магазина» (на случай
    пересборки БД приложения: полный реконсиляционный проход `GET /catalog/products?include=custom_fields`
    постранично, чтобы восстановить карту). Реконсиляция — фолбэк, не горячий путь.
- **Никогда** не искать по `name`/`url`/`slug` (нестабильны), §5.1.
- **SKU-политика (§5.2):** `Product.sku` = `article` (только если непустой). `public_id` в SKU **НЕ** класть.
  BigCommerce **требует уникальности sku** (как Woo, в отличие от Shopify) → при коллизии sku с другим товаром
  **не перезаписывать**, логировать (§5.2).
- Легаси-фолбэк (§5.1): приложение новое — старого ключа нет; но саму схему «БД-карта + Custom Field/Metafield
  зеркало» фиксируем как единственный ключ, чтобы будущая миграция имела опору.

### 3. Характеристики `options[]` → Custom Fields / Metafields — **НЕ варианты, НЕ модификаторы!** (критично)

Центральная архитектурная развилка относительно стандарта §3 (где характеристика = «глобальный атрибут + термины»).

На BigCommerce **произвольных таксономий атрибутов товара НЕ существует**. Доступны:
- **Variant Options** (`/catalog/products/{id}/options`) — порождают **варианты** (Размер/Цвет). Использовать
  их под «характеристики-факты» НЕЛЬЗЯ — это взорвёт число вариантов (декартово произведение).
- **Modifiers** (`/catalog/products/{id}/modifiers`) — поля выбора покупателем при заказе (надбавки к цене).
  Тоже **не** для характеристик-фактов.
- **Custom Fields** (name/value на товаре, до 200) — простой «характеристика → значение». **Базовый выбор.**
- **Metafields** (типизированные namespace/key/value, есть permission/visibility) — для технических/скрытых
  или структурированных характеристик.

**Решение:**
- Каждую характеристику OneCatalog класть в **Custom Field**: `name = specification_label`, `value =`
  значение по типу:
  - `text` → `specification_option_name` (как есть);
  - `numeric` → `numeric_option` (+ при наличии единицы дописать подпись, напр. «12 мм»);
  - `boolean` → значение для true/false из настроек маппинга (§5.6: null/false → «значение false»).
- ⚠️ **Проблема стабильного ключа:** Custom Field матчится по `name` (переводимый label!). Если label
  меняется при смене языка/правке в OneCatalog — связь рвётся, плодятся дубли полей. Поэтому **технический
  ключ-якорь храним в Metafield** с `key = "spec_<specification_id>"` (стабильный `specification_id`), а
  Custom Field используем как «человекочитаемую витрину». При ручном маппинге (см. ниже) — опираться на
  `specification_id`, а Custom Field перезаписывать по соответствию через карту в БД приложения (id Custom
  Field'а товара хранить рядом), а не искать по name.
- **Идемпотентность характеристик:** Custom Fields товара **не** имеют стабильного внешнего ключа в API
  (только числовой `id`, выданный BigCommerce). Чтобы при переимпорте обновлять, а не плодить:
  - либо при апдейте читать `GET .../custom-fields`, матчить по `name` (с риском из-за смены label),
  - либо хранить в БД приложения карту `public_id + specification_id → custom_field_id` и слать `PUT`.
  Рекомендую второй путь (надёжнее) — это ещё один довод за БД приложения.
- **«Атрибут + термины» как фасет:** на BigCommerce storefront-фильтрация (faceted search) работает по
  **Product Filters** в storefront-настройках, и фильтры можно строить по **Custom Fields** (на планах,
  где доступен Product Filtering). То есть «характеристики-фасеты» достижимы через Custom Fields, но это
  **не** переиспользуемая таксономия термов — у значения нет своей сущности-страницы (см. «невозможно»).
- **Ручной маппинг (§5.4, строгий режим):** UI приложения «характеристика OneCatalog (`specification_id`) →
  целевой Custom Field name / Metafield key». При ВКЛ — только сопоставленные; для `boolean` — значения
  true/false. При ВЫКЛ — авто: Custom Field по `specification_label`, ключ-якорь в Metafield по
  `specification_id`.

### 4. Категории `categories[]` → нативное дерево Categories (плюс относительно Shopify)

- BigCommerce имеет **нативное дерево категорий**: `/catalog/trees` + `/catalog/trees/{tree_id}/categories`
  (или legacy `/catalog/categories`), у категории есть **`parent_id`** → **иерархия из OneCatalog ложится
  один-в-один**, в отличие от плоских Shopify-коллекций.
- Товар привязывается массивом id: `Product.categories = [id, ...]`.
- **Идемпотентность категории:** искать по **имени (`menutitle`) в пределах того же `parent_id`** (§5.1 —
  по имени, не по слагу/url). API: `GET /catalog/trees/{tree}/categories?name=...&parent_id=...`. Если нет —
  создать. **Хранить карту `category public_id/slug → bc_category_id` в БД приложения** (как у товаров —
  поиск по внешнему ключу напрямую API не делает; есть фильтр по name, но он не гарантирует уникальность
  при совпадающих именах в разных ветках).
- **Multi-Storefront:** в BigCommerce деревья категорий привязаны к channel(ам). Если магазин мультивитринный,
  при создании дерева/категорий указывать нужный `tree_id`/channel — настройка приложения (см. вопросы).
- Порядок создания: родители раньше детей (топологическая сортировка ветки) — иначе `parent_id` не на что
  сослаться.

### 5. Бренд `brand` → нативный Brand (плюс относительно Shopify)

- BigCommerce имеет **нативную сущность Brand** (`/catalog/brands`), товар ссылается через **`brand_id`**.
  Это сильно лучше Shopify (`vendor` — просто строка).
- **Идемпотентность бренда:** искать по **имени (`brand.menutitle`)** — `GET /catalog/brands?name=...`
  (имя бренда уникально). Нет → `POST /catalog/brands`. Карту `brand.id (OneCatalog) → bc_brand_id`
  держать в БД приложения.
- Богатые поля бренда: у BigCommerce Brand есть `name`, `page_title`, `meta_keywords`, `meta_description`,
  `image_url`, `custom_url`, **`custom_fields`/metafields бренда**. Туда ложатся `description_text`,
  `foundation_year`, `website`, `country`, логотип (`images_urls` → `image_url` бренда, дедуп §5.3).
- Развилка стандарта «атрибут / таксономия / CPT» на BigCommerce → **нативный Brand** (по умолчанию) либо
  Custom Field (если кто-то хочет бренд как фасет-поле, а не сущность). По умолчанию — Brand.

### 6. Страна `country` → Custom Field / Metafield

- В ядре товара BigCommerce **нет поля «страна происхождения»** (есть `Product.tax_class_id`, но это про
  налоги, не про страну). Поэтому страна → **Custom Field `oc_country`** (видна на витрине) или **Metafield**
  (если скрытая). Развилка стандарта «атрибут / таксономия» здесь = «Custom Field (витринный фасет) / Metafield
  (технический)».
- Если страна нужна как **фасет** — Custom Field + Product Filter по нему.
- ⚠️ Не путать с возможными налогово-таможенными полями; страна происхождения OneCatalog — справочная, не для
  таможни.

### 7. Коллекции `collections[]` → Category-ветка или Metafield (настройка)

- Нативной сущности «коллекция» у BigCommerce нет (в отличие от Shopify Collections). Развилка стандарта
  «тип объекта + цель»:
  - **Category** — создать отдельную ветку дерева «Коллекции» (или подкатегории) и привязывать товар как к
    обычной категории. Плюс: навигация/фильтрация из коробки. Минус: смешивается с товарными категориями
    (решается отдельным `tree`/корневой категорией «Коллекции»).
  - **Metafield / Custom Field** — коллекция как поле товара (если коллекция не нужна как раздел навигации).
- Богатые поля коллекции (`link_3d`, `link_official_site`, логотип) — в Custom Fields/Metafields категории
  (у Category тоже есть `metafields`) либо в metafields товара.
- Выбор «Category | Metafield» + цель — настройка приложения.

### 8. Теги `tags[]` → нет нативных тегов

- ⚠️ **У товара BigCommerce НЕТ нативной сущности «теги»** (в отличие от Shopify `Product.tags` и Woo
  `product_tag`). Ближайшие варианты:
  - **`Product.search_keywords`** (строка ключевых слов для поиска по магазину) — семантически близко к тегам,
    но это поле поиска, не фасет и не кликабельный термин.
  - **Custom Field `oc_tags`** со списком (CSV) — можно сделать фасетом через Product Filter.
  - **Metafield** (`list`-подобное хранение строкой) — техническое.
  - **Category-ветка «Теги»** — если теги должны быть навигационными разделами (тяжеловесно).
- По умолчанию (как в стандарте) теги **выключены**; при включении — целевое хранилище выбирается настройкой
  (search_keywords | Custom Field | Category). См. вопрос.

### 9. Медиа `images_urls` / `files[]` → Product Images by URL (§5.3)

- ФС магазина недоступна (SaaS). Но BigCommerce умеет **импорт картинки по URL**: `POST
  /catalog/products/{id}/images` с телом `{ "image_url": "https://…" }` — **BigCommerce сам скачивает файл
  по URL на свою сторону**. Это снимает проблему «URL без расширения / MIME» из §2.3 (определение типа —
  на стороне BigCommerce), и **не требует staged uploads**, как у Shopify.
  - Альтернатива — `image_file` (multipart upload файла), если по URL не выходит (напр. нужен токен в
    заголовке, которого BigCommerce при скачивании не пошлёт). Тогда приложение само качает файл (как в §2.3:
    ручное скачивание + MIME) и отдаёт BigCommerce как multipart. **Для приватных OneCatalog-URL с токеном в
    пути это рабочий путь; для публичных `min` — можно просто `image_url`.** Развилку зафиксировать.
- **Обложка vs галерея:** у изображения товара есть флаг **`is_thumbnail`** — первый/обложка ставит
  `is_thumbnail=true`, остальные — галерея. Порядок — `sort_order`.
- **Трекинг качества (§5.3):** выбранный размер (`min|middle|max`) фиксировать в БД приложения рядом с записью
  о картинке (BigCommerce поля «качество» не хранит). На переимпорте: если доступен размер лучше (появился
  токен) — удалить старое изображение (`DELETE .../images/{id}`) и загрузить новое; иначе не трогать.
- **Идемпотентность галереи по имени файла (§5.3):** в БД приложения карта `public_id → [{content_key,
  bc_image_id, size}]`. URL подписан/меняется — сравнивать по **контент-ключу** (из base64-префикса
  `media_files` — `path`, §5.3), не по URL.
- **Дедуп по контент-ключу (§5.3):** один файл у многих товаров. Но ⚠️ **в BigCommerce изображение
  привязано к конкретному товару** (нет общей медиа-библиотеки с переиспользованием одной записи на много
  товаров, как WP attachments). Значит «не качать повторно» означает: BigCommerce всё равно скачает файл для
  каждого товара (по `image_url`), но **приложение не должно дублировать обращения и может переиспользовать
  тот же `image_url`** (BigCommerce кэширует CDN сам). «Общие файлы не удалять» (§5.3) здесь означает: при
  замене качества у одного товара не трогать изображения других товаров — что и так выполняется (они отдельные
  записи). См. «невозможно» про отсутствие общей медиатеки.

### 10. Вес и габариты → нативные поля товара (плюс относительно Shopify)

- **Вес:** `Product.weight` — нативное поле товара (а не вариант, как в Shopify). Единица — **глобальная из
  Store settings** (`weight_units`: LB/KG/…), у товара единицу не задать → конвертировать источник в единицу
  магазина (§5.6, конвертер ниже).
- **Габариты:** `Product.width` / `Product.height` / `Product.depth` — **нативные поля в ядре товара**
  (у Shopify их нет вообще → metafields). Маппинг `sizes`: `length → depth` (или по договорённости — см.
  вопросы про соответствие осей), `width → width`, `height → height`. Единица — глобальная
  `dimension_units` (IN/CM/…) из Store settings.
- Это **главное преимущество BigCommerce над Shopify** для нашего каталога (плитка/керамика — габариты важны).

## Фоновая очередь (rate limits), медиа, настройки, i18n, жизненный цикл

### Фоновая очередь и rate limits (§6) — на стороне приложения

- У магазина **нет серверного cron** — вся фоновая работа в инфраструктуре приложения (свой worker + БД/Redis
  очередь). Это как Shopify.
- Список `public_id` режется на **порции по «шагу импорта» (минимум 10)**; порции — фоновые задачи; UI ставит в
  очередь и **опрашивает статус** (pending/running + лог). Синхронный фолбэк — для маленьких списков.
- **Rate limits BigCommerce:** REST V3 отдаёт заголовки **`X-Rate-Limit-Requests-Left`,
  `X-Rate-Limit-Time-Reset-Ms`, `X-Rate-Limit-Requests-Quota`, `X-Rate-Limit-Time-Window-Ms`** (квота на
  окно, зависит от плана — Standard/Plus/Pro/Enterprise разные). При исчерпании — **HTTP 429** с
  `Retry-After`. Worker обязан:
  - читать `X-Rate-Limit-*` и **притормаживать проактивно** (адаптивный throttle: если `Requests-Left` мал —
    спать до `Time-Reset-Ms`), а не ждать 429;
  - на 429 — экспоненциальный backoff с уважением `Retry-After`;
  - **конкурентность 1–2 на store_hash** (квота на магазин, не на эндпоинт), чтобы не выжигать окно.
- **Стоимость импорта 1 товара в запросах** высока (товар + N изображений + custom fields + категория/бренд
  поиск/создание). На больших списках это и есть главный практический лимит — оценивать throughput по квоте
  плана (см. вопросы). GraphQL для чтения снижает число round-trip (батч), но запись всё равно REST.
- **Webhooks vs очередь:** входящие Webhooks (напр. `store/product/updated`) приложение принимает на свой
  endpoint и кладёт в свою же очередь — наружу магазина произвольных событий нет (§8 — только in-process).

### Медиа

См. §9 выше: Product Images by URL (`image_url`) — основной путь; multipart `image_file` — для приватных
URL с токеном; трекинг качества и идемпотентность по контент-ключу — в БД приложения (BigCommerce их не
хранит).

### Настройки приложения (§7)

Хранятся в БД приложения (per `store_hash`), UI — внутри iframe-панели приложения в админке BigCommerce.

| Настройка | BigCommerce-реализация | По умолчанию |
|---|---|---|
| Токен Wiki API | `X-API-Key`; **env приоритетнее поля** | пусто |
| Язык товаров | `lang` (en/ru/ar/zh/kk) | en |
| Шаг импорта | размер порции очереди, **минимум 10** | 10 |
| Статус новых товаров | → `is_visible` (true=publish; pending/draft/private → false); **только при создании** | visible |
| Импортировать коллекции + хранить как | **Category-ветка \| Metafield** + цель | Category |
| Импортировать бренд + хранить как | **нативный Brand** \| Custom Field | вкл, Brand |
| Импортировать страну + хранить как | **Custom Field \| Metafield** | выкл, Custom Field |
| Импортировать теги + хранить как | **search_keywords \| Custom Field \| Category** | выкл |
| Ручной маппинг характеристик | вкл/выкл; страница: `specification_id` → Custom Field name/Metafield key; для boolean — true/false | выкл |
| Целевой channel/tree (Multi-Storefront) | id дерева/канала для категорий | дефолтный channel |
| Конвертер единиц | читать `weight_units`/`dimension_units` из Store settings, конвертировать г→ед., мм→ед. | авто из Store |

### Конвертер единиц (§5.6) — обязателен

- OneCatalog отдаёт **вес в граммах, размеры в миллиметрах** (подпись локализована — «мм»/«г»).
- BigCommerce хранит **в единицах магазина** (`weight_units`: `LB`/`KG`/`OZ`/`G`; `dimension_units`:
  `IN`/`CM`/… — читать через `GET /v2/store` или Settings API).
- Конвертер: таблица коэффициентов к базовой единице (вес→граммы, длина→миллиметры) + нормализация
  локализованных подписей источника (`sizes.*_unit`) → перевод в единицу магазина перед записью. Единицу
  источника брать из payload, иначе мм/г; переопределяемо событием приложения (§8). Без этого 1200 мм
  превращаются в 1200 см.

### i18n (§9) и мультиязычность

- **UI приложения** (его iframe-панель/настройки) — английские исходники + переводы (механизм фреймворка
  приложения, напр. i18next), text-domain единый. Строки фронт-скриптов — из бэкенда локализованными.
- **Мультиязычность каталога (Multi-Storefront / Translations):** BigCommerce поддерживает **Multi-Storefront**
  (несколько channel) и **Translations API** для перевода полей товара/категории на язык канала. Маппинг
  `lang` OneCatalog → язык/локаль конкретного channel — настройка приложения. Базовый импорт пишет поля на
  язык магазина по умолчанию; перевод на доп. локали — отдельным проходом через Translations API (если у
  магазина включён Multi-Storefront и план это позволяет). См. вопросы — это нетривиально и плановозависимо.

### Жизненный цикл приложения (§10) — install/uninstall, scopes

- **OAuth-установка:** Single-Click App из Marketplace (или custom app в Dev Portal). Поток: пользователь
  ставит → BigCommerce редиректит на `auth_callback` с `code/scope/context` → приложение меняет `code` на
  **`access_token` + `store_hash`**, сохраняет в БД. **`load_callback`** (вход из админки) и
  **`uninstall_callback`** (удаление → стереть данные магазина) обязательны.
- **Scopes (минимально необходимые):**
  - `store_v2_products` / `store_v2_products_read_write` — товары, варианты, изображения, custom fields;
  - доступ к **Categories/Brands** идёт под товарным scope (Catalog) — проверить точное имя scope в Dev
    Portal (см. вопросы);
  - `store_v2_information_read_only` — чтение Store settings (единицы веса/размеров);
  - `store_content` — если используем Content/виджеты (вероятно не нужно);
  - `store_v2_default` / webhooks scope — для подписки на Webhooks;
  - **запрашивать минимум** (принцип наименьших привилегий; лишние scopes отпугивают при установке).
- Манифест/версия — **SemVer**, CHANGELOG (Keep a Changelog), бэклог = «Не выпущено»; поднимать версию перед
  каждой сборкой; новые строки UI — переводить.
- **Миграции** схемы БД приложения (per-store настройки/карты) — стандартные миграции приложения.
- **Деградация:** при отзыве токена/удалении приложения — корректно останавливать очередь магазина, чистить
  данные на `uninstall_callback`.

### Точки расширения (§8)

- **Внутри приложения** — in-process шина: `before_import_product`, `product_imported(bc_id, payload,
  report)` (сайтовый слой дозаполняет custom fields/metafields, alt картинок, ед.изм., area_per_package,
  product_shape, upsells — всё вне маппинга ядра), `collection_imported`, `brand_imported`,
  `country_imported`, `queue_enqueued`. Фильтры: base URL, токен, аргументы запроса, язык, шаг, payload,
  SKU, цена (по умолчанию не задаётся → но см. блокер цены!), порядок размеров изображений, дедуп медиа,
  справочник характеристик, флаги импорта, origin виджета.
- **Наружу магазина** — только предопределённые **Webhooks** (`store/product/*`, `store/category/*` и т.п.),
  не произвольные хуки. Приложение их принимает и реагирует у себя.

### Виджет-пикер (§2.4) в админке BigCommerce

- Кнопка/панель приложения рендерится в **iframe внутри админки BigCommerce** (BigCommerce App UI — само
  приложение и так iframe в админке). OneCatalog picker (`tools.onecatalog.net/picker.html`) встраивается
  как **вложенный iframe** внутри iframe приложения — это **iframe-в-iframe**.
  - Подводные камни: sandbox `allow-scripts allow-forms allow-same-origin`; origin виджета задавать явно
    (не `window.location.origin`); проверять `event.origin` в `postMessage`; импорт по **`productPublicIds`**.
  - ⚠️ BigCommerce может ограничивать CSP/встраивание в своей админ-панели — проверить, что вложенный
    `tools.onecatalog.net` грузится (см. вопросы). Фолбэк — импорт по списку ID, вставленному вручную, или
    picker в отдельном окне (`window.open`) с `postMessage` через opener.

## ❓ Вопросы и сомнения (ответить ДО старта)

1. **Цена обязательна — блокер.** BigCommerce **требует `price` при создании товара** (как минимум для
   корректного physical-товара; создание без цены даёт 0/ошибку валидации в ряде сценариев). Стандарт §5.6
   **запрещает синтезировать цену**, которой нет в OneCatalog. Как разрешаем конфликт: (а) создавать с
   `price = 0` и `is_visible = false` до ручной простановки? (б) обязательный фильтр-источник цены до импорта?
   (в) импортировать как **draft/скрытые** и требовать прайс-лист? Нужна продуктовая политика.
2. **`weight` обязателен для physical.** Аналогично: physical-товар в BigCommerce обычно требует `weight`
   (иначе ломается расчёт доставки). Если в payload `sizes.weight` пуст — ставить 0? Делать товар
   `is_visible=false`? Брать из события сайта?
3. **Поиск по public_id невозможен через API → БД приложения обязательна.** Подтверждаем, что приложение
   ведёт собственную БД-карту `public_id → bc_product_id` (а не полагается только на Custom Field/Metafield)?
   Это влияет на инфраструктуру (хостинг с persistent БД) и на стоимость.
4. **Уникальность `name` и `sku`.** BigCommerce требует уникальное `name` (и `sku`, если задан). Если в
   OneCatalog два товара с одинаковым `menutitle` — как разруливаем коллизию имени при создании (суффикс?
   отказ + лог?)? При коллизии `article` (sku) — §5.2 говорит не перезаписывать; подтверждаем «создать без
   sku + лог»?
5. **Custom Field vs Metafield для технических ключей.** Финализировать: `oc_public_id` и якоря характеристик
   — в **Metafields** (скрыты от витрины, есть permission) или **Custom Fields** (видны на странице товара)?
   Рекомендация: технический ключ → Metafield, человекочитаемые характеристики → Custom Field. Согласовать.
6. **Теги: целевое хранилище.** Нативных тегов нет. По умолчанию теги выключены; при включении — что цель по
   умолчанию: `search_keywords` (поиск), Custom Field (фасет) или Category (навигация)? (Рекомендую Custom
   Field как наиболее «теговое».)
7. **Маппинг осей габаритов.** OneCatalog `sizes` даёт `length/width/height`; BigCommerce — `width/height/
   depth`. Подтвердить соответствие: `length → depth`? Или `length → height`? Для плитки это влияет на смысл.
8. **Категории и Multi-Storefront / trees.** Магазин одно- или мультивитринный? В какой `tree`/channel
   создавать категории? Использовать существующее дерево или отдельную ветку «OneCatalog»?
9. **REST V3 vs GraphQL как основной транспорт.** Подтверждаем: запись — REST V3 (полнее по images/custom
   fields/variants), чтение/поиск — GraphQL (батч)? Или всё на одном (проще, но дороже по round-trip)?
10. **Rate limits и пропускная способность.** Какой план магазина (Standard/Plus/Pro/Enterprise)? От него
    зависит квота `X-Rate-Limit` и реальный throughput. Нужно для оценки времени импорта больших списков и
    настройки throttle/конкурентности.
11. **Product Filtering (фасеты) доступен?** Storefront-фильтры по Custom Fields доступны не на всех планах/
    темах. Если «характеристики как фасеты» — требование, проверить план и тему (Stencil + Product Filtering).
12. **Картинки: `image_url` vs multipart.** OneCatalog отдаёт `max` только с токеном в URL. Сможет ли
    BigCommerce скачать такой приватный URL сам (`image_url`), или приватные качаем мы и шлём multipart
    (`image_file`)? Нужно протестировать на реальном URL (разведка §0).
13. **Мультиязычность каталога.** Нужен ли перевод полей товара/категории на несколько локалей через
    Translations API (Multi-Storefront), или достаточно одного `lang` на язык магазина по умолчанию? Это
    сильно меняет объём работы и зависит от плана.
14. **Picker iframe-в-iframe в админке BigCommerce.** Допустит ли CSP админки BigCommerce вложенный iframe
    `tools.onecatalog.net`? Если нет — фолбэк (отдельное окно / импорт по списку ID). Проверить на стенде.
15. **Идемпотентность Custom Fields / изображений.** Подтверждаем, что карту `public_id+specification_id →
    custom_field_id` и `public_id → [image_id+content_key+size]` ведём в БД приложения (BigCommerce стабильных
    внешних ключей для этих под-сущностей не даёт)?
16. **Scopes — точные имена.** Зафиксировать минимальный набор OAuth-scopes в Dev Portal (Catalog
    read/write, Store information read, Webhooks). Уточнить актуальные имена под текущую версию API.
17. **Объём метаданных vs лимиты.** До **200 Custom Fields на товар** и ограничения на Metafields. У товаров
    OneCatalog с большим числом характеристик — уложимся? Нужен ли отбор/маппинг обязательных характеристик?

## 🚫 Сложно / невозможно на BigCommerce

**Чем BigCommerce ЛУЧШЕ Shopify (важно зафиксировать):**
- **Нативные Brands** (`/catalog/brands`, `brand_id`) — бренд это сущность с полями/логотипом, а не строка
  `vendor`. Прямой маппинг `brand` OneCatalog.
- **Нативное дерево Categories** (`parent_id`, `/catalog/trees`) — иерархия OneCatalog ложится один-в-один,
  без «расплющивания» как у плоских Shopify Collections.
- **Габариты товара в ядре** (`width`/`height`/`depth`) и `weight` на товаре — для плитки/керамики критично;
  у Shopify габаритов нет вовсе (metafields), а вес — у варианта.
- **Импорт картинок по URL** (`image_url`) — BigCommerce качает сам, не нужны staged uploads как у Shopify;
  и проблема «URL без расширения/MIME» решается на их стороне.

**Чем BigCommerce ХУЖЕ / что сложно или невозможно:**

1. **Нет произвольных таксономий/атрибутов товара.** Как и Shopify, BigCommerce не имеет аналога Woo
   `pa_*` + термины или Bitrix свойств-списков. Характеристики → **Custom Fields/Metafields** (key-value,
   без сущности-страницы у значения). *Обход:* Custom Fields + Product Filters (фасеты), Metafields для
   технических полей; «термин как объект» — только эмуляция через категории, что искажает навигацию.

2. **Поиск товара по Custom Field/Metafield НЕвозможен через API.** `/catalog/products` не фильтруется по
   значению Custom Field/Metafield → `find_by_external_id(public_id)` за один API-запрос **недостижим**.
   *Обход (обязательный):* **собственная БД приложения** с картой `public_id → bc_product_id`; Custom Field/
   Metafield на товаре — лишь зеркало для реконсиляции (полный постраничный проход). Это жёстче, чем Shopify
   (там поиск по metafield в `query` есть), и поднимает требования к инфраструктуре приложения.

3. **Нет нативных тегов товара.** В отличие от Shopify (`Product.tags`) и Woo (`product_tag`). *Обход:*
   `search_keywords` / Custom Field / Category — но ни один не является «настоящим тегом» (кликабельный
   переиспользуемый термин). По умолчанию теги выключены.

4. **Нет общей медиа-библиотеки с переиспользованием одной записи на много товаров.** Изображение привязано
   к конкретному товару; дедуп «одно вложение на много товаров» (как WP attachments, §5.3) **невозможен** —
   каждый товар получает свою запись изображения. *Обход:* переиспользовать тот же `image_url` (BigCommerce
   кэширует на CDN), вести контент-ключи в БД приложения для идемпотентности галереи; «общие файлы не
   удалять» сводится к «не трогать изображения других товаров» (они и так отдельные).

5. **Цена обязательна при создании** (блокер vs §5.6 «не синтезировать»). *Обход:* создавать скрытыми
   (`is_visible=false`) с `price=0` и/или требовать прайс-источник через фильтр; продуктовое решение нужно
   до старта (вопрос 1). На Shopify цена варианта тоже по сути нужна, но политика «черновик без цены» там
   мягче.

6. **Нет серверного cron и доступа к ФС магазина** (общее SaaS-ограничение, как Shopify). *Обход:* вся
   фоновая работа и хранилище — в инфраструктуре приложения (свой worker/очередь/БД).

7. **Rate limits (квота на магазин, зависит от плана) + 429.** Импорт 1 товара = много запросов (товар +
   images + custom fields + поиск/создание категорий/бренда). На больших списках это главный практический
   потолок. *Обход:* проактивный адаптивный throttle по `X-Rate-Limit-*`, backoff по `Retry-After`,
   конкурентность 1–2 на магазин, минимизация round-trip (GraphQL для чтения). Хуже, чем у плагина-внутри-CMS
   (Woo) — там лимитов нет; сопоставимо с Shopify (но у Shopify калькуляция по «cost», у BigCommerce — по
   числу запросов в окне).

8. **Идемпотентность под-сущностей (Custom Fields, изображения) без стабильных внешних ключей.** API даёт им
   только числовой `id` BigCommerce, нет места под наш стабильный ключ внутри самой под-сущности (Custom
   Field — это name/value без namespace). *Обход:* карты в БД приложения; технический якорь характеристики —
   в Metafield (там есть namespace/key).

9. **Мультиязычность каталога — плановозависима и нетривиальна.** Translations API/Multi-Storefront есть, но
   доступность и объём работы зависят от плана; «просто перевести поля» как в одной CMS не выйдет. *Обход:*
   базовый импорт на язык магазина по умолчанию, переводы доп. локалей — отдельным проходом, только если
   магазин мультивитринный.

10. **iframe-пикер в админке (CSP).** Само приложение — iframe в админке BigCommerce; OneCatalog picker
    становится iframe-в-iframe и может быть заблокирован CSP админки. *Обход:* проверить на стенде; фолбэк —
    отдельное окно (`window.open` + `postMessage` через opener) или импорт по вставленному списку ID.

### Резюме для исполнителя

BigCommerce — **самая «каталожно-родная» из SaaS-целей** для этого стандарта: нативные **Brands**,
**дерево Categories** и **габариты+вес в ядре товара** дают прямой маппинг там, где Shopify требует
костылей (metafields/коллекции/вариант). Но три вещи делают реализацию жёстче ядра-в-CMS: **(1)** поиск по
`public_id` невозможен через API → **БД приложения обязательна**; **(2)** **цена и вес обязательны** при
создании, что конфликтует с §5.6 «не выдумывать»; **(3)** нет нативных тегов и произвольных таксономий —
характеристики/теги/страна живут в **Custom Fields/Metafields**. Плюс общие SaaS-ограничения: rate limits
(429, квота на магазин), отсутствие cron/ФС, всё фоновое — в приложении.
