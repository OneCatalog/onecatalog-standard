> 👏 Обзор точный: фичи-через-тип товара (классическая ловушка), отсутствие меты → свои таблицы, фон без

## Главное про данные (правит Q9 и закрывает Q13)

🟢 **Иерархия категорий ЕСТЬ — апгрейд к §«Категории» и Q9.** Вы написали «categories[] приходит плоским
→ импортируем плоско, иерархию из `/categories/` достроить best-effort (не блокер)». Поправка: плоский —
только эмбед `categories[]` в товаре. Справочный **`/categories/` отдаёт `parent_category`** (id родителя
или `null` = корень). Поскольку `shop_category` — **nested sets** (нативное дерево), это идеальная пара:
тяните `/categories/?limit=1000`, стройте полноценное дерево (родители раньше детей), find-or-create по
имени в пределах родителя. Не best-effort, а **рекомендованный путь** — получится точнее эталона (эталон
строит плоско).

🟢 **ВАРИАЦИЙ ТОВАРА НЕТ ВООБЩЕ — закрывает Q13.** Один `public_id` = один готовый товар, один SKU
(=`article`). Все `options[]` идут как **обычные (неселектируемые) фичи**. Подтверждаю ваше ключевое
предупреждение: НЕ включать `selectable`/не разворачивать комбинации — иначе Shop-Script размножит SKU по
декартову произведению. Это вы поймали верно.

🟢 **Характеристики многозначны** — «Цвет» = 2 значения, «Помещение» = **6**. Справочник
`/specifications/` отдаёт флаг **`multitude`** (true = многозначная; boolean `false`). → фичу под
многозначную характеристику заводить как **multiple-value** (фича с несколькими значениями у товара),
кардинальность по `multitude`.

🟢 **`numeric` в этом инстансе не используется** (все 13 характеристик text/boolean) — но код пишите
универсально (numeric → feature `double`/`dimension`).

🟢 **Всего ~13 характеристик** — типу товара назначается немного фич, схема не разрастётся.

## Ответы на вопросы

**1. Тип товара для импорта.** 🟡 Платформенное, но критичное (фичи привязаны к типу — без него
«невидимы»). Рекомендация: **настройка «тип товара по умолчанию»** = выбор существующего ИЛИ авто-создание
выделенного типа «OneCatalog»; все импортируемые фичи/бренд/страна назначать этому типу при создании.
Ваша «классическая ловушка» (блокер 2) — верно, это надо в дефолт. **Подтвердить, создавать ли тип
автоматически.**

**2. Хранилище `public_id`.** 🟢🟡 Гибрид верный: **таблица карты (источник истины для
`find_by_external_id`) + фича `_oc_public_id` (редактируемость в админке, §5.2 требует «отдельное
редактируемое поле») + легаси-фолбэк SKU**. Источник истины при поиске — таблица, фолбэк фича→SKU. Это
правильный перенос (у Shop-Script нет `wp_postmeta`). Согласен.

**3. Коллекция = `shop_set`?** 🟡 Платформенное `(проверить)`: допускают ли подборки (а) программную
привязку товаров, (б) идемпотентный find-or-create, (в) хранение ключей. Рекомендация: если да — `shop_set`
дефолт; если нет — дефолт «категория». Доп. поля коллекции (`link_3d`/`link_official_site`/изображения)
ядро не хранит — из эмбеда `collections[]` через событие `collection_imported`. Развилку «тип объекта +
цель» (set/категория/feature) сохраняете. **Подтвердить возможности `shopSetModel`.**

**4. Цена = 0.** 🟢 В источнике цены НЕТ — не синтезировать (§5.6). 🟡 `0.00` допустимо для каталога-
визитки; цену подставляет сайт через фильтр/событие. Если тема не должна продавать нулёвки — статус
`draft`/скрытие до простановки. Цена/статус — **только при создании**, не перезатирать. **Подтвердить
поведение темы при цене 0.**

**5. Cron доступен?** 🟡 Платформенное/хостинг. Рекомендация (ваша): **AJAX-степпер — основной режим**
(детерминирован, не зависит от хостинга), cron на `cli.php` — опциональное ускорение. Эталонный Action
Scheduler тоже визит/cron-зависим — не минус Shop-Script. **Подтвердить наличие системного cron.**

**6. Точка входа пикера.** 🟡 Платформенное `(проверить)`: есть ли backend-хук на тулбар списка товаров.
Фолбэк (всегда работает) — **отдельная страница плагина «Импорт из OneCatalog»** с пикером + поле «список
public_id». Стандарт §2.4 допускает импорт по списку. `event.origin` проверять, origin/parentOrigin
задавать явно, импорт по `productPublicIds`. **Подтвердить наличие хука.**

**7. Публикация событий из плагина.** 🟡 Платформенное `(проверить)` — может ли плагин ПУБЛИКОВАТЬ
событие, на которое подпишется mu-слой, и в каком неймспейсе. Контракт со стороны стандарта: нужна точка
`product_imported(product_id, payload, report)`, где сайт дозаполняет неядровые поля (полный payload).
Фолбэк (ваш) разумный: префиксный неймспейс shop ИЛИ DI-колбэк/конфиг-хук. **Подтвердить на версии.**

**8. «Фильтры» (возврат значения).** 🟡 У Webasyst нет `apply_filters`. Ваш приём — **событие с
изменяемым параметром по ссылке** (`$params['value']`) — рабочий и идиоматичный для Webasyst. Перечень
фильтров (base URL, токен, lang, шаг, payload, SKU, цена, порядок размеров, дедуп, флаги, карта маппинга,
origin) зафиксировать контрактом в README. Согласен.

**9. Иерархия категорий.** 🟢 **Строить дерево из `/categories/` (`parent_category`)** — см. «главное про
данные». `shop_category` nested sets подходит идеально; общий корень «OneCatalog» — опционально настройкой,
но настоящая иерархия лучше плоской под корнем.

**10. Мультиязычность.** 🟢🟡 API отдаёт **один язык за запрос**. **v1 — один язык** (настройка `lang`) —
вы это верно зафиксировали как осознанное сокращение (блокер 8). Мультиязык (N запросов с разным `lang` +
запись в локали контента) — отдельная фича позже. Эталон тоже один язык за прогон. **Подтвердить, нужен ли
мультиязык.**

**11. Единицы магазина.** 🟢 Источник: **вес в граммах, размеры в миллиметрах** (подпись локализована
«мм»/«г» на языке `lang`). **`sizes.weight` часто `null`** (у OC.IND.1 веса нет) — не выдумывать. 🟡 Куда
писать вес/габариты в Shop-Script (глобальная единица магазина vs feature `dimension` с явной единицей) —
платформенное. **Рекомендация: feature типа `dimension`** (число + единица) — удобно, единица хранится
рядом. Конвертер г/мм → целевая единица обязателен (иначе 1200 мм → 1200 см); единицу источника брать из
`sizes.*_unit`, иначе мм/г; нормализовать обе раскладки подписей («мм/mm», «г/g»). **Подтвердить целевое
поле для `sizes`.**

**12. Лимиты API источника.** 🟢 Справочники (`/specifications/`, `/categories/`, `/countries/`) — **одним
запросом `limit=1000`**; дефолтный лимит инстанса молча обрезает → сверяться с **`meta.counts`** (общее
число) и дострелить пагинацией, если больше. Жёсткого rate-limit на `X-API-Key` в тестах не наблюдалось.
Товары — по одному `/products/{public_id}/`. Порция очереди ≥10.

**13. Артикул-варианты.** 🟢 **Подтверждаю: вариаций НЕ создаём** (см. «главное про данные») — один
`article` → один SKU, `options[]` = обычные неселектируемые фичи. Никаких selectable/комбинаций.

**14. Объём выборки.** 🟡 От объёма — размер порции и нужность cron vs степпер. Рекомендация: степпер с
порцией ≥10 покрывает любой объём; на больших списках cron ускоряет. **Подтвердить порядок (десятки/
тысячи).**

**15. Версия Shop-Script.** 🟡 Платформенное — сверить наличие `shop_set`, сигнатуры моделей, состав
backend-событий, формат CLI-cron, install-механизм плагина. **Подтвердить версию `shop`** (шаг сверки по
помеченным `(не уверен)` местам).

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id`** (таблица карты + фича для UI), `article → sku` отдельно (только
   непустой; коллизия — не перезаписывать чужой). Искать ВСЕГДА по public_id, не по slug/имени.
2. **Справочники — поиск по имени (label), без учёта регистра, не по `code`/слагу** — критично:
   Shop-Script генерит url/code транслитом → разные слаги для одного имени → дубли (в эталоне был баг
   «Толщина» tolschina/tolshhina). Категория по имени в пределах родителя; фича/термин по `name`.
   Стабильный ключ маппинга — `specification_id`/category id, не переводимый label.
3. **Характеристики:** обычные (неселектируемые) фичи; назначать типу товара (Q1); многозначные
   (`multitude`) → multiple-value фича; ручной маппинг по `specification_id`, строгий режим (только
   сопоставленные). Вариаций НЕ создавать (Q13).
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие =
   false**; значения true/false из маппинга. (`multitude=false` у boolean.)
5. **Медиа:** ручное скачивание (`waNet`/curl) + **MIME по содержимому** (URL без расширения); дедуп
   `sha1(path#size)` (своя таблица); галерея по имени файла, не по URL; трекинг качества (min/middle/max
   оригинала, не путать с thumbs Shop-Script); общие файлы не удалять. NB: бинарный `media_files`-URL
   **самодостаточен** — токен зашит в base64-хеш, качается без `X-API-Key` (в эталоне `Media::sideload`
   берёт его обычным GET); `X-API-Key` нужен только JSON-вызовам `/products/` для получения max-URL.
6. **Не выдумывать цену** (`0.00`/статус draft, не перезатирать); статус — только при создании; `weight`
   бывает `null`.
7. **Единицы:** конвертер г/мм → единица магазина (feature `dimension` с явной единицей — удобно);
   `measurement_unit` (м²/шт) — продажная единица, не габарит → событие.
8. **Категории — дерево** из `/categories/` (`parent_category`, nested sets идеально). **`/collections/{slug}/`
   отдаёт 500 — не дёргать**, всё из эмбеда `collections[]`. **Справочники — `limit=1000` + сверка
   `meta.counts`.**
9. **Прочий payload** (`measurement_unit`, `areas_per_package`, `product_shape`, alt/описание картинок,
   детали бренда, `upsells/cross_sells`) — через событие `product_imported` (mu-слой сайта; полный
   payload). Прямой аналог mu-плагина эталона.
10. **SemVer + CHANGELOG** (Keep a Changelog), бэклог = «Не выпущено»; версию в `plugin.php` поднимать
    перед сборкой; новые UI-строки — в `locale/*.po` + пересборка `.mo` (gettext нативен — совпадает с
    эталоном).
