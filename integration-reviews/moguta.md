# Интеграционный план: Moguta.CMS

Реализация стандарта **OneCatalog Import** (`INTEGRATION-STANDARD.md`) на
**Moguta.CMS** — российской коробочной CMS для интернет-магазинов (PHP, своя
MVC-обвязка, таблицы `mg_*`). Эталон — WooCommerce; ниже — перенос внешних
контрактов §2 и инвариантов §5 на реальные механизмы Moguta: модели
`Models_Product` / `Models_Catalog`, характеристики (userProperty), система хуков
`MG::createHook` / `mgAddAction`, плагины `mg-plugins/*`, штатный импорт (CSV/XLS/YML,
в т.ч. «YML импорт по CRON»).

> **Допущения о платформе (зафиксированы, чтобы не выдумывать).** Целимся в
> актуальную линейку **Moguta.CMS 10.x–11.x** (в 11.0 заметные изменения ядра).
> Префикс таблиц по умолчанию `mg_`, в коде — только через штатную абстракцию БД
> (`Models_DB` / `MG::getDB()` — *не уверен по Moguta, имя класса проверить в
> исходниках/документации*). Движок каталога у Moguta встроенный (отдельного
> «движка магазина» поверх CMS, как Woo поверх WP, нет) — проверка §10 «активация
> без движка магазина» вырождается в проверку версии Moguta.CMS.
>
> **Что подтверждено по документации** (ссылки внизу): модель товара и её хуки,
> 7 типов характеристик, система хуков `MG::createHook`/`mgAddAction`, структура
> плагина (`index.php`, `Pactioner.php`, `pageplugin.php`, `readme.txt`),
> `getOption`/`setOption`, штатный API `Controllers_Api`, AJAX-роутер
> `Controllers_Ajaxrequest`, существование плагина «YML импорт по CRON».
> **Что НЕ подтверждено и помечено `(не уверен по Moguta)`:** нативный бренд как
> отдельная сущность, страна происхождения, коллекции, мультиязычность, точные
> сигнатуры загрузки изображений, наличие в ядре планировщика/очереди, хук на
> установку плагина.

Сводная таблица (§3 стандарта → реальные таблицы/механизмы Moguta). Имена таблиц —
без префикса; в коде через абстракцию БД.

|---|---|---|---|

| options[] text | Характеристика | характеристика типа «Строка»/«Набор» (`mg_property` + значения) + привязка к товару (`userProperty`) | `userProperty` в `addProduct`/`updateProduct` |
| options[] boolean | Характеристика-флаг | характеристика «Набор»/«Чекбокс» со значениями «Да»/«Нет» (из маппинга true/false) | нет нативного типа boolean — см. §boolean |

| tags[] | Теги | `mg_product.meta_keywords` (CSV) ИЛИ характеристика-«Набор» «Теги» (настройка) | нет нативной таблицы тегов (блокер B4) |

### Слои (перенос §4 на Moguta) — оформляем как **плагин** `mg-plugins/onecatalogimport/`

Стандартная структура плагина Moguta (подтверждено докой): `index.php` (манифест +
регистрация хуков и логики), `Pactioner.php` (класс `Pactioner extends Actioner` —
обработчик AJAX-запросов плагина из админки/паблика), `pageplugin.php` (разметка
страницы настроек), `readme.txt`. Ядро-логику разносим по PHP-классам внутри
плагина (Moguta не навязывает слои, но и не мешает):

```
mg-plugins/onecatalogimport/
  index.php                  — манифест (Plugin Name/Version/...), MG::addAction(...) на хуки + меню
  pageplugin.php             — UI настроек + кнопка «Выбрать товары» (пикер)
  Pactioner.php              — приём списка public_id, шаг очереди, статус/лог (REST/AJAX-слой)
  lib/Api.php                — клиент Wiki API: base URL/token/lang/GET/обёртка ответа/ошибки→null
  lib/Units.php              — конвертер источник→единицы магазина (вес г→кг; размеры мм→…)
  lib/Media.php              — скачивание по URL/MIME-детект/обложка-галерея/трекинг качества/дедуп
  lib/Taxonomies.php         — find-or-create характеристик/категорий ПО ИМЕНИ (label)
  lib/CollectionImporter.php — коллекция → категория/характеристика/кастом (по настройке)
  lib/BrandImporter.php      — бренд → характеристика/категория (по настройке)
  lib/CountryImporter.php    — страна → характеристика/категория (по настройке)
  lib/ProductImporter.php    — оркестратор (вызывает Api/Media/Taxonomies/* в порядке §5.4)
  lib/Queue.php              — порции по «шагу» (≥10), фоновый прогон, лог/статус
  assets/picker-loader.js    — iframe + postMessage (productPublicIds)
  assets/admin-import.js      — постановка в очередь + поллинг прогресса
  lang/                      — i18n (en/ru) — формат проверить (§i18n)
```

Зависимости направлены вниз (как в эталоне): `ProductImporter` оркестрирует
`Api/Media/Taxonomies/*`; `Queue` вызывает `ProductImporter`; `Pactioner` вызывает
`Queue`; UI зависит только от `Pactioner`/настроек. Циклов нет.

### Идемпотентность (§5.1) — критично

- **`public_id` НЕ кладём в `code` (артикул)** — это разные поля (§5.2). Храним в
  **своей таблице** `mg_oc_product_map`. Перед импортом —
  `findProductByPublicId(public_id)`: нашли → `updateProduct`/`fastUpdateProduct`,
  нет → `addProduct` и записать связь.
- **Легаси-фолбэк:** если до перехода на свою таблицу public_id временно лежал в
  `code` — при не-найденном в карте искать по `code`, найти → создать запись в
  карте, очистить `code` если там был чужеродный ключ (миграция §10).
- **Справочники — find-or-create ПО ИМЕНИ (label), не по слагу/url.** Категория,
  значение характеристики, бренд, страна, коллекция — матчить по
  человекочитаемому имени без учёта регистра; url/slug — только для создания
  нового. Это прямо предотвращает дубли от разных транслитераций (Moguta сам
  генерит `url` транслитом → поиск по url плодил бы дубли, как «tolschina» vs
  «tolshhina» в эталоне).
- ⚠️ **Кэш товара.** `getProduct($id, $getProps=true, $disableCashe=false)` —
  при повторном чтении в рамках одного прогона передавать `$disableCashe=true`,
  иначе можно прочитать устаревший снимок.

### Артикул / SKU (§5.2)

- `mg_product.code` ← **только `article`** из payload, если он непустой; иначе
  оставить пустым. `public_id` туда **не писать**.
- При коллизии `code` с другим товаром — не перезаписывать (артикул трактуем как
  уникальный бизнес-идентификатор; в Moguta уникальность `code` не гарантирована
  на уровне схемы — проверять перед записью самим).

### Характеристики (§5.4) — `userProperty`, 7 типов

Подтверждено: Moguta поддерживает 7 типов характеристик — **Строка, Текстовое
поле, Набор (список значений, опц. с наценкой), Чекбоксы, Файл, Цвет, Размер**.
Характеристика **привязывается к категориям** — «чтобы значения были видны в
карточке товара, нужна привязка к разделам каталога». Значения товара лежат как
`userProperty` (массив) при `addProduct`/`updateProduct`; на чтении —
`thisUserFields` / `characteristics[id]` (для последнего нужно `CATALOG_PROP=0` в
`config.ini`).

Маппинг типов источника → Moguta:
- `text` → характеристика **«Строка»** или **«Набор»**. Рекоменд. «Набор», т.к.
  даёт фасетную фильтрацию из коробки и переиспользование значений между товарами.
- `numeric` → **«Строка»** (значение = `numeric_option` + опц. единица из payload).
- `boolean` → **«Набор»**/«Чекбокс» со ЗНАЧЕНИЯМИ «Да»/«Нет» из маппинга
  (для `true` и `false`). Источник не даёт текстового значения булева — без явного
  правила «true/false → значение» характеристика молча теряется (§5.6, §boolean).
- `color`/`size` нативные типы — не используем для общего ядра (источник их не
  размечает как таковые); при желании — через ручной маппинг (§ручной маппинг).

**Ручной маппинг (строгий режим, §5.4).** Отдельная вкладка настроек:
`specification_id (OneCatalog) → существующая характеристика Moguta`. Ключ привязки
— **стабильный `specification_id`**, не переводимое название. При ВКЛ импортируем
ТОЛЬКО сопоставленные характеристики; для `boolean` в маппинге задаются значения
для true/false (выбор из значений «Набора»). При ВЫКЛ — «по имени/создать».

⚠️ **Привязка характеристики к категориям обязательна.** В Moguta характеристика
без привязки к разделу не отображается в товаре. Импортёр после find-or-create
характеристики ОБЯЗАН ещё и привязать её к категориям импортируемого товара
(иначе значения «не видны»). Это дополнительный шаг, которого нет в Woo-эталоне
(там глобальные атрибуты видны всегда).

### Категории (§5.4)

- `mg_product.cat_id` — основная категория; `inside_cat` — массив дополнительных.
  Из `categories[]` payload: первую (или по правилу) → `cat_id`, остальные →
  `inside_cat`.
- find-or-create по имени; дерево плоское-ли/иерархическое — проверить, как
  `Models_Catalog` хранит родителя (поле parent в `mg_category`).
- После создания категории привязать к ней характеристики товара (см. выше).

### Бренд / Страна / Коллекция — «тип объекта + цель» (§3/§7)

Нативных отдельных сущностей «бренд»/«страна происхождения»/«коллекция» в Moguta
**не подтверждено** (блокеры B1–B3). Поэтому, как требует стандарт, даём **выбор
«тип объекта + цель»**, но набор типов у Moguta беднее, чем в Woo (нет CPT/
таксономий):

- **Бренд** (по умолчанию ВКЛ): тип = **характеристика «Набор» «Бренд»** (рекоменд.)
  | **категория-ветка** «Бренды/{name}». Цель — конкретная характеристика/корневая
  категория. Доп. поля бренда (`foundation_year`, `website`, `country`, логотип) —
  ядро не пишет, отдаёт в событие `oc_brand_imported`.
- **Страна** (по умолчанию ВЫКЛ): характеристика «Набор» «Страна производства»
  (рекоменд.) | категория-ветка.
- **Коллекция** (по умолчанию ВКЛ): категория-ветка «Коллекции/{Бренд}» (рекоменд.,
  даёт страницу-листинг) | характеристика «Набор» | кастом-таблица. Поля коллекции
  (`link_3d`, `link_official_site`, логотип) — ядро не пишет, отдаёт в событие
  `oc_collection_imported`.

UI настроек скрывает нерелевантные поля цели по выбранному типу.

### Теги (§3)

Нет нативной таблицы тегов (блокер B4). Настройка «Импортировать теги» (по
умолч. ВЫКЛ): при ВКЛ → характеристика «Набор» «Теги» (find-or-create по имени)
ИЛИ `mg_product.meta_keywords` (CSV). «Поиск по имени» сохраняется только в
варианте «Набор».

### Медиа (§2.3 / §5.3)

- **URL без расширения** (`…/media_files/<base64hash>`) → штатные функции вывода
  (`mgImageProduct`/`mgGalleryProduct`/`mgImageProductPath`) — это **только
  рендер**, не загрузчики. Скачиваем **вручную** (cURL/file_get_contents),
  определяем **MIME по содержимому** (`finfo`), сохраняем файл с расширением в
  каталог изображений товаров, прописываем имя в `image_url` (обложка) и в
  доп. изображения (галерея).
- **Трекинг качества:** рядом с файлом (в `mg_oc_product_map.quality_json` или
  своей таблице медиа) фиксируем выбранный размер `min|middle|max`. На повторном
  импорте: доступен размер лучше (например, появился токен) → перекачать/заменить,
  старый удалить (кроме дедуплицированных — см. ниже); не лучше → не трогать.
- **Галерея идемпотентна по ИМЕНИ файла** (подписи в URL меняются между запросами).
- **Дедуп по контент-ключу:** из base64-префикса `media_files` извлекаем стабильный
  `path`, ключ = `hash(path # size)`; перед скачиванием ищем уже загруженный файл
  по ключу (один файл встречается у многих товаров и как логотип бренда/коллекции).
  ⚠️ Дедуплицированные (общие) файлы при апгрейде качества **НЕ удалять** — сломает
  другие товары.
- ⚠️ Точные пути/функции сохранения картинки товара в Moguta — *проверить в
  исходниках* (`Models_Product` упоминает «copying product images to a new storage
  structure»); возможно есть служебный метод, который надо переиспользовать, а не
  писать файл напрямую.

### Единицы (§5.6) — конвертер обязателен

- Источник отдаёт в наименьших единицах: **вес — граммы, размеры — миллиметры**
  (подпись локализована: «мм»/«г»). Единицу брать из `sizes.*_unit`, иначе мм/г.
- `mg_product.weight` — *проверить, в каких единицах магазин ожидает вес* (обычно
  кг). Конвертер: таблица коэффициентов к базовой единице + нормализация
  локализованных подписей; источник→единицы магазина перед записью.
- **Габариты товара (length/width/height):** у `mg_product` **подтверждённых полей
  габаритов нет** (в сигнатуре `addProduct` есть `weight`, габаритов — нет, блокер
  B6). Варианты: писать в характеристики-«Строка» «Длина/Ширина/Высота» (с
  конверсией мм→см/мм по настройке) ИЛИ отдавать целиком в событие
  `oc_product_imported`. По умолчанию — характеристики (чтобы попадали в фильтры).

### Идемпотентность справочников и порядок (§5.4)

Порядок импорта одного товара — строго по §5.4:
```
payload → addProduct/update (title, code=article, count=in_stock, unit; цена НЕ
          задаётся; activity/статус — только при создании)
        → категории (find-or-create по имени → cat_id + inside_cat)
        → характеристики (резолв: ручной маппинг по specification_id ИЛИ по имени/
          создать; boolean → значение true/false; привязать значения; ПРИВЯЗАТЬ
          характеристику к категориям товара)
        → сохранить → получить product_id
        → public_id в mg_oc_product_map (своё поле)
        → бренд (характеристика/категория по настройке)
        → страна (характеристика/категория по настройке)
        → теги (если вкл)
        → обложка + галерея (скачивание/MIME/трекинг качества/дедуп)
        → коллекция (категория/характеристика/кастом по настройке, привязать, поля)
        → событие oc_product_imported(product_id, payload, report)  ← сайт дозаполняет
```

## Фоновая очередь, медиа, настройки, i18n, жизненный цикл

### Фоновая очередь (§6) — у ядра Moguta готового планировщика, похоже, НЕТ

Подтверждено: в ядре **нет встроенного механизма cron/планировщика** (в
справочнике разработчика не упоминается); CRON-импорт реализован **отдельным
платным плагином** «YML импорт товаров по CRON» (требует Moguta ≥10.10.5, ведёт
лог и статус) — то есть это паттерн «cron хостинга дёргает endpoint», а не очередь
ядра. Вывод — реализуем **AJAX-степпер** (как нативный импорт CSV/YML, который
заведомо пошаговый, иначе упёрся бы в таймаут):

- Список public_id режем на **порции по «шагу импорта»** (минимум 10).
- `Pactioner` отдаёт по одному вызову обработку очередной порции + прогресс/лог;
  `assets/admin-import.js` ставит в очередь и **опрашивает статус** до готовности.
- Состояние очереди храним в своей таблице/опции (`setOption`), чтобы переживать
  перезагрузку страницы.
- **Опциональный CRON-режим** (как у штатного YML-плагина): endpoint в `Pactioner`,
  дёргаемый хостинговым cron по расписанию для регулярной синхронизации — но это
  уже синхронизация по списку, а не пакет из пикера.
- Фолбэк без степпера: синхронная обработка маленького списка с тем же отчётом.

### Медиа

См. §Медиа выше: ручное скачивание + MIME-детект (URL без расширения), обложка +
галерея, трекинг качества с авто-апгрейдом, дедуп по контент-ключу, общие файлы не
удалять.

### Настройки (§7) — `getOption`/`setOption`

Подтверждено: настройки плагина хранятся через `getOption`/`setOption` (таблица
конфигурации). UI — `pageplugin.php`. Минимум (§7):

| Настройка | Назначение | По умолчанию |
|---|---|---|
| Токен Wiki API | `X-API-Key`; приоритет env над полем | пусто |
| Язык товаров | `lang` из `en/ru/ar/zh/kk` | `en` (для рос. магазина обычно `ru`) |
| Шаг импорта | размер порции, **≥10** | 10 |
| Статус новых товаров | `activity` 1/0 (опц. «новинка»/«рекомендуемый»); **только при создании** | активен (1) |
| Импортировать коллекции + тип+цель | категория / характеристика / кастом + цель | вкл, категория |
| Импортировать бренд + тип+цель | характеристика / категория + цель | вкл, характеристика «Набор» |
| Импортировать страну + тип+цель | характеристика / категория + цель | выкл |
| Импортировать теги | характеристика «Набор» / meta_keywords | выкл |
| Ручной маппинг характеристик | вкл/выкл; страница `specification_id`→характеристика, boolean true/false | выкл |

Валидация и кламп; приоритет env над токеном; язык — только из списка. Для
«справочных» сущностей — единый паттерн «тип + цель», UI скрывает нерелевантные
поля.

### i18n (§9)

- Исходные строки — английские, единый text-domain (как в эталоне) — НО Moguta
  ориентирована на русский UI и её родной механизм локализации/языковых файлов
  *нужно проверить* (формат `lang/*.php`? массивы констант?). Практика проекта
  (см. память по onecatalog-import): UI **по-русски**, строки в коде EN + перевод.
- Строки JS-скриптов (`picker-loader`, `admin-import`) передавать из PHP
  локализованными, не хардкодить в JS.
- ⚠️ Если у Moguta нет gettext-инфраструктуры (как у WP `.po/.mo`) — переводы
  ведём родным механизмом Moguta или простым PHP-массивом строк по локали.

### События / точки расширения (§8) — модель хуков Moguta

Подтверждено: хуки создаются `MG::createHook('name', $result, func_get_args())`,
обработчик регистрируется `mgAddAction('hook', 'handler', $argc, $priority)`.
Тип хука «сложный» (с `$result`) фактически РАБОТАЕТ КАК ФИЛЬТР — обработчик
получает `['result'=>…, 'arg'=>…]` и **возвращает изменённый результат**. То есть
Moguta даёт и фильтры, и действия одним механизмом — это покрывает §8.

Наш плагин **сам публикует** свои хуки `MG::createHook(...)` в ключевых точках
(имена — наши, по аналогии с §8):
- **Фильтры:** `oc_api_base_url`, `oc_api_token`, `oc_request_args`, `oc_lang`,
  `oc_queue_step`, `oc_product_payload`, `oc_sku`, `oc_price` (по умолч. не
  задаётся), `oc_image_sizes_order`, `oc_media_dedup_key`, `oc_gallery_files`,
  `oc_specifications`, флаги импорта и карта маппинга, `oc_widget_origin`.
- **Действия/события:** `oc_before_import_product(payload, existing_id)`,
  **`oc_product_imported(product_id, payload, report)`** ← сайт дозаполняет
  `unit`, `areas_per_package`, `product_shape`, alt картинок, габариты, детали
  бренда, upsells; `oc_collection_imported`, `oc_brand_imported`,
  `oc_country_imported`, `oc_queue_enqueued`.
- Сайтовые/оформительские поля витрины живут в **отдельном сайтовом плагине** на
  `oc_product_imported` — ядро о витрине не знает (граница «универсальное↔проектное»).

Штатные хуки Moguta, которые можно использовать дополнительно: `mg_start`/`mg_end`
(страница), `models_product_addproduct`/`models_product_updateproduct` (можно
навесить пост-обработку при создании/обновлении товара любым путём, не только
нашим). Полный список штатных хуков — на странице «Хуки» справочника (проверить).

### Виджет-пикер (§2.4)

- В `pageplugin.php` — кнопка «Выбрать товары»; по клику `assets/picker-loader.js`
  открывает **iframe** `https://tools.onecatalog.net/picker.html?token=…&parentOrigin=…`.
- `postMessage`: `{type:'ONECATALOG_SELECTED', productPublicIds}` → ставим в очередь
  по **`productPublicIds`** (стабильные ключи, не числовые id); `{type:'ONECATALOG_CLOSE'}`
  → закрыть.
- Sandbox `allow-scripts allow-forms allow-same-origin`; origin виджета задавать
  явно (фильтр `oc_widget_origin`), проверять `event.origin`.
- ⚠️ Админка Moguta может иметь свой CSP/обёртку iframe — проверить, что iframe на
  внешний домен не блокируется (блокер B7). Фолбэк — импорт по списку ID без пикера.

### Установка / обновление (§10)

- Манифест плагина — блочный комментарий в `index.php`: `Plugin Name`, `Version`,
  `Description`, `Author` (+ требование версии Moguta — *способ декларации
  проверить*). **SemVer** + `CHANGELOG.md` (Keep a Changelog), как заведено в
  проекте onecatalog-import.
- **Установка таблицы `mg_oc_product_map`** (+ опц. таблица медиа-карты): у Moguta
  **подтверждённого хука install/activate/uninstall плагина НЕТ** (блокер B5) →
  создаём таблицы **лениво** (CREATE TABLE IF NOT EXISTS при первом вызове/при
  загрузке `index.php` под флагом в `setOption('oc_schema_ver')`). Миграции схемы
  — по тому же флагу версии.
- **Миграция легаси:** если public_id был в `code` — фолбэк-поиск + перенос в карту,
  флаг «миграция выполнена».
- Активация не должна падать без движка магазина → вырождается в проверку версии
  Moguta и наличия `Models_Product` (если класса нет — деградируем, не падаем).

### Права

- AJAX/REST-приём (`Pactioner`) — только из админки и только для авторизованного
  администратора; Moguta-роутер AJAX (`Controllers_Ajaxrequest`) передаёт `action`
  в `Pactioner`. Проверять текущего пользователя/роль (механизм проверки прав
  Moguta — *уточнить*; в эталоне это capability + nonce).
- CSRF: использовать штатный токен Moguta для админ-AJAX (если есть; иначе —
  собственный токен в опции) — блокер/вопрос Q ниже.
- Штатный `Controllers_Api` (token+secret, `importProduct`/`importCategory`) —
  **можно переиспользовать как транспорт**, но он «фиксированный контроллер, не
  расширяемый хуками» (по доке) и заточен под формат Moguta-API, а нам нужен приём
  списка public_id и оркестрация — поэтому свой `Pactioner` предпочтительнее;
  `Controllers_Api::importProduct` можно дёргать ВНУТРИ как низкоуровневый
  upsert, если он удобнее `Models_Product` напрямую (оценить).

## 🚫 Сложно / невозможно на Moguta.CMS (как написано в стандарте)

- **B1. Бренд как богатая сущность.** Стандарт допускает «атрибут / таксономия /
  CPT» для бренда. У Moguta нет CPT/таксономий и (вероятно) нет нативной сущности
  «бренд» с собственной страницей и полями (`foundation_year`, `website`, логотип).
  *Обход:* бренд = характеристика «Набор» (фильтруемая) ИЛИ категория-ветка
  «Бренды/{X}» (даёт страницу-листинг); доп. поля бренда — в событие
  `oc_brand_imported` (сайтовый слой/кастом-таблица). Полноценную карточку бренда
  «из коробки» получить нельзя.

- **B2. Страна происхождения** — нет нативного поля (отдельного от стран доставки).
  *Обход:* характеристика «Набор» «Страна производства» (по умолчанию ВЫКЛ).

- **B3. Коллекции** — нет CPT/таксономии. *Обход:* категория-ветка (рекоменд., есть
  листинг) ИЛИ характеристика ИЛИ кастом-таблица + событие. «Тип+цель» в настройке
  сохраняем, но набор типов беднее, чем в Woo.

- **B4. Теги** — нет нативной таблицы тегов с «поиском по имени». *Обход:*
  характеристика «Набор» «Теги» (сохраняет поиск по имени) ИЛИ
  `meta_keywords`-CSV (поиск по имени теряется). По умолчанию ВЫКЛ.

- **B5. Нет хука установки/деинсталляции плагина** (не подтверждён). Нельзя
  «при активации создать таблицу `mg_oc_product_map`». *Обход:* ленивое
  `CREATE TABLE IF NOT EXISTS` + версия схемы в `setOption`; миграции по флагу.
  Деинсталляция/чистка таблиц — вручную/скриптом (риск осиротевших таблиц).

- **B6. Габариты товара (length/width/height)** — нет подтверждённых полей у
  `mg_product` (есть только `weight`). Конвертер мм→см «в родное поле габаритов»
  негде применить. *Обход:* писать габариты в характеристики-«Строка» (с конверсией)
  или целиком в событие `oc_product_imported`; «нативной» доставки по габаритам
  из этих данных не будет.

- **B7. iframe-пикер в админке.** Админ-панель Moguta может оборачивать страницы
  плагина так, что внешний iframe (`tools.onecatalog.net`) упрётся в CSP/верстку
  (не подтверждено). *Обход:* проверить рендеринг iframe в `pageplugin.php`;
  гарантированный фолбэк — импорт по списку public_id (вставка ID/загрузка файла),
  без пикера. Контракт `postMessage` остаётся, если iframe всё же работает.

- **B8. Фоновая очередь уровня ядра** — нет (Action Scheduler-аналога нет; cron —
  только через хостинг/платный YML-плагин). *Обход:* AJAX-степпер с шагом ≥10 и
  своим хранением состояния + опц. хостинговый cron-endpoint; это рабочий, но более
  ручной механизм, чем в эталоне (нет ретраев/изоляции задач из коробки).

- **B9. Расширяемость штатного `Controllers_Api`** — «фиксированный контроллер, не
  расширяемый хуками» (по доке). Нельзя добавить свой метод `importByPublicId` в
  штатный API. *Обход:* свой `Pactioner` (AJAX-роутер плагина это штатно
  поддерживает) как точка приёма списка public_id; штатный API — только как
  опциональный низкоуровневый upsert внутри.

- **B10. boolean-характеристики** — у Moguta нет «boolean-типа значения» в общем
  смысле (есть «Чекбоксы» как тип характеристики, но источник не размечает булевы
  как чекбоксы). *Обход (как в стандарте):* «Набор» со значениями «Да»/«Нет»; в
  маппинге задать значения для true/false; отсутствие/`null`/`false` → «значение
  для false». Без явного правила bool-характеристика теряет значение.
