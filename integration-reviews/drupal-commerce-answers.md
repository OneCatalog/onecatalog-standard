## Главное про данные (правит §4/блокер 8 и закрывает Q4)

🟢 **ВАРИАЦИЙ ТОВАРА НЕТ ВООБЩЕ** — один `public_id` = один готовый товар. Это **прямой ответ на ваш Q4 и
блокер 3**: покупательских вариаций (цвет/размер → разные SKU/цены) **не будет**. Значит:
- модель **single-variation product** (1 product + 1 variation) — правильная и окончательная;
- характеристики → **taxonomy** (Вариант B), а Commerce **attribute использовать НЕ нужно** (он ради
  генерации вариаций; у нас их нет — это был бы abuse, вы верно это описали). Дефолт `taxonomy`
  подтверждён авторитетно, не догадка.

🟢 **Иерархия категорий ЕСТЬ — поправка к §4 (строка 146) и блокеру 8.** Вы написали «иерархии у
categories из payload нет (плоский список)». Это так **только для эмбеда `categories[]` в товаре**. Но
справочный эндпоинт **`/categories/` отдаёт `parent_category`** (id родителя или `null` = корень). Пример:
«Керамогранит» → «Плитка» → корень. → тяните `/categories/?limit=1000`, стройте **иерархию термов**
(`parent` у taxonomy term), родители раньше детей. Drupal taxonomy нативно иерархичен — используйте это,
получится **точнее эталона** (эталон строит плоско).

🟢 **Характеристики многозначны** — «Цвет» = 2 значения, «Помещение» = **6**. Справочник
`/specifications/` отдаёт флаг **`multitude`** (true = многозначная; boolean `false`). → entity_reference
поле характеристики должно быть **multi-value** (cardinality по `multitude`).

🟢 **`numeric` в этом инстансе не используется** (все 13 характеристик text/boolean) — код пишите
универсально (Вариант C: decimal/string-поле под numeric).

## Ответы на вопросы

**1. Пустой `article` vs обязательный SKU Commerce.** 🟢 Факт: `article` бывает пуст/отсутствует.
Конфликт с обязательным уникальным `sku` реален. 🟡 Рекомендация для Drupal: **sku = `public_id`, если
`article` пуст** (помечен как авто-технический; `public_id` гарантированно есть и уникален — см. Q17), при
непустом — sku = `article`. «Артикул производителя для покупателя» при этом можно вынести в отдельное
информационное поле (не sku). Это вынужденный Drupal-специфичный компромисс к §5.2 (там Woo разрешает
пустой sku — Commerce нет). Коллизия `article` с чужим товаром — не перезаписывать, фолбэк на
public_id-sku + лог.

**2. Цена не задаётся (§5.6) vs обязательное `price`.** 🟢 В источнике цены НЕТ — не синтезировать
(§5.6). 🟡 Рекомендация: **отдельный variation type с НЕобязательной ценой** (снять required с field
price) — самый чистый путь, не выдумывает цену. Альтернатива — плейсхолдер `0` + product `unpublished` до
простановки. Цену подставляет сайт через событие (см. §8). Выбор влияет на схему variation type —
**подтвердить**.

**3. Привязка к `commerce_store`.** 🟡 Чисто Drupal/Commerce-требование (эталон Woo его не знает).
Рекомендация: **настройка «целевой магазин(ы)»**, по умолчанию default store. Без store товар не
покупаем/не виден. **Подтвердить целевой store.**

**4. Характеристики: attribute vs taxonomy vs field.** 🟢 **`taxonomy` — дефолт, подтверждено
авторитетно:** вариаций нет (см. «главное про данные»), Commerce attribute не нужен. Многозначные
(`multitude`) → multi-value entity_reference. Режим `attribute|taxonomy|field` оставьте опцией, но дефолт
taxonomy правильный. Ключ маппинга — `specification_id`, не переводимый label.

**5. Статусы новых товаров.** 🟢🟡 У Drupal product «из коробки» только boolean `status`. Маппинг (как у
вас): **`publish→published`, всё остальное (pending/draft/private) → unpublished**. Это корректный перенос
(те статусы — WP-понятия). Опциональная интеграция с `content_moderation` — если на сайте есть workflow.
Статус задавать **только при создании**, при переимпорте не трогать. **Подтвердить, нужен ли
content_moderation.**

**6. Physical/Measurement модуль.** 🟢 Источник: **вес в граммах, размеры в миллиметрах** (подпись
локализована «мм»/«г» на языке `lang`). **`sizes.weight` часто `null`** (у OC.IND.1 веса нет) — не
выдумывать. 🟡 Если `physical` установлен — пишите `Weight/Length` (конвертация и хранение единицы из
коробки) — лучший путь. Если нет — свой конвертер `Units` + фиксированная целевая единица. **Подтвердить
наличие `physical` и целевые единицы магазина.** Конвертер обязателен (иначе 1200 мм → 1200 см); единицу
источника брать из `sizes.*_unit`, иначе мм/г; нормализовать обе раскладки подписей («мм/mm», «г/g»).

**7. СУБД (collation).** 🟡 Чисто проектное. Для регистронезависимого find-or-create термов по `name`:
MySQL utf8mb4_*_ci — ок; PostgreSQL case-sensitive → `LOWER()`/функциональный индекс. **Подтвердить СУБД.**
(Поиск термов по name — критичный инвариант §5.1, см. «золото» п.2.)

**8. Гарантированность фона.** 🟡 Проектное. Рекомендация: **Queue API + cron — основной путь**, Batch —
синхронный фолбэк/CLI. Без системного cron/`drush` штатный Drupal-cron зависит от трафика — **тот же изъян,
что WP-Cron эталона** (вы верно отметили — не минус Drupal). **Подтвердить системный cron/drush.**

**9. Мультиязычность контента.** 🟡 API отдаёт **один язык за запрос**. **Фаза 1 — один язык** (`langcode`
из настройки). Мультиязык (`addTranslation` по `langcode` + переводы термов) = N запросов на товар —
отдельная итерация, стандарт это не специфицирует (общий пробел, отметили все агенты). Эталон тоже один
язык за прогон. **Подтвердить, нужен ли многоязычный каталог.**

**10. Media entity vs прямой file.** 🟢🟡 Рекомендую **media** (bundle image) — ваш план с полями
`field_oc_content_key`/`field_oc_size`/`field_oc_source_path` для дедупа и трекинга качества **точный**
(у file/media нет произвольной меты — поля на bundle = правильный аналог эталонной меты вложения).
**Подтвердить, использует ли проект Media Library.** Полезный факт: бинарный `media_files`-URL
**самодостаточен** — токен зашит в base64-хеш, файл качается **без заголовка `X-API-Key`** (в эталоне
`Media::sideload` берёт его обычным GET и получает max при наличии токена). URL без расширения → Guzzle
GET + MIME по содержимому, расширение присваивать самим.

**11. Бренд/коллекция как node или term.** 🟢🟡 Дефолты разумны: **бренд → vocabulary `brands`**
(нативной `product_brand` в Drupal НЕТ, создаёте сами — верно, блокер 7), **коллекция → node
`oc_collection`** (даёт страницу с `link_3d`/`link_official_site`/изображениями). Богатые поля бренда
(`description_text`/`foundation_year`/`website`/`country`) на term требуют доп. полей на vocabulary — ок,
либо в событие `brand_imported`. Развилку «тип объекта + цель» сохраняете. **Подтвердить дефолты.**

**12. CSP / iframe-пикер.** 🟡 Проектное. Drupal 10.1+/seckit/csp может блокировать `frame-src
tools.onecatalog.net`. **Фолбэк (всегда работает) — импорт по списку `public_id`** (стандарт §2.4
допускает; textarea). Рекомендую сначала ID-list, пикер опционально. `event.origin` проверять,
origin/parentOrigin задавать явно, импорт по `productPublicIds`.

**13. Лимиты/пагинация источника.** 🟢 Справочники (`/specifications/`, `/categories/`, `/countries/`) —
**одним запросом `limit=1000`**; дефолтный лимит инстанса молча обрезает → сверяться с **`meta.counts`**
(общее число) и дострелить пагинацией, если больше. Жёсткого rate-limit на `X-API-Key` в тестах не
наблюдалось. Товары — по одному `/products/{public_id}/`. Размер порции очереди ≥10.

**14. `public_id` как base field.** 🟢 Согласен — **base field** (`hook_entity_base_field_info` для
`commerce_product`): индексируемо, переносимо между bundle, быстрее в `entityQuery`. Возражений нет.

**15. Удаление при апгрейде качества + дедуп.** 🟢 Согласен — перед удалением старого file/media
проверять usage (контент-ключ не используется другими товарами): **общие файлы НЕ удалять** (ломает
другие товары, §5.3). Политика: при апгрейде качества — перекачать лучший размер, заменить; если файл
общий — не трогать. Проверка через `file_usage`/обратный entityQuery по `field_oc_content_key`.

**16. upsells/cross_sells.** 🟢 Согласен — **задача сайтового слоя** (subscriber на `PRODUCT_IMPORTED`),
ядро не маппит. Как в эталоне (всё, чего нет в ядре — `measurement_unit`, `areas_per_package`,
`product_shape`, alt картинок, детали бренда, upsells — в событие с полным payload).

**17. Уникальность `public_id`.** 🟢 Гарантируем один product на `public_id`: идемпотентность —
`entityQuery` по `field_oc_public_id` `range(0,1)` ПЕРЕД импортом (нашли → update, нет → create). `public_id`
всегда присутствует и уникален в источнике. Опциональный constraint — плюс, но проверка в импортёре
обязательна.

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id`** (base field, `range(0,1)`); `article → sku` (фолбэк sku=public_id
   при пустом article). Искать ВСЕГДА по public_id, не по sku/title/slug.
2. **Справочники — поиск по имени (label/title), case-insensitive** в нужном vocabulary; в Drupal у
   термов нет slug (плюс — нет транслит-дублей на уровне термов), но машинные имена автосоздаваемых
   словарей/полей транслитерировать через `\Drupal::transliteration()`. Стабильный ключ маппинга —
   `specification_id`/category id/brand id, не переводимый label.
3. **Характеристики → taxonomy** (вариаций нет); многозначные (`multitude`) → multi-value
   entity_reference; ручной маппинг по `specification_id`, строгий режим (только сопоставленные).
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие =
   false**; термы true/false из маппинга. (`multitude=false` у boolean.)
5. **Медиа:** media (bundle image) + поля трекинга (`field_oc_content_key`/`field_oc_size`); дедуп
   `sha1(path#size)`; Guzzle GET + MIME по содержимому (URL без расширения); галерея по имени файла, не по
   URL; общие файлы не удалять; URL самодостаточен (токен в хеше).
6. **Не выдумывать цену** (variation type без required price); статус только при создании; `weight` бывает
   `null` — не синтезировать.
7. **Единицы:** `physical` (`Weight/Length`) если есть, иначе свой конвертер г/мм → единица магазина;
   `measurement_unit` (м²/шт) — продажная единица, не габарит → событие.
8. **Категории — дерево термов** из `/categories/` (`parent_category`, родители раньше детей). **`/collections/{slug}/`
   отдаёт 500 — не дёргать**, всё из эмбеда `collections[]` товара. **Справочники — `limit=1000` + сверка
   `meta.counts`.**
9. **Прочий payload** (`measurement_unit`, `areas_per_package`, `product_shape`, alt/описание картинок,
   детали бренда, `upsells/cross_sells`) — через `OneCatalogEvents::PRODUCT_IMPORTED` (subscriber в
   отдельном модуле сайта; полный payload). Прямой аналог mu-плагина эталона.
10. **SemVer + CHANGELOG** (Keep a Changelog), бэклог = «Не выпущено»; версия в `.info.yml`; новые
    UI-строки — `t()`/`TranslatableMarkup` + `.po` в `translations/`, JS — `Drupal.t()` из бэка.
