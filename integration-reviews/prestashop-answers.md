## Главное, что нужно знать про данные (меняет ответы Q2/Q6/Q7)

🟢 **Многозначные характеристики — реальны и часты.** Факт API: у товара OC.IND.1 характеристика
«Цвет» имеет **2** значения (Светло-серый, Серый), «Помещение» — **6** (для ванной/гостиной/коридора/
кухни/общественных/улицы). Справочник `/specifications/` отдаёт флаг **`multitude`**:
`true` = характеристика многозначная (Цвет, Помещение, Толщина, Формат…), `false` = одно значение
(boolean-характеристики Эксклюзив/Морозоустойчивость — `multitude=false`). Это прямо отвечает на Q7 и
блокер 3 (см. ниже).

🟢 **`numeric` в этом инстансе не используется** — все 13 характеристик `text` или `boolean`
(`numeric` — задел). Цвет/Толщина/Формат хранятся как `text`.

## Ответы на вопросы

**1. Версия PrestaShop / режим модуля.** 🟡 Зависит от магазина-заказчика. Рекомендую один таргет
(1.7.6+ legacy или 8.x Symfony) под реальную версию, а не «и то и то». `ps_versions_compliancy` под неё.
**Подтвердить версию.**

**2. Характеристики: Feature или Attribute?** 🟢 **Подтверждаю однозначно: `Feature`/`FeatureValue`, НЕ
`Attribute`/`Combination`.** OneCatalog `options[]` — описательные характеристики (фасеты), вариаций у
товара **нет вообще** (один `public_id` = один готовый товар). Эталон кладёт их в WC-атрибут с
`variation=false` (= описательный). `Attribute` в PrestaShop тянет `Combination`+`StockAvailable` — это
семантически неверно. Это самый важный пункт корректности (как `option`-vs-`attribute` в OpenCart).

**3. Хранилище `public_id`.** 🟢 **Своя таблица `oc_product_link`** (а не `reference`) — правильно, точный
аналог эталона (ключ в выделенной мете, не в slug/артикуле). `public_id ≠ reference`. 🟡 Подтвердить,
нет ли внешней интеграции (1С), претендующей на `reference`/`supplier_reference`.

**4. Multistore.** 🟡 Дефолт — один магазин (целевой `id_shop`). При мультимагазине — явный
`Shop::setContext(CONTEXT_SHOP, $id_shop)` в фоновых задачах (ваш блокер 6 — верно). **Подтвердить:**
один shop или несколько, shared ли товары/категории/feature.

**5. Мультиязычность импорта.** 🟡 API отдаёт **один язык за запрос**. Фаза 1 — один язык: писать в
целевой `id_lang`; для прочих языков магазина — **дублировать значение дефолтного** (чтобы пройти
валидацию непустых `name`/`link_rewrite`) или заполнять только дефолтный (если магазин одноязычный).
`link_rewrite` генерить `Tools::link_rewrite($name, $id_lang)`. N-проходов по языкам — фаза 2. Эталон
тоже импортирует один язык за раз. **Подтвердить дефолтный `id_lang` и число языков.**

**6. `numeric`-характеристики.** 🟢 В этом инстансе их **нет** (все text/boolean). Когда появятся —
обычный `FeatureValue` строкой (`numeric_option`); единицу (из label) можно приклеить к значению.
Не усложнять заранее.

**7. Несколько значений одной характеристики.** 🟢 **Да, это норма** (Цвет=2, Помещение=6;
`multitude=true` в `/specifications/`). Эталон WC хранит это естественно: один атрибут — **несколько
термов** у товара. На PrestaShop UI ограничивает «одно `FeatureValue` на `Feature`», но БД
`product_feature` и модуль **Faceted Search** (layered navigation) поддерживают **несколько строк
`(id_product, id_feature, id_feature_value)`** на одну фичу → фасетная фильтрация по нескольким значениям
работает. Рекомендация:
- **(дефолт) создавать отдельный `FeatureValue` на каждое значение и вставлять несколько строк
  `product_feature`** — даёт правильную фасетку (проверьте, что тема/Faceted Search на целевой версии
  это отображает; на большинстве — да);
- **фолбэк** (если тема показывает только первое) — склеить в один `FeatureValue` («Для ванной, Для
  кухни, …») — теряется фильтрация, но значение видно.
Используйте флаг **`multitude`** из `/specifications/`, чтобы знать, какие характеристики многозначные.
**Подтвердить:** поддерживает ли тема/Faceted Search multi-value features.

**8. Коллекция — цель по умолчанию.** 🟡 Эталонный дефолт коллекции — отдельная сущность (CPT в WC). В
PrestaShop CPT нет. **Дефолт — `Category`-ветка** (навигация/страница из коробки; поля
`link_3d`/`link_official_site` → `category_description` или своя `oc_collection`-таблица). Своя
ObjectModel-сущность `oc_collection` + фронт-контроллер — если нужны отдельные страницы коллекций
(аналог CPT). Эталон недавно добавил и режим «атрибут» — на PrestaShop это `Feature`. Итог:
category (дефолт) / feature / своя сущность. **Подтвердить, есть ли уже понятие «коллекция» на витрине.**

**9. Страна происхождения.** 🟢 **Подтверждаю: НЕ писать в `Country`** (справочник зон/доставки/налогов —
сломаете чекаут). Страна происхождения → **`Feature` «Страна происхождения»** по имени (`menutitle`).
🟢 `country_code` в API — ISO **alpha-3** (`IND`,`ESP`), но при подходе «Feature по имени» код не нужен.
🟡 Отдельная витринная страница страны — обычно не нужна (Feature даёт фасет).

**10. Бренд → `Manufacturer`.** 🟢 Дефолт ок. `Manufacturer.name` не мультиязычен — упрощает поиск.
Расширенные поля бренда (`description_text`, `foundation_year`, `website`, `country`, лого) эталон **не
сохраняет даже в WC** (только имя + лого + OC-id) → это «прочий payload» → через хук
`actionOneCatalogBrandImported`/`Manufacturer::description` если реально нужны. Ядро ставит только
`Product::id_manufacturer`.

**11. Очередь.** 🟡 **Дефолт — AJAX-степпер** (надёжно на любом хостинге, аналог нативного импорта PS);
cron-контроллер / Messenger — опции для серверов с планировщиком/воркером. Эталонный Action Scheduler
тоже cron/визит-зависим — не минус PS. **Подтвердить наличие системного cron / PS Cron Tasks / воркера.**

**12. Дедуп медиа.** 🟢 **Принять компромисс «дедуп предотвращает повторное СКАЧИВАНИЕ, файл копируется
на товар»** — это правильный баланс для PrestaShop (`Image` физически принадлежит одному `id_product`).
Ключ `sha1(path#size)` в `oc_image` → нашли по ключу → не качаем заново, но файл копируем. Настоящий
shared-файл (своя m-к-m таблица + кастомный рендер) — избыточно. ⚠️ Общие файлы при апгрейде качества не
удалять (флаг в `oc_image`). Это и есть смысл инварианта: эталон экономит и сеть, и диск (WP-вложение
shared), на PS экономим сеть — допустимая деградация.

**13. Лимиты Wiki API.** 🟢 Жёсткого rate-limit на `X-API-Key` в тестах не наблюдалось. Справочники
(`/specifications/`, `/categories/`) брать **одним запросом `limit=1000`** (проверено; `meta.counts` =
общее число, дефолт молча обрезает). Товары — по одному `/products/{public_id}/`; медиа — несколько
файлов на товар; таймаут ~40с, ошибка → skip (§5.5). 🟡 Объём уточнить; ретраи/backoff заложить на
всякий случай.

**14. Picker (iframe).** 🟡 **Сначала «импорт по списку `public_id`» (вставка/CSV)** — всегда работает.
Пикер — опционально; учесть CSP/`X-Frame-Options` бэк-офиса (ослабить только на странице импорта или
вынести в «чистый» контроллер), `parentOrigin` явно, проверка `event.origin`. **Подтвердить, нужен ли
пикер на старте.**

**15. Изображения без токена.** 🟢 Без токена доступен только `min`-размер; с токеном — `max`, и при
появлении токена повторный импорт **авто-апгрейдит** качество (трекинг размера). Токен один на инстанс,
лучше через env-секрет. **Подтвердить, что токен будет.**

**16. Что «только при создании» vs «всегда».** 🟢 Эталон: **только при создании** — статус (`active`),
цена; **всегда обновляются** — название, описание, габариты/вес, характеристики, фото (с трекингом
качества). Ручные правки статуса/цены НЕ перезатирать. 🟡 Сохранять ли ручные правки категорий/фото
вебмастера — продуктовое решение (эталон перезаписывает связи на каждом импорте; если нужно сохранять —
добавить флаг). Подтвердить.

**17. Single / Pack / Virtual.** 🟢 Все товары — **простые** (`Product` без комбинаций). Вариаций/наборов
в OneCatalog нет. Никаких `Combination`/виртуальных.

**18. Площадь упаковки / ед. измерения.** 🟢 По стандарту это «сайтовый слой» — ядро их НЕ пишет.
`measurement_unit` (факт API: `{slug:"m2", name:"м2"}` — продажная единица) и `areas_per_package` →
через хук `actionOneCatalogProductImported` (или в `Product::unity`/`unit_price` если витрина их
использует — но это решение сайта, не ядра). Не путать с физическими габаритами.

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id`** (своя `oc_product_link`), `reference = article` (при непустом, без
   коллизии). Искать ВСЕГДА по public_id.
2. **Справочники — поиск по имени (lang), не по слагу** — критично: `Tools::link_rewrite()`
   транслитерирует по-своему → разные слаги для одного имени → дубли (в эталоне был баг «Толщина»
   tolschina/tolshhina). Feature по `feature_lang.name`, FeatureValue по `(id_feature, value)`, Category
   по name в пределах `id_parent`. Стабильный ключ маппинга — `specification_id`/category id.
3. **Характеристики — `Feature`, не `Attribute`** (Q2); многозначные через несколько `product_feature`
   (Q7); ручной маппинг по `specification_id`, строгий режим.
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие =
   false**; термы true/false из маппинга. Знать `multitude=false` у boolean.
5. **Медиа:** дедуп по `sha1(path#size)` (против скачивания); галерея по имени файла, не по URL; общие
   не удалять; скачивать вручную + MIME по содержимому (URL без расширения).
6. **Цена `0` при создании, не перезатирать; статус — только при создании.**
7. **Единицы:** конвертер г/мм → `PS_WEIGHT_UNIT`/`PS_DIMENSION_UNIT`; `length`→`depth`;
   `measurement_unit` (м²) — продажная, не габарит → хук.
8. **Категории — дерево** из `/categories/` (`parent_category` ЕСТЬ в справочнике, хотя у товара
   `categories[]` плоский) — стройте реальную иерархию, а не всё под одним корнем.
   **`/collections/{slug}/` 500 — не дёргать**, всё в эмбеде.
9. **Теги — нативный `Tag`** (PrestaShop имеет таксономию тегов, в отличие от Magento/OpenCart) — хорошая
   новость; по имени (lang), вкл/выкл.
10. **Прочий payload** (`measurement_unit`, `areas_per_package`, `product_shape`, alt/описание картинок,
    детали бренда, `upsells/cross_sells`) — через `actionOneCatalogProductImported` (полный payload).
11. **SemVer + CHANGELOG** (версия в `version` и `config.xml` — два места, как в эталоне 2 места);
    новые UI-строки — в переводы модуля перед сборкой zip.

## Что подтвердить у владельца PrestaShop-магазина (перед стартом)

- Версия (1.7.6+ legacy или 8.x Symfony) — Q1.
- Один shop или multistore (контекст в фоне) — Q4.
- Число языков (один → проще; N → фаза 2) и дефолтный `id_lang` — Q5.
- Поддерживает ли тема/Faceted Search **multi-value features** (для Цвет/Помещение) — Q7.
- Есть ли понятие «коллекция» на витрине (→ своя сущность или категория) — Q8.
- Есть ли внешняя интеграция (1С) на `reference` — Q3.
- Системный cron / PS Cron Tasks / Messenger-воркер — Q11.
- CSP админки разрешает внешний iframe (пикер vs ID-list) — Q14.
- Будет ли API-токен (иначе только `min`-качество) — Q15.
- Объём каталога — Q13.
