# Интеграционный план: NetCat

Реализация стандарта `INTEGRATION-STANDARD.md` (OneCatalog Import) на CMS **NetCat**
(российская PHP-CMS с модулем «Магазин» / NetShop). Эталон — WooCommerce.

Документ написан как разработчиком на NetCat. Везде, где механизм платформы не
подтверждён документацией на 100 %, стоит пометка **(не уверен по NetCat,
проверить в документации)** — выдумок нет, только то, что удалось свериться на
[netcat.ru/developers/docs](https://netcat.ru/developers/docs/) + явные допущения.

Ключевые подтверждённые факты о платформе, на которых строится план:

- Данные хранятся как **объекты** (записи) внутри **компонентов**; у каждого
  компонента своя SQL-таблица `MessageXX` (XX = ID компонента), с системными
  полями (даты, автор). Поля компонента имеют типы (строка, число, текст, **список**,
  файл, дата, связь-объект). Источник: [Данные](https://netcat.ru/developers/docs/components/data/).
- Класс объекта — `nc_Message extends nc_Essence`; чтение `nc_Message::get_by_id()`.
  Источник: [Класс nc_Message](https://netcat.ru/developers/docs/nc-core/nc-message-class/).
- Модуль «Магазин»: стандартный компонент «Товар» с полями `Name`, `ItemID`
  (артикул/SKU), `Price`, `Currency`, `PriceMinimum`, `Units` (из справочника
  `ShopUnits`), `StockUnits`, `VAT`, **`ImportSourceID`** — «служебное поле,
  используется при импорте из внешних источников» (1С и т. п.). Источник:
  [Компоненты товаров](https://netcat.ru/developers/docs/modules/module-netshop-old/products-cc/).
- Категории каталога — **разделы** (subdivisions/`Subdivision`); инфоблок товаров
  привязывается к разделу. Источник: [Прикрепление инфоблоков](https://netcat.ru/developers/docs/components/infoblocks/).
- Событийная модель: `addMessage / updateMessage / dropMessage / checkMessage /
  uncheckMessage` (и `*Subdivision`), параметры `site_id, subdivision_id,
  infoblock_id, component_id, object_id`; подписка `$nc_core->event->bind(...)` или
  `add_listener($event, $cb)` (v5.6+); свои события — `register_event()`.
  Источники: [Система событий](https://netcat.ru/developers/docs/events/),
  [Системные события](https://netcat.ru/developers/docs/events/system-events/),
  [Прикрепление](https://netcat.ru/developers/docs/events/attaching/).
- Файлы/изображения: `$nc_core->files->save_file()` /
  `$nc_core->files->field_save_file()`; класс трансформации картинок `nc_ImageTransform`.
  Источники: [save_file](https://netcat.ru/developers/docs/api/nc_core_files_save_file.html),
  [field_save_file](https://netcat.ru/developers/docs/api/nc_core_files_field_save_file.html),
  [ncImageTransform](https://netcat.ru/developers/docs/tools/ncimagetransform/).
- Cron: системный планировщик «Управление задачами» вызывает скрипты по URL
  (с `secret_key`); требует НАСТОЯЩИЙ системный cron (`netcat/admin/crontab.php`
  каждую минуту). Внутреннего тика без ОС-cron нет. Источник:
  [Управление задачами (cron)](https://netcat.ru/developers/docs/tools/cron/).

## Маппинг сущностей и реализация

### Таблица соответствий OneCatalog → NetCat

| OneCatalog | Абстракция | Реализация в NetCat |
|---|---|---|
| product | Товар | **объект** кастомного компонента «Товар OneCatalog» (на базе стандартного «Товар» NetShop), запись в `MessageXX` |
| public_id (`OC.IND.1`) | Внешний ключ (идемпотентность) | **отдельное пользовательское поле** `OneCatalogPublicID` (строка, индекс). НЕ `ImportSourceID` по умолчанию — см. вопрос 1 |
| article | Артикул производителя | стандартное поле `ItemID` (SKU) — только если `article` непустой |
| categories[] | Категория | **разделы** (`Subdivision`) каталога: find-or-create по имени, привязка объекта/инфоблока к разделу |
| options[] (text/numeric) | Характеристика | пользовательские поля компонента типа **«список»** (enum) ИЛИ единый репозиторий «характеристик» (компонент-справочник + связь). См. §«Характеристики» и вопрос 4 |
| options[] (boolean) | Булева характеристика | поле-список со значениями «Да/Нет» или bool-поле; `null/false → false` |
| brand | Бренд | по настройке: поле-список / отдельный компонент-справочник «Бренды» (объект+связь) / раздел. По умолчанию — компонент-справочник |
| country | Страна | по настройке: поле-список / компонент-справочник «Страны». По умолчанию — поле-список |
| collections[] | Коллекция | по настройке: компонент «Коллекции» (объект) / раздел / поле-список. По умолчанию — компонент-объект |
| tags[] | Теги | поле-список «Теги» (вкл/выкл), поиск по имени |
| files/images_urls | Медиа | поля-файлы компонента (обложка + множественное поле «галерея») через `$nc_core->files->*` |
| measurement_unit, sizes, areas_per_package, product_shape, … | прочее | стандартные `Units`/`StockUnits` где есть; остальное — через **своё событие** `onecatalog_product_imported` (сайтовый слой) |

### Архитектура (слои стандарта → NetCat)

NetCat — не Composer-PSR-проект «из коробки», но классы PHP подключать можно.
Предлагаю **модуль** (папка `netcat/modules/onecatalog/`) с автозагрузкой классов
по соглашению имён `nc_onecatalog_*` **(не уверен по NetCat, проверить: как именно
регистрируется автозагрузчик пользовательского модуля — см. вопрос 12)**. Слои 1:1
со стандартом:

```
nc_onecatalog (фасад, get_object(), жизненный цикл/миграции)
  ├─ nc_onecatalog_api         — клиент Wiki API (base URL, X-API-Key, lang, GET, обёртка data/success)
  ├─ nc_onecatalog_units       — конвертер г/мм → единицы магазина
  ├─ nc_onecatalog_media       — выбор размера, скачивание, MIME-by-content, обложка/галерея, трекинг качества, дедуп
  ├─ nc_onecatalog_taxonomies  — характеристики (поля-список/справочник), категории→разделы (find-or-create по имени)
  ├─ nc_onecatalog_collection  — коллекция → компонент/раздел/список
  ├─ nc_onecatalog_brand       — бренд → справочник/список/раздел
  ├─ nc_onecatalog_country     — страна → справочник/список
  ├─ nc_onecatalog_product     — оркестратор импорта одного товара
  ├─ nc_onecatalog_queue       — порционная очередь (cron-степпер/AJAX)
  ├─ nc_onecatalog_endpoint    — приём списка ID + статус (admin-action скрипт)
  └─ nc_onecatalog_settings    — опции модуля + UI настроек
admin UI: кнопка «Импорт из OneCatalog» + iframe-пикер + поллинг прогресса
```

Сетевой слой: запросы делать **через cURL напрямую** (так контролируем заголовок
`X-API-Key`, бинарную загрузку медиа и таймауты), не полагаясь на встроенные
утилиты — это переносимо и предсказуемо.

### Товар → объект каталога (CRUD)

- **Создание/обновление объекта.** В NetCat объекты обычно создаются «единым
  механизмом управления компонентами», но публичной, прямой и стабильной функции
  вида `nc_objects_add($fields)` мне в документации **подтвердить не удалось**
  (см. вопрос 2 — это потенциальный блокер качества кода). Рабочие варианты по
  убыванию надёжности:
  1. **Использовать внутренний механизм добавления объекта компонента** (тот же,
     что вызывает форма добавления) с массивом значений полей, ключи = системные
     имена полей. Документация подтверждает «единый механизм», но не даёт чистого
     API → нужно свериться по исходникам ядра нужной версии.
  2. **Прямой INSERT/UPDATE в `MessageXX`** + ручная отправка событий
     `addMessage/updateMessage` через `$nc_core->event->execute(...)`. Работает, но
     это «ниже» уровня абстракции: не получим автоматических побочных эффектов
     (индексация поиска, пересчёт связей) → хрупко при апдейтах CMS. Допустимо как
     фолбэк, но НЕ как основной путь.
- **Идемпотентность.** Перед импортом: `SELECT ID FROM MessageXX WHERE
  OneCatalogPublicID = :public_id` (поиск по СТАБИЛЬНОМУ ключу, не по slug/имени).
  Нашли → update (с учётом «поля только при создании»), нет → insert. Легаси-фолбэк:
  если в проекте `public_id` исторически клали в `ImportSourceID`/`ItemID` — искать
  и там, мигрировать в `OneCatalogPublicID`.
- **Цена / SKU / статус.** `Price` НЕ задаётся (в источнике цены нет, §5.6) — поле
  цены оставляем пустым/0, подстановка — через фильтр/событие. `ItemID` = `article`
  только если непуст. Статус (поле активности объекта) задаём только при СОЗДАНИИ,
  при переимпорте не трогаем.
- **Раздел/привязка.** Объект должен жить в разделе каталога; для много-категорийных
  товаров NetCat-объект, как правило, привязан к ОДНОМУ инфоблоку/разделу →
  «множественные категории» решаем не разделами, а дополнительным полем-связью/
  списком (см. вопрос 6).

### Характеристики (options[])

- При **выключенном** ручном маппинге: для каждой характеристики
  `find-or-create поле-список` по человекочитаемому `specification_label` (НЕ по
  слагу — §5.1, защита от дублей при разных транслитах), затем find-or-create
  значение-терм по `specification_option_name`.
  - **Подводный камень NetCat:** поля-список — это **структура компонента**, а не
    данные. Программно «на лету» создавать новое поле компонента под каждую новую
    характеристику = модифицировать схему `MessageXX` (ALTER + запись в служебных
    таблицах полей). Это технически возможно, но (а) рискованно, (б) может требовать
    очистки кэша структуры, (в) плохо переносится между версиями. См. вопрос 4 и
    раздел «Сложно/невозможно». **Рекомендация:** не плодить поля динамически, а
    хранить характеристики в **отдельном компоненте-справочнике** «Характеристики»
    (объект = пара «характеристика+значение») со связью «многие-ко-многим» к товару.
- `boolean`: значение из `bool_option`; `null/false → false`; в ручном маппинге
  задаются термы для true/false.
- При **включённом** ручном маппинге: импортируются ТОЛЬКО сопоставленные
  характеристики; ключ привязки — стабильный `specification_id` → конкретное поле/
  справочник платформы (карта в настройках).

### Бренд / страна / коллекции (настройка «тип объекта + цель»)

Стандарт требует выбор хранилища. В терминах NetCat «тип объекта» = одно из:
**поле-список** | **компонент-справочник (объект+связь)** | **раздел**. Реализуем
три импортёра с общим интерфейсом resolve(payload)→target_ref:

- **Бренд** — по умолчанию компонент-справочник «Бренды» (есть доп. поля: год
  основания, сайт, страна, логотип → их кладём при создании). find-or-create по
  `menutitle`.
- **Страна** — по умолчанию поле-список (легче). find-or-create по `menutitle`;
  `country_code` — в доп. поле, если цель = справочник.
- **Коллекции** — по умолчанию компонент-объект «Коллекции» (есть `link_3d`,
  `link_official_site`, картинки). Все поля коллекции приходят в эмбеде товара —
  отдельный запрос `/collections/{slug}/` НЕ делать (он 500, §2.1).

### Конвертер единиц

`nc_onecatalog_units`: источник отдаёт вес в **граммах**, размеры в **мм**
(подпись локализована «мм»/«г»). Магазин хранит в своих (`Units`/доп. поля). Таблица
коэффициентов к базовой единице + нормализация локализованных подписей; единицу
источника брать из `sizes.*_unit`, иначе мм/г; переопределение — фильтром. Без этого
1200 мм → 1200 см.

### Точки расширения (события NetCat)

Регистрируем СВОИ события через `register_event()` и эмитим `execute()`:

- `onecatalog_before_import_product(payload, existing_id)`
- `onecatalog_product_imported(object_id, payload, report)` ← **сайт дозаполняет**
  ед. изм., площадь упаковки, `product_shape`, alt картинок, детали бренда, upsells —
  всё, чего нет в ядре (полный payload в событии)
- `onecatalog_collection_imported / brand_imported / country_imported`
- `onecatalog_queue_enqueued`

«Фильтры» (меняющие значение) в NetCat нативно — это та же шина событий, но НетКат
events возвращают значения **(не уверен по NetCat, проверить: умеет ли
`event->execute` собирать модифицированное значение от обработчиков, как WP
`apply_filters`)** — см. вопрос 9. Если нет — фолбэк: собственный реестр коллбэков
`add_filter()/apply_filter()` внутри модуля.

## Фоновая очередь, медиа, настройки, i18n, жизненный цикл

### Фоновая очередь

- Список public_id режется на порции ≥10 («шаг импорта»). Состояние очереди —
  отдельная служебная таблица модуля (`nc_onecatalog_queue`: batch_id, ids json,
  status, log) **(не уверен по NetCat: есть ли удобный API создания таблиц модуля
  при установке — вероятно ручной CREATE в install-скрипте; вопрос 13)**.
- Обработчик — скрипт `netcat/modules/onecatalog/cron.php?secret_key=…`,
  регистрируемый в «Управлении задачами» (минимум 1 мин). На каждый тик берёт одну
  pending-порцию, импортирует элементы, пишет лог.
- **Важное ограничение:** у NetCat нет внутреннего «тика» — встроенный планировщик
  работает ТОЛЬКО если хостинг реально дёргает `crontab.php` системным cron. Если
  cron недоступен → нужен **AJAX-степпер** (как нативный импорт): админ-страница в
  цикле шлёт запросы «обработай следующую порцию» и поллит прогресс. Реализуем оба
  пути (cron + синхронный/AJAX-фолбэк) — это и есть «синхронный фолбэк» стандарта.
- UI ставит в очередь и опрашивает статус (pending/running + последние N записей лога).

### Медиа

- URL `…/media_files/<base64hash>` **без расширения** → стандартные NetCat-загрузчики
  по URL могут не определить тип. **Скачиваем cURL во временный файл, MIME — по
  содержимому** (`finfo`), затем `$nc_core->files->save_file()` /
  `field_save_file()` для записи в поле-файл объекта.
- Размер: с токеном — `max`; без — `min`/превью.
- **Трекинг качества:** рядом с файлом хранить выбранный размер (доп. поле/мета
  вложения). При переимпорте: лучше доступного → перекачать и заменить; не лучше —
  не трогать.
- **Галерея** идемпотентна по ИМЕНИ файла (подписи в URL меняются).
- **Дедуп по контент-ключу:** из base64-префикса достать `path`, ключ =
  `hash(path#size)`, хранить на вложении, перед загрузкой искать существующее →
  переиспользовать без скачивания. **Общие файлы НЕ удалять** при апгрейде качества.
  - ⚠️ Дедуп требует «реестра вложений с произвольной метой». В NetCat файл обычно
    привязан к полю объекта, отдельной «таблицы медиатеки с произвольными метаданными»
    как в WP нет → **завести свою таблицу** `nc_onecatalog_media (content_key, size,
    file_path/ID)` для дедупа и трекинга качества (см. вопрос 7).

### Настройки

Минимум из §7 стандарта: токен (env приоритетнее поля), язык (`en/ru/ar/zh/kk`),
шаг (≥10), статус новых (только при создании), вкл/выкл коллекций/брендов/страны/
тегов, «тип объекта + цель» для коллекции/бренда/страны, ручной маппинг
характеристик (+ true/false для boolean). Хранить — в опциях модуля
**(не уверен по NetCat: точный механизм — аналог `COption`/таблица настроек модуля;
вероятно своя таблица `nc_onecatalog_settings`; вопрос 11)**. Валидация и кламп.

### i18n

- Исходные строки — английские, единый «домен». NetCat использует **языковые файлы**
  (`lang/` модуля) — не gettext .po/.mo. Перенос концепции стандарта: PHP-массивы
  переводов `lang/english.php`, `lang/russian.php`. Строки фронтовых JS — отдавать из
  бэкенда локализованными, не хардкодить. **(не уверен по NetCat: точная структура
  языковых файлов модуля — проверить; вопрос 14)**.

### Жизненный цикл

- Манифест/заголовок модуля с версией; **SemVer + CHANGELOG** (Keep a Changelog),
  бэклог = «Не выпущено», поднимать версию перед сборкой (как требует MEMORY проекта
  для onecatalog-import).
- Установка: создать кастомный компонент «Товар OneCatalog» (или дать инструкцию/
  мастер привязки к существующему), служебные таблицы (queue, media, settings),
  зарегистрировать cron-задачу, события.
- Миграция опций/полей при смене схемы (перенос `ImportSourceID`/`ItemID` →
  `OneCatalogPublicID`, флаг «миграция выполнена»).
- Активация НЕ должна падать без модуля «Магазин»: функции импорта проверяют его
  наличие.
- **Права:** импорт/настройки — только админ/разработчик; cron-скрипт защищён
  `secret_key`; admin-action — проверка сессии/прав NetCat + CSRF-токен
  **(не уверен по NetCat: штатный механизм CSRF-токена в админке — проверить; вопрос 10)**.

## ❓ Вопросы и сомнения (ответить ДО старта)

1. **public_id: `ImportSourceID` или отдельное поле?** `ImportSourceID` —
   штатное «служебное поле для импорта», но его может занимать импорт из 1С (конфликт
   двух источников). Решение: класть `public_id` в собственное поле
   `OneCatalogPublicID`. Подтвердить, что это приемлемо и не ломает штатные сценарии
   синхронизации.
2. **Чистый API создания/обновления объекта.** Есть ли в целевой версии NetCat
   стабильная функция/метод добавления объекта компонента с массивом полей (аналог
   `nc_objects_add`)? Если нет — придётся либо вызывать внутренний механизм формы,
   либо INSERT в `MessageXX` + ручные события. От ответа зависит надёжность всего
   импортёра. **(не уверен по NetCat — критично проверить по исходникам версии.)**
3. **Версия NetCat и редакция.** Какая версия (5.x?) и редакция (E-commerce/NetShop

## 🚫 Сложно / невозможно на NetCat

1. **Динамическое создание полей характеристик (схема на лету).** В WooCommerce
   глобальные атрибуты `pa_*` и термины создаются «налету» дешево. В NetCat «поле
   компонента» — это изменение СТРУКТУРЫ компонента (правка `MessageXX` + служебных
   таблиц полей + сброс кэша структуры). Делать это на каждый новый `specification`
   при импорте — рискованно и плохо переносимо между версиями.
   **Обходной путь:** не плодить поля, а хранить характеристики в отдельном
   **компоненте-справочнике** «Характеристики» (объект «характеристика→значение») со
   связью к товару; ручной маппинг по `specification_id` — на существующие поля.

2. **«Медиатека с произвольными метаданными» (дедуп + трекинг качества).** В WP
   вложение — самостоятельный пост с произвольной метой (`content_key`, `size`). В
   NetCat файл привязан к полю объекта; единого реестра вложений с мета-полями нет.
   **Обходной путь:** собственная таблица `nc_onecatalog_media (content_key, size,
   file_ref, owner)`; дедуп/апгрейд качества — через неё; общие файлы не удалять.

3. **Несколько категорий на один товар.** Модель «объект ↔ раздел/инфоблок» тяготеет
   к одной категории. **Обходной путь:** основной раздел + доп. поле-связь «доп.
   категории», или таблица связи «товар↔раздел»; навигацию витрины строить с её учётом.

4. **Нет встроенного «тика» очереди.** Action Scheduler (WP) самозапускается на
   трафике; в NetCat фон зависит от РЕАЛЬНОГО системного cron (`crontab.php`). На
   хостинге без cron автоматического фона не будет.
   **Обходной путь:** AJAX-степпер из админки (браузер дёргает обработку порций +
   поллит) как полноценный синхронный фолбэк; плюс инструкция по настройке cron.

5. **«Фильтры» (изменяющие значение) как в WP.** Шина событий NetCat — про реакцию;
   возврат модифицированного значения из `execute()` не гарантирован
   **(проверить — вопрос 9)**. **Обходной путь:** собственный лёгкий реестр
   `add_filter/apply_filter` внутри модуля для точек «payload/SKU/цена/размеры/origin».

6. **Чистый высокоуровневый CRUD объектов под вопросом.** Отсутствие подтверждённого
   публичного `nc_objects_add/update` означает риск опуститься до прямых SQL +
   ручных событий, теряя побочные эффекты ядра (поиск, связи). **Обходной путь:**
   приоритетно использовать внутренний механизм добавления объекта компонента;
   прямой SQL — только осознанный фолбэк с ручной эмиссией `addMessage/updateMessage`.

7. **gettext .po/.mo не нативен.** Стандарт описывает POT/PO/MO; NetCat — языковые
   PHP-файлы. **Обходной путь:** держать переводы в `lang/`-файлах модуля; концепцию
   «исходник EN + перевод RU» сохранить, формат адаптировать.

8. **iframe сторонних доменов в админке.** Возможны CSP/политики панели против
   стороннего iframe/`postMessage` (вопрос 15). **Обходной путь:** если пикер
   заблокирован — импорт по списку public_id (поле ввода/вставка из пикера в
   отдельной вкладке), стандарт это допускает как альтернативу.
