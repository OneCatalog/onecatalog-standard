# Ответы BigCommerce-разработчику — OneCatalog Import (App)

Ответы на [bigcommerce.md](bigcommerce.md) (17 вопросов + 10 блокеров). Маркировка:
🟢 — авторитетно (проверено на живом Wiki API / поведении эталона WooCommerce `onecatalog-import` v1.7.0);
🟡 — продуктовое решение/зависит от магазина и плана, рекомендация + что подтвердить.

> ⚠️ Эталонная витрина `ceram-online` — headless WordPress+Nuxt, не BigCommerce. Вопросы про «этот
> магазин/приложение» (план, Multi-Storefront, scopes, CSP) — про новое приложение; по ним рекомендация +
> «подтвердить». API/логику стандарта отвечаю точно.

> 👏 Отчёт верный, в т.ч. вывод «BigCommerce — самая каталожно-родная SaaS-цель»: нативные **Brands**,
> **дерево Categories** (`parent_id` ложится на `/categories/` `parent_category` 1:1) и **габариты+вес в
> ядре товара** — там, где Shopify требует костылей. Ниже подтверждаю и закрываю часть открытых пунктов
> фактами данных/эталона.

## Главное про данные

🟢 **Вариаций товара НЕТ вообще** — один `public_id` = один готовый товар. Variant Options/Modifiers под
характеристики не нужны (вы это верно зафиксировали).

🟢 **Характеристики многозначны** — у товара «Цвет» = 2 значения, «Помещение» = **6**. Справочник
`/specifications/` отдаёт флаг **`multitude`** (true = многозначная; boolean `false`). На BigCommerce
многозначную кладите как **несколько записей Custom Field с одинаковым `name`** (массив `custom_fields`
имена не уникализирует) — Product Filters это поддерживают. Используйте `multitude`.

🟢 **`numeric` в этом инстансе не используется** (все 13 характеристик text/boolean).

🟢 **Лимит 200 Custom Fields/товар не страшен** — характеристик всего 13; даже с многозначными
(Помещение=6 записей) выйдет ~20–30 полей на товар. Запас огромный (Q17).

## Ответы на вопросы

**1. Цена обязательна (блокер).** 🟢 В источнике цены НЕТ → **`price = 0` при создании, не перезатирать при
апдейте** (эталон: цена только при создании). 🟡 Чтобы «нулёвки» не висели — рекомендую при создании
`is_visible = false` (скрытый/черновик) до простановки цены сайтом; реальную цену даёт сайт через фильтр-
источник (аналог `filterOneCatalogProductPrice`). Это вынужденное отклонение от §5.6 «не задавать» = «0 и
не трогать», не «отсутствует». Кросс-платформенно (Magento/PrestaShop/Shopify/Drupal). **Подтвердить:
публиковать сразу или держать скрытыми до цены.**

**2. `weight` обязателен для physical.** 🟢 **Важно: в источнике `weight` часто `null`** (у OC.IND.1
`sizes.weight = null`, `weight_unit = null` — вес есть не у всех товаров). → если веса нет, ставить `0`
(или из события сайта). Габариты (`length/width/height`) у OC.IND.1 есть (1200/600/85 мм). 🟡 Если
0-вес ломает расчёт доставки — `is_visible=false` до заполнения, как с ценой.

**3. БД приложения обязательна.** 🟢 **Подтверждаю** — `/catalog/products` НЕ фильтруется по Custom Field/
Metafield, поиск по `public_id` через API за один запрос невозможен. БД-карта `public_id → bc_product_id`
— единственный быстрый `find_by_external_id`. Custom Field/Metafield на товаре — зеркало для реконсиляции
(полный постраничный проход `?include=custom_fields`). Это жёстче Shopify — закладывайте persistent БД.

**4. Уникальность `name`/`sku`.** 🟡 BigCommerce требует уникальное `name` (и `sku`, если задан). 🟢
`menutitle` в OneCatalog теоретически может совпасть у двух товаров → при коллизии имени добавлять суффикс
(`article`/`public_id`) или лог+отказ. `sku = article` (при непустом); при коллизии sku — **не
перезаписывать чужой**, создать без sku + лог (§5.2). Идемпотентность ВСЕГДА по `public_id` (БД-карта), не
по name/sku. **Подтвердить политику коллизии имён.**

**5. Custom Field vs Metafield для ключей.** 🟢 Рекомендация (как у вас): **технический ключ `oc_public_id`
и якоря характеристик (`spec_<specification_id>`) → Metafields** (есть namespace/key/permission, скрыты от
витрины); человекочитаемые характеристики → **Custom Fields** (видны на странице, идут в Product Filters).
Согласовать.

**6. Теги: целевое хранилище.** 🟡 Нативных тегов нет. Теги по умолчанию ВЫКЛ. При включении —
**рекомендую Custom Field `oc_tags`** (можно сделать фасетом), `search_keywords` — если нужен только поиск,
Category — если теги навигационные (тяжело). Эталон кладёт теги в нативную таксономию WC; на BigCommerce
прямого аналога нет.

**7. Маппинг осей габаритов.** 🟢 Факт API: `sizes` = `length=1200, width=600, height=85` (мм; height =
толщина плитки). 🟡 BigCommerce — `width/height/depth` (для габаритов отгрузки объём L×W×H не зависит от
имён осей). Рекомендация-конвенция: **`length → depth`, `width → width`, `height → height`** (как у
PrestaShop). Уточнить с владельцем, важна ли семантика осей для отображения; для доставки — не критично.

**8. Категории и Multi-Storefront / trees.** 🟡 Дефолт — одно дерево/канал. 🟢 Иерархия из `/categories/`
(`parent_category`) ложится на `parent_id` **1:1** — создавать родителей раньше детей (топологическая
сортировка ветки), find-or-create по имени в пределах `parent_id`, карту `category id → bc_category_id` в
БД. 🟡 Мультивитринность — настройка `tree_id`/channel. **Подтвердить одно/мультидерево.**

**9. REST V3 vs GraphQL.** 🟡 Рекомендация (как у вас): **запись — REST V3** (полнее по images/custom
fields), **чтение/поиск — GraphQL** (батч, меньше round-trip → экономия квоты). Согласен.

**10. Rate limits / план.** 🟡 Квота `X-Rate-Limit-*` зависит от плана (Standard/Plus/Pro/Enterprise) →
**подтвердить план**. Проактивный адаптивный throttle по `Requests-Left`/`Time-Reset-Ms`, backoff по
`Retry-After`, конкурентность 1–2 на `store_hash`. 🟢 Со стороны Wiki API жёсткого лимита в тестах не
наблюдалось; справочники — `limit=1000`. Главный потолок — квота BigCommerce.

**11. Product Filtering (фасеты).** 🟡 Доступность storefront-фильтров по Custom Fields зависит от плана/
темы (Stencil). **Подтвердить, нужна ли фасетка** по характеристикам и поддерживает ли её план/тема.

**12. Картинки: `image_url` vs multipart.** 🟢 **Хорошая новость: `image_url` работает и для max-качества.**
Бинарный URL `…/media_files/<base64hash>` **самодостаточен** — токен зашит в base64-хеш (`size|path|token`),
и сама загрузка файла идёт **без заголовка `X-API-Key`** (в эталоне `Media::sideload` качает его обычным
GET без auth и получает max при наличии токена). То есть **BigCommerce сервер-сайд скачает URL сам** для
того качества, что URL кодирует — multipart НЕ нужен. (Токен `X-API-Key` нужен только для JSON-вызовов
каталога `/products/`, чтобы они ВЕРНУЛИ max-URL; сам бинарь по URL открыт.) → используйте `image_url`,
multipart — только фолбэк, если BigCommerce не достучится до URL.

**13. Мультиязычность.** 🟡 Фаза 1 — **один язык** (язык магазина по умолчанию; API отдаёт один `lang` за
запрос). Multi-Storefront + Translations API на доп. локали — отдельный проход, плановозависимо. Эталон
тоже один язык за прогон. **Подтвердить целевые локали.**

**14. Picker iframe-в-iframe.** 🟡 Приложение само — iframe в админке BigCommerce; вложить ещё
`tools.onecatalog.net` — может упереться в CSP. **Фолбэк — отдельное окно (`window.open` + postMessage
через opener) или импорт по списку `public_id`** (стандарт §2.4 допускает). Рекомендую сначала ID-list.

**15. Идемпотентность под-сущностей.** 🟢 Подтверждаю — карты `public_id+specification_id → custom_field_id`
и `public_id → [image_id+content_key+size]` вести в БД приложения (BigCommerce стабильных внешних ключей
под-сущностям не даёт). Технический якорь характеристики — в Metafield (namespace/key).

**16. Scopes.** 🟡 Минимум: Catalog read/write (товары/категории/бренды/изображения/custom fields), Store
information read (единицы), Webhooks. Точные имена — в Dev Portal под текущую версию API. Least-privilege.

**17. Объём метаданных vs лимиты.** 🟢 Уложимся с запасом — 13 характеристик, даже с многозначными ~20–30
Custom Fields, лимит 200 (см. «главное про данные»).

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id`** (БД-карта + Metafield-зеркало), `sku = article` (уникален, как Woo);
   искать ВСЕГДА по public_id, не по name/sku.
2. **Справочники — поиск по имени (label):** категория по `name` в пределах `parent_id`; бренд по `name`
   (`/catalog/brands?name=`). Стабильный ключ маппинга — `specification_id`/category id/brand id; человеко-
   читаемый label НЕ ключ (он переводимый). Якорь характеристики — Metafield `spec_<id>`.
3. **Характеристики:** Custom Fields (+ Metafield-якорь); многозначные (`multitude`) → несколько записей с
   одним `name`; ручной маппинг по `specification_id`, строгий режим. Variant Options/Modifiers НЕ
   использовать.
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие =
   false**; значения true/false из маппинга. (`multitude=false` у boolean.)
5. **Медиа:** `image_url` (работает и для max — Q12); `is_thumbnail=true` для обложки; контент-ключ
   `sha1(path#size)` и трекинг качества — в БД приложения; галерея по контент-ключу/имени, не по URL.
6. **Цена `0` при создании, не перезатирать**; `weight=0` если в источнике `null`; статус (`is_visible`) —
   только при создании.
7. **Единицы:** конвертер г/мм → единицы магазина (`weight_units`/`dimension_units` из Store settings);
   `measurement_unit` (м²/шт) — продажная единица → Custom Field/событие, не габарит.
8. **Категории — дерево** из `/categories/` (`parent_category`) → `parent_id` 1:1, родители раньше детей.
   **Бренд — нативный `/catalog/brands`** по имени. **`/collections/{slug}/` 500 — не дёргать**, всё в
   эмбеде `collections[]`. **`/specifications/`, `/categories/` — `limit=1000`.**
9. **Прочий payload** (`measurement_unit`, `areas_per_package`, `product_shape`, alt/описание картинок,
   детали бренда `description_text/foundation_year/website` → Brand custom_fields, `upsells/cross_sells`)
   — через событие приложения `product_imported` (полный payload).
10. **SemVer + CHANGELOG** на стороне приложения + **зафиксировать версию/дату API** (BigCommerce
    эволюционирует — плановое обновление).
