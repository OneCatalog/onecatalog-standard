# Интеграционный план: RetailCRM

> Перенос стандарта `INTEGRATION-STANDARD.md` (импорт каталога из OneCatalog Wiki API)
> на **RetailCRM** — российскую CRM/систему обработки заказов и товарного каталога
> (omnichannel, заказы/клиенты, товарный каталог, ICML-импорт, REST API v5,
> модули-интеграции маркетплейса, MG). Эталон стандарта — WooCommerce.
>
> **Принципиальная рамка (читать до всего остального).** RetailCRM — это **CRM/каталог
> для обработки заказов и маркетинга**, а НЕ витринная CMS со своим storefront.
> У неё нет фронтенда-витрины, taxonomy-страниц, тем, Gutenberg, SEO-страниц
> коллекций/брендов, фасетного поиска для покупателя. Каталог в RetailCRM — это
> **товары (products) и торговые предложения (offers)**, **товарные группы
> (product-groups / категории)** и **свойства (properties)**. Свойства каталога
> приходят в RetailCRM преимущественно из ICML-фида и/или через `store/*` API;
> они НЕ равны кастомным полям заказа/клиента (`customFields`) — это разные миры
> (см. §«Сложно/невозможно», п.1 — самая частая ошибка интеграторов).
> Поэтому значительная часть инвариантов стандарта, завязанных на витрину
> (страницы коллекций/брендов, фасеты, alt-тексты картинок, SEO, статусы
> публикации draft/pending), на стороне RetailCRM **просто не живут**: их место —
> внешний сайт/CMS, который синхронизируется с RetailCRM. RetailCRM в этой связке —
> «бэкенд заказов», витрина — отдельно.

## Что такое целевая поставка (два пути)

Стандарт описывает «модуль импорта в товарную систему платформы». На RetailCRM
есть **два принципиально разных способа** наполнить каталог, и выбрать надо
осознанно:

- **Путь A — генератор ICML-фида (декларативный).** Мы поднимаем отдельный сервис,
  который тянет OneCatalog Wiki API и отдаёт **один XML-файл формата ICML**
  (`*.xml`) по публичному URL. В админке RetailCRM («Администрирование →
  Магазины → Каталог товаров») этот URL прописывается как источник, и **RetailCRM
  сам по своему расписанию скачивает и парсит фид** (проверено: ICML — штатный
  формат загрузки каталога RetailCRM, импорт по URL с периодичностью на стороне
  RetailCRM). Идемпотентность — встроенная: товар/offer матчится по `externalId` в
  фиде; повторная загрузка обновляет, а не дублирует (проверено: ICML использует
  атрибуты `id`/`externalId`/`productId` как ключи).

- **Путь B — императивный пуш через REST API v5 (`store/*`).** Наш сервис сам
  создаёт/обновляет товарные группы и товары через эндпоинты каталога:
  `store/product-groups/create`, `store/products/create`/`/edit`, `store/products`
  (выборка) и т.п. (не уверен в точном наборе/именах методов конкретной версии —
  проверить по API v5 раздел «Справочник товаров / store»: исторически создание
  товаров через API было ограничено и часть каталога заводилась именно ICML-фидом;
  набор `store/products/*` методов записи зависит от версии — **проверить**).

**Основной путь — A (ICML-фид).** Обоснование:

1. ICML — это «родной» и наиболее полный канал загрузки каталога в RetailCRM:
   через него штатно заводятся товарные группы, товары, торговые предложения,
   картинки (по URL — RetailCRM сам качает), свойства (properties), цены, остатки
   (проверено: состав ICML — `categories`, `offers` с `<picture>`, `<param>`,
   `<price>`, `<vendor>` и т.п.).
2. Идемпотентность, дедуп, частичное обновление и расписание — на стороне
   RetailCRM, нам не надо строить очередь/трекинг (см. §«Фоновая очередь»).
3. Декларативность совпадает с философией стандарта «источник истины внешний,
   локально — проекция»: фид и есть проекция.

Путь B держим как **дополнение** для того, что в ICML выразить нельзя или неудобно
(точечное обновление одного offer, чтение существующего каталога для сверки,
программное создание групп) — но не как основной механизм наполнения.

**Целевые версии:** REST API **v5** (актуальная линейка RetailCRM/Simla.com,
проверено — текущая публичная версия API). Формат каталога — **ICML** (RetailCRM
Markup Language, проверено). Аутентификация интеграции — **apiKey** (X-API-Key/
`apiKey`) и/или **OAuth-модуль маркетплейса** для публичной интеграции (см.
§«Жизненный цикл»).

### Таблица соответствий: OneCatalog → абстракция → RetailCRM

### product → product/offer (§3, §5.1)

В модели RetailCRM «товар» (`product`) — это карточка, а покупается/заказывается
**торговое предложение (`offer`)**; у простого товара одно предложение (проверено:
заказы ссылаются на offers, не на абстрактный товар). У OneCatalog **нет вариаций**
(`options[]` — это характеристики-фасеты, а не SKU/варианты), поэтому маппим
**один товар → один offer**. В ICML это `<offer>` внутри `<offers>`; в API —
`product` с единственным offer.

⚠️ Важно: то, что в WooCommerce «один продукт без вариаций», в RetailCRM всё равно
порождает пару product↔offer. Заказы и аналитика RetailCRM работают на уровне
offer — это нормально и ожидаемо.

### public_id → externalId (центральный вопрос идемпотентности, §5.1)

- `public_id` (`OC.IND.1`) пишем в **`externalId`** товара и/или offer. Это
  **штатный внешний ключ RetailCRM** для матчинга при импорте: повторная загрузка
  с тем же `externalId` обновляет существующую запись (проверено: `externalId` —
  сквозной идентификатор связи внешней системы с RetailCRM во всём API и ICML).
- В ICML offer задаётся `<offer id="OC.IND.1" productId="OC.IND.1">` (точные имена
  атрибутов — `id` обязателен; есть ли требование на пару product↔offer
  externalId — **проверить** по спецификации ICML конкретной версии). RetailCRM
  матчит по этим ключам сам — **очередь find-or-create нам не нужна** (в отличие от
  императивных платформ стандарта).
- ⚠️ **`externalId` должен быть уникален и стабилен.** `public_id` это
  гарантирует (бизнес-ключ источника). НЕ использовать для ключа `slug`/название
  (нестабильны) — полностью совпадает с §5.1 стандарта.
- Уникальность `externalId` на стороне RetailCRM: **(не уверен, проверить)** —
  обеспечивает ли платформа жёсткую уникальность externalId товара/offer на уровне
  БД, или матчинг «по соглашению при импорте». При ICML коллизий не будет, если
  ключи в фиде уникальны (наша забота — не выдать два offer с одним id).
- Легаси-фолбэк (§5.1): при первой загрузке прежнего ключа нет — это новый
  каталог, миграция не требуется. Если позже схема ключа поменяется — фолбэк через
  поиск `store/products?filter[externalId]` (имя фильтра **проверить**).

### article → артикул offer (§5.2)

- `public_id` ≠ `article`. `article` (артикул производителя, часто пуст) кладём в
  **штатное поле артикула offer** (`article`/`vendorCode` — точное имя в ICML/API
  **проверить**). `externalId` — отдельно (см. выше). Полное соответствие §5.2.
- Если `article` пуст — поле не заполняем (не синтезировать). Артикул в RetailCRM
  не обязан быть уникален — но по стандарту чужой не перезаписываем (наш генератор
  фида и так пишет по одному товару за раз).

### options[] → свойства каталога (НЕ customFields заказа!) (§5.4)

Это самый тонкий пункт переноса. Характеристики товара в RetailCRM выражаются как
**`<param>` каталога** (свойства торгового предложения), которые становятся
**property** товарной группы/каталога (проверено: ICML `<param name code>value`
формирует свойства каталога, по ним строятся фильтры/выгрузки внутри RetailCRM).

- Для каждой `options[]`: `<param name="specification_label" code="oc_spec_<id>">
  specification_option_name</param>`. **`code` — стабильный ключ** (из
  `specification_id`), `name` — человекочитаемая подпись (label). Это покрывает
  инвариант §5.1 «матчить по стабильному ключу, не по транслиту слага».
- **boolean-характеристики (§2.2, §5.6):** `specification_option_name = null`,
  значение в `bool_option`. В RetailCRM нет «булева свойства» — кладём текстовое
  значение терма: `bool_option === true` → «Да», иначе (null/false/нет) → «Нет»
  (§5.6: false по умолчанию). Подписи Да/Нет — из настройки маппинга (локализуемо).
  Без этого правила boolean молча теряется (как и предупреждает стандарт).
- **numeric-характеристики:** `numeric_option` → значение `<param>` числом
  (с единицей или без — по настройке). RetailCRM хранит свойства как строки/числа
  по типу свойства — **проверить**, типизируются ли params автоматически.
- **Ручной маппинг (строгий режим, §5.4):** при ВКЛ генерируем `<param>` только
  для сопоставленных `specification_id`; ключ привязки — `code=oc_spec_<id>`.
  При ВЫКЛ — все характеристики по имени/коду.

🚫 **Категорически НЕ путать с `customFields`.** В RetailCRM `customFields` и
`customDictionary` — это **кастомные поля и словари для заказов, клиентов,
компаний** (поля формы заказа), а НЕ свойства товара (проверено: customFields
относятся к сущностям order/customer/company, конфигурируются в «Администрирование →
Системные настройки → Дополнительные поля»). Товарные характеристики OneCatalog
**должны идти в свойства каталога (`<param>`/properties)**, не в customFields
заказа. Это разные подсистемы; смешение — классическая ошибка (см. §«Сложно», п.1).

### categories[] → product-groups (дерево) (§5.4)

- ICML: блок `<categories>` с `<category id="…" parentId="…">menutitle</category>`,
  у offer — `<categoryId>…</categoryId>` (один или несколько). RetailCRM строит
  **товарные группы (product-groups)** из этих категорий (проверено: ICML
  `<categories>` → дерево групп каталога).
- Идемпотентность групп — по `id` категории (стабильный из OneCatalog) как
  externalId группы; повторная загрузка обновляет (RetailCRM сам). Фолбэк-поиск по
  имени (label) — как в §5.1 — при ICML не нужен (ключ — id).
- **Иерархия:** payload `categories[]` в товаре — плоский (id, menutitle, slug)
  **без явного parent** (см. §2.2). Чтобы построить дерево, нужен отдельный вызов
  `/categories/` Wiki API и его поле родителя — **разведка API обязательна**
  (шаг 0 стандарта). Если родитель не приходит — кладём плоско под корнем
  (`parentId` пуст).
- Через API (путь B): `store/product-groups/create`/`edit` (имена методов
  **проверить** по API v5).

### brand → vendor / property (§3, §7)

- RetailCRM имеет **нативное поле производителя `<vendor>`** в ICML offer
  (проверено: `<vendor>` — стандартный элемент). Это ближайший аналог «нативной
  таксономии брендов» из дефолтов стандарта. **Основной вариант: brand → `<vendor>`.**
- Доп. поля бренда (страна, год основания, сайт, лого из `brand.images_*`) в каталог
  RetailCRM не ложатся (нет сущности «карточка бренда») → это **витрина/событие**
  на стороне сайта. Честно: «страница бренда» в RetailCRM не живёт.
- Альтернатива (настройка «тип объекта + цель» из §3/§7): brand → `<param "Бренд">`
  (свойство), если `vendor` занят/не подходит. Реализуем выбор, дефолт — `vendor`.

### country → property (§3, §7)

- **Нет нативного поля «страна происхождения товара»** в каталоге RetailCRM
  (страна есть у клиента/адреса доставки — это другое!). → `<param "Страна">`
  (`country.menutitle`). Дефолт — выключено (как в §7). Витринная «страница страны»
  не живёт.

### collections[] → property / группа (§3, §7)

- Нет нативной сущности «коллекция». Два варианта (настройка «тип объекта + цель»):
  - `<param "Коллекция">` (свойство) — простой путь, дефолт.
  - дополнительная **категория-группа** «Коллекции/<имя>» (коллекция как товарная
    группа) — если нужна навигация по коллекциям внутри RetailCRM.
- Доп. поля коллекции (`link_3d`, `link_official_site`, лого) — в каталог RetailCRM
  не ложатся → витрина/событие. Поля берём из эмбеда `collections[]` товара
  (стандарт §2.1: `/collections/{slug}/` отдаёт 500 — НЕ дёргать).
- **Витринные страницы коллекций/брендов с фасетами и SEO в RetailCRM
  отсутствуют** — это функция витрины-сайта, не CRM (см. §«Сложно», п.3).

### tags[] → property (§7)

- Нет нативных «тегов товара» в каталоге (теги в RetailCRM — у заказов/клиентов,
  это другое). → множественный `<param "Теги">` (вкл/выкл, дефолт выкл). Матчинг по
  `title` (label). Семантика «фасет-тег витрины» в RetailCRM не воспроизводится.

### media → картинки offer по URL (§2.3, §5.3)

- ICML: `<picture>https://api.onecatalog.net/wiki/media_files/<hash></picture>`
  (несколько `<picture>` на offer — обложка первой). **RetailCRM сам скачивает
  картинки по URL** при импорте фида (проверено: ICML `<picture>` — RetailCRM
  загружает изображение по ссылке). Это снимает с нас бОльшую часть §2.3/§5.3.
- ⚠️ **URL без расширения** (§2.3) — может оказаться проблемой: загрузчик RetailCRM
  способен не определить тип файла по URL без расширения (как WP `media_sideload`).
  **(не уверен, проверить)** — глотает ли RetailCRM `media_files/<base64hash>` без
  расширения, или нужен прокси-эндпоинт нашего сервиса, отдающий картинку с
  корректным `Content-Type` и «псевдо-расширением» в пути. **Скорее всего нужен
  наш прокси** (наш сервис качает из OneCatalog с токеном → отдаёт RetailCRM по
  чистому URL с расширением). Это снимает и проблему токена/качества (см. ниже).
- **Качество/токен (§2.3):** размеры `min|middle|max`; `max` только с токеном.
  Наш сервис (генератор фида) держит токен и подставляет URL нужного размера —
  RetailCRM токена OneCatalog не знает. Если фид отдаёт прямые URL OneCatalog без
  токена, RetailCRM получит только `min`. → ещё один аргумент за **прокси
  картинок** в нашем сервисе (качаем `max` с токеном → отдаём RetailCRM).
- **Трекинг качества/дедуп по контент-ключу (§5.3):** на стороне RetailCRM этим
  управлять нельзя (импорт картинок — чёрный ящик платформы). Дедуп/выбор качества
  делает **наш сервис** до отдачи фида (один URL на один контент-ключ
  `hash(path#size)`; в фид всегда лучший доступный размер). Галерея
  идемпотентна на стороне RetailCRM по набору `<picture>` в offer (перезагрузка
  фида перезаписывает список картинок offer — **проверить** поведение «удаляет ли
  RetailCRM картинки, исчезнувшие из фида»).

### sizes/weight → вес/габариты offer; единицы (§5.6) — Units

- OneCatalog отдаёт **вес в граммах, размеры в миллиметрах** (подписи
  локализованы; ед. из `sizes.*_unit`, иначе мм/г) — §5.6.
- **Вес:** RetailCRM offer имеет поле веса — **(не уверен в единице, проверить)**:
  в кг или граммах. Конвертер `Units` обязателен (нормализация подписи →
  коэффициент → единица RetailCRM). В ICML — `<weight>` (поддержка/единица
  версии — **проверить**).
- **Габариты:** поддержка length/width/height в каталоге RetailCRM/ICML —
  **(не уверен, проверить)**. Если нативных полей нет — габариты идут в `<param>`
  (как свойства), без участия в логистических расчётах. Это деградация, но не блокер.
- **measurement_unit (м²/шт):** RetailCRM имеет справочник **единиц измерения**
  (`unit` у offer) — **(проверить имя/механизм)**. Маппим `measurement_unit.slug`
  на единицу RetailCRM (find-or-create/справочник). Это **продажная** единица —
  не путать с единицей габаритов (физической). Полностью созвучно предупреждению
  §5.6 «не путать ед. измерения товара с ед. габаритов».

### Цена (§5.6) — но RetailCRM-каталог обычно требует цену

В источнике **цены нет** (§5.6: не синтезировать). Но **в ICML/каталоге RetailCRM
цена у offer фактически обязательна** — это система обработки **заказов**, offer без
цены не оформляется в заказ и может быть невалиден при импорте фида **(проверить,
обязателен ли `<price>` в ICML конкретной версии — скорее да)**. Это прямой
конфликт стандарта с природой RetailCRM (см. §«Сложно», п.2). Варианты:

1. `<price>0</price>` — формально валидно, но 0 «течёт» в заказы (плохо).
2. Цену подставляет **наш сервис/событие** из СВОЕГО источника (прайс) — но это
   нарушает §5.6 «не выдумывать». Делаем только если у заказчика есть реальный прайс.
3. Цена приходит в RetailCRM **из другого источника** (1С/сайт), а наш фид несёт
   только описательную часть — но ICML загружает offer целиком, частичный «без
   цены» фид проблематичен.

Решение — **вопрос к заказчику (см. §Вопросы, п.5)**: нужен ли вообще каталог для
**заказов** (тогда цена обязательна и нужен её источник) или только справочный
каталог/обогащение карточек на витрине (тогда RetailCRM, возможно, вообще не
целевая система для этих товаров).

## Фоновая очередь, медиа, настройки, i18n, жизненный цикл

### Как RetailCRM импортирует каталог: расписание на их стороне (§6)

- **Путь A (ICML):** очередь/фоновая обработка стандарта (§6) нам **не нужна** в
  том виде, как для императивных платформ. RetailCRM сам по своему расписанию
  скачивает фид по URL, парсит, идемпотентно обновляет каталог (проверено:
  периодический импорт каталога по URL — штатная функция). У клиента **серверного
  cron для нашего кода нет** — но он и не требуется: «cron» живёт на стороне
  RetailCRM. Наш сервис лишь **держит URL фида актуальным** (генерирует/кэширует
  XML). Генерация фида (тянуть OneCatalog постранично, §2.1, с высоким `limit` для
  справочников) — это **наша фоновая задача в нашем сервисе** (наш cron/worker), а
  не в RetailCRM.
- **Частота:** периодичность импорта фида настраивается в RetailCRM **(проверить
  гранулярность — раз в N часов/сутки)**. Для «по кнопке здесь и сейчас» ICML не
  предназначен — это батч по расписанию. Если нужен мгновенный точечный импорт —
  путь B (API).
- **Путь B (API-пуш):** если выбираем императив, очередь стандарта (§6) реализуем
  **в нашем сервисе** (любая job-очередь: Redis/DB + worker; порции ≥10; статус/
  лог; синхронный фолбэк) — RetailCRM здесь приёмник, а оркестрация на нас.
  ⚠️ **rate-limit API v5** существует (проверено: у RetailCRM есть ограничение
  частоты запросов к API; **конкретный лимит/окно — проверить** по версии) — порции
  и паузы обязательны, иначе 503/429.

### Где хранить настройки/токен (§7)

- **Наш сервис** хранит: токен OneCatalog (`X-API-Key`), язык (`lang`), шаг/частоту
  генерации фида, маппинг характеристик, флаги (бренд/страна/коллекции/теги), URL
  RetailCRM + его apiKey (для пути B), правила цены. Хранилище — БД/конфиг нашего
  сервиса (не в RetailCRM).
- **Со стороны RetailCRM** при оформлении как модуль маркетплейса настройки модуля
  отдаются через **manifest/настроечную форму интеграции** (OAuth/apiKey, URL
  нашего сервиса) — **(механика конфигурации модуля — проверить по docs
  «Интеграционные модули»)**. Токен OneCatalog в RetailCRM не светим — он наш.
- Приоритет env над полем токена (§7) — соблюдаем в нашем сервисе.

### Картинки (§2.3) — RetailCRM качает по URL, но см. прокси

См. блок media выше: RetailCRM сам качает `<picture>` по URL; из-за отсутствия
расширения и токена OneCatalog **рекомендуется прокси картинок в нашем сервисе**
(качаем `max` с токеном → раздаём RetailCRM чистым URL с расширением). Трекинг
качества/дедуп (§5.3) — целиком на нашей стороне (RetailCRM этим не управляет).

### Мультиязычность (§9)

- Wiki API отдаёт контент по `lang` (`ru/en/ar/zh/kk`). RetailCRM **не является
  мультиязычной витриной** — каталог в одной языковой проекции. → генерируем фид
  на **одном языке** (для ceram — `ru`). Многоязычный каталог в одной системе
  RetailCRM штатно не предусмотрен **(проверить, есть ли вообще локали каталога —
  скорее нет)**. Несколько языков = несколько фидов/инстансов — отдельная задача.
- i18n самого модуля/сервиса (строки UI настроек) — по стандарту EN-исходник +
  переводы; но это UI **нашего сервиса**, не RetailCRM. POT/PO/MO (WP-механизм) —
  неприменимы; используем механизм нашего стека.

### Жизненный цикл модуля-интеграции (§10)

- **Если поставка — публичный модуль маркетплейса RetailCRM:** нужен **manifest
  интеграции** (название, описание, иконка, список нужных прав/scope), регистрация
  в **маркетплейсе RetailCRM**, и аутентификация через **OAuth-поток установки
  модуля** или **apiKey** (проверено: модули-интеграции RetailCRM регистрируются в
  маркетплейсе и используют apiKey/токен для доступа к API; точная схема
  manifest/registration — **проверить по docs «Создание интеграционного модуля»**).
- **Если поставка — приватный сервис «только для ceram»:** маркетплейс не нужен;
  достаточно apiKey конкретного аккаунта RetailCRM + публичный URL ICML-фида,
  прописанный в настройках магазина. **Это, скорее всего, наш случай** (один сайт,
  не тиражируемый продукт).
- **SemVer + CHANGELOG (§10, и по практике проекта):** ведём для нашего сервиса
  (а не для «плагина RetailCRM», которого как файла-плагина нет). Версия в манифесте
  модуля маркетплейса (если он есть) + CHANGELOG (Keep a Changelog), бэклог = «Не
  выпущено», поднимать версию перед релизом.
- **Активация без падений (§10):** наш сервис при недоступности OneCatalog или
  RetailCRM деградирует (§5.5): отдаёт прошлый валидный фид/логирует, не роняет
  весь импорт из-за одного битого товара.

## 🚫 Сложно / невозможно на RetailCRM

1. **Свойства товара ≠ кастомные поля заказа (`customFields`/`customDictionary`).**
   В стандарте «характеристика» — единая абстракция (атрибут+термины). В RetailCRM
   **товарные свойства** (`<param>`/properties каталога) и **customFields**
   (доп. поля order/customer/company) — **разные подсистемы**. Характеристики
   OneCatalog обязаны идти в свойства каталога, НЕ в customFields. *Обход:*
   жёсткое правило в маппинге; ревью, чтобы никто не «удобно» сложил характеристики
   в customFields заказа. Самая частая и дорогая ошибка интегратора.

2. **Цены нет в источнике, но offer в RetailCRM обычно требует цену.** §5.6 «не
   синтезировать», но RetailCRM — система **заказов**: offer без валидной цены
   неполноценен/может не импортироваться. *Обход:* источник цены (1С/прайс) или
   `<price>0</price>` (с риском «нулей» в заказах), либо признать, что для заказов
   эти товары RetailCRM не обрабатывает. Прямой конфликт инварианта с природой
   платформы — решать с заказчиком (см. Вопросы 5–6).

3. **Нет витрины: страниц коллекций/брендов, фасетов, SEO в RetailCRM не
   существует.** Инварианты стандарта про «коллекция/бренд как
   страница/таксономия/CPT с полями, alt картинок, фасеты» в RetailCRM **не живут** —
   это функция витрины-сайта, не CRM. *Обход:* в RetailCRM brand→`<vendor>`,
   collection/country/tags→`<param>` (просто значения свойств для фильтров заказов/
   аналитики). Витринные страницы/SEO/фасеты — на стороне сайта-витрины,
   синхронизируемого с RetailCRM. Честно: бОльшая часть «витринного» слоя стандарта
   на RetailCRM не реализуется и не должна.

4. **Нет управляемой клиентом фоновой очереди/cron внутри RetailCRM (для нашего
   кода).** §6 (очередь, шаг, статус, прогресс) на императивных платформах — наш
   код. В RetailCRM импорт ICML — «чёрный ящик по расписанию платформы»; своего
   серверного cron у клиента нет. *Обход:* оркестрация (генерация фида, пагинация
   OneCatalog, трекинг качества/дедуп) живёт в **нашем сервисе** с собственным
   worker'ом; «прогресс импорта» в RetailCRM мы не контролируем (видим лог импорта
   фида в админке RetailCRM, но не управляем порциями). Для пути B очередь —
   полностью на нас + rate-limit API.

5. **Трекинг качества картинок и дедуп по контент-ключу (§5.3) — не на стороне
   RetailCRM.** Импорт картинок RetailCRM — чёрный ящик (качает по URL, своя
   логика). *Обход:* выбор качества (`max` с токеном), дедуп `hash(path#size)`,
   «лучшее доступное» — делает **наш сервис** до отдачи фида; вероятен **прокси
   картинок** (URL без расширения + токен `max`). RetailCRM получает уже «правильный»
   URL.

6. **Идемпотентность держится на `externalId` и дисциплине ключей в фиде.**
   В WP — уникальная мета. В RetailCRM при ICML уникальность — «по соглашению»
   (наш фид не должен выдавать дубль `externalId`); жёсткость на уровне БД —
   **(проверить)**. *Обход:* единственный источник `externalId` = `public_id`,
   валидация уникальности при генерации фида, никаких ключей по slug/имени (§5.1).

7. **Ограничения ICML как формата.** Декларативный батч: нет точечного «обновить
   один offer сейчас», частота — на стороне RetailCRM; набор полей ограничен
   спецификацией ICML версии (габариты/булевы свойства/множественные единицы могут
   не иметь нативных полей → деградируют в `<param>`). *Обход:* для точечных
   операций — путь B (REST API v5, `store/*`, **набор методов проверить**); описательная
   часть — ICML. Гибрид A+B, но это два канала записи в один каталог — следить за
   тем, чтобы не конфликтовали по `externalId`.

8. **Статусы публикации (publish/pending/draft/private) и «поля только при
   создании» (§5.6) плохо ложатся.** В RetailCRM у offer нет 4 витринных статусов;
   есть активность/доступность offer **(проверить модель)**. ICML — декларативный
   «полный слепок», поэтому «не перезатирать ручные правки вебмастера при
   переимпорте» (§5.6) на ICML **трудно**: повторная загрузка фида перезаписывает
   поля offer значениями фида. *Обход:* поля, которые правят вручную в RetailCRM, в
   фид не включать (тогда RetailCRM их не трогает — **проверить, действительно ли
   отсутствие поля в фиде = «не трогать», а не «очистить»**); либо путь B с
   точечным edit только нужных полей. Полной семантики «только при создании» ICML не
   даёт.

## Резюме для исполнителя

- **Основной путь — генератор ICML-фида** (декларативно): наш сервис тянет
  OneCatalog Wiki API и отдаёт ICML по URL, RetailCRM сам качает/парсит/обновляет по
  расписанию. Идемпотентность, дедуп товаров, расписание и загрузка картинок — на
  стороне RetailCRM. REST API v5 (`store/*`) — дополнение для точечных операций
  (набор методов записи **проверить по версии**).
- **Ключ идемпотентности:** `public_id` → **`externalId`** товара/offer. Бренд →
  нативный `<vendor>`; категории → product-groups (`<category parentId>`);
  характеристики → **`<param>`/properties каталога** (НЕ customFields заказа!);
  страна/коллекция/теги → `<param>` (нативных сущностей нет). Медиа → `<picture>`
  по URL (вероятен прокси из-за URL без расширения + токена `max`). Вес/габариты →
  поля offer/`<param>` через конвертер г/мм → единицы RetailCRM.
- **Главная рамка:** RetailCRM — каталог **для заказов/маркетинга**, не витрина.
  Витринные инварианты стандарта (страницы коллекций/брендов, фасеты, SEO, alt,
  статусы публикации) на RetailCRM **не живут** — это сайт-витрина, синхронизируемый
  с CRM.
- **Самые острые вопросы:** (1) ICML vs API как основной; (2) ЦЕНА — её нет в
  источнике, а RetailCRM для заказов её требует (источник цены?); (3)
  характеристики строго в свойства каталога, не в customFields заказа; (4) прокси
  картинок (URL без расширения + токен `max`) — нужен ли.
- **Главные «невозможно/сложно»:** (1) свойства товара ≠ customFields заказа —
  не перепутать; (2) цена обязательна для offer, но в источнике её нет; (3) нет
  витрины — страницы коллекций/брендов/фасеты/SEO отсутствуют как класс; (4)
  трекинг качества/дедуп картинок и «поля только при создании» (§5.3, §5.6) на
  декларативном ICML на стороне RetailCRM не контролируются — переносятся в наш
  сервис, либо деградируют.
