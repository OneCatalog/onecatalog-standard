🟢 — авторитетно (проверено на живом Wiki API / решение эталона WooCommerce `onecatalog-import` v1.7.0);
🟡 — зависит от редакции/версии/схемы PHPShop или продуктовое решение; рекомендация + что подтвердить (фаза 0).

## Главное про данные (влияет на формат `vendor_array` и категории)

🟢 **Вариаций товара НЕТ вообще** — один `public_id` = один готовый товар. Характеристики — факты/фасеты,
не «торговые предложения». Никаких вариантов в `vendor_array`.

🟢 **Характеристики многозначны** — у товара «Цвет» = 2 значения, «Помещение» = **6**. Справочник
`/specifications/` отдаёт флаг **`multitude`** (true = многозначная; boolean `false`). **Это прямо влияет
на ваш Q2/B1:** при реверс-инжиниринге формата `vendor_array` берите дамп товара, у которого есть
**многозначная** характеристика (не только одиночные), иначе не увидите, как платформа кодирует
МАССИВ значений одной характеристики. Используйте `multitude` для кардинальности.

🟢 **`numeric` в этом инстансе не используется** (все 13 характеристик text/boolean) — но код пишите
универсально (numeric → строка-значение + единица).

🟢 **Иерархия категорий ЕСТЬ** — у товара `categories[]` плоский, но `/categories/` отдаёт
`parent_category` (или `null` = корень). Влияет на B7/Q6.

🟢 **Всего ~13 характеристик** — формат `vendor_array` не разрастётся; даже с многозначными немного.

## Ответы на вопросы

**1. Точная схема БД товара.** 🟡 Платформенное — **фаза 0** (`DESCRIBE` живой инсталляции). Контракт со
стороны данных: должны лечь `name` ← `menutitle`; `uid` ← `article` (только непустой); цена — **не
задаётся** (см. Q ниже про цену); вес/габариты — через конвертер; `public_id` — в свою таблицу-карту, не
в поле товара. Имена таблиц брать через `$GLOBALS['SysValue']['base'][...]`, не хардкодом — согласен.

**2. Формат `vendor_array`.** 🟡 Реверс-инжиниринг обязателен (доки схему не дают). 🟢 **Важная вводная:
снимайте дамп с товара, имеющего многозначную характеристику** (см. «главное про данные» — «Помещение» с
несколькими значениями), чтобы поймать кодирование массива значений. Безопасный приём (ваш) верный:
создать товар с 2–3 характеристиками (одна — многозначная, одна — boolean) через админку, сдампить
`vendor_array`, скопировать формат. Ключ привязки в маппинге — стабильный `specification_id`, не
переводимый label.

**3. `sort_categories` и наборы.** 🟡 Платформенное — фаза 0 (связь набор↔раздел↔товар, можно ли создать
характеристику программно так, чтобы она показывалась в фильтрах). Контракт: справочник характеристик/
значения **предсоздавать ДО прогона** (на admin-странице маппинга или пред-сканом `/specifications/`),
очередь только присваивает значения — схему/наборы в фоне не плодить (как у Magento/UMI). **Подтвердить,
требуется ли «общий» набор, не привязанный к разделу.**

**4. Тип «Бренд».** 🟡 Платформенное (как хранится `type`, логотип; программное создание/привязка). 🟢
Согласен — **нативный «Бренд» (сквозной фильтр) — лучший таргет**, лучше чем у OpenCart/Magento.
find-or-create по имени `brand.menutitle`; логотип (`brand.images_urls`) — как медиа с дедупом; доп. поля
(`description_text/website/foundation_year/country`) — через хук, ядро их не пишет. **Подтвердить
программное API брендов.**

**5. `insert()` → last insert id.** 🟡 Платформенное — фаза 0. Контракт §5.4: id товара нужен ДО привязок
(характеристики/медиа/карта). Фолбэк (ваш) верный — `mysqli_insert_id()` через соединение ядра или
повторный `getOne` по уникальному полю.

**6. Множественные категории + иерархия.** 🟢 `categories[]` действительно **массив**, и **иерархия ЕСТЬ**
(`/categories/` `parent_category`) → стройте дерево разделов (родители раньше детей), find-or-create по
**имени в пределах родителя**. 🟡 Если движок держит товар в ОДНОМ разделе (поле «Размещение») — основной
раздел = первая/листовая категория, прочие через «Виртуальный каталог»/доп. привязку (если поддерживается)
или лог-скип. Эталон строит плоско — PHPShop с деревом разделов может **точнее**. **Подтвердить
множественную привязку.**

**7. Картинки товара.** 🟡 Платформенное (каталог файлов, поля `pic_small`, вызов ресайзера) — фаза 0.
🟢 Полезный факт: бинарный `media_files`-URL **самодостаточен** — токен зашит в base64-хеш, файл качается
**без заголовка `X-API-Key`** (в эталоне `Media::sideload` берёт его обычным GET и получает max при
наличии токена). URL **без расширения** → скачивать вручную (cURL) + **MIME по содержимому** (`finfo`),
расширение присваивать самим. `X-API-Key` нужен только JSON-вызовам `/products/`, чтобы они вернули
max-URL.

**8. Cron/фон.** 🟡 Платформенное (есть ли штатный планировщик). 🟢 Рекомендация: **AJAX-степпер —
основной** (детерминирован, совпадает с пошаговым CommerceML-обменом); cron/системный crontab — вторичный.
Эталонный Action Scheduler тоже визит/cron-зависим — не минус PHPShop. Порция ≥10; битый `public_id` →
лог, не прерывать пакет (§5.5).

**9. Редакция/версия.** 🟡 Платформенное — наборы характеристик и `PHPShopOrm` есть не во всех редакциях
(нужен Pro/Enterprise 5.x–6.x). **Подтвердить редакцию/версию.** Активация не должна падать без движка
магазина (проверять классы/таблицы).

**10. Сосуществование с CommerceML/1С.** 🟡 Реальный риск (B6/B10). Рекомендация: `public_id` — строго в
**своей таблице** `oc_product_map`, НЕ в `uid` (артикул бизнес-уникален, его может занять 1С как ключ
обмена); добавить опцию **«не трогать товары с 1С-GUID»**; правила §5.6 «поля только при создании не
перезаписывать» снижают конфликт. **Подтвердить, активен ли обмен и какое поле он использует ключом.**

**11. Module API / админ-UI.** 🟡 Платформенное, публично слабо документировано (ваш B11 — главный риск
сроков) — фаза 0 (снять рабочий пример штатного модуля). Фолбэк (ваш) разумный: точка входа
`phpshop/core/onecatalog.class.php` (`PHPShopCore`→`/onecatalog/`) + идемпотентный `ensure schema`
(`CREATE TABLE IF NOT EXISTS`) вместо install-хука.

**12. i18n.** 🟡 Платформенное — у PHPShop нет gettext/.po. Ваш фолбэк (собственный словарь
`lang/<code>.php` + хелпер `t()`, строки фронт-JS из PHP) — правильный и переносимый. Концепцию стандарта
(исходник EN + перевод RU) сохраняете, формат адаптируете — это нормально.

**13. Права/сессия/CSRF.** 🟡 Платформенное — взять из ядра (как штатные админ-модули проверяют вход) +
CSRF-токен на запись (аналог nonce эталона), cron-скрипт за `secret_key`. **Подтвердить механизм.**

**14. Единицы магазина.** 🟢 Источник: **вес в граммах, размеры в миллиметрах** (подпись локализована
«мм»/«г»). **`sizes.weight` часто `null`** (у OC.IND.1 вес отсутствует) — не выдумывать (0/пусто). 🟡 В
каких единицах PHPShop хранит вес/габариты — платформенное, **подтвердить**. Конвертер г/мм → единицы
магазина обязателен (иначе 1200 мм → 1200 см); единицу источника брать из `sizes.*_unit`, иначе мм/г.

**15. Статусы товара.** 🟡 Платформенное (B9) — у PHPShop меньше статусов, чем publish/pending/draft/
private у Woo. Маппинг «статуса новых» на реальные флаги (опубликован/скрыт/под заказ), задаётся **только
при создании**, при переимпорте не трогать. **Подтвердить набор статусов.**

> **Про цену (вы зафиксировали в B-таблице и §5.6, отдельным вопросом не вынесли):** 🟢 в источнике цены
> НЕТ — это факт API, не упущение. Импорт `price..price5` **не задаёт** (§5.6, не синтезировать). Если
> валидация PHPShop требует цену — создавать товар скрытым до простановки; цену подставляет сайт через
> хук/фильтр `onecatalog_product_price`. Не перезаписывать при переимпорте.

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id`** (своя таблица `oc_product_map`, не `uid`/`vendor_array`-скан),
   `article → uid` отдельно (только непустой; коллизия — не перезаписывать чужой). Искать ВСЕГДА по
   public_id, не по slug/имени.
2. **Справочники — поиск по имени (label), не по слагу** — критично: автотранслит даёт разные слаги для
   одного имени → дубли (в эталоне был баг «Толщина» tolschina/tolshhina). Категория по `name` (в пределах
   родителя), характеристика/набор по имени в `sort_categories`. Стабильный ключ маппинга —
   `specification_id`/category id/brand id, не переводимый label.
3. **Характеристики:** предсоздание справочника/наборов вне очереди (Q3); многозначные (`multitude`) →
   массив значений в `vendor_array`; ручной маппинг по `specification_id`, строгий режим (только
   сопоставленные).
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие =
   false**; термы true/false из маппинга. (`multitude=false` у boolean.)
5. **Медиа:** ручное скачивание + **MIME по содержимому** (URL без расширения); дедуп `sha1(path#size)`
   (своя `oc_media_map`); обложка + галерея; галерея идемпотентна по имени файла, не по URL; трекинг
   качества (`min|middle|max`) в `oc_product_map.quality`; общие файлы при апгрейде НЕ удалять; прогон
   через штатный ресайзер. URL самодостаточен (токен в хеше).
6. **Не выдумывать цену** (`price..price5` не задавать, скрытый до простановки если валидация требует);
   статус — только при создании, не перезатирать.
7. **Единицы:** конвертер г/мм → единицы магазина (настройкой); `measurement_unit` (м²/шт) — продажная
   единица, не габарит → хук; `weight` бывает `null`.
8. **Категории — дерево** из `/categories/` (`parent_category`). **`/collections/{slug}/` отдаёт 500 —
   не дёргать**, всё из эмбеда `collections[]` товара. **`/specifications/`, `/categories/` — одним
   запросом `limit=1000` + сверка `meta.counts`** (дефолтный лимит инстанса молча обрезает).
9. **Прочий payload** (`measurement_unit`, `areas_per_package`, `product_shape`, alt/описание картинок,
   детали бренда, `upsells/cross_sells`) — через хук/реестр `product_imported` (полный payload).
10. **SemVer + CHANGELOG** (Keep a Changelog), бэклог = «Не выпущено»; версию поднимать во всех местах
    манифеста перед сборкой; новые UI-строки — в `lang/`-файлы модуля.
