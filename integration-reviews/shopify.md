# Интеграционный план: Shopify (App)

> Реализация стандарта `INTEGRATION-STANDARD.md` (OneCatalog Wiki API) на платформе **Shopify**.
>
> **Принципиальное отличие от эталона (WooCommerce):** на Shopify это **НЕ плагин внутри CMS**, а
> **внешнее приложение** (public или custom App), которое работает с магазином через **Admin API**
> (GraphQL — приоритетно; REST — legacy/частично) по OAuth. У приложения **свой сервер, своя БД,
> своя очередь, свой cron**. Внутрь рантайма магазина инжектить код нельзя; «хуки/события» стандарта
> §8 реализуются исключительно **на стороне приложения** (in-process событийная шина приложения),
> а вовне магазина — через **Webhooks** Shopify (только предопределённые темы, не произвольный
> `do_action`). Доступа к ФС магазина нет — медиа только через **Files API / staged uploads**.
>
> Версия стандарта, на которую опирается план: документ от 2026 (разделы §1–§12).
> API Shopify: ориентир на стабильную версию Admin API `2025-01`+ (точную версию зафиксировать, §жизненный цикл).

## Маппинг сущностей и реализация

Сводная таблица «OneCatalog → абстракция → Shopify», затем детализация по каждой строке.

| OneCatalog | Абстракция | Shopify (механизм) |
|---|---|---|
| `product` | Товар | `Product` (GraphQL `productCreate`/`productUpdate`/`productSet`) |
| `public_id` (`OC.IND.1`) | Внешний ключ (идемпотентность) | **Metafield** `oc.public_id` (namespace `oc`, type `single_line_text_field`) + поиск через `products(query:"metafield:oc.public_id=...")` |
| `article` | Артикул производителя | `ProductVariant.sku` (только если непустой) |
| `options[]` (specifications) | Характеристики | **Metafields / Metaobjects** товара — **НЕ product options, НЕ варианты** |
| `categories[]` | Категория | `Collection` (manual) + опц. `Product.category` (Shopify Standard Product Taxonomy) |
| `brand` | Бренд | `Product.vendor` (строка) + детали бренда → **metaobject** `brand` (ссылка metafield) |
| `country` | Страна происхождения | **Metafield** `oc.country` (`single_line_text_field` или ссылка на metaobject); НЕ путать с `CountryHarmonizedSystemCode`/таможней |
| `collections[]` | Коллекция | `Collection` (manual) + поля коллекции → metafields коллекции / metaobject |
| `tags[]` | Теги | `Product.tags` (массив строк) |
| `images_urls` / `files[]` | Медиа | **Files API + staged uploads** → `productCreateMedia` (обложка = первый media; галерея = остальные) |
| `sizes.weight` + `weight_unit` | Вес | `ProductVariant`: `inventoryItem.measurement.weight { value, unit }` (вес — у варианта!) |
| `sizes.{length,width,height}` | Габариты | **Metafields** (`dimension` type) — **в ядре Shopify размеров товара НЕТ** |
| `measurement_unit` (м²/шт) | Единица измерения | **Metafield** + опц. `unitPriceMeasurement` (если магазин использует unit pricing) |
| `product_shape`, `areas_per_package`, alt картинок, детали бренда, upsells/cross_sells | Прочее | Metafields/metaobjects + событие приложения `product_imported` (см. ниже) |

### 1. Товар → `Product`

- Создание/обновление через GraphQL: `productCreate` / `productUpdate`, либо **`productSet`** (декларативный
  upsert — удобен для идемпотентности, может принять полный желаемый стейт товара с вариантами).
- Базовые поля стандарта (§5.4): `title` ← `menutitle`; `status` ← настройка «статус новых» (`ACTIVE`/`DRAFT`,
  **только при создании**, §5.6); `descriptionHtml` ← `description_text` (часто null → не трогать); `vendor` ← бренд.
- Маппинг `status` стандарта (publish/pending/draft/private) на Shopify (`ACTIVE`/`DRAFT`/`ARCHIVED`): см. вопросы.
- `productSet` поддерживает идемпотентный апдейт по идентификатору, но «идентификатор» у Shopify — это его GID.
  Связь `public_id ↔ GID` приложение держит **в своей БД** (см. идемпотентность).

### 2. `public_id` → metafield + идемпотентность (§5.1, §5.2)

- `public_id` (`OC.IND.1`) — стабильный бизнес-ключ. Хранить в **metafield** товара:
  `namespace="oc"`, `key="public_id"`, `type="single_line_text_field"`.
- **Поиск перед импортом (find-or-update):** `products(first:1, query:"metafield:oc.public_id=OC.IND.1")`.
  Shopify поддерживает поиск товаров по metafield в `query` — это и есть `find_by_external_id`.
  Дополнительно дублировать связь `public_id → product GID` в **БД приложения** (быстрее, надёжнее, не зависит
  от индексации поиска, которая у Shopify асинхронная и может отставать на секунды).
- **Никогда** не искать по `title`/`handle` (нестабильны), §5.1.
- **SKU-политика (§5.2):** `ProductVariant.sku` = `article` (только если непустой). `public_id` в SKU **НЕ** класть.
  При коллизии sku с другим товаром — Shopify **не требует** уникальности sku (в отличие от Woo), но для гигиены
  логировать коллизию и не перезаписывать (см. вопросы про дубль-sku).
- Metafield для поиска желательно объявить как **`pinned` definition** и (на новых версиях) включить ему
  индекс/`access`/`capabilities` — без определения metafield поиск по нему всё равно работает, но определение
  даёт типизацию и стабильность.
- Легаси-фолбэк (§5.1): на Shopify «старого способа» нет (приложение новое) — пункт неактуален, но саму схему
  «metafield + БД-зеркало» зафиксировать как единственный ключ, чтобы будущая миграция имела фолбэк.

### 3. Характеристики `options[]` → Metafields/Metaobjects — **НЕ варианты!** (критично)

Это центральная архитектурная точка расхождения с WooCommerce и со стандартом §3.

- В стандарте характеристика = «глобальный атрибут `pa_*` + термины» (Woo) или «свойство-список» (Bitrix).
  На Shopify **произвольных таксономий/атрибутов товара НЕ существует**. Есть только:
  - **product options** (максимум **3** опции, максимум **100/2048** вариантов на товар) — это для
    создания **вариантов** (Размер/Цвет), их НЕЛЬЗЯ использовать как «характеристики-факты», иначе плодятся варианты;
  - **metafields** товара (типизированные поля);
  - **metaobjects** (кастомные структурированные записи + ссылки).
- **Решение:** каждую характеристику OneCatalog класть в **metafield** товара, тип по `specification_type`:
  - `text` → `single_line_text_field` (или `list.single_line_text_field` если несколько значений);
  - `numeric` → `number_decimal` / `number_integer` (+ при наличии единицы → `dimension`/`weight`/`volume`);
  - `boolean` → `boolean` (§5.6: null/false → `false`).
- **Ключ metafield = стабильный `specification_id`**, а не переводимый label (иначе при смене языка ключи «поедут»).
  Пример: `namespace="oc_spec"`, `key="spec_<specification_id>"`. Человекочитаемое имя (`specification_label`)
  — в **name** соответствующего **metafield definition** (создаётся один раз, find-or-create по `specification_id`).
- **Справочные характеристики (большой словарь значений) → Metaobjects.** Если нужно, чтобы значения
  характеристики были общими сущностями (фильтрация/переиспользование/локализация значений), завести **metaobject
  definition** на характеристику, значения = записи metaobject, а metafield товара = **metafield reference**
  (`metaobject_reference` / `list.metaobject_reference`). Это ближайший аналог «атрибут + термины».
  Find-or-create записи metaobject — **по `specification_option_id`** (стабильный ключ значения), а не по имени.
- **Ручной маппинг (§5.4, строгий режим):** UI приложения «характеристика OneCatalog (`specification_id`) →
  целевой metafield definition / metaobject definition магазина». При ВКЛ импортируются только сопоставленные;
  для `boolean` — задаются значения для true/false (если целевой тип не `boolean`, а, скажем, ссылка/строка).
  При ВЫКЛ — авто: find-or-create definition по `specification_id`, тип по `specification_type`.
- **Фасетная фильтрация в витрине** работает на Shopify через **Search & Discovery** app (фильтры по metafields
  с включённым `capabilities`/storefront-доступом) — то есть «характеристики-фасеты» достижимы, но это **не**
  таксономия, а metafield-фильтры. Это надо включить и для каждого definition разрешить storefront-доступ.

### 4. Категории `categories[]` → Collections (+ Standard Taxonomy)

- На Shopify нет древовидной «product_cat» как сущности товара. Категории мапим на **Collections** (manual collections):
  `collectionCreate` (find-or-create **по metafield коллекции** `oc.category_id`/`oc.public_id`, не по title/handle),
  затем привязка товара `collectionAddProducts`.
- **Иерархия категорий не нативна:** Shopify-коллекции **плоские** (нет parent/child). Дерево OneCatalog придётся
  либо «расплющить», либо эмулировать через metafield `parent` + навигацию темы (см. «невозможно»).
- Дополнительно (опц.) — проставлять `Product.category` из **Shopify Standard Product Taxonomy** (фиксированный
  глобальный справочник Shopify). Это **не** наши категории — это нормализованная категория Shopify; автоматический
  маппинг «категория OneCatalog → taxonomy node» нетривиален (нужна таблица соответствий, см. вопросы).
- Smart collections (по правилам) можно использовать как альтернативу: создать smart collection по правилу
  «metafield/тег = категория», тогда привязка товаров автоматическая. Развилку «manual vs smart» вынести в настройку.

### 5. Бренд `brand` → `vendor` + metaobject

- Простой путь: `Product.vendor = brand.menutitle` (строка). Этого достаточно для «бренд как фасет» (по vendor
  можно фильтровать и строить smart collection).
- Богатый путь (детали бренда: `description_text`, `foundation_year`, `website`, `country`, логотип): завести
  **metaobject definition `brand`**, find-or-create запись по стабильному ключу (`brand.id`/`slug`), товар ссылается
  через **metafield reference** `oc.brand` (`metaobject_reference`). Логотип бренда (`images_urls`) — Files API,
  дедуп по контент-ключу (§5.3).
- Развилка стандарта «атрибут / таксономия / CPT» на Shopify превращается в развилку **`vendor` (строка) /
  metaobject (структура)** — таксономии и CPT нет. Это и есть настройка «тип объекта + цель» в Shopify-редакции.

### 6. Страна `country` → metafield (+ опц. metaobject)

- Страна происхождения → **metafield** `oc.country` (`single_line_text_field`), либо ссылка на metaobject `country`
  (если нужны `country_code`, локализация названия).
- ⚠️ **Не путать** с `inventoryItem.countryCodeOfOrigin` / `harmonizedSystemCode` — это **таможенные** поля Shopify
  (для международной доставки), формат — ISO-страна/HS-код, и они про логистику, а не про «страну-производителя
  как характеристику витрины». Если бизнес хочет именно таможенную страну — можно дополнительно мапить туда
  `country.country_code` (ISO-2), но это отдельное решение (см. вопросы).

### 7. Коллекции `collections[]` → Shopify Collections

- Прямое соответствие: OneCatalog «коллекция» → Shopify **Collection** (manual). Find-or-create по metafield
  `oc.collection_id`. Поля коллекции (`link_3d`, `link_official_site`, изображения) → **metafields коллекции**
  и/или изображение коллекции (`Collection.image` через Files API).
- Поскольку у нас и категории, и коллекции мапятся в один тип `Collection`, **разнести их по namespace/префиксу
  handle** и/или по metafield `oc.kind = category|collection`, чтобы не путались и чтобы тема могла их различать.

### 8. Теги `tags[]` → `Product.tags`

- Прямое соответствие: `tags[].title` → элементы массива `Product.tags` (до 250 тегов на товар, каждый ≤255 симв.).
- Включается настройкой (§7). Идемпотентность тегов тривиальна (это строки в массиве; перезаписывать набор
  аккуратно — не затирать теги, проставленные вручную: лучше **merge**, через `tagsAdd`/`tagsRemove`, а не полную
  замену; либо хранить «наши» теги отдельным префиксом).

### 9. Медиа → Files API / staged uploads (§5.3) — детально в следующем разделе

### 10. Вес и размеры — ключевая особенность Shopify

- **Вес — у варианта**, не у товара. Шор записывается в `inventoryItem.measurement.weight { value, unit }`
  (GraphQL; единицы `GRAMS|KILOGRAMS|OUNCES|POUNDS`). OneCatalog отдаёт **граммы** → можно писать `unit: GRAMS`
  напрямую (без потери точности) ИЛИ конвертировать в единицу магазина (см. конвертер). Подпись единицы из
  payload (`sizes.weight_unit`) локализована («г») — нормализовать (§5.6).
- **Размеров товара (length/width/height) в ядре Shopify НЕТ.** Их кладём в **metafields** товара
  (тип `dimension` — value+unit, либо просто `number_decimal` + отдельный metafield единицы). OneCatalog отдаёт
  **миллиметры** → конвертер «мм → нужная единица» (см. §5.6). Без размеров не работает расчёт габаритного
  объёма/доставки по размеру (Shopify считает доставку по весу/кол-ву, не по габаритам из коробки).

## Фоновая очередь (rate limits), медиа (Files API), настройки, i18n, жизненный цикл

### Фоновая очередь и rate limits (§6)

На Shopify **нет серверного cron у магазина**. Фон — целиком на стороне приложения:

- **Очередь приложения:** Redis/DB-очередь (BullMQ / Sidekiq / SQS / Cloud Tasks — что в стеке приложения).
  Список `productPublicIds` режется на порции по «шагу импорта» (минимум 10, §6). Каждый товар — отдельная job
  (или порция-job). UI приложения (embedded, App Bridge) **ставит в очередь** и **поллит статус** (можно через
  свой REST/GraphQL приложения; внутри iframe App Bridge poll/long-poll или websockets приложения).
- **Cron приложения** (не магазина): для регулярной ресинхронизации/доимпорта — планировщик на стороне приложения
  (system cron / Cloud Scheduler / Temporal). Webhooks Shopify (`products/update`, `app/uninstalled` и т.п.)
  обрабатываются отдельным эндпоинтом приложения, **верифицируя HMAC**.
- **Rate limits — обязательный инвариант (аналог §5.5 «деградация без падений»):**
  - **GraphQL Admin API — стоимостная модель (calculated query cost):** bucket по умолчанию **1000 points**,
    восстановление **50 points/сек** (для стандартных планов; Plus выше). Каждый ответ возвращает
    `extensions.cost { throttleStatus { currentlyAvailable, maximumAvailable, restoreRate } }`. Воркер обязан:
    читать `throttleStatus`, при недостатке очков **ждать** (token-bucket на стороне клиента), а на `THROTTLED`/429
    — **экспоненциальный backoff + повтор** (идемпотентность позволяет безопасно ретраить).
  - **REST Admin API (если где-то используется) — leaky bucket** (40 запросов, +2/сек на стандартном плане),
    заголовок `X-Shopify-Shop-Api-Call-Limit`. Лучше держаться GraphQL.
  - **Bulk operations:** для массового **чтения** каталога/справочников (когда тянем большие списки из Shopify)
    использовать **`bulkOperationRunQuery`** (асинхронный экспорт в JSONL) вместо постраничного перебора — дешевле
    по cost. Для массовой **записи** — `bulkOperationRunMutation` (загрузка JSONL мутаций через staged upload).
  - **`productSet`/`productCreate`** с большим числом вариантов/медиа дороже по cost — учитывать в планировании
    конкурентности воркеров (не запускать 50 параллельных тяжёлых мутаций — выест bucket).
  - **Деградация (§5.5):** один битый товар → job помечается failed, лог, **пакет не падает**; сетевые/HTTP-ошибки
    одного запроса → null/skip + ретрай по политике, не исключение наверх.

### Медиа: Files API + staged uploads (§5.3)

Доступа к ФС магазина нет — **sideload невозможен**. Поток загрузки изображения:

1. Приложение **скачивает** бинарь у OneCatalog: `…/media_files/<base64hash>` (URL **без расширения** → определять
   **MIME по содержимому**, §2.3; с токеном тянуть `max`, без токена доступен только `min`).
2. **`stagedUploadsCreate`** → получить временный `url` + параметры формы (`resourceUrl`/`parameters`) на CDN Shopify.
3. **PUT/POST** бинаря на staged URL (multipart по параметрам).
4. **`fileCreate`** (или сразу `productCreateMedia` с `originalSource = resourceUrl`) → Shopify асинхронно
   обрабатывает медиа (`MediaImage` проходит `status: UPLOADED → READY/FAILED`). **Дождаться `READY`** (поллинг
   `node(id){ ... on MediaImage { status } }` или webhook), прежде чем считать импорт картинки успешным.
5. **Обложка** = первый media товара (порядок media определяет «главное» изображение); **галерея** = остальные.

Инварианты §5.3 на Shopify:

- **Трекинг качества (`min|middle|max`)**: хранить выбранный размер **в БД приложения** (рядом с записью о медиа),
  т.к. у Shopify-файла нет места под «наш» произвольный признак, кроме `alt`/имени. На повторном импорте: появился
  токен/лучший размер → **перекачать**, заменить media (`productDeleteMedia` старого + создать новый), записать новый
  размер. Если не лучше — не трогать.
- **Дедуп по контент-ключу:** ключ = `hash(path # size)`, извлекаем из base64-префикса `media_files` (стабилен между
  запросами и сменой токена, §5.3). Хранить реестр `contentKey → Shopify File GID` в **БД приложения**. Перед
  загрузкой искать по ключу: нашли → **переиспользовать существующий File GID** (`productCreateMedia` ссылается на уже
  загруженный `MediaImage`/`File` по его `originalSource`/привязке; либо привязать тот же File к товару). ⚠️ Общие
  (дедуплицированные) файлы **НЕ удалять** при апгрейде качества — это сломает другие товары (§5.3).
- **Идемпотентность галереи по имени файла** (подписи в URL меняются, §5.3): сравнивать по сохранённому имени/контент-
  ключу в БД, не по URL.
- Лимиты медиа Shopify: до **250 media** на товар; учесть при больших галереях.

### Настройки приложения (§7)

Все настройки — в **БД приложения** (per-shop), редактируются в **embedded admin UI** (App Bridge + Polaris):

| Настройка стандарта | Реализация на Shopify |
|---|---|
| Токен Wiki API | per-shop запись в БД приложения; приоритет env приложения над полем (§7) |
| Язык товаров (`lang`) | enum из `en/ru/ar/zh/kk`; per-shop |
| Шаг импорта (≥10) | размер порции очереди приложения |
| Статус новых товаров | маппинг на `ACTIVE/DRAFT/ARCHIVED`, **только при создании** |
| Импортировать коллекции + «хранить как + цель» | на Shopify цель ограничена: **manual/smart Collection** (CPT/таксономии нет) → развилка сужается |
| Бренд «хранить как» | **`vendor` (строка) / metaobject** (вместо атрибут/таксономия/CPT) |
| Страна «хранить как» | **metafield / metaobject** (+ опц. таможенный `countryCodeOfOrigin`) |
| Импортировать теги | `Product.tags` merge |
| Ручной маппинг характеристик | UI: `specification_id` → metafield/metaobject definition; для boolean — значения true/false |

Дополнительно к стандарту (специфика Shopify): выбор **версии Admin API**, **scopes** (см. жизненный цикл), выбор
**цели категории** (Collection manual/smart и/или Standard Taxonomy), флаг «писать таможенную страну».

### Конвертер единиц (§5.6)

- Универсальный конвертер «источник → единицы цели» (таблицы коэффициентов к базовой единице + нормализация
  локализованных подписей «мм»/«г»).
- **Вес:** OneCatalog = граммы. Shopify принимает `GRAMS` напрямую → дефолтно писать в граммах **без конверсии**
  (минимум потерь). Если витрина/тема ждёт кг — конвертировать в `KILOGRAMS`.
- **Размеры:** OneCatalog = миллиметры. Metafield `dimension` хранит value+unit; писать `{ value, unit: "MILLIMETERS" }`
  или конвертировать в см/м по настройке. Единицу источника брать из `sizes.*_unit`, иначе мм/г; переопределяемо
  «фильтром» (= хук приложения, см. ниже).

### i18n (§9)

- **Строки UI приложения** (embedded admin): обычная i18n-библиотека приложения (i18next/Polaris i18n), исходник EN +
  переводы. Это **строки приложения**, не магазина — здесь Shopify не участвует.
- **Контент товара на разных языках** (важное отличие от Woo): на Shopify мультиязычность товаров — это
  **Translate & Adapt / Markets + Translations API** (`translationsRegister` на полях `title`, `body_html`,
  metafields с `locale`). То есть `lang` стандарта (один язык за импорт) на Shopify может разворачиваться в:
  (а) импорт в основной язык магазина, либо (б) импорт нескольких языков OneCatalog → регистрация переводов
  Shopify по локалям магазина. Базовый план — (а); (б) — опционально (см. вопросы).
- Локализованные **значения характеристик** (metaobject записи) тоже переводятся через Translations API, если нужны
  на нескольких языках витрины.

### Жизненный цикл приложения (§10)

- **OAuth + scopes** (минимально необходимые):
  `write_products`, `read_products` (товары, коллекции, варианты, медиа-привязки),
  `write_inventory`/`read_inventory` (вес — на `inventoryItem`),
  `write_files` (Files API / staged uploads),
  `write_metaobjects`/`read_metaobjects` (характеристики/бренды как metaobjects),
  `write_translations` (если включаем мультиязычность),
  при работе с публикациями/каналами — `write_publications`.
  Точный список зафиксировать и запросить **минимум** (принцип least-privilege; лишние scopes тормозят ревью в App Store).
- **Install:** OAuth-флоу, сохранение offline-токена per-shop, создание **metafield/metaobject definitions** при
  установке (idempotent), регистрация **webhooks** (`app/uninstalled`, `products/delete`, опц. `products/update`),
  миграция схемы (создать недостающие definitions).
- **Uninstall:** webhook `app/uninstalled` → пометить shop неактивным, **остановить очередь/cron** для этого магазина,
  по GDPR-вебхукам (`customers/redact`, `shop/redact`, `customers/data_request`) — обработать (даже если приложение не
  трогает покупателей, эндпоинты обязательны для App Store).
- **Версионирование:** SemVer + CHANGELOG (§10) — на стороне **приложения** (релизы приложения), плюс зафиксированная
  **Admin API version** (Shopify депрекейтит версии ежеквартально → плановое обновление, иначе ломается).
- **«Активация не должна падать без движка магазина»** (§10): на Shopify аналог — приложение должно корректно
  работать при отсутствии данных/прав, не падать на пустом каталоге, проверять наличие scopes/токена перед операциями.

### «События/хуки» стандарта §8 → на стороне приложения

- На Shopify **нельзя инжектить `do_action`/`apply_filters` в рантайм магазина** — чужой рантайм недоступен.
- **«Фильтры»** (меняют значение: URL, токен, payload перед импортом, SKU, цена, порядок размеров, дедуп, флаги,
  origin виджета) → **точки расширения внутри приложения**: конфигурируемые функции/плагины/мидлвары импорт-пайплайна
  приложения (in-process). Это код приложения, а не магазина.
- **«События»** (`product_imported`, `collection_imported`, …) → **внутренняя событийная шина приложения**
  (in-process emitter / очередь доменных событий). «Сайт дозаполняет поля здесь» → на Shopify это значит:
  обработчик приложения дозаполняет **metafields/metaobjects/переводы/публикации** товара (то, чего нет в базовом
  маппинге: `measurement_unit`, `areas_per_package`, `product_shape`, alt картинок, детали бренда, upsells/cross_sells).
  Граница «универсальное ↔ проектное» сохраняется: ядро импорта пишет маппинг §3, расширения приложения — остальное.
- **Вовне магазина** доступны только **Webhooks** Shopify (предопределённый список тем) — это НЕ аналог произвольных
  событий ядра, а уведомления магазина приложению. Использовать для ресинхронизации/реакции на ручные правки
  (`products/update`), но нельзя «подписать сайт» на наше `product_imported` (такого события у магазина нет).

### Picker (§2.4)

- Виджет OneCatalog (`tools.onecatalog.net/picker.html`) встраивается как **iframe**. В embedded Shopify-приложении
  (само уже iframe в Shopify Admin через App Bridge) получается **вложенный iframe** → проверить CSP/`frame-ancestors`
  Shopify Admin и sandbox/`postMessage`-origin (см. «невозможно»: ограничения embedded iframe).
- Связь `postMessage` (`ONECATALOG_SELECTED { productPublicIds }`) — как в стандарте; импорт **по `productPublicIds`**.
  `parentOrigin` задавать явно (origin страницы приложения), проверять `event.origin`.
- Альтернатива (надёжнее в embedded): открыть picker **во всплывающем окне/новой вкладке приложения** (вне вложенного
  iframe) или реализовать **свой** выбор товаров средствами приложения (тянуть `/products_ids/` и `/products/` Wiki API
  и рисовать список в Polaris) — тогда вложенного iframe нет вовсе.

## 🚫 Сложно / невозможно на Shopify

Развёрнуто — где платформа принципиально не повторяет стандарт/эталон и почему.

### 1. Нет произвольных таксономий и Custom Post Types
- WooСommerce-маппинг §3 опирается на **произвольные таксономии** (`product_cat`, `product_brand`, теги-таксономии,
  глобальные атрибуты `pa_*`) и **CPT** (для коллекций/брендов). **На Shopify их нет в принципе.**
- Следствия: «коллекция/бренд/страна как CPT» — **невозможно** (нет CPT → используем metaobjects, но это другая
  модель, без архивных URL/шаблонов «из коробки»). «Характеристика как атрибут `pa_*` + термины» — **невозможно как
  таксономия**; ближайший аналог — metaobject + metafield reference, но это не таксономия (нет нативных
  архивов/иерархии терминов, фильтрация — только через Search & Discovery по metafields).
- Развилка стандарта «тип объекта (атрибут/таксономия/CPT) + цель» **схлопывается**: на Shopify целей реально две —
  **строковое поле/metafield** или **metaobject**; плюс для категорий/коллекций — **Collection**. Это надо честно
  отразить в настройках (не предлагать несуществующие «таксономию/CPT»).

### 2. Характеристики НЕЛЬЗЯ класть в «атрибут + термин» как варианты
- product options ограничены: **максимум 3 опции** и **до 100 (исторически) / 2048 (новый лимит) вариантов** на товар,
  причём каждый набор значений опций порождает **вариант** (с ценой/инвентарём/sku). Характеристик у плитки бывает
  много (формат, толщина, поверхность, ректификация, PEI, морозостойкость…) — положить их в options **невозможно**:
  это создаст комбинаторный взрыв вариантов и упрётся в лимиты.
- Поэтому характеристики идут в **metafields/metaobjects** — это **не** «атрибут+термин» эталона, и фасетная
  фильтрация требует **отдельного** механизма (Search & Discovery), которого в ядре нет.

### 3. Размеров товара (length/width/height) в ядре нет
- В отличие от WooCommerce (`length/width/height` — нативные поля товара), у Shopify **габаритов товара нет вообще**.
  Только **вес** (и то на `inventoryItem`/варианте). Размеры приходится хранить в **metafields** (`dimension`), и они
  **не участвуют** в нативных расчётах доставки/упаковки. Габаритную доставку считает только сторонний
  carrier-service app — из коробки невозможно.

### 4. Вес — на варианте, не на товаре; единицы ограничены enum'ом
- Концептуально иначе, чем в стандарте (где вес — у товара). Записывать в `inventoryItem.measurement.weight`.
  Единицы — фиксированный enum (`GRAMS/KILOGRAMS/OUNCES/POUNDS`); граммы OneCatalog ложатся хорошо, но произвольную
  единицу измерения товара (`measurement_unit` = м²/шт) **в вес положить нельзя** — это не масса.

### 5. `measurement_unit` (м²/шт) и unit pricing — частично/сложно
- «Единица измерения товара» (продаём м²) в ядре Shopify представлена только как **unit pricing**
  (`unitPriceMeasurement` — цена за единицу) и доступна не во всех странах/настройках. Полноценной «продажи по м² с
  пересчётом упаковок» из коробки **нет** — это кастом в теме + metafields (`areas_per_package`), не нативная функция.

### 6. Нет серверного cron у магазина; нельзя инжектить хуки в рантайм
- В магазине **нет cron** и **нет доступа к рантайму** (нельзя зарегистрировать `do_action`/`apply_filters`,
  нельзя выполнить свой PHP/JS в контексте магазина). Весь фон, очередь, планировщик, событийная шина — **только в
  приложении**. «Хуки стандарта §8» воспроизводятся как точки расширения **приложения**, а не магазина.
- Единственный канал «магазин → приложение» — **Webhooks** с **предопределённым** списком тем (нельзя создать
  произвольную тему-событие). Это не эквивалент произвольных `do_action` ядра.

### 7. Нет доступа к файловой системе магазина (sideload невозможен)
- WooCommerce `media_sideload_image`-подход неприменим: **ФС магазина недоступна**. Любое медиа — только через
  **staged uploads → fileCreate/productCreateMedia**, с асинхронной обработкой (`UPLOADED → READY/FAILED`) и
  ожиданием готовности. URL без расширения и MIME-by-content — как в стандарте, но грузим **не в ФС, а в Shopify CDN**.
- Трекинг качества и контент-дедуп **нельзя** хранить «рядом с файлом» в произвольных метаданных файла — храним в
  **БД приложения** (у Shopify File нет нашего произвольного поля; `alt` ненадёжен для машинных ключей).

### 8. Идемпотентный поиск по metafield — асинхронный индекс
- Поиск `products(query:"metafield:oc.public_id=...")` работает, но индекс Shopify **обновляется асинхронно**: только
  что созданный товар может **не найтись** сразу. Поэтому единственным источником идемпотентности его делать
  **нельзя** — нужна **БД приложения** как primary-зеркало `public_id ↔ GID` (Shopify-поиск — вторичный/reconcile).

### 9. Rate limits / GraphQL cost — жёсткий бюджет
- В отличие от локального плагина (пишет в свою БД без лимитов), приложение работает **сквозь API с квотой**: GraphQL
  bucket 1000 pts, восстановление 50 pts/сек; тяжёлые мутации (`productSet` с медиа/вариантами) дорогие. Массовый
  импорт **нельзя** гнать «в лоб» — обязателен token-bucket по `throttleStatus`, backoff на `THROTTLED`, и Bulk
  Operations для массовых чтений/записей. Это формирует **верхнюю границу скорости импорта**, которой у плагина нет.

### 10. Embedded iframe / App Bridge ограничения (и вложенный picker)
- Приложение живёт в **iframe внутри Shopify Admin** (App Bridge), под строгим **CSP `frame-ancestors`**. Встроить
  ещё и **picker OneCatalog как вложенный iframe** проблематично: вложенный iframe со сторонним origin под CSP админки,
  sandbox-флаги, `postMessage` между тремя уровнями (Admin → приложение → picker), cookie/`SameSite`-ограничения.
  Часто это **невозможно/ненадёжно как есть** → решение: свой пикер в Polaris (тянуть Wiki API самим) или отдельное
  окно. Прямой перенос iframe-контракта §2.4 «как в Woo-плагине» в embedded-приложение **не гарантирован**.

### 11. Коллекции плоские — иерархия категорий не нативна
- Shopify-коллекции **не имеют parent/child**. Дерево категорий OneCatalog **невозможно** воспроизвести нативно:
  либо «плоский» список коллекций, либо эмуляция иерархии через metafield `parent` + кастомная навигация темы
  (рендер дерева — задача темы, не платформы). Хлебные крошки/вложенные архивы «из коробки» не получатся.

### 12. Цена обязательна — расходится с «не выдумывать данные»
- Стандарт §5.6: цену **не синтезировать** (в источнике её нет). Но Shopify-вариант **требует** `price` (хотя бы
  `0.00`). Полностью «не задавать цену» **невозможно** — придётся ставить `0.00` (или из своего источника через хук
  приложения). Это вынужденное отклонение от §5.6, требующее явного решения (вопрос 8).

### 13. Публикация на каналы — отдельный шаг
- Созданный `productCreate` товар может быть **не опубликован** ни на одном канале (не виден в Online Store). Нужен
  явный `publishablePublish`/назначение публикаций — в эталоне (Woo) «опубликован» = статус товара, здесь это
  **отдельная сущность** (publications), легко упустить.

### Итоговая граница «универсальное ↔ проектное» на Shopify

Ядро импорт-**приложения** реализует контракты §2 и инварианты §5 (идемпотентность через metafield+БД, find-or-create
справочников по стабильным ключам, трекинг качества/дедуп медиа через Files API+БД, конвертер единиц, очередь с
rate-limit-бюджетом, не-перезатирание полей «только-при-создании»). Всё «витринное» (рендер характеристик, иерархия
категорий-коллекций, калькулятор м²/упаковок, мультиязычность витрины) — задача **темы Online Store + metafields/
metaobjects/переводов**, дозаполняемых через событийную шину приложения (`product_imported`-аналог) — но **внутри
приложения**, а не магазина, поскольку инжектить код в рантайм магазина нельзя.
