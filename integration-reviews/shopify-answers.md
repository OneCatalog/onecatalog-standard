## Главное про данные (закрывает блокер 2 и уточняет Q9, характеристики)

🟢 **Вариаций товара НЕТ вообще** — один `public_id` = один готовый товар. Поэтому **всегда один
default variant**, а лимит «3 опции / 100 вариантов» к нашим данным **не относится** (он сработал бы,
только если класть характеристики в options — чего мы корректно НЕ делаем). Комбинаторного взрыва нет
по построению.

🟢 **Характеристики многозначны** — у товара «Цвет» = 2 значения, «Помещение» = **6**. Справочник
`/specifications/` отдаёт флаг **`multitude`** (true = многозначная). На Shopify это ложится **чисто**:
многозначная характеристика → **list-metafield** (`list.single_line_text_field` или
`list.metaobject_reference`), одно-значная → обычный metafield. Используйте `multitude`, чтобы выбрать
тип definition. (Это лучше, чем у PrestaShop, где Feature держит одно значение.)

🟢 **`numeric` в этом инстансе не используется** (все 13 характеристик — text/boolean).

## Ответы на вопросы

**1. Характеристики — фасеты или просто поля.** 🟡 Для плитки фильтрация по характеристикам (формат,
поверхность, цвет, помещение) — обычно нужна → **metafields со storefront-доступом + Search & Discovery**.
Многозначные (`multitude=true`) — list-metafields; для общих фильтруемых значений с локализацией —
metaobjects + `list.metaobject_reference`. **Подтвердить, нужна ли фасетка** (если только показ — простые
metafields).

**2. Категории: Collection / smart / Standard Taxonomy.** 🟡 Дефолт — **manual Collection**, find-or-create
по metafield `oc.category_id` (не по title/handle). Smart-коллекции — опция (правило по metafield/тегу).
`Product.category` из Standard Taxonomy — отдельный опциональный слой (см. Q3). 🟢 Иерархия в данных ЕСТЬ
(`/categories/` `parent_category`), но Shopify-коллекции плоские → см. блокер 11.

**3. Маппинг на Shopify Standard Product Taxonomy.** 🟡 Авто-маппинга «категория OneCatalog → taxonomy
GID» нет; нужна ручная таблица соответствий. **Рекомендую на старте НЕ делать** (опционально позже);
категории → свои Collections достаточно для навигации/фасетки. Если нужен — таблицу даёт владелец/
контент-менеджер.

**4. Бренд: vendor или metaobject.** 🟢 Один товар — **всегда один бренд** (`brand` — объект, не список) →
`Product.vendor` подходит. Богатые поля бренда (`description_text`, `foundation_year`, `website`,
`country`, лого) эталон **не сохраняет даже в WC** (только имя+лого+id) → это «прочий payload». 🟡
Рекомендация: `vendor` (дефолт, даёт фасет/smart-collection) + опц. metaobject `brand`, если нужна
карточка бренда с логотипом. Развилка стандарта «атрибут/таксономия/CPT» на Shopify = **vendor / metaobject**.

**5. Страна: витринная или таможенная.** 🟢 По умолчанию — **витринная**: metafield `oc.country` (или
metaobject) по `menutitle`. ⚠️ НЕ путать с таможенными `inventoryItem.countryCodeOfOrigin`/HS-код. 🟢
Важно: `country.country_code` в API — ISO **alpha-3** (`IND`, `ESP`), а таможенное поле Shopify ждёт
**alpha-2** → если решите писать таможню, нужна конверсия alpha-3→alpha-2 (и не все страны покроются).
При metafield-по-имени код не нужен. **Дефолт — metafield-характеристика, таможню только если бизнес
просит.**

**6. Статус-маппинг.** 🟢 Эталон использует свою настройку «статус новых», **только при создании**. На
Shopify: `publish → ACTIVE`, `draft/pending/private → DRAFT` (у Shopify нет pending/private; `ARCHIVED`
оставить для снятых). 🟢 API даёт `status_publish: "publish"` — можно опц. уважать, но эталонный путь —
своя настройка, применяемая только при создании, не перезатирать при апдейте.

**7. Публикация на каналы.** 🟢/🟡 Да — `productCreate` может НЕ опубликовать товар ни на один канал →
нужен явный `publishablePublish` на Online Store (и нужные каналы) после создания. В эталоне (WC)
«опубликован» = статус; на Shopify это отдельная сущность publications — **не забыть** (блокер 13).
🟡 Публиковать сразу или оставлять DRAFT до проверки (особенно из-за цены 0, Q8) — продуктовое решение.

**8. Цена.** 🟢 В источнике цены НЕТ. Shopify `ProductVariant.price` обязателен → **`0.00` при создании,
не трогать при апдейте** (эталон: цена только при создании, ручные правки не перезатирать). Реальную —
из своего источника через точку расширения приложения (аналог `filterOneCatalogProductPrice`). Это
вынужденное отклонение от §5.6 «не задавать», но «не задавать» = «0 и не перезатирать», не «null».
Кросс-платформенно (то же у Magento/PrestaShop/Drupal). 🟡 Если 0₽ на витрине нежелательно — держать
товар `DRAFT`/неопубликованным до подстановки цены (Q7).

**9. Варианты.** 🟢 **Всегда один вариант** (вариаций нет) → создавать default variant. Отдельный
маппинг под options/комбинации НЕ нужен; лимит 3/100 не упираемся (см. «главное про данные»).

**10. Единица измерения / unit pricing.** 🟢/🟡 `measurement_unit` (факт API: `{slug:"m2", name:"м2"}`) —
продажная единица. Нативный механизм «цена за м²» на Shopify — **unit pricing** (`unitPriceMeasurement`),
но он ограничен (не во всех странах, требует включения) → для плитки «цена за м² + пересчёт упаковок»
обычно делается **темой + metafields**, не ядром. **Рекомендация:** хранить `measurement_unit` и
`areas_per_package` в metafields (через событие приложения), unit pricing — опционально, если бизнес
требует и доступен. Это «сайтовый слой».

**11. Площадь упаковки `areas_per_package`.** 🟢 Факт API: поле есть, но **часто `null`** (у образца
пусто). Хранить в metafield `oc.areas_per_package`; калькулятор «упаковок на площадь» — кастом темы, не
ядро. Через событие приложения.

**12. Размеры товара.** 🟢 В ядре Shopify габаритов НЕТ → **metafields** (`dimension`, мм → конвертер или
писать в мм). Для плитки 600×1200 размер важен как данные → хранить. В нативной доставке габариты не
участвуют (Shopify считает по весу) — габаритная доставка только через сторонний carrier app. 🟡
Подтвердить, нужны ли габариты для доставки (обычно нет — только показ).

**13. Public App или Custom App.** 🟡 Если для **одного** магазина (как ceram-сценарий) — **Custom App**
проще (без App Store ревью/биллинга; GDPR-вебхуки желательны, но не обязательны как для публичного).
Public App — только если планируете распространять. **Подтвердить.**

**14. Мультиязычность.** 🟡 Фаза 1 — **один язык** (основной язык магазина; API отдаёт один `lang` за
запрос). Несколько языков → Translations API (`translationsRegister`) + Markets — отдельная фаза. Эталон
тоже один язык за прогон. **Подтвердить целевые локали.**

**15. Picker во вложенном iframe.** 🟢/🟡 В embedded-приложении (само iframe под App Bridge) вложить ещё
`tools.onecatalog.net/picker.html` — **ненадёжно** (CSP `frame-ancestors`, тройной `postMessage`, cookie/
SameSite). **Рекомендация — свой пикер в Polaris** (тянуть `/products_ids/` + `/products/` Wiki API и
рисовать список) ИЛИ отдельное окно; чистый прямой перенос iframe-контракта §2.4 не гарантирован.
Фолбэк всегда — импорт по списку `public_id`. (стандарт §2.4 это допускает.)

**16. Дедуп файлов между товарами.** 🟢/🟡 Реестр `contentKey (sha1(path#size)) → File GID` в БД
приложения; перед загрузкой ищем по ключу → нашли → **не качаем заново**, привязываем тот же File/
MediaImage как media к товару. ⚠️ Общие файлы НЕ удалять при апгрейде качества. 🟡 Детально проверить на
целевой версии Admin API, что один File можно привязать media к нескольким Product (поведение Files API);
если нет — дедуп всё равно экономит **скачивание** (как компромисс на PrestaShop). Главное — не качать
повторно.

**17. Своя БД приложения (зеркало).** 🟢 **Да, обязательно.** Поиск по metafield у Shopify **асинхронный**
(только что созданный товар может не найтись сразу) → БД приложения = primary-зеркало `public_id ↔ GID`
и реестр медиа; Shopify-поиск — вторичный/reconcile. Это ключевой момент идемпотентности на Shopify
(блокер 8).

**18. Webhook на ручные правки.** 🟡 Полезно слушать `products/update`, чтобы при ресинхронизации не
затирать ручные правки вебмастера (перекликается с §5.6 «поля только-при-создании не перезатирать»).
Рекомендую на старте: не перезатирать статус/цену (только при создании); полноценный reconcile по
webhook — улучшение.

**19. Объёмы и лимиты.** 🟢 Жёсткого rate-limit на Wiki API в тестах не наблюдалось; справочники
(`/specifications/`, `/categories/`) — `limit=1000`. На стороне Shopify — **GraphQL cost-бюджет** (1000
pts / 50 pts/сек) обязателен: token-bucket по `throttleStatus`, backoff на `THROTTLED`, **Bulk
Operations** для массовых чтений/записей. 🟡 Ожидаемый объём уточнить — он задаёт верхнюю границу скорости.

**20. Тема/витрина.** 🟡 Верно: приложение кладёт данные (metafields/metaobjects/collections), **рендер —
задача темы** (Liquid + metafields: характеристики, дерево категорий-коллекций, калькулятор м²/упаковок).
Это вне ядра импорта — граница «универсальное ↔ проектное». **Подтвердить, кто делает тему.**

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность:** primary — **БД приложения** (`public_id ↔ GID`), metafield `oc.public_id` —
   вторичный/reconcile (поиск асинхронный). Искать ВСЕГДА по public_id, не по title/handle.
2. **Справочники (collections, metaobject-значения) — find-or-create по стабильному ключу OneCatalog**
   (`category_id`, `specification_id`, `specification_option_id`, `collection_id`), а human-readable
   label/имя — в name definition. Это естественно решает проблему дублей при транслитерации (на Shopify
   handle тоже автогенерится — не матчить по handle).
3. **Характеристики:** metafield key = `spec_<specification_id>` (не label); многозначные (`multitude`)
   → list-metafields; ручной маппинг по `specification_id`, строгий режим.
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие =
   false**; для boolean-типа metafield пишем `false`. (`multitude=false` у boolean.)
5. **Медиа:** дедуп по `sha1(path#size)` (реестр в БД), общие File не удалять; галерея по имени файла, не
   по URL; ждать `READY`; URL без расширения → MIME по содержимому.
6. **Цена `0` при создании, не перезатирать; статус — только при создании** (`ACTIVE`/`DRAFT`).
7. **Вес — граммы напрямую** (`unit: GRAMS`); размеры — metafield (мм → конвертер).
   `measurement_unit`/`areas_per_package`/`product_shape`/alt/детали бренда/upsells — через событие
   приложения (полный payload), не в ядро.
8. **Категории:** иерархия из `/categories/` (`parent_category`) — эмулировать через metafield parent.
   **`/collections/{slug}/` 500 — не дёргать**, всё в эмбеде `collections[]` товара.
   **`/specifications/`, `/categories/` — `limit=1000`.**
9. **Теги — `Product.tags` merge** (не полная замена, чтобы не затирать ручные).
10. **SemVer + CHANGELOG** на стороне приложения + **зафиксировать Admin API version** (Shopify
    депрекейтит версии ежеквартально → плановое обновление).
