## Главное про данные (закрывает Q7-масштаб, Q9, Q19)

🟢 **Вариаций товара НЕТ ВООБЩЕ** — один `public_id` = один готовый товар. → Commerce: **простой
`Product`, без `VariantProduct`**; S/4: один материал. `options[]` — характеристики, не оси вариативности.

🟢 **Иерархия категорий ЕСТЬ — ответ на Q19.** Эмбед `categories[]` в товаре плоский, но справочный
**`/categories/` отдаёт `parent_category`** (id родителя или `null`). → Commerce: дерево `Category`
(supercategories); S/4: product hierarchy/группа. Не плоско под один корень — есть настоящая иерархия.

🟢 **Масштаб классификации мал — снимает остроту блокера 2/Q7.** Характеристик **всего ~13** (text/
boolean; `numeric` в этом инстансе не используется). Предсоздать 13 характеристик класса (CT04/S/4 или
ClassificationAttribute/Commerce) транспортом/ImpEx — **вполне посильно**; «строгий ручной маппинг» здесь
не тяжёлая ноша, а разовая настройка. Тяните `/specifications/?limit=1000` один раз — получите полный
список для предсоздания.

🟢 **Характеристики многозначны** — «Цвет» = 2 значения, «Помещение» = **6**. `/specifications/` отдаёт
флаг **`multitude`** (true = многозначная; boolean `false`). → характеристику класса делать **multi-value**
(multiple-values flag в S/4 `CABN` / multivalued в `ClassAttributeAssignment` Commerce). Используйте
`multitude`.

🟢 **`country_code` — ответ на Q9: ISO alpha-3** (трёхбуквенный). SAP `T005`/`HERKL` обычно оперирует
**alpha-2** → нужен маппинг alpha-3 → alpha-2 (или матч по имени страны). Не предполагайте совпадение
форматов.

🟢 **Единицы — ответ на Q8 (вход):** OneCatalog отдаёт **вес в граммах, размеры в миллиметрах**;
продажная `measurement_unit` — **м²/шт**. Есть справочный эндпоинт **`/measurement-units/`** (тяните
`limit=1000`) — основа для ISO-маппинга (г→`GRM`, мм→`MMT`, м²→`MTK`, шт→`PCE`/`H87`). **`sizes.weight`
часто `null`** (у OC.IND.1 веса нет) — не выдумывать.

🟢 **Цены в источнике НЕТ** (факт API) — S/4 материал без цены продажи нормален; Commerce `PriceRow` не
создаём.

## Ответы на вопросы

**1–4. Какой SAP / цель / связка S/4↔Commerce / релиз.** 🟡 Стратегическое, решает заказчик — **блокер
№0, согласен**. Контракт со стороны данных: OneCatalog отдаёт **готовые каталожные товары для покупателя**
(описания, картинки, фасет-характеристики, коллекции) — это семантически ближе к **витрине (путь B,
Commerce Cloud)**. Если цель — учётная запись материала (закупки/склад) — путь A. Связка «писать в S/4 →
штатная репликация S/4→Commerce» — валидна (тогда наш модуль целится в S/4). **Определить до кода.**

**5. Нумерация материала (S/4).** 🟡 Критично для идемпотентности (§5.1). 🟢 Факт: `public_id` —
стабильный бизнес-ключ, гарантированно уникален и присутствует. Рекомендация: (а) **внешняя нумерация**:
`public_id` = номер материала (`BAPI_MATERIAL_SAVEDATA` делает upsert по номеру) — лучший случай; (б)
внутренняя: `public_id` → доп. поле/характеристика + **кросс-таблица `public_id↔номер`** в middleware. В
Commerce проблемы нет — `Product.code = public_id`, ImpEx `code[unique=true]`. **Подтвердить политику
нумерации.**

**6. Тип материала / отрасль / базовая единица.** 🟡 Конфиг заказчика. Контракт: базовая единица `MEINS`
← `measurement_unit` через ISO-маппинг (Q8); виды (Basic Data + при необходимости Sales) — минимально под
каталог. **Подтвердить material type/отрасль.**

**7. Классификация — предсоздание.** 🟢 **Подтверждаю: создавать характеристики на лету нельзя →
строгий ручной маппинг (§5.4) обязателен** (как Magento/UMI). НО масштаб мал — **всего ~13 характеристик**
(см. «главное про данные»), предсоздание разовое и посильное. Ключ маппинга — стабильный
`specification_id`, не переводимый label. Импорт только проставляет значения. **Согласовать, кто/когда
заводит классы транспортом.**

**8. Единицы → ISO.** 🟢🟡 Вход: г/мм/м²/шт (+ `/measurement-units/?limit=1000`). Маппинг г→`GRM`,
мм→`MMT`, м²→`MTK`, шт→`PCE`/`H87` → внутренние коды `T006`. ⚠️ Две разные операции: **маппинг кода
единицы** (на ISO) и **конверсия величины** (§5.6: г→кг, мм→см) — не путать. **Подтвердить полноту
ISO-аналогов и активацию единиц.**

**9. Страна происхождения.** 🟢 `country_code` OneCatalog = **ISO alpha-3** (ответ на ваш вопрос) +
локализованное имя. SAP `HERKL`/`T005` — обычно alpha-2 → **маппинг alpha-3→alpha-2 или матч по имени**.
Нативное поле страны происхождения — сильная сторона (используйте). 🟡 Точное поле по релизу + покрытие
стран — **подтвердить**.

**10. Вес/габариты.** 🟢 Вес в **граммах** → `BRGEW`/`NTGEW` + `GEWEI` (конверсия г→кг, §5.6);
**`weight` бывает `null`** — не писать. Габариты length/width/height в **мм** — раздельных числовых полей
в стандартном material master нет → **характеристики классификации (numeric с unit)** или `GROES` текстом;
в Commerce — numeric classification attributes с unit. 🟡 Брутто vs нетто из `sizes.weight` — **уточнить**
(вероятно оба = вес товара, если нетто/брутто не разделены источником).

**11. Артикул (`article`).** 🟢 `article` ≠ `public_id` (§5.2) — отдельно (S/4: `BISMT`/«номер
производителя»; Commerce: отдельный атрибут/`manufacturerAID`). Только непустой; коллизия — не
перезаписывать чужой. 🟡 Точное поле — **подтвердить**.

**12. Медиа.** 🟢🟡 Рекомендация (ваша): **S/4 — картинки не грузить** (DMS тяжёл, material master их на
витрине не использует), отдать витрине (Commerce). Commerce: `Media`/`MediaContainer`, **`Media.code` =
контент-ключ `sha1(path#size)`** + ImpEx `INSERT_UPDATE` — отличный дедуп (ваша идея верна). Полезный
факт: бинарный `media_files`-URL **самодостаточен** — токен зашит в base64-хеш, файл доступен **без
заголовка `X-API-Key`** (в эталоне `Media::sideload` так и качает max). → если ImpEx/Media API умеет
**загрузку по внешнему URL** — отдаёте прямой max-URL, SAP скачает max сам, токен не нужен; иначе middleware
качает (URL без расширения → MIME по содержимому) и кладёт файлом. Трекинг качества (min/middle/max) — в
атрибуте медиа/middleware. **Подтвердить URL-загрузку в Commerce.**

**13. Цена.** 🟢 В источнике цены НЕТ — **не синтезировать** (§5.6). S/4 — материал без цены нормален.
Commerce — `PriceRow` не создаём (продукт без цены / «цена по запросу» / ставит другая система). Эталон
цену не задаёт (фильтр `onecatalog_product_price`) — тот же принцип. **Подтвердить.**

**14. Мультиязычность.** 🟢🟡 API отдаёт **один язык за запрос** → несколько прогонов с разным `lang`
дозаполняют `MAKT` по `SPRAS` (S/4, нативно — сильная сторона) / локализованные атрибуты (Commerce). Языки
OneCatalog: `ru/en/ar/zh/kk`. Для ceram базово — **`ru`**. ⚠️ Проверить активацию редких языков в SAP
(особенно `kk` — казахский). **Подтвердить набор языков.**

**15–16. Middleware / транспорты / мандант / авторизации.** 🟡 Конфиг ландшафта. Контракт: контракты §2
и инварианты §5 неизменны; «куда складываем» (S/4/Commerce) и «через что» (CPI/PI vs прямой OData/Service
Layer) — ваше решение. Справочники OneCatalog кэшировать в middleware (value mapping), не дёргать на каждый
товар. **Подтвердить наличие CPI/PI и процесс транспортов.**

**17. Batch / throughput / объёмы.** 🟡 Конфиг. 🟢 Со стороны OneCatalog: справочники одним запросом
`limit=1000` + сверка `meta.counts` (дефолтный лимит инстанса молча обрезает); товары по одному
`/products/{public_id}/`; жёсткого rate-limit на `X-API-Key` в тестах не было. SAP-сторона: async IDoc-
пакеты (S/4) предпочтительнее синхронного BAPI на объёме; Commerce — ImpEx-партии + sync Staged→Online +
реиндекс Solr **после батча, не на товар**. **Подтвердить объём.**

**18. Каталог-версии Commerce.** 🟡 Писать в `Staged`, публиковать sync→`Online` (вручную/по расписанию,
после партии). **Подтвердить целевой каталог/версию.**

**19. Категории: иерархия или плоско.** 🟢 **Дерево** — `/categories/` отдаёт `parent_category` (ответ).
Commerce: `Category` supercategories; S/4: product hierarchy/группа. Родители раньше детей.

**20. Бренд/коллекции/теги.** 🟢🟡 Нативных «бренд/коллекция/тег» нет (S/4 и base Commerce) → «тип объекта
+ цель» (§7): Commerce — `Category`/classification/атрибут; S/4 — характеристика классификации/группа
материалов. Дефолты согласовать. Детали бренда/коллекции (логотип/год/сайт/`link_3d`) — из эмбеда
`collections[]`/`brand`; витринная часть — Commerce, не material master. **Подтвердить дефолты.**

**21. Удаление/снятие.** 🟢 Импорт по эталону **upsert, не удаляет**. Пропавший из источника — осознанная
политика (флаг удаления/статус материала/снятие с продажи), не удаление по умолчанию. **Подтвердить.**

**22. Picker / UI оператора.** 🟢 Пикер/приём `public_id` — **вне SAP** (отдельный веб-UI оператора); или
импорт по списку `public_id` (стандарт §2.4 допускает). UI не часть SAP.

**23. Роль SAP: мастер или приёмник.** 🟢🟡 Если SAP — мастер каталога, переимпорт затирает; если
приёмник с ручной доработкой — **«поля только при создании» (§5.6) защищать от перезаписи**. На IDoc
`MATMAS` это требует слать **только дельту** / исключать «ручные» поля (блокер 5). В Commerce ImpEx
управляемее (только нужные колонки). **Определить роль.**

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id`:** Commerce — `Product.code` (ImpEx `code[unique=true]`, лучший
   случай); S/4 — внешняя нумерация или кросс-таблица. `article` отдельно (только непустой). Не по
   name/slug (§5.1).
2. **Справочники — стабильный ключ:** классификация/категории по предсогласованным кодам/именам, не по
   транслиту слага (в эталоне был баг дублей «Толщина»). Ключ маппинга характеристики — `specification_id`.
3. **Характеристики → классификация, предсозданная** (~13, посильно); многозначные (`multitude`) →
   multi-value характеристика; строгий ручной маппинг по `specification_id`.
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие =
   false**; характеристика со значениями true/false. (`multitude=false` у boolean.)
5. **Медиа:** Commerce только (`Media.code`=`sha1(path#size)`, ImpEx `INSERT_UPDATE` = дедуп); URL
   самодостаточен (токен в хеше) → URL-загрузка прямым max-URL если поддержана, иначе middleware качает
   (MIME по содержимому, URL без расширения); трекинг качества min/middle/max в атрибуте медиа. S/4 медиа
   не грузить.
6. **Не выдумывать цену** (S/4 без цены ок; Commerce без `PriceRow`); `weight` бывает `null`; «поля только
   при создании» — дельта/исключение ручных полей при репликации.
7. **Единицы:** ISO-маппинг кода (г→`GRM`/мм→`MMT`/м²→`MTK`/шт→`PCE`) **+** конверсия величины г→кг/мм→
   (§5.6) — две операции. `measurement_unit` (м²/шт) — продажная единица (`MEINS`/ISO), не путать с
   единицей габаритов.
8. **Категории — дерево** из `/categories/` (`parent_category`): Commerce `Category` supercategories / S/4
   product hierarchy. **`/collections/{slug}/` = 500 — не дёргать**, всё из эмбеда `collections[]`.
   **Справочники (`/specifications/`, `/measurement-units/`, `/countries/`, `/categories/`) — `limit=1000`
   + сверка `meta.counts`.**
9. **Прочий payload** (`areas_per_package`, `product_shape`, alt/описания картинок, детали бренда/коллекции,
   `upsells/cross_sells`) — где есть нативное поле (alt-единицы `MARM`, страна `HERKL`), туда; остальное —
   витринный слой (Commerce/внешний сайт), ядро не обязано.
10. **SemVer + CHANGELOG** (Keep a Changelog) для пакета iFlow/extension; **этап 0 «настройка SAP»** до
    первого товара (нумерация, классы, ISO-единицы, языки, медиа-форматы).
