## Главное про данные

🟢 **Вариаций товара НЕТ ВООБЩЕ** — один `public_id` = один готовый товар. **Это прямо подтверждает вашу
центральную развилку (Q2/блокер 1):** `options[]` → **`attributes` (доп. поля)**, НЕ `characteristics`
(иначе расплодятся модификации-variants, которых в источнике нет). Тип товара — **product**, не variant/
bundle/service.

🟢 **Иерархия категорий ЕСТЬ — ответ на Q11.** Эмбед `categories[]` в товаре плоский, но справочный
**`/categories/` отдаёт `parent_category`** (id родителя или `null`). → стройте **`productfolder` дерево**
(нативная вложенность — ваша сильная сторона), родитель по ссылке. Не плоско.

🟢 **Характеристики многозначны** — «Цвет» = 2 значения, «Помещение» = **6**. `/specifications/` отдаёт
флаг **`multitude`** (true = многозначная; boolean `false`). ⚠️ Нюанс под МойСклад: типизированные
attributes обычно **одно-значные** на определение. Многозначную характеристику → либо **attribute-
справочник с множественным значением** (если поддерживается — проверить), либо несколько значений строкой
через разделитель, либо отдельные attribute-определения. Используйте `multitude`, чтобы выбрать стратегию.

🟢 **`numeric` в этом инстансе не используется** (все 13 характеристик text/boolean) — код универсально.

🟢 **`country_code` — ответ на Q5: ISO alpha-3** (трёхбуквенный). → матчите `entity/country` **по имени
(label) как основной ключ**, alpha-3 — вторично (если формат кода МойСклада совпадёт). Всего **~13
характеристик** — attributes немного.

🟢 **Цены в источнике НЕТ** (факт API); **`sizes.weight` часто `null`** (у OC.IND.1 веса нет);
`measurement_unit` (м²/шт) — **продажная** единица (→ `uom`), не путать с единицей габаритов.

## Ответы на вопросы

**1. Идемпотентность по `externalCode` — уникальность.** 🟢 `public_id` → `externalCode` — верный ключ
(§5.1), фильтр `?filter=externalCode=...` → найден → PUT, иначе POST. Не по `name`/`code`. 🟡 Гарантия
уникальности на уровне платформы vs «по соглашению» — **проверить**; наш сервис в любом случае единственный
писатель в OC-товары + валидация уникальности (он же генерирует externalCode из public_id, дублей не
создаёт).

**2. ХАРАКТЕРИСТИКИ → attributes, НЕ characteristics.** 🟢 **Подтверждаю авторитетно** — вариаций нет
(см. «главное про данные»), characteristics породили бы модификации. Это центральная развилка, и она
решена верно. Ключ маппинга — `specification_id` (определение attribute по нему/по имени, не по слагу).

**3. Типы attributes + создание через API.** 🟡 Платформенное. Контракт: нужны string/число/boolean
(+справочник для переиспользуемых фасетов — предпочтительно). Если создание определений через API
недоступно — **преднастройка attributes в аккаунте + строгий ручной маппинг** по `specification_id` (как
на Magento/UMI — справочники предсоздаются вне импорта). **Подтвердить типы и создание через API.**

**4. Бренд — доп. поле или справочник.** 🟢🟡 Нативного бренда нет → attribute. **Рекомендую справочник**
(переиспользуемые значения, аналог таксономии, годится для фильтрации) по умолчанию имя бренда; доп. данные
бренда (страна/год/сайт/лого) — витрина (внешний сайт), не ядро. Развилка «тип объекта + цель» сохраняется.

**5. Country — формат кода.** 🟢 **`country_code` OneCatalog = ISO alpha-3** (ответ на ваш вопрос). →
`country` → `entity/country`, матч **по имени (label) основной**, alpha-3 вторично. 🟡 Формат кода и
наличие предзаполненного справочника стран в МойСкладе — **проверить** (скорее find существующую, create —
фолбэк). Это сильная сторона — нативная страна, не атрибут.

**6. UOM — find-or-create.** 🟢🟡 `measurement_unit` → `entity/uom` (нативно, сильная сторона). Матч по
имени единицы («м²», «шт»). 🟡 Состав предзаполненного справочника + возможность создания недостающих —
**проверить** (скорее find). `measurement_unit` — продажная единица, не путать с единицей габаритов (§5.6).

**7. ЦЕНА.** 🟢 В источнике цены НЕТ — **не синтезировать** (§5.6). Дефолт — `salePrices` пуст/0, поле
**только при создании**, при переимпорте не затирать (1С/вебмастер могли проставить). 🟡 Если нужен прайс
— из реального источника заказчика через фильтр/событие в нужный `pricetype`. Эталон цену не задаёт (фильтр
`onecatalog_product_price`) — тот же принцип. **Решать с заказчиком (Q8).**

**8. Роль МойСклада: учёт/заказы или справочно?** 🟡 Стратегический вопрос. Если ведёт остатки/заказы →
цена/единицы/вес обязательны (полная интеграция). Если справочник → мягче. **Определить до старта.**

**9. Вес/габариты — единицы.** 🟢 Источник: **вес в граммах, размеры в мм** (подпись локализована;
`sizes.*_unit`, иначе мм/г). **`weight` бывает `null`** — не выдумывать. 🟡 Единица веса МойСклада
(вероятно **кг**) → **конвертер г→кг обязателен** (иначе 1.2 кг станут 1200). Габариты: нет 3 раздельных
нативных полей → **доп. поля «Длина/Ширина/Высота, мм»** (точные значения) + опционально расчётный
`volume`. **Подтвердить единицу веса и нативные поля габаритов/объёма.**

**10. Медиа — по URL или multipart.** 🟢 **Поправка/упрощение:** бинарный `media_files`-URL
**самодостаточен** — токен зашит в base64-хеш, файл доступен **без заголовка `X-API-Key`** (в эталоне
`Media::sideload` качает его обычным GET и получает max). Значит ваше опасение «max только с токеном →
обязаны качать сами» **снимается частично**: если МойСклад поддерживает загрузку картинки **по URL**
(`{url}`), можно отдать ему прямой max-URL OneCatalog — он скачает max сам, токен не нужен. Если URL-
загрузка не поддержана — fallback: наш сервис качает (URL без расширения → MIME по содержимому) и грузит
multipart/base64. 🟡 **Проверить, поддержан ли URL-upload и лимиты числа/размера картинок.** Трекинг
качества/дедуп `sha1(path#size)` — в нашей БД-проекции (платформа не помогает).

**11. Мульти-категория + иерархия.** 🟢 **Иерархия ЕСТЬ** (`/categories/` `parent_category`) → дерево
`productfolder`. `categories[]` — массив, но товар в МойСкладе в **одной** `productFolder` → основная
категория (листовая/первая) в `productFolder`, остальные → доп. поле-справочник «доп. категории». **Подтвердить
одиночность `productFolder`.**

**12. Rate-limit и батч.** 🟡 Платформенное — **проверить** числа (запросов/сек, параллельность, размер
батча). 🟢 Со стороны OneCatalog жёсткого лимита в тестах не было. Используйте **батч (массив в одном
запросе)** + троттлинг/бэкофф на 429 + очередь шагом ≥10. Порядок зависимостей (uom/country/productfolder/
определения attributes → товар → картинки) соблюдаем сами.

**13. `code` товара — роль.** 🟡 Платформенное. Рекомендация: `externalCode` = `public_id` (ключ),
`article` = артикул производителя, `code` — оставить автогенерации МойСклада (он в документах, уникален).
`public_id` в `code` не класть. **Подтвердить уникальность/роль `code`.**

**14. Аутентификация/тиражируемость.** 🟡 На старте — **приватный Bearer-токен** (один аккаунт ceram).
Vendor-приложение (маркет, OAuth-флоу) — отдельная фаза, если тиражируемо. **Подтвердить.**

**15. Поля «только при создании».** 🟢 Подтверждаю — при `PUT` НЕ затирать: цену (`salePrices`), статус/
архив, возможно `code` и ручные правки вебмастера в attributes. Идемпотентность ≠ «затереть заново»:
читаем существующий товар, сливаем поля (реализуем в нашем сервисе).

**16. Мультиязычность.** 🟢🟡 API отдаёт **один язык за запрос**. Для ceram — **`ru`, один язык на
старте**. МойСклад одноязычен; мультиязык = несколько аккаунтов/проекций (отдельная фаза). Эталон тоже
один язык за прогон. **Подтвердить.**

**17. Коллекции — доп. поле или productfolder.** 🟡 Дефолт — **доп. поле-справочник «Коллекция»**. Ветка
productfolder — если нужна навигация по коллекциям в учёте (смешивать с деревом категорий нежелательно).
Поля коллекции из эмбеда `collections[]` (`/collections/{slug}/` = 500, не дёргать). **Подтвердить.**

**18. Удаление/архивация.** 🟢 Импорт по эталону **не удаляет** — только добавляет/обновляет. Пропавший
из набора товар — осознанная политика (ничего/архив), не удаление по умолчанию. **Подтвердить политику.**

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id` → `externalCode`** (нативно), `article` отдельно (только непустой),
   `code` — автогенерация. Никогда не по name/slug (§5.1).
2. **Справочники — поиск по имени (label), не по слагу:** productfolder по имени в пределах родителя;
   определение attribute по `specification_id`/имени; country/uom по имени. Стабильный ключ маппинга —
   `specification_id`/category id, не переводимый label (в эталоне был баг дублей «Толщина» из транслита).
3. **Характеристики → attributes** (НЕ characteristics); многозначные (`multitude`) → multi-value
   attribute/справочник (см. нюанс выше); ручной маппинг по `specification_id`, строгий режим.
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие =
   false**; attribute типа boolean. (`multitude=false` у boolean.)
5. **Медиа:** URL самодостаточен (токен в хеше) → URL-upload прямым max-URL если поддержан, иначе
   download+multipart (MIME по содержимому, URL без расширения); дедуп `sha1(path#size)` и трекинг
   качества — в нашей БД-проекции; галерея по имени файла, не по URL.
6. **Не выдумывать цену** (`salePrices` пуст, create-only); `weight` бывает `null`; статус/архив — только
   при создании.
7. **Единицы:** конвертер **г→кг** (вес) и **мм→** (габариты в доп. поля); `measurement_unit` (м²/шт) →
   нативный `uom`, не путать с габаритами.
8. **Категории — нативный `productfolder` дерево** из `/categories/` (`parent_category`, 1:1).
   **`/collections/{slug}/` = 500 — не дёргать**, всё из эмбеда `collections[]`. **Справочники —
   `limit=1000` + сверка `meta.counts`.**
9. **Прочий payload** (`areas_per_package`, `product_shape`, alt/описание картинок, детали бренда/коллекции,
   `upsells/cross_sells`, витринные страницы/SEO/статусы) — **слой внешнего сайта**, не МойСклад. В нашем
   сервисе — через событие/хук, если витрина потребляет.
10. **SemVer + CHANGELOG** (Keep a Changelog) для нашего сервиса; батч-операции; деградация §5.5 — битый
    товар/429 логируем и пропускаем.
