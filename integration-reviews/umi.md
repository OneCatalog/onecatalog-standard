# Интеграционный план: UMI.CMS

Платформо-независимый стандарт (`INTEGRATION-STANDARD.md`, эталон — WooCommerce)
перенесён на **UMI.CMS** (российская CMS, объектная модель данных, PHP).
Цель — модуль импорта товаров из OneCatalog Wiki API: идемпотентно, в фоне, без
оформительских полей сайта в ядре.

Все внешние контракты §2 и инварианты §5 стандарта **неизменны** (API, public_id,
трекинг качества картинок, конверсия единиц, «не выдумывать цену», boolean→false).
Меняется только слой «как это лечь в UMI» — он и описан ниже.

**Уровень уверенности по UMI:** средний. Объектная модель (`umiObjectsCollection`,
`umiObjectType`, `umiHierarchy`, поля-`relation`/справочники), событийная модель
(`umiEventListener`/`events.php`) и cron (`cron.php` + слушатель события `cron`)
подтверждены по docs.umi-cms.ru. Конкретные имена точек вызова при сохранении
объекта, поведение `commit()`, AJAX в админке, multidomain-выборки —
помечены «(не уверен, проверить)».

> Терминология UMI, используемая ниже:
> **тип объекта / тип данных** (`umiObjectType`) — аналог «схемы»; задаёт набор
> полей (`umiField`). **Объект** (`umiObject`) — экземпляр типа. **Справочник
> (`guide`)** — это тип объекта, чьи объекты используются как значения
> поля-связи (аналог «таксономии/словаря»). **Элемент иерархии**
> (`umiHierarchyElement`) — страница в дереве сайта; у каждой страницы есть
> привязанный объект (`getObject()`), хранящий поля. **Модуль `catalog`** —
> товары как страницы-объекты в дереве разделов; **модуль `emarket`** — магазин
> (цены/корзина/заказы) поверх каталога.

## Маппинг сущностей и реализация

### Сводная таблица соответствий (§3 стандарта → UMI)

| OneCatalog | Абстракция | Реализация в UMI.CMS |
|---|---|---|
| product | Товар | Страница `catalog` (`umiHierarchyElement`) типа объекта `catalog object` + её `umiObject` с полями |
| public_id | Внешний ключ (идемпотентность) | **Отдельное кастомное поле** `oc_public_id` (string) на типе товара. Поиск через `Selector`/`umiSelection` по значению поля. НЕ slug, НЕ имя |
| article | Артикул производителя | Кастомное поле `article` (или штатное поле артикула типа товара, если есть) — только если пришёл |
| options[] (характеристики) | Характеристика | Поле-`relation` на тип-справочник «Характеристика» + объекты-значения; ИЛИ отдельное поле на тип товара (см. ниже — две стратегии) |
| categories[] | Категория | **Разделы** иерархии `catalog` (`umiHierarchyElement` тип `catalog category`), вложенность по дереву |
| brand | Бренд | Поле-`relation` на справочник «Бренды» (тип объекта) — настраиваемо |
| country | Страна происхождения | Поле-`relation` на справочник «Страны» — настраиваемо |
| collections[] | Коллекция | Справочник «Коллекции» (relation) ИЛИ раздел иерархии ИЛИ отдельный тип-страница — настройка |
| tags[] | Теги | Поле-`relation` на справочник «Теги» (вкл/выкл, поиск по имени) |
| images_urls/files | Медиа | Поля типа `img_file` (обложка) + `relation`/множественное `file`-поле или галерея модуля «Фотогалереи» |
| measurement_unit, sizes, areas_per_package, product_shape, … | прочее | Через событие `oc_product_imported` (сайтовый слой), §8 |

### 1. Товар → объект каталога

В UMI товар каталога — это **страница в дереве** (`umiHierarchyElement`) с
привязанным объектом нужного **типа** (`umiObjectType`, унаследованного от базового
типа товара модуля `catalog`). Создание:

```php
$hierarchy = umiHierarchy::getInstance();
// $catalogObjectHierType — id иерархического типа "catalog object" (объект каталога)
// $parentSectionId — раздел-категория (создаётся ниже)
$pageId = $hierarchy->addElement($parentSectionId, $catalogObjectHierType, $name /* h1 */, $name /* title */);
$element = $hierarchy->getElement($pageId);
$object  = $element->getObject();
$object->setValue('oc_public_id', $publicId);
$object->setValue('article', $article);
// ... остальные поля
$object->commit();   // (не уверен: нужен ли $element->commit() и переиндексация — проверить)
```

- **Тип объекта товара.** Модуль должен либо использовать существующий тип «Товар»
  магазина клиента, либо при установке создать собственный тип-наследник и добавить
  на него служебные поля (`oc_public_id`, `article`, поля характеристик). Создание
  типа/полей — через `umiObjectTypesCollection` + `umiFieldsCollection`/`umiField`
  с `setGuideId()` для полей-связей. (не уверен в точных сигнатурах добавления поля —
  проверить `umiFieldsCollection::addField`.)
- **Базовые поля при создании vs апдейте.** Имя/`h1`/title — всегда. Статус
  публикации (`is_active`/публикация страницы) — **только при создании** (§5.6); при
  переимпорте не трогать (вебмастер мог снять с публикации).
- **Цена НЕ задаётся** (§5.6) — в источнике её нет. В UMI цена живёт в `emarket`
  (поле `price` объекта или таблица цен). Ядро её не пишет; подстановка — через
  событие/фильтр.

### 2. Идемпотентность (§5.1)

- Перед импортом: найти товар по полю `oc_public_id == public_id` через `Selector`:

```php
$sel = new selector('objects');
$sel->types('object-type')->name('catalog', 'object'); // тип товара
$sel->where('oc_public_id')->equals($publicId);
$sel->option('return')->value('id');
$sel->limit(0, 1);
$found = $sel->result(); // нашли → обновляем, нет → создаём
```
(не уверен в точном fluent-API `selector` конкретной версии — проверить; в старых
версиях — `umiSelection`/`umiSelectionsParser`.)

- **Справочники (бренд/страна/коллекция/теги/характеристики) — find-or-create по
  ИМЕНИ (label), не по слагу** (§5.1, критично). В UMI у объектов справочника
  есть `name`; искать `selector ... where('name')->equals($label)` без учёта
  регистра. Слаг (`alt-name`/`name` страницы) генерируется самой CMS — на разных
  инстансах транслит разный → поиск по слагу плодит дубли. То же для разделов-категорий.
- **Легаси-фолбэк.** Если на проекте идентификатор раньше лежал в другом поле
  (напр. артикуле) — поддержать поиск по нему как фолбэк и мигрировать в `oc_public_id`.

### 3. Характеристики (options[]) — самый спорный узел

Две стратегии (выбор настройкой, аналог §3 «тип объекта + цель»):

**(А) Справочник на характеристику (рекомендуется как «по умолчанию»).** Для каждой
`specification_label` — свой тип-справочник, чьи объекты = значения (`text`-термы).
На типе товара — поле-`relation` (множественное) к этому справочнику. Минус:
динамическое создание типов/полей под каждую новую характеристику — тяжёлая операция
(меняет схему БД, требует прав разработчика, см. блокеры).

**(Б) Один универсальный «склад» характеристик.** Тип-справочник «Характеристика»
с полями `spec_id`, `label`, `value`; на товаре — множественная связь. Гибко, не
плодит типы, но фасетная фильтрация в штатном каталоге UMI по такому полю — слабее.

- **`numeric` → `numeric_option`** пишется в числовое поле/значение терма.
- **`boolean` (§2.2/5.6):** `specification_option_name = null`, значение в
  `bool_option`. Без явного правила «true/false → имя терма» значение теряется. В
  ручном маппинге (см. ниже) задаются объекты-значения справочника для true и false;
  `null`/отсутствие/`false` → **false**.
- **Ручной маппинг (строгий режим, §5.4):** отдельная admin-страница
  «specification_id → существующий тип/поле/справочник UMI». Ключ привязки —
  стабильный `specification_id`, а не переводимый `label`. При ВКЛ импортируются
  ТОЛЬКО сопоставленные; при ВЫКЛ — «по имени/создать».

### 4. Категории → разделы иерархии

`categories[]` создаются как разделы (`umiHierarchyElement` типа `catalog category`)
через `addElement($parentId, $categoryHierType, $label)`. Идемпотентность —
find-or-create раздела **по имени** в пределах родителя. Иерархия из payload плоская
(`categories[]` — массив без вложенности?) — **уточнить** (вопрос №4). Товар
привязывается размещением страницы в соответствующем разделе; для нескольких
категорий — основная как родитель + дополнительные через поле-связь «доп. разделы»
или модульный механизм «товар в нескольких разделах» (`catalog` это поддерживает
через служебную таблицу/поле — не уверен в имени, проверить).

### 5. Бренд / страна / коллекция — «тип объекта + цель» (§3, обязательная настройка)

Для каждой — настройка хранилища:
- **relation на справочник** (по умолчанию для бренда/страны) — объект-значение
  find-or-create по имени, связь на товаре.
- **раздел иерархии** (вариант для коллекции — коллекция как раздел дерева).
- **отдельный тип-страница** (CPT-аналог — для коллекции с богатыми полями: `link_3d`,
  `link_official_site`, логотип). UMI «CPT» = свой тип объекта + свой иерархический тип.

Коллекция приходит **в эмбеде товара** (`collections[]`), `/collections/{slug}/`
отдаёт 500 — все поля брать из эмбеда (§2.1). После привязки — событие
`oc_collection_imported`.

### 6. Теги

Если включено — справочник «Теги», relation, find-or-create по имени (`tags[].title`).

### 7. Изображения → поля-картинки/галерея (§5.3, §2.3)

- Обложка → поле `img_file` объекта товара. Галерея → множественное `file`-поле
  ИЛИ объекты модуля «Фотогалереи» (`photoalbum`), привязанные к товару.
- **URL без расширения** (`/media_files/<base64hash>`) → скачивать вручную, MIME по
  содержимому, класть во временный файл и регистрировать через файловую подсистему
  UMI (`umiImageFile`/`umiFile` из временного пути). НЕ полагаться на загрузчик «по
  URL с расширением».
- **Размеры min|middle|max**: с токеном — `max`; без — `min`. **Трекинг качества:**
  рядом с файлом хранить выбранный размер (доп. поле/служебная мета объекта-вложения
  или таблица модуля). Повторный импорт: лучше доступен — перекачать и заменить.
- **Галерея идемпотентна по имени файла** (подписи в URL меняются), не по URL.
- **Дедуп по контент-ключу** (`hash(path#size)` из base64-префикса). В UMI нет
  встроенной «медиатеки с метаданными» как в WP → **собственная таблица модуля**
  `oc_media_index(content_key, umi_file_path/object_id, size)`; перед загрузкой искать
  по ключу, нашли — переиспользовать путь без скачивания. **Общие файлы не удалять**
  при апгрейде качества (сломает другие товары).

### 8. Единицы измерения (§5.6, обязательно)

Конвертер `Units` переносится 1:1 (он самодостаточен — таблицы коэффициентов + алиасы
RU/EN, см. эталон `class-units.php`). Источник: вес — граммы, размеры — мм
(`sizes.*_unit`, иначе мм/г). Цель — единицы поля магазина UMI. Отличие от WC: в UMI
нет глобальной опции `woocommerce_weight_unit` → **целевые единицы задаются настройкой
модуля** (`oc_weight_unit`, `oc_dimension_unit`) или берутся из конфигурации полей.
`measurement_unit` (м²/шт) товара — это НЕ вес/размер; это поле «единица товара»,
заполняется сайтом через событие (нет прямого аналога в ядре, §2.2).

## Фоновая очередь, медиа, настройки, i18n, жизненный цикл

### Фоновая очередь (§6)

- **Механизм UMI: `cron.php`** + слушатель события `cron` в `events.php` модуля
  (`new umiEventListener('cron', 'onecatalog', 'processQueue')`). Это
  подтверждённый штатный путь фоновых задач в UMI (аналог Action Scheduler / WP-Cron).
- **Очередь как собственная таблица модуля** `oc_import_queue(id, identifier, status,
  attempts, batch, log, ts)`. Список выбранных public_id режется на порции по «шагу
  импорта» (минимум 10, §6). Cron-обработчик берёт N pending, импортирует, пишет лог.
- **UI-степпер (фолбэк/прогресс):** так как `cron.php` запускается по расписанию
  хостинга (минуты/часы), для отзывчивого UX — **AJAX-степпер из админки**
  (`module=onecatalog&method=...&ajax`), обрабатывающий по порции за запрос и
  опрашивающий статус. (не уверен в штатном AJAX-роутинге админ-методов модуля —
  проверить `__call`/`ajax`-суффикс; точно есть обычные публичные методы модуля.)
- **Синхронный фолбэк** без планировщика — как требует §6.
- ⚠️ Риск: на shared-хостинге `cron.php` могут запускать редко/не запускать вовсе →
  основным сделать **AJAX-степпер**, cron — вторичным. (вопрос №6.)

### Медиа

См. §7 выше: собственная таблица индекса медиа, ручное скачивание + MIME, трекинг
качества, дедуп по контент-ключу, общие файлы не удалять.

### Настройки (§7)

Хранилище — **настройки модуля** (реестр UMI / `regedit`-ветка модуля
`//modules/onecatalog/...`), редактируются на странице конфигурации модуля.
Минимум (как в стандарте): токен (env приоритетнее поля), язык (`en/ru/ar/zh/kk`,
только из списка), шаг импорта (кламп ≥10), статус новых (только при создании),
вкл/выкл коллекций + «тип объекта + цель», бренд/страна/коллекция «тип объекта +
цель», теги вкл/выкл, ручной маппинг характеристик (вкл/выкл + страница карты,
true/false для boolean), целевые единицы веса/размера. Валидация и кламп —
обязательно.

### Точки расширения (§8) — события UMI

Универсальное ядро публикует **кастомные точки вызова** через
`umiEventsController::getInstance()->callEvent('oc_...', $params)`, на которые сайт
вешает `umiEventListener` в своём `events.php` (mu-плагин-аналог = отдельный модуль
сайта). Аналог `do_action`/`apply_filters`:

- `oc_before_import_product(payload, existingId)`
- `oc_product_imported(objectId, payload, report)` ← **сайт дозаполняет** ед. изм.,
  `areas_per_package`, `product_shape`, alt картинок, детали бренда, upsells — всё,
  чего нет в маппинге ядра. Полный payload в событие.
- `oc_collection_imported`, `oc_brand_imported`, `oc_country_imported`,
  `oc_queue_enqueued`.
- **Фильтры** (меняют значение): UMI-события возвращают изменённые `$params` (через
  ссылку/возврат) — base URL, токен, аргументы запроса, язык, шаг, payload, SKU,
  **цена** (по умолчанию не задаётся), порядок размеров картинок, дедуп медиа, файлы
  галереи, справочник характеристик, флаги импорта и карта маппинга, origin виджета.

⚠️ В UMI событие-«фильтр» и событие-«экшен» — один механизм (`callEvent`); чтобы
«менять значение», обработчик правит переданный по ссылке `umiEventPoint`-параметр и
ядро читает его обратно. Это работает, но менее идиоматично, чем WP `apply_filters`.

### i18n (§9)

- Исходные строки — **английские**, единый text-domain. На UMI родной механизм —
  **языковые файлы модуля** (`i18n.php`/`lang/`-словари, `getLabel()`/`__()` UMI).
- Строки фронтовых JS (пикер, степпер) передавать из бэкенда локализованными, не
  хардкодить в JS.
- ⚠️ Расхождение со стандартом: §9 говорит про POT/PO/MO (gettext) — это WP-специфика.
  В UMI это **языковые PHP-массивы модуля**. Принцип (EN-исходник + перевод) сохранён,
  формат — родной UMI. (Соответствует памяти проекта: «строки в коде англ + перевод».)

### Жизненный цикл (§10)

- **Манифест модуля UMI** (`__module`/инсталлятор): версия, требования (версия
  UMI.CMS, наличие модулей `catalog` и — для цены — `emarket`), путь языковых файлов.
- **Install/update** (метод установки модуля): создать служебный тип объекта
  (или поля на существующем типе товара: `oc_public_id`, `article`, поля
  характеристик/связей), создать справочники (Бренды/Страны/Коллекции/Теги), создать
  таблицы `oc_import_queue` и `oc_media_index`, зарегистрировать `events.php`,
  настройки по умолчанию.
- **SemVer + CHANGELOG** (Keep a Changelog), как в памяти проекта; поднимать версию
  перед сборкой дистрибутива; новые UI-строки — переводить.
- **Активация не падает без `emarket`** — функции цены проверяют наличие модуля;
  каталог-импорт работает и без магазина (товары как объекты каталога).
- **Миграция** при смене схемы ключа (старое поле → `oc_public_id`, флаг
  «миграция выполнена»).

### Права (§11)

Контроллер приёма списка ID и admin-методы — под проверкой прав UMI (`permissions`
модуля: право `onecatalog/import`) + CSRF-токен админки (`csrf`/`secret` UMI) вместо
WP-nonce.

## ❓ Вопросы и сомнения (ответить ДО старта)

1. **Тип объекта товара.** Импортировать в **существующий** тип «Товар» магазина
   клиента (и просто добавить на него поля `oc_public_id`/`article`/характеристики)
   или создавать **собственный тип-наследник**? Первое — меньше дублей и совместимо
   с витриной/`emarket`, но мы навязываем поля чужому типу; второе — изолированно,
   но товары не «увидит» штатный магазин. (Влияет на всю схему.)
2. **Цена и `emarket`.** Цены в источнике нет (§5.6) — ядро её не пишет. Но в UMI
   товар без цены в `emarket` может не показываться/не продаваться. Нужна ли вообще
   интеграция с `emarket` (корзина/заказы), или это «каталог-витрина» без продаж?
   Если продажи нужны — кто и как проставляет цену (событие `oc_product_price`?).
3. **Характеристики: стратегия А (тип-справочник на каждую характеристику) или Б
   (один универсальный склад)?** А — лучше для фасетной фильтрации штатного каталога,
   но **динамически меняет схему БД** (создание типов/полей на лету — операция уровня
   разработчика, см. блокеры). Б — безопаснее, но фильтрация слабее. Что важнее:
   фасеты или устойчивость?
4. **Множественные категории.** `categories[]` товара — это плоский список или
   иерархия? Товар может лежать в нескольких разделах одновременно — использовать
   штатный механизм «товар в нескольких разделах» модуля `catalog` (как он называется
   в текущей версии?) или поле-связь «доп. категории»? Какая категория — основной
   родитель страницы?
5. **Версия UMI.CMS и API-стиль выборок.** Какая версия (это влияет на fluent
   `selector` vs `umiSelection`/`umiSelectionsParser`, на сигнатуры
   `addField`/`setGuideId`, на доступность кастомных `callEvent`). Commercial-сборка
   или PRO? Есть ли доступ к коду модулей (`classes/modules/...`) для install-хуков?
6. **Фон: cron или AJAX-степпер как основной?** Запускается ли `cron.php` на хостинге
   клиента регулярно (раз в N минут)? Если нет/редко — основным делаем AJAX-степпер,
   cron вторичным. Какой реальный объём импорта (десятки/тысячи товаров)?
7. **Iframe-пикер (§2.4).** Открывается ли админка UMI по HTTPS и допускает ли
   встраивание стороннего iframe (`tools.onecatalog.net/picker.html`) с `postMessage`?
   Нет ли CSP/`X-Frame-Options`, ломающих виджет? Если запрещено — фолбэк «импорт по
   списку public_id вручную» (стандарт это допускает).
8. **Медиа-хранилище.** Использовать модуль «Фотогалереи» (`photoalbum`) для галереи
   товара или множественное `file`/`relation`-поле на объекте? Где хранить трекинг
   качества и контент-ключ — служебная таблица модуля (предлагается) или поля
   объекта-вложения (если у файлов UMI есть произвольные метаданные)?
9. **Мультиязычность.** Сайт многоязычный (домены/языки UMI)? Если да — импорт
   создаёт версии страницы товара на каждый язык (`addElement(..., $domainId,
   $langId)` и перевод полей через повторный запрос API с другим `lang`), или один
   язык? Стандарт качает по одному `lang` за импорт — мультиязык = отдельный проход.
10. **Лимиты API и пагинация справочников (§2.1).** Подтвердить: справочник
    (`/specifications/` и т.п.) тянем одним запросом с высоким `limit`, сверяясь с
    `meta.counts`; на каком инстансе работаем (URL базового API переопределяем)?
11. **Артикул.** Есть ли у типа товара клиента **штатное** поле «Артикул», или
    заводим своё `article`? Уникальность артикула на стороне UMI обеспечивается? При
    коллизии — не перезаписывать (§5.2).
12. **Права/роли.** Кто запускает импорт (роль), и нужно ли отдельное право модуля
    `onecatalog/import` в системе прав UMI?

## 🚫 Сложно / невозможно на UMI.CMS

1. **Динамическое создание типов объектов и полей под характеристики (стратегия А)
   — тяжело и рискованно.** В WooCommerce глобальный атрибут `pa_*` + термины
   создаётся дешёвой API-операцией. В UMI добавление **поля** на тип — это изменение
   схемы (новая колонка/служебная запись `umiField` + кэш типов), а создание нового
   **типа** под каждую новую характеристику — операция install-уровня. Делать это на
   лету в фоновой очереди — медленно и чревато гонками/порчей кэша типов.
   **Обход:** предпочесть стратегию Б (универсальный склад характеристик), либо
   создавать недостающие типы/поля **только при включённом ручном маппинге и заранее**
   (на admin-странице маппинга, вне очереди), а в очереди — только привязывать
   значения. Сбрасывать кэш типов после изменения схемы.

2. **Нет единой «медиатеки с произвольными метаданными» как в WP.** Дедуп по
   контент-ключу и трекинг качества (§5.3) в WP опираются на `post_meta` вложения.
   В UMI файлы — это значения полей/объекты «Фотогалерей», без удобного слоя
   произвольной меты на файл. **Обход:** собственная таблица модуля
   `oc_media_index(content_key, path/object_id, size)` — индекс существует независимо
   от того, к скольким товарам привязан файл; общие файлы при апгрейде качества не
   удаляем.

3. **«Фильтры» как у WP (`apply_filters`) — не нативны.** Событийная модель UMI
   (`callEvent` + `umiEventListener`) реагирует, но «вернуть изменённое значение»
   требует правки параметра по ссылке в `umiEventPoint`. **Обход:** ядро после
   `callEvent` перечитывает мутируемый параметр; задокументировать контракт. Работает,
   но менее очевидно, чем `apply_filters`.

4. **Цена/витрина: товар без `emarket` может «не существовать» как продаваемый.**
   Стандарт запрещает выдумывать цену (§5.6). На UMI товар каталога без цены `emarket`
   — это просто страница-объект; для продаж нужна цена, которой нет. **Обход:** цена —
   через событие `oc_product_price` (сайт подставляет из своего источника), как
   «поле только при создании»; либо режим «каталог-витрина без продаж». Решается
   вопросом №2.

5. **Фоновая надёжность зависит от хостинга (`cron.php`).** В WC есть Action Scheduler
   (БД-очередь, async-runner). В UMI штатный фон — `cron.php` по расписанию хостинга;
   на shared-хостинге может не вызываться → очередь «зависнет». **Обход:** основной
   путь — **AJAX-степпер** из админки (детерминирован, не зависит от cron), `cron` —
   как дополнительный авто-добор; синхронный фолбэк для маленьких пакетов (§6).

6. **iframe-пикер может быть заблокирован админкой/CSP.** Не гарантировано, что
   админка UMI допустит сторонний iframe + `postMessage` (CSP/`X-Frame-Options`,
   sandbox). **Обход:** §2.4 допускает работу без пикера — импорт по списку
   `public_id` (вставка/загрузка списка), пикер опционален.

7. **Создание поля-связи на справочник с нужной кардинальностью** (множественные
   характеристики/теги/доп.категории) — поддерживается (`setGuideId()` +
   `getIsMultiple()` «составной тип»), но **точные сигнатуры и поведение `commit()`/
   переиндексации различаются по версиям** (не уверен — проверить на целевой версии).
   **Обход:** инкапсулировать всю работу со схемой в install/update модуля и в слой
   `Taxonomies`-аналога, протестировать на 1–3 товарах до пакетного импорта (порядок
   прототипирования из §1 стандарта).

8. **Мультиязычный импорт — отдельная сложность.** UMI хранит языковые версии как
   связанные страницы; один проход API = один `lang`. Полноценный мультиязык требует
   N проходов и аккуратной привязки версий. **Обход:** MVP — один язык (настройка);
   мультиязык — отдельная итерация (вопрос №9).
