# Ответы Magento-разработчику — OneCatalog Import

Ответы на [magento2.md](magento2.md) (22 вопроса + 12 блокеров). Маркировка:
🟢 — авторитетно (проверено на живом Wiki API / решение эталонной реализации WooCommerce
`onecatalog-import` v1.7.0); 🟡 — продуктовое решение/зависит от инсталляции, даю рекомендацию +
что подтвердить.

> ⚠️ Эталонная витрина `ceram-online` — headless WordPress+Nuxt, не Magento. Поэтому вопросы про
> «эту Magento-инсталляцию» (AMQP, store views, объём, CSP) — это про новое развёртывание; по ним
> рекомендация + «подтвердить». API/логику стандарта отвечаю точно.

## Главная рекомендация по Magento (читать первой)

🟢 **Ручной маппинг характеристик должен быть ДЕФОЛТНЫМ режимом на Magento** (в отличие от WooCommerce,
где дефолт — авто-создание по названию). Причина — ваш же блокер 3: динамическое создание EAV-атрибутов
в рантайме дорого и опасно (кэш, attribute set, реиндекс, гонки consumers). Эталон специально сделал
режим строгого ручного маппинга по стабильному `specification_id`. На Magento:
1. Один раз тянем `/specifications/` (`limit=1000`), **предсоздаём** все нужные EAV-атрибуты + опции
   data-патчем или одноразовым admin-действием (не на лету по каждому товару).
2. Импорт товара только **проставляет option_id** на заранее заведённых атрибутах — никаких
   `AttributeRepository::save()` в горячем пути.
Это снимает блокер 3 и делает импорт масштабируемым. Авто-создание-по-имени оставить как опцию «для
малых каталогов», но не дефолт.

## Ответы на вопросы

**1. SKU при отсутствии `article`.** 🟢 Эталон: SKU = `article`, если пришёл, иначе пусто (в WC можно).
В Magento пусто нельзя → **фолбэк SKU = `public_id`** (например `OC.IND.1`, или с префиксом `OC-…` если
боитесь коллизий с реальными артикулами). Идемпотентность ВСЕГДА по `public_id` (атрибут/таблица), НЕ
по сгенерированному SKU. При коллизии `article` с чужим товаром — не перезаписывать чужой SKU.
Запрещать импорт без article — НЕ надо (потеряете товары). Рекомендация: `article ?: public_id`.

**2. `public_id`: EAV-атрибут vs своя link-таблица.** 🟡 Для каталога в сотни–тысячи (типично для
плитки) хватит индексированного `varchar`-EAV-атрибута + поиск через `SearchCriteria`. Своя
`onecatalog_product_link` (`public_id` UNIQUE → `entity_id`) — оправдана от ~десятков тысяч SKU или при
частых прогонах. Эталон обходится одной метой. **Рекомендация: начать с EAV-атрибута; link-таблицу —
если объём большой** (подтвердить объём, см. Q17/общий чек-лист).

**3. Тип товара.** 🟢 **Только `simple`, configurable НЕ нужен.** OneCatalog не имеет вариаций: один
`public_id` = один готовый товар; `options[]` — характеристики-фасеты (цвет/формат/поверхность), не
SKU-варианты. Эталон импортирует `WC_Product_Simple`. Никаких `is_configurable`.

**4. Attribute set.** 🟡 **Выделенный set `OneCatalog`** — да, рекомендую (изоляция EAV-атрибутов
характеристик от Default, проще чистить/мигрировать). Это не требование стандарта, а гигиена Magento.

**5. Один атрибут на `specification_label`?** 🟢 В эталоне ключ привязки — стабильный
`specification_id` (не label): **одна характеристика OneCatalog = один атрибут**. Факт API: в текущем
инстансе все 13 характеристик имеют **различные** label (Цвет, Форма, Материал, Толщина, …) — коллизия
«разные spec_id с одинаковым именем» не наблюдается. В авто-режиме (по label) теоретически склеит, но
на практике этого нет. **Используйте ручной маппинг по `specification_id`** (он и так дефолт на Magento,
см. главную рекомендацию) — вопрос склейки снимается.

**6. `numeric` как фасет.** 🟢 Факт API: **`numeric`-характеристик в этом инстансе НЕТ** (все 13 —
text/boolean). Так что это задел. Когда появятся — `decimal`-атрибут (фильтр-диапазон в layered nav,
`is_filterable_for_range`), не текст и не опции. Эталон numeric не спецобрабатывает.

**7. Категории: дерево или плоско?** 🟢 **Стройте дерево — иерархия в API ЕСТЬ.** Факт API: у товара
`categories[]` плоский (`{id, menutitle, slug}`), но справочник **`/categories/` отдаёт
`parent_category: {id, slug, menutitle}`** (или `null` = корень). Пример: id=3 «Керамогранит» →
parent id=1 «Плитка» → `null`. Алгоритм: тянем `/categories/?limit=1000`, строим карту `id→parent`,
при привязке резолвим/создаём всю цепочку под выбранной root-категорией импорта (find-or-create по
**name**, регистронезависимо, не по `url_key`). 🟡 root-категорию импорта вынести в настройку (store
root по умолчанию). Эталон WC строит плоско — Magento может **лучше**.

**8. Бренд/страна/коллекция — дефолты.** 🟢/🟡 Эталонные дефолты (адаптировать под Magento):
- **бренд → EAV-атрибут `select`** (в WC дефолт — нативная таксономия `product_brand`, которой в
  Magento нет → атрибут). Опция «категория» для бренда-как-страницы.
- **страна → свой `select`-атрибут по label** (НЕ `country_of_manufacture`, см. Q9).
- **коллекция → категория** (богатые поля; своя сущность+grid не нужна на старте).
Все три — настраиваемы «store_as + target», как в эталоне.

**9. `country_code` / `country_of_manufacture`.** 🟢 **ВАЖНО: формат не совпадает.** Факт API:
OneCatalog `country_code` — **ISO 3166-1 alpha-3** (`IND`, `ESP`, `ABW`…), а Magento
`country_of_manufacture` — **alpha-2** (`IN`, `ES`…). Прямой маппинг НЕВОЗМОЖЕН без конверсии alpha-3→
alpha-2. Варианты:
- **(рекомендую) свой `select`-атрибут `onecatalog_country`**, опции find-or-create по `menutitle`
  (label) — просто, без ISO-возни, покрывает любые страны источника (есть и `/countries/` справочник
  в том же формате).
- Либо `country_of_manufacture` + **таблица alpha-3→alpha-2** (IND→IN, ESP→ES…) — но страны вне
  ISO-списка Magento выпадут. Лишняя сложность. **Дефолт — свой атрибут по label.**

**10. Габариты (`length/width/height`).** 🟢/🟡 В ядре Magento их нет (только `weight`). Эталон пишет
их (в WC есть нативные length/width/height). На Magento: **завести свои decimal-атрибуты**
(`onecatalog_length/width/height`) патчем и заполнять в импортёре — если габариты реально нужны (для
плитки — да, размер 600×1200 важен). Если нужны для расчёта доставки — использовать `ts_dimensions_*`
(MSI Shipping). 🟡 Подтвердить: нужны ли габариты как данные (рекомендую да) и для доставки ли.

**11. Единицы веса/размеров магазина.** 🟢 OneCatalog: вес — **граммы**, размеры — **миллиметры**
(подписи локализованы: «мм», вес/`weight_unit` бывают `null`). В Magento нет ядрового
`weight_unit` → **целевую единицу берёте из настройки модуля** (рекомендую дефолт кг + см, или г + мм
если так удобнее витрине — на ваш выбор). Конвертер: таблица коэффициентов к базе (вес→г, длина→мм) +
нормализация локализованной подписи; единица источника из `sizes.*_unit`, иначе мм/г; переопределяемо.
Коэффициенты (эталонные): вес к г — `g=1, kg=1000, oz=28.3495, lb=453.592`; длина к мм —
`mm=1, cm=10, m=1000, in=25.4`.

**12. MSI / наличие.** 🟢 Факт API: есть `in_stock: true/false`, **количества (qty) в источнике НЕТ**
(никаких qty-полей). Рекомендация: писать `SourceItem` на `_default` source: `status = in_stock ? 1:0`;
qty — эвристика, т.к. источник её не даёт → **при in_stock ставить условное qty>0** (например 100 или
из настройки), при out — 0; либо включить backorders. 🟡 Подтвердить: нужен ли вообще stock при
импорте или только статус; на каком source. Если MSI отключён — legacy `cataloginventory_stock_item`.

**13. Цена = 0.00 / отображение.** 🟢/🟡 Эталон: цена НЕ задаётся (в WC товар без цены валиден); в
Magento `price` обязателен → **`0.00` при создании, не трогать при апдейте**, реальную цену даёт сайт
на `product_imported`. 🟡 Чтобы «нулёвки» не висели на витрине — рекомендую при создании ставить
`visibility = Not Visible Individually` ИЛИ `status = Disabled` до подстановки цены, и публиковать
после. Это продуктовое решение — согласовать с владельцем (синхронно ли сайт ставит цену).

**14. Мультиязычность (store views).** 🟡 Фаза 1 — **один язык** (один прогон, scope `default`/нужный
store). API отдаёт один `lang` за запрос; multi-store (значения per store view) + идемпотентность
опций между scope — отдельная крупная фаза (общий пробел стандарта, его отметили все агенты). Если
мультиязык нужен: резолв атрибутов/опций вести по `admin`-scope label, переводы писать на store view
отдельными прогонами. Подтвердить одноязычность на старте.

**15. Очередь: AMQP или DB-queue.** 🟡 Зависит от инсталляции. Поддержите оба: AMQP (RabbitMQ) если
есть, иначе DB-коннектор (`queue.xml` connection=`db`) + cron-consumer; синхронный фолбэк для малых
пакетов. Эталон (Action Scheduler) тоже по сути cron/async. **Дефолт — DB-queue + cron** (меньше
требований к окружению). Подтвердить наличие RabbitMQ.

**16. Статус/прогресс UI.** 🟡 REST-поллинг job-таблицы достаточно (как в эталоне: эндпоинт статуса +
лог последних N). SSE избыточен. N ~ 30–100 последних результатов (эталон — 100).

**17. Лимиты API.** 🟢 Жёсткого rate-limit на `X-API-Key` в тестах не наблюдалось (высокий объём
отрабатывал). Справочники брать одним запросом с `limit=1000` (проверено; `meta.counts` = общее число,
дефолтный лимит инстанса молча обрезает). Товары — по одному `/products/{public_id}/`. Ретраи/таймаут
(~40с) + skip на ошибке (§5.5). 🟡 Реальный объём — уточнить (влияет на шаг порции; для Magento
рекомендую шаг 20–50, а не 10).

**18. Iframe-пикер и CSP.** 🟡 adminhtml имеет CSP (`csp_whitelist.xml`) → нужен whitelist
`frame-src https://tools.onecatalog.net` (+ возможно `connect-src`) и проверка `event.origin`. Жёсткие
инсталляции могут запрещать внешний iframe. **Фолбэк (всегда работает) — «импорт по вставленному списку
public_id»** (стандарт §2.4 это допускает). Рекомендую сначала сделать ID-list, пикер — опционально.

**19. Где кнопка импорта.** 🟡 **Отдельная страница «Импорт из OneCatalog» в админ-меню** (+ пикер/
ID-list), НЕ mass-action на product grid (там локальные товары, а выбираем внешние). Согласен с вашим
выводом.

**20. Дедуп медиа: шарить файл или дублировать.** 🟢 Стандарт требует дедуп → **шарить один физический
файл** между товарами (ключ `sha1(path#size)`). Эталон именно так. ⚠️ Главное правило: **общий
(дедуплицированный) файл НЕ удалять при апгрейде качества/смене** — иначе «удалил у одного — пропало у
всех». В Magento работать аккуратно с `media_gallery` + `..._value_to_entity`; вести флаг shared/
счётчик ссылок в своей `onecatalog_media_quality`. Дублировать на каждый товар — проще, но против
стандарта; идём в шаринг.

**21. Индексация.** 🟡 На пакетах — индексаторы в `schedule`-режим + **реиндекс после батча**
(`indexer:reindex`), один `save()` на товар. on-save на 1000+ медленный. Согласен с вашим выводом —
это правильный Magento-паттерн.

**22. Удаление/снятие исчезнувших товаров.** 🟢 Стандарт и эталон — **только upsert (создать/обновить),
синхронизации удалений НЕТ**. Это сознательный пробел: источник может временно не отдать товар, и
автоудаление опасно. Рекомендация: на старте — только upsert; «disable пропавших» — отдельная
опциональная фаза (по полному списку `/products_ids/` сверять и `status=Disabled`, не delete). Не
закладывать удаление в ядро.

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id`** (атрибут/таблица), SKU = `article ?: public_id` — но искать
   ВСЕГДА по public_id, не по SKU.
2. **Справочники — поиск по label (регистронезависимо), не по слагу/`attribute_code`** (защита от
   дублей при разных транслитах; в эталоне был баг «Толщина» tolschina/tolshhina). Стабильный ключ —
   id OneCatalog (`specification_id`, category id, option id).
3. **Ручной маппинг по `specification_id`, строгий режим — ДЕФОЛТ на Magento** (главная рекомендация),
   атрибуты предсозданы. boolean: значения опций для true/false из маппинга.
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие
   = false**.
5. **Медиа:** дедуп по `sha1(path#size)`; общие файлы НЕ удалять при апгрейде; галерея по имени файла,
   не по URL.
6. **Не выдумывать данные:** цены нет → `0.00` только при создании, не перезатирать; статус — только
   при создании.
7. **Единицы:** конвертер g/mm → ед. магазина (настройка); `measurement_unit` (м²/шт) — продажная, не
   габарит; `areas_per_package`/`product_shape` — через `product_imported`.
8. **Категории:** дерево из `/categories/` (`parent_category`). **`/collections/{slug}/` отдаёт 500 —
   не дёргать**, всё в эмбеде `collections[]` товара.
9. **`/specifications/`, `/categories/`, `/countries/` — одним запросом `limit=1000`.**
10. **Прочий payload** (`measurement_unit`, `areas_per_package`, `product_shape`, alt/описание картинок,
    `description_text/foundation_year/website` бренда, `upsells/cross_sells` через
    `ProductLinkRepository`) — не в ядро, а в событие `onecatalog_product_imported` (полный payload).
11. **SemVer + CHANGELOG** (версия в `composer.json`), новые строки UI — в оба `i18n/*.csv`.

## Что подтвердить у владельца Magento-инсталляции (перед стартом)

- Объём каталога (EAV-атрибут vs link-таблица; размер порции) — Q2/Q17.
- Нужны ли габариты как данные и для доставки (`ts_dimensions_*` vs свои) — Q10.
- Целевые единицы веса/размеров магазина — Q11.
- Stock: писать ли наличие, на какой source, политика qty — Q12.
- Политика «нулёвок» (скрывать до цены?) и ставит ли сайт цену синхронно — Q13.
- Одноязычность (или multi-store фаза 2) — Q14.
- RabbitMQ есть? (AMQP vs DB-queue) — Q15.
- CSP разрешает внешний iframe (пикер) — Q18.
