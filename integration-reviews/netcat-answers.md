## Главное про данные (меняет Q4/Q5/Q6)

🟢 **Вариаций товара НЕТ вообще** — один `public_id` = один готовый товар. Характеристики — это факты/
фасеты, не варианты. Никаких «торговых предложений».

🟢 **Характеристики многозначны** — у товара «Цвет» = 2 значения, «Помещение» = **6**. Справочник
`/specifications/` отдаёт флаг **`multitude`** (true = многозначная; boolean-характеристики `false`).
Это **прямо отвечает на ваш Q5**: одиночное поле-список НЕ годится → нужна модель «многие-ко-многим»
(справочник + связь). Используйте `multitude` для выбора кардинальности.

🟢 **`numeric` в этом инстансе не используется** (все 13 характеристик text/boolean) — но код пишите
универсально (numeric → строка со значением + единица).

🟢 **Иерархия категорий ЕСТЬ** — у товара `categories[]` плоский, но `/categories/` отдаёт
`parent_category` (или `null` = корень). Меняет Q6: разделы NetCat можно строить деревом.

## Ответы на вопросы

**1. public_id: `ImportSourceID` или своё поле.** 🟢 Согласен — **собственное поле `OneCatalogPublicID`**
(индекс), не `ImportSourceID` (его может занять импорт из 1С — конфликт двух источников). Идемпотентность
ВСЕГДА по этому стабильному ключу, не по slug/имени. Легаси-фолбэк (искать в `ImportSourceID`/`ItemID` +
мигрировать) — верно. `public_id` в SKU/артикул НЕ класть.

**2. Чистый API создания/обновления объекта.** 🟡 Это NetCat-внутреннее (версионно) — **шаг 0
обязателен**. Контракт со стороны стандарта: нужен путь, который даёт **побочные эффекты ядра**
(индексация поиска, пересчёт связей). Ваша приоритизация верна: (1) внутренний механизм добавления
объекта компонента → (2) прямой INSERT в `MessageXX` + ручная эмиссия `addMessage/updateMessage` ТОЛЬКО
как осознанный фолбэк. Эталон пишет через нативный CRUD платформы (`wp_insert_post`/`wc_*`), не голым
SQL — держитесь того же принципа. **Подтвердить наличие стабильного метода по исходникам версии.**

**3. Версия/редакция NetCat.** 🟡 Инстанс-specific (`add_listener` v5.6+, набор событий, наличие
NetShop). **Подтвердить версию и присутствие модуля «Магазин».** Активация не должна падать без NetShop
(проверять наличие).

**4. Характеристики: динамические поля vs справочник.** 🟢🟡 **Рекомендация — справочник, как у вас в
блокере 1, и это правильный путь** (совпадает с рекомендацией для Magento/UMI). Причины авторитетные:
- многозначность (`multitude`, см. «главное про данные») требует «многие-ко-многим» — одиночное поле-
  список не подходит;
- динамически менять СТРУКТУРУ компонента (`ALTER MessageXX` + служебные таблицы полей + сброс кэша) на
  каждый новый `specification` — рискованно и плохо переносимо между версиями (вы верно это пометили).

**Решение:** отдельный **компонент-справочник «Характеристики»** (объект = «характеристика+значение») со
связью к товару. Справочники/значения **предсоздавать ДО прогона** (на admin-странице маппинга или пред-
сканом `/specifications/?limit=1000`), очередь только **привязывает значения** — схему в фоне не трогаем.
Ключ маппинга — стабильный `specification_id`, не переводимый `specification_label`.

**5. Множественные значения характеристики/категории.** 🟢 **Да, обязательно** — «Помещение» = 6
значений; категорий у товара может быть несколько. → модель «многие-ко-многим» (справочник+связь для
характеристик; см. Q6 для категорий). Одиночный enum не годится.

**6. Несколько категорий + иерархия.** 🟢 **Иерархия ЕСТЬ** (`/categories/` `parent_category`) → стройте
**дерево разделов** (`Subdivision`), родители раньше детей, find-or-create по **имени в пределах
родителя**. 🟡 Товар в N категориях: основной раздел = родитель страницы объекта, дополнительные —
штатным «товар в нескольких инфоблоках/разделах» (имя механизма сверить) ИЛИ доп. поле-связь «доп.
категории»/таблица связи. Эталон строит плоско — NetCat может **лучше** (нативное дерево разделов).
**Подтвердить целевую модель навигации.**

**7. Медиа-реестр для дедупа/трекинга.** 🟢 Согласен — **своя таблица** `nc_onecatalog_media
(content_key, size, file_ref, owner)`. У NetCat файл привязан к полю объекта, «медиатеки с произвольной
метой» как у WP-вложений нет → реестр обязателен (прямой аналог эталонной меты `_onecatalog_file_key`).
Дедуп по `sha1(path#size)`; общие файлы при апгрейде качества НЕ удалять.

**8. Цена.** 🟢 **Подтверждаю §5.6: импорт цену НЕ задаёт** (в источнике её нет — это факт API, не
упущение). Поле `Price` — пустое/`0`, заполнение вручную/через событие. 🟡 Если валидация NetShop требует
цену/`PriceMinimum` — создавать товар **неактивным** (статус только при создании) до простановки, цену
подставляет сайт через событие/фильтр `onecatalog_product_price`. Не синтезировать. **Подтвердить
поведение валидатора NetShop при пустой цене.**

**9. Фильтры (меняющие значение).** 🟡 NetCat-внутреннее — **проверить, возвращает ли `event->execute()`
модифицированное значение**. Контракт: нужны точки, меняющие `payload/SKU/цену/размеры/origin`. Если
шина значение не собирает — ваш фолбэк (собственный реестр `add_filter/apply_filter` внутри модуля)
**правильный**; задокументируйте контракт каждой точки. Это аналог `apply_filters` эталона.

**10. CSRF admin-эндпоинта.** 🟡 NetCat-внутреннее — использовать штатный токен/сессию админки на каждый
POST/AJAX (аналог nonce эталона) + `secret_key` на cron-скрипте. **Подтвердить штатный механизм.**

**11. Хранение настроек модуля.** 🟡 NetCat-внутреннее — API опций модуля (аналог `COption`) или своя
таблица `nc_onecatalog_settings`. Со стороны стандарта важно лишь: per-site, валидация+кламп (шаг ≥10,
язык из набора, токен — env приоритетнее поля). **Подтвердить механизм.**

**12. Регистрация/автозагрузка модуля.** 🟡 NetCat-внутреннее (бутстрап, навеска событий) — шаг 0.
Контракт: обработчики событий должны навешиваться на каждом запросе (где сайт дозаполняет данные).

**13. Служебные таблицы при установке.** 🟡 install-хук модуля или ручной `CREATE` через `$nc_core->db`;
откат при удалении. Со стороны стандарта: нужны таблицы `queue`, `media`, `settings`, карты маппинга.

**14. Языковые файлы модуля.** 🟡 NetCat — PHP-массивы `lang/english.php`/`lang/russian.php`, не
gettext. Концепцию стандарта (исходник EN + перевод RU, строки фронт-JS из бэкенда локализованными)
сохраняете, формат адаптируете. Это нормально — у каждой CMS свой механизм i18n.

**15. iframe-пикер / CSP.** 🟡 Не гарантировано, что админка NetCat пустит сторонний iframe +
`postMessage`. **Фолбэк (всегда работает) — импорт по списку `public_id`** (стандарт §2.4 допускает).
Рекомендую сначала ID-list, пикер опционально. `event.origin` проверять явно, origin виджета задавать
явно (не `window.location.origin`), импорт по `productPublicIds`.

**16. Лимиты API и справочник.** 🟢 Справочники (`/specifications/`, `/categories/`) — **одним запросом
`limit=1000`**; дефолтный лимит инстанса молча обрезает, поэтому сверяться с **`meta.counts`** (общее
число) и дострелить пагинацией, если вдруг больше. Жёсткого rate-limit на `X-API-Key` в тестах не
наблюдалось. Товары — по одному `/products/{public_id}/`. Базовый URL API переопределяем (настройка).

**17. Единицы магазина.** 🟢 Источник: **вес в граммах, размеры в миллиметрах** (подпись локализована
«мм»/«г»). **Важно: `sizes.weight` часто `null`** (у OC.IND.1 вес отсутствует) — если нет, не выдумывать
(0/пусто). 🟡 Куда писать длину/ширину/высоту/вес в NetShop (`Units`/`StockUnits` или доп. поля) —
инстанс-specific, **подтвердить**. Конвертер г/мм → единицы магазина обязателен (иначе 1200 мм → 1200 см).

**18. Стабильность slug коллекции / полнота эмбеда.** 🟢 `/collections/{slug}/` отдаёт **500 — не
дёргать**. Все поля коллекции (картинки, `link_3d`, `link_official_site`) берём из **эмбеда `collections[]`
в payload товара** — он полный. Доп. запрос не нужен.

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id`** (поле `OneCatalogPublicID`), `article → ItemID` отдельно (только
   при непустом; коллизия артикула — не перезаписывать чужой, лог). Искать ВСЕГДА по public_id.
2. **Справочники — поиск по имени (label), не по слагу/alt-name** — критично: автотранслит даёт разные
   слаги для одного имени → дубли (в эталоне был баг «Толщина» tolschina/tolshhina). Объект справочника/
   раздел по `name` (в пределах родителя). Стабильный ключ маппинга — `specification_id`/category id/
   brand id, не переводимый label.
3. **Характеристики:** справочник + связь, предсоздание вне очереди (Q4); многозначные (`multitude`) →
   много-ко-многим; ручной маппинг по `specification_id`, строгий режим (только сопоставленные).
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие =
   false**; термы true/false из маппинга. (`multitude=false` у boolean.)
5. **Медиа:** скачивать cURL + **MIME по содержимому** (URL без расширения); дедуп `sha1(path#size)`
   (своя `nc_onecatalog_media`); обложка/галерея; галерея идемпотентна по имени файла, не по URL; общие
   файлы не удалять. NB: бинарный `media_files`-URL **самодостаточен** (токен зашит в base64-хеш,
   качается без заголовка `X-API-Key`) — `X-API-Key` нужен только JSON-вызовам `/products/`, чтобы они
   вернули max-URL.
6. **Не выдумывать цену** (`Price` пусто/0, статус неактивный до простановки если надо); статус — только
   при создании, не перезатирать.
7. **Единицы:** конвертер г/мм → единицы магазина (`oc_weight_unit`/`oc_dimension_unit` настройкой);
   `measurement_unit` (м²/шт) — продажная единица, не габарит → событие; `weight` бывает `null`.
8. **Категории — дерево** из `/categories/` (`parent_category`). **`/collections/{slug}/` 500 — не
   дёргать**, всё из эмбеда `collections[]`. **`/specifications/`, `/categories/` — `limit=1000` +
   сверка `meta.counts`.**
9. **Прочий payload** (`measurement_unit`, `areas_per_package`, `product_shape`, alt/описание картинок,
   детали бренда `description_text/foundation_year/website`, `upsells/cross_sells`) — через своё событие
   `onecatalog_product_imported` (полный payload).
10. **SemVer + CHANGELOG** (Keep a Changelog), бэклог = «Не выпущено»; версию поднимать перед сборкой;
    новые UI-строки — в `lang/`-файлы модуля.
