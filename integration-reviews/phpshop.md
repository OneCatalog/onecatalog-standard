# Интеграционный план: PHPShop

Реализация стандарта **OneCatalog Import** (`INTEGRATION-STANDARD.md`) на
**PHPShop** — российской коробочной CMS интернет-магазина (PHP, разработчик
phpshop.ru с 2004 г.). Эталон — WooCommerce; ниже инварианты §2/§5 переложены на
реальные механизмы PHPShop: ядро-фреймворк (`PHPShopCore`/`PHPShopObj`), ORM
(`PHPShopOrm`, `class/orm.class.php`), модель товара (поля `uid`, `vendor_array`,
`price`…), характеристики (`vendor_array` + таблица `sort_categories`), штатный
обмен **CommerceML** (1С) и дизайн-хуки.

> **Допущения о платформе (зафиксированы, чтобы не выдумывать).** Целюсь в
> **PHPShop Enterprise/Pro 5.x–6.x** (линейка с `PHPShopOrm` и наборами
> характеристик; наборы характеристик появились ~Enterprise 2.1.78, ORM —
> в 6.X-доке). PHPShop — нишевая закрытая CMS; публичная документация
> (`wiki.phpshop.ru`, `doc.phpshop.ru`) фрагментарна, исходники под лицензией и
> в репозитории проекта их НЕТ (проект Ceram — WP+Nuxt; PHPShop-таргет
> greenfield). **Поэтому значительная часть схемы БД восстановлена по докам и
> косвенным признакам — каждое такое место помечено «(не уверен по PHPShop,
> проверить в исходниках/у вендора)».** Точные имена таблиц/полей нужно сверить в
> `phpshop/class/*.class.php` и в БД конкретной инсталляции до кода.
>
> **Принципиальное отличие от Woo/Bitrix:** у PHPShop нет ни CPT/таксономий, ни
> EAV-таблицы свойств. Характеристики товара хранятся **сериализованным PHP-массивом
> в одном поле `vendor_array`** записи товара, а словарь характеристик/наборов —
> в `sort_categories`. Это меняет почти всё в маппинге (§5.4) — см. блокеры B1–B3.

## Маппинг сущностей и реализация

Сводная таблица (§3 стандарта → механизмы PHPShop). Имена таблиц даю по докам;
в коде брать через `$GLOBALS['SysValue']['base'][<ключ>]` (PHPShop держит карту
имён таблиц там, напр. `$GLOBALS['SysValue']['base']['sort_categories']`), а не
хардкодом.

| OneCatalog | Абстракция | PHPShop: хранилище | Механизм |
|---|---|---|---|
| product | Товар | запись каталога (таблица товаров, ключ `id`); поля `name`, `uid`, `price..price5`, `content`, `vendor_array`, `items`/`sklad`, `pic_small`, `newtip`/`spec`, `datas` | `PHPShopOrm` на таблицу товаров |
| public_id | Внешний ключ | **своё поле/таблица сопоставления** `oc_product_map(product_id, public_id, source, quality)` (нет «меты на объект») — см. «Идемпотентность» | прямой `PHPShopOrm` на свою таблицу |
| article | Артикул произв. | **`uid`** (родной артикул товара) | поле записи товара (блокер B6) |
| options[] text | Характеристика | значение в `vendor_array` (сериализ. массив) + словарь в `sort_categories` (набор «Фильтр») | `PHPShopSortSearch` / прямой апдейт `vendor_array` (B1) |
| options[] boolean | Характеристика-флаг | то же `vendor_array`; значение = строка-терм для true/false из маппинга | нет нативного boolean-типа |
| options[] numeric | Характеристика-число | то же `vendor_array`; число (+опц. единица) текстом | — |
| categories[] | Категория | раздел каталога (дерево разделов, своя таблица каталогов) | `PHPShopCategory` / `PHPShopOrm` (B7: товар обычно в ОДНОМ разделе) |
| brand | Бренд | **характеристика типа «Бренд» (сквозной фильтр)** — нативный механизм PHPShop | `sort_categories` + `vendor_array` |
| country | Страна происх. | **характеристика типа «Фильтр»** (нет родной сущности «страна товара») | как обычная характеристика (B4) |
| collections[] | Коллекция | **«Виртуальный каталог»/доп. раздел** ИЛИ характеристика-фильтр (настройка) | нет CPT (B3) |
| tags[] | Теги | **нет родной таксономии тегов** → характеристика-фильтр ИЛИ `keywords`/meta (настройка, вкл/выкл) | B5 |
| images_urls (обложка) | Медиа-обложка | главное фото товара (файл в каталоге картинок + `pic_small`/превью) | загрузка файла + апдейт поля (B8) |
| files[] (галерея) | Медиа-галерея | доп. фото товара (отдельная таблица/папка доп. изображений) | дедуп по имени файла |
| measurement_unit | Ед. измерения | поле «Единица измерения» товара (если есть) ИЛИ хук | сайтовый слой / хук |
| sizes.weight | Вес | поле «Вес» товара | конвертер г→ед. магазина (§5.6) |
| sizes.length/width/height | Габариты | поля «Габариты» (Д/Ш/В) | конвертер мм→ед. магазина |
| areas_per_package, product_shape, upsells, alt картинок… | прочее | дизайн-хук `product_imported` (сайтовый слой) | граница «универсальное↔проектное» |

> Все имена таблиц/полей в этой секции — **(не уверен по PHPShop, проверить)**:
> доки дают имена `vendor_array`, `uid`, `sort_categories`, `name`, `price`, но не
> приводят полную DDL. Перед кодом снять реальную схему c инсталляции
> (`SHOW TABLES LIKE 'phpshop_%'`, `DESCRIBE <товары>`).

### Слои (перенос §4 на PHPShop)

PHPShop — самописный MVC-фреймворк: ядро `PHPShopCore`/`PHPShopObj`
(`loadClass()`), модель через `PHPShopOrm`, представление через шаблоны `.tpl` и
дизайн-хуки. Своя логика подключается классом в `/phpshop/core/`
(`coretest.class.php` → `PHPShopCoretest`, доступен по `/coretest/`), либо
админ-модулем. Раскладка слоёв стандарта:

```
phpshop/core/onecatalog.class.php          — внешняя точка/REST-приём ID + статус (Rest), extends PHPShopCore
phpshop/modules/onecatalog/                — админ-модуль: настройки + кнопка/пикер (Settings + AdminUI)
phpshop/class/onecatalog/Api.class.php          — клиент Wiki API (base URL/token/lang/GET/обёртка) — extends PHPShopObj
phpshop/class/onecatalog/Units.class.php        — конвертер источник→ед. магазина
phpshop/class/onecatalog/Media.class.php        — скачивание/MIME/обложка-галерея/качество/дедуп
phpshop/class/onecatalog/Taxonomies.class.php   — характеристики/категории find-or-create по ИМЕНИ (vendor_array + sort_categories)
phpshop/class/onecatalog/CollectionImporter.class.php
phpshop/class/onecatalog/BrandImporter.class.php
phpshop/class/onecatalog/CountryImporter.class.php
phpshop/class/onecatalog/ProductImporter.class.php  — оркестратор
phpshop/class/onecatalog/Queue.class.php        — порции + cron-обработчик + лог в своей таблице
templates/<tpl>/js/onecatalog-picker.js         — iframe + postMessage
templates/<tpl>/js/onecatalog-admin.js          — постановка в очередь + поллинг
sql/install.sql                                  — свои таблицы (map, queue/log)
```

Зависимости вниз: `Queue → ProductImporter → Api/Media/Taxonomies/*Importer`.
Классы наследуют `PHPShopObj` и грузятся через `loadClass()` — прямой аналог DI
эталона. Циклов нет.

> **(не уверен по PHPShop)** Каноничная структура «модуля» (manifest, авто-меню в
> админке, install/update-хук) в публичной доке не описана детально (статья на
> Хабре «Спотыкашки» про шаблоны, не про модуль-лайфцикл). Регистрация админ-пункта
> и установка таблиц — кандидаты на блокер B11; до кода уточнить у вендора, как
> штатные модули PHPShop объявляют себя в админ-меню и ставят свою схему.

### Идемпотентность (§5.1) — критично

У PHPShop **нет аналога WP-меты** (произвольный ключ на объекте). Два пути:

1. **Рекомендуемый — своя таблица сопоставления** `oc_product_map`:

```sql
CREATE TABLE `phpshop_oc_product_map` (
  `product_id` INT NOT NULL,
  `public_id`  VARCHAR(64) NOT NULL,
  `source`     VARCHAR(32) NOT NULL DEFAULT 'onecatalog',
  `quality`    TEXT NULL,            -- JSON: трекинг качества картинок (§5.3)
  `created` DATETIME, `updated` DATETIME,
  PRIMARY KEY (`public_id`,`source`),
  UNIQUE KEY (`product_id`,`source`)
);
```
`find_by_external_id(public_id)` = `PHPShopOrm(map).getOne(['product_id'],
['public_id'=>"='$pid'", 'source'=>"='onecatalog'"])`.

2. **Альтернатива** — положить `public_id` в `vendor_array['_oc_public_id']` товара
(сериализованный массив всё равно читается/пишется). Минус: поиск «найти товар по
public_id» становится `LIKE`-сканом по `vendor_array` (нет индекса) → медленно на
больших каталогах. **Своя таблица предпочтительнее.**

- ⚠️ **`uid` (артикул) НЕ использовать как ключ идемпотентности** — это артикул
  производителя (§5.2), он может быть пуст/неуникален.
- **Справочники find-or-create по ИМЕНИ (label), не по слагу** (§5.1): категории
  матчить по `name` (`LOWER(name)=LOWER(?)`); характеристики/наборы — по
  человекочитаемому имени в `sort_categories`. У PHPShop русские имена → URL могут
  транслитерироваться по своей схеме, поэтому матч по слагу плодит дубли — матчим
  по имени, слаг только для создания.
- Легаси-фолбэк не нужен (greenfield), но если на этой инсталляции уже шёл обмен
  CommerceML — есть риск **двух источников истины** на один товар (см. B10):
  предусмотреть опцию «не трогать товары, у которых есть 1С-GUID».

### Артикул / SKU (§5.2)

- `article` (артикул производителя) → **`uid`** товара, если пришёл; иначе оставить
  пустым.
- `public_id` хранить отдельно (своя таблица, §выше), **не** в `uid`.
- Коллизия `uid` с другим товаром: проверить занятость другим `product_id`,
  при коллизии не перезаписывать (артикул — бизнес-уникален).
- ⚠️ **(не уверен по PHPShop)** уникальность `uid` на уровне БД и его роль в
  CommerceML-обмене проверить: если `uid` используется обменом как ключ — наш
  импорт и 1С будут конфликтовать за это поле (B10).

### Характеристики (§5.4) — сериализованный `vendor_array`, не EAV

Главное отличие платформы. Характеристика товара = элемент массива в поле
`vendor_array` (товар), словарь характеристик и наборов — таблица
`sort_categories` (наборы привязаны к разделам каталога; значения присваиваются
товарам). Типы PHPShop: **Фильтр / Опция / Бренд / Виртуальный каталог /
Переключатель**. Маппинг типов OneCatalog:

- `text` → характеристика типа **«Фильтр»** (значение = терм).
- `numeric` → «Фильтр», значение числом (+опц. единица текстом).
- `boolean` → «Фильтр», значение = строка-терм для true/false из ручного маппинга;
  отсутствие/`null`/`false` → значение «false» (§5.6).

Алгоритм импорта одной характеристики (`Taxonomies`):
1. Резолв набора/характеристики: ручной маппинг по `specification_id` (строгий
   режим, §5.4) ИЛИ find-or-create в `sort_categories` по **имени** характеристики.
   Набор привязывается к разделу каталога товара — **(не уверен по PHPShop)** как
   именно набор связан с разделом (`category_id` в `sort_categories`?), уточнить;
   возможно, понадобится «общий» набор, не зависящий от раздела.
2. Прочитать `vendor_array` товара (`unserialize`), записать/обновить значение
   характеристики, `serialize` обратно, `PHPShopOrm(products).update(...)`.
   ⚠️ Формат `vendor_array` (ключ = id характеристики? имя? вложенность массива
   значений) **обязательно реверс-инжинирить на реальных данных** — доки дают
   только «сериализованный массив значений», не схему. Безопаснее: создать товар с
   парой характеристик через админку, сдампить `vendor_array`, скопировать формат.

⚠️ **boolean без текстового значения** (§2.2): из источника
`specification_option_name=null`, реальное значение в `bool_option`. Без правила
«true/false → терм» характеристика молча теряется — поэтому в UI маппинга для
boolean задаются термы true/false (§7).

### Категории (§5.4)

- Дерево разделов каталога; find-or-create по **имени** в пределах родителя.
- Создание: `PHPShopOrm(categories).insert([...])`; получить новый id (см.
  «Insert и id» ниже), пересчёт служебных полей дерева (left/right или
  `parent`-путь) — **(не уверен по PHPShop)**: уточнить, поддерживает ли PHPShop
  вложенные множества или простой `parent_id`; от этого зависит код вставки.
- ⚠️ **Товар обычно принадлежит ОДНОМУ разделу** (поле «Размещение»), тогда как у
  OneCatalog `categories[]` — массив. Стратегия: основной раздел = первая
  категория; остальные — либо «Виртуальные каталоги»/доп. привязка (если движок
  поддерживает множественную привязку — проверить), либо игнор с логом (B7).

### Бренд — нативный механизм (§3/§7)

PHPShop **имеет родной «Бренд»** — это характеристика-тип «Бренд» (сквозной фильтр
по всему магазину, с логотипом). Это лучший таргет, чем у OpenCart/Magento:
- `brand.menutitle` → значение характеристики «Бренд» (find-or-create по имени).
- `brand.images_urls` (логотип) → картинка бренда (механизм логотипа бренда у
  PHPShop есть) — грузить как медиа с дедупом (§5.3).
- Доп. поля бренда (`description_text`, `website`, `foundation_year`, `country`) —
  через хук `brand_imported`/`product_imported` (сайтовый слой), ядро их не пишет.
- Настройка «тип объекта + цель»: по умолчанию «характеристика типа Бренд»;
  альтернатива — обычный «Фильтр» (если витрина не хочет сквозной бренд).

### Страна происхождения (§3/§7)

Родной сущности «страна происхождения товара» у PHPShop нет (есть страны
доставки/валюты — это другое). Реализация: обычная **характеристика «Фильтр»
«Страна»** (find-or-create по имени `country.menutitle`). Настройка вкл/выкл,
по умолчанию выкл (B4).

### Коллекции (§3/§7)

Нет CPT. Варианты цели (настройка «тип объекта + цель»):
- **Виртуальный каталог** (нативный тип характеристики — «кнопки каталога»,
  лендинги) — семантически ближе всего к «коллекции/линейке».
- **Доп. раздел каталога** «Коллекции/{Бренд}» — если нужна страница-список.
- **Характеристика-фильтр** — если коллекция = просто фасет.
Доп. поля коллекции (`link_3d`, `link_official_site`, изображения) — в HTML
описания целевого объекта или через хук `collection_imported` (B3).

### Теги (§3)

Родной таксономии тегов нет. Опция (вкл/выкл, по умолчанию выкл): теги →
характеристика-фильтр «Теги» (поиск по имени) ИЛИ строка `keywords`/meta товара.
«Поиск по имени» при втором варианте вырождается в работу со строкой (B5).

### Медиа (§2.3/§5.3)

- URL OneCatalog **без расширения** → `media_sideload`-аналога нет; **скачивать
  вручную** (cURL/`file_get_contents`), MIME определять по содержимому
  (`finfo_buffer`), расширение присваивать самим.
- Обложка → главное фото товара (положить файл в каталог изображений PHPShop,
  прогнать через ресайз движка, записать имя/превью в поле товара —
  **(не уверен)** точные поля `pic_small`/папка `userfiles`/`products` уточнить).
- Галерея → доп. фото (отдельная таблица/папка доп. изображений товара). Дедуп
  **по имени файла** (подписи в URL меняются — по URL нельзя).
- **Трекинг качества** (§5.3): размер (`min|middle|max`) хранить в `quality`
  (JSON) своей `oc_product_map`. На переимпорте: лучше записанного → перекачать и
  заменить; не лучше → не трогать.
- **Дедуп по контент-ключу** (§5.3): из base64-префикса `media_files` извлечь
  `path`, ключ `= hash(path # size)`, хранить рядом с файлом (своя таблица
  `oc_media_map(content_key, file_path, size)`); перед загрузкой искать по ключу,
  нашли → переиспользовать. Общие файлы при апгрейде качества **не удалять**.
  (У PHPShop нет «вложений» как объектов — ведём свою таблицу медиа, B8.)

### Единицы (§5.6) — конвертер

OneCatalog отдаёт вес в граммах, размеры в миллиметрах (подпись локализована).
PHPShop хранит вес/габариты в своих единицах товара (**(не уверен)** какие именно
— кг? г? см? мм? — зависит от настроек/версии; уточнить). `Units`-класс:
таблицы коэффициентов к базовой единице + нормализация локализованных подписей
(«мм»/«г»/«mm»/«g»), вход — `sizes.*_unit` из payload, иначе мм/г; цель — единицы
магазина из настроек PHPShop. Переопределяемо хуком. Без конвертера 1200 мм
станут 1200 см.

### Идемпотентность справочников и порядок (§5.4)

Порядок внутри импорта товара — строго по §5.4: базовые поля (статус/цена только
при создании) → категории → характеристики (резолв набора + термы, boolean) →
сохранить товар (получить id) → public_id в `oc_product_map` → бренд → страна →
теги → обложка+галерея (качество+дедуп) → коллекция → хук `product_imported`
(сайтовый слой дозаполняет ед.изм., площадь упаковки, вид товара, alt и т.д.).

### Insert и получение id (тонкость ORM)

`PHPShopOrm::insert($value, $prefix, …)` доступен, но **(не уверен по PHPShop)**:
документация явно отмечает, что возврат `last_insert_id` у `insert()` не описан.
Кандидаты: метод базового слоя `PHPShopBase`/`mysqli_insert_id`, либо повторный
`getOne` по только что вставленному уникальному полю. До кода проверить, что
`insert()` возвращает id (или взять id штатным `mysqli_insert_id()` через
соединение ядра). Это влияет на весь порядок §5.4 (нужен id товара до привязок).

## Фоновая очередь, медиа, настройки, i18n, жизненный цикл

### Фоновая очередь (§6) — нет встроенного job-движка

У PHPShop **нет аналога Action Scheduler**. Варианты:
- **AJAX-степпер (основной)**: контроллер в `phpshop/core/onecatalog.class.php`
  принимает список public_id, режет на порции ≥10, фронт (`onecatalog-admin.js`)
  крутит цикл «обработай порцию → опроси статус/лог → следующая». Состояние и лог —
  в своей таблице `oc_queue`. Это совпадает с подходом штатного CommerceML-обмена,
  который тоже идёт пошагово по файлам/документообороту.
- **Cron (опц.)**: PHPShop умеет cron-задачи (планировщик для рассылок/обмена —
  **(не уверен)**, проверить наличие штатного cron-механизма); если есть — cron
  дёргает обработчик порции. Иначе системный crontab на URL контроллера с токеном.
- Фолбэк без планировщика — синхронная обработка с тем же отчётом (стандарт §6).

Лог последних N результатов (статус/имя/ошибка) — в `oc_queue`/`oc_log`, отдаётся
в поллинг. Один битый public_id помечается ошибкой, **не прерывает** пакет (§5.5).

### Медиа

См. §«Медиа» выше: ручное скачивание + MIME, своя таблица медиа-карты для дедупа
и качества (у PHPShop нет объектов-вложений). Прогон через штатный ресайзер
изображений PHPShop, чтобы превью/`pic_small` совпадали с остальным каталогом.

### Настройки (§7)

PHPShop хранит настройки в `$GLOBALS['SysValue']` / таблице настроек системы
(**(не уверен)** точное имя — `system`/`config`; в штатных модулях есть свои
опции). Реализация: своя таблица `oc_settings(name, value)` ИЛИ страница опций
модуля. Минимум (§7):

| Настройка | По умолчанию | Заметка PHPShop |
|---|---|---|
| Токен Wiki API | пусто (env приоритетнее) | env через `getenv()` поверх поля |
| Язык товаров (`en/ru/ar/zh/kk`) | `en` | только из списка API |
| Шаг импорта (≥10) | 10 | размер AJAX-порции |
| Статус новых (опубл./скрыт/под заказ) | опубликован | маппинг на флаги PHPShop, **только при создании** (B9: меньше статусов, чем у Woo) |
| Импортировать коллекции + цель | вкл; виртуальный каталог | «тип объекта (вирт.каталог/раздел/характеристика) + цель» |
| Импортировать бренд + цель | вкл; характеристика «Бренд» | нативный бренд PHPShop |
| Импортировать страну + цель | выкл; характеристика «Фильтр» | нет родной сущности |
| Импортировать теги | выкл | характеристика/keywords |
| Ручной маппинг характеристик | выкл | отдельная страница: specification_id→характеристика; boolean→термы true/false |

Валидация/кламп — в контроллере перед сохранением; приоритет env-токена; язык —
только из поддерживаемых.

### i18n (§9)

- Исходные строки — **английские**, единый «text-domain».
- ⚠️ У PHPShop **нет gettext/.po-механизма** как у WP/PrestaShop. Локализация —
  **(не уверен по PHPShop)** через языковые массивы шаблона/системы
  (`$GLOBALS['CONFIG']`/языковые файлы шаблона) либо собственный массив
  `lang/en.php`, `lang/ru.php` в модуле. Реализуем свой простой словарь
  `t($key)` с массивами по языкам — переносимо и не зависит от наличия в PHPShop
  полноценного i18n. Строки фронт-скриптов передавать из PHP (локализованными),
  не хардкодить в JS (§9).

### События / точки расширения (§8) — дизайн-хуки

PHPShop имеет **дизайн-хуки** (`$addHandler`-массивы в
`phpshop/templates/<tpl>/php/hook/`, маппинг метод→функция; «база дизайн-хуков»
в доке). Это аналог `do_action`/`add_action`, но ориентирован на вывод/шаблоны.
Для доменных событий импорта:
- Публикуем свои хуки через тот же механизм `$addHandler` (если он допускает
  произвольные точки) ИЛИ собственный мини-реестр коллбэков
  `OneCatalog::on('product_imported', $cb)` в ядре модуля.
- События: `before_import_product`, **`product_imported(product_id, payload,
  report)`** (сайт дозаполняет ед.изм./площадь/вид/alt/upsells — полный payload),
  `collection_imported`, `brand_imported`, `country_imported`, `queue_enqueued`.
- Фильтры (меняют значение): base URL, токен, аргументы запроса, язык, шаг,
  payload перед импортом, SKU, цена (по умолч. не задаётся), порядок размеров
  картинок, дедуп медиа, файлы галереи, справочник характеристик, флаги импорта и
  карта маппинга, origin виджета.
- ⚠️ **(не уверен по PHPShop)** допускают ли штатные дизайн-хуки произвольные
  кастомные точки вне предопределённой «базы хуков». Если нет — реализуем
  **собственный реестр коллбэков** в модуле (не зависит от шаблонной системы) —
  это безопаснее и переносимее (B12). Оформительские поля витрины Ceram живут в
  обработчике `product_imported` сайта, ядро о витрине не знает (граница §8).

### Виджет-пикер (§2.4)

- Кнопка на странице товаров админки → iframe
  `https://tools.onecatalog.net/picker.html?token=…&parentOrigin=…`.
- `postMessage`: `ONECATALOG_SELECTED {productPublicIds}` → импорт по **public_id**;
  `ONECATALOG_CLOSE` → закрыть. Проверять `event.origin`; origin виджета задавать
  явно; sandbox `allow-scripts allow-forms allow-same-origin`.
- Фолбэк: textarea со списком public_id (импорт без виджета).
- ⚠️ Встроить кнопку в админ-страницу товаров PHPShop — **(не уверен)** есть ли
  штатная точка расширения админ-UI товаров; вероятно через свой админ-модуль/
  пункт меню, а не врезку в чужой шаблон (B11).

### Жизненный цикл (§10)

- Манифест модуля: версия (SemVer), требования (версия/редакция PHPShop — наборы
  характеристик и ORM есть не во всех), путь языковых файлов.
- **CHANGELOG** (Keep a Changelog), бэклог = «Не выпущено»; поднимать версию во
  всех местах манифеста перед сборкой; новые строки — переводить.
- **Установка/обновление**: выполнить `sql/install.sql` (свои таблицы map/queue/
  media), зарегистрировать пункт админ-меню. ⚠️ **(не уверен по PHPShop)** штатный
  механизм install/uninstall/update модуля — уточнить у вендора (B11); как фолбэк —
  идемпотентный «ensure schema» при первом обращении (CREATE TABLE IF NOT EXISTS).
- **Миграция** опций/полей при смене схемы (флаг «миграция выполнена» в `oc_settings`).
- Активация не должна падать без движка магазина: функции импорта проверяют
  наличие классов `PHPShopOrm`/таблиц товаров.

### Права

- REST/контроллер импорта — только для авторизованного админа PHPShop; проверять
  сессию админки (`PHPShopCore`-механизм авторизации) + **CSRF-токен** на запись
  (аналог nonce). Пикер и постановка в очередь — за админ-сессией.
- ⚠️ **(не уверен)** имя/механизм проверки прав админки PHPShop — взять из ядра
  (как штатные админ-модули проверяют вход).

## 🚫 Сложно / невозможно на PHPShop

**B1. Характеристики — сериализованный `vendor_array`, не EAV/термины.**
Значения характеристик лежат PHP-массивом в одном поле товара, без нормализованной
таблицы «термов». Последствия: (а) запись характеристики = read-modify-write
сериализованного поля (риск гонок при параллельных порциях — порции делать
последовательными по одному товару); (б) «идемпотентность термов» и «поиск по
имени» работают через словарь `sort_categories`, а не через объекты-термины;
(в) формат массива не документирован — обязателен реверс-инжиниринг (вопрос 2).
*Обход:* строгий read-modify-write одного товара за раз; формат снять с дампа;
по возможности дублировать «сырьё» характеристик в свою таблицу для отладки.

**B2. Нет «меты на объект».** Негде хранить `public_id`, трекинг качества,
контент-ключи медиа «рядом с объектом». *Обход:* свои таблицы `oc_product_map`,
`oc_media_map`, `oc_queue` (как у OpenCart). Допустимо, но это +DDL и своя
install-логика (см. B11).

**B3. Нет CPT/таксономий — коллекции негде «жить» нативно.** Нет ни типа контента
«Коллекция», ни богатой таксономии. *Обход:* «Виртуальный каталог» (ближе всего),
доп. раздел каталога или характеристика-фильтр — по настройке; доп. поля
коллекции (3D-ссылка, офиц.сайт) в HTML описания или через хук.

**B4. Нет родной «страны происхождения товара».** *Обход:* обычная
характеристика-фильтр «Страна» (по умолчанию выкл).

**B5. Нет таксономии тегов.** *Обход:* характеристика-фильтр «Теги» или строка
`keywords` (вкл/выкл); при строке «поиск по имени» вырождается в работу со строкой.

**B6. `uid` (артикул) может пересекаться с ролью ключа.** Если на сайте активен
CommerceML, `uid`/внутренний код участвует в обмене → наш импорт и 1С спорят за
поле и за сам товар. *Обход:* строго `public_id` в своей таблице (не в `uid`);
опция «не трогать товары с 1С-GUID»; согласовать с владельцем, кто «источник
истины» по каким полям (B10).

**B7. Товар — обычно в одном разделе.** `categories[]` (массив) не ложится 1:1.
*Обход:* основной раздел = первая категория; прочие — виртуальные каталоги/
характеристика/лог-скип (по настройке). Если движок всё же поддерживает
множественную привязку — использовать её (вопрос 6).

**B8. Нет объектов-вложений и стандартного загрузчика по URL.** Нет
`media_sideload`, нет «attachment» с метой. URL OneCatalog без расширения → штатные
загрузчики (если есть) их отвергнут. *Обход:* ручное скачивание + MIME по
содержимому + свой `oc_media_map` для дедупа/качества; прогон через ресайзер
движка вручную.

**B9. Меньше статусов товара, чем у Woo.** Нет полного набора
publish/pending/draft/private. *Обход:* маппинг на реальные флаги PHPShop
(опубликован/скрыт/под заказ — вопрос 15); статус задаётся **только при создании**.

**B10. Риск двух источников истины (OneCatalog + CommerceML/1С).** Если оба
активны, при переимпорте затрут правки друг друга. *Обход:* §5.6 «поля только при
создании не перезаписывать»; явная опция игнорировать 1С-товары; документировать
разграничение полей.

**B11. Module API / lifecycle PHPShop слабо документирован публично.** Регистрация
админ-меню, install/uninstall/update схемы, врезка кнопки в страницу товаров — не
описаны детально в открытой доке (нишевая закрытая CMS). *Обход:* до кода снять
рабочий пример штатного модуля с инсталляции/у вендора; как фолбэк — точка входа
через `phpshop/core/onecatalog.class.php` (документированный путь
`PHPShopCore`→`/onecatalog/`) и идемпотентный `ensure schema` вместо install-хука.
**Это главный риск сроков — заложить разведку.**

**B12. Событийная модель ориентирована на шаблоны.** Дизайн-хуки PHPShop — про
вывод/шаблоны, а не доменные события «до/после импорта». *Обход:* собственный
мини-реестр коллбэков в ядре модуля (`OneCatalog::on()/emit()`), не зависящий от
шаблонной системы — переносимо и даёт чистую границу §8.

**B13. Нет gettext/.po.** *Обход:* собственный словарь `lang/<code>.php` +
хелпер `t()`; строки фронта прокидывать из PHP.

**B14. Закрытый код и фрагментарная документация в целом.** Многие имена
таблиц/полей/механизмов восстановлены косвенно и помечены «(не уверен)». *Обход:*
обязательная фаза 0 «разведка на живой инсталляции» (DESCRIBE/дампы/пример
модуля) ПЕРЕД любым кодом; пока разведка не закрыла вопросы 1–7 и 11, оценку
сроков считать предварительной.

### Уровень уверенности

- **Высокая:** наличие `PHPShopOrm` (select/insert/update/delete/query), хранение
  характеристик в `vendor_array` + `sort_categories`, нативный «Бренд» как
  сквозной фильтр, точка входа `phpshop/core/*.class.php`, AJAX-степпер как путь
  фоновой обработки, отсутствие CPT/таксономий/меты.
- **Средняя/низкая (помечено «не уверен»):** точная DDL и формат `vendor_array`,
  возврат id у `insert()`, множественность категорий, module/install API,
  механизм переводов и проверки прав, единицы веса/габаритов, штатный cron. Эти
  пункты закрываются только разведкой на инсталляции (B14, вопросы 1–15).
