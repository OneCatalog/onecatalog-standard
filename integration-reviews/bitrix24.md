# Интеграционный план: Битрикс24

> Перенос стандарта `INTEGRATION-STANDARD.md` (импорт каталога из OneCatalog Wiki API)
> на **Битрикс24 (Bitrix24)** — облачную/коробочную **CRM-платформу** для продаж и
> совместной работы, у которой есть **Торговый каталог CRM** (товары, разделы,
> свойства товара, единицы измерения, цены). Эталон стандарта — WooCommerce.
> Внешние контракты (§2) и инварианты (§5) стандарта неизменны; меняется только
> слой «как это лечь в Битрикс24».

## ВАЖНО: Битрикс24 ≠ «1С-Битрикс: Управление сайтом»

Это **два разных продукта одного вендора**, их постоянно путают — и это ломает
интеграцию на корню:

- **«1С-Битрикс: Управление сайтом» (БУС / Bitrix CMS)** — это **CMS-витрина**:
  свой публичный фронтенд, инфоблоки (`iblock`), модуль `catalog`, шаблоны,
  компоненты, ЧПУ/SEO-страницы разделов, фасетный (умный) фильтр для покупателя.
  Идемпотентность там — `XML_ID` элемента инфоблока. Под этот продукт в этом же
  каталоге лежит **отдельный** план — `integration-reviews/bitrix.md`. ЭТОТ
  документ — НЕ про него.
- **Битрикс24 (Bitrix24)** — это **CRM/портал совместной работы** (облако
  `*.bitrix24.ru` или коробка «1С-Битрикс24»). Ядро — лиды/сделки/счета/CRM-
  аналитика. У него есть **Торговый каталог CRM** — справочник товаров и услуг,
  которые добавляются в сделки, счёта, КП. Это НЕ витрина: у CRM-каталога нет
  публичных SEO-страниц коллекций/брендов, нет фасетов покупателя, нет тем и
  шаблонов вывода. (Опционально к порталу можно подключить модуль «Интернет-
  магазин + CRM» / «Магазины (Sites)», и тогда тот же каталог покажется на сайте —
  но **ядро всё равно CRM**, витрина — отдельный слой, см. «🚫 Сложно».)

**Следствие для нас.** Цель импорта в Битрикс24 — наполнить **CRM-каталог** товарами,
чтобы менеджеры подбирали их в сделки/счета и чтобы работала CRM-аналитика по
номенклатуре. Витринные инварианты стандарта (SEO-страницы коллекций/брендов,
фасеты, alt-тексты ради SEO, статусы публикации draft/pending как у поста) на
CRM-каталоге **не живут** — их место либо в подключённом модуле магазина, либо во
внешнем сайте/CMS, который синхронизируется с порталом. Идемпотентность —
**`XML_ID`** товара/раздела (та же модель внешнего ключа, что и в Битрикс-CMS).

## Целевые API

Битрикс24 даёт REST через **вебхуки** и **OAuth-приложения** (Маркет). Для каталога
есть два поколения методов:

- **`catalog.*` — приоритет (современный торговый каталог).** *(проверено: раздел
  «Торговый каталог» apidocs.bitrix24.ru — `catalog.product.*`, `catalog.section.*`,
  `catalog.productProperty.*`, `catalog.productPropertyEnum.*`, `catalog.measure.*`,
  `catalog.productImage.*`, `catalog.catalog.list`, `catalog.price.*`.)* Это наша
  основная цель: модель «товар + разделы + свойства + единицы + картинки + цены».
- **`crm.product.*` / `crm.catalog.*` — легаси.** *(проверено: в apidocs помечены
  как `outdated` — раздел `crm/outdated/...`.)* Простой плоский товар CRM без полной
  модели свойств/разделов современного каталога. Использовать ТОЛЬКО как фолбэк,
  если на конкретном портале/тарифе `catalog.*` недоступен *(не уверен по тарифам —
  проверить, см. «❓ Вопросы»)*.

**Аутентификация (два режима, выбор — это ЖЦ-решение, §«Фоновая очередь…»):**

- **Входящий вебхук (inbound webhook)** — *(проверено)* ключ вида
  `https://<portal>.bitrix24.ru/rest/<user_id>/<token>/` с правами выдавшего
  пользователя, **без срока истечения**. Подходит для **одного** портала, которым
  мы управляем (внутренняя интеграция «один клиент»). Проще всего для нашего
  сервиса-оркестратора.
- **OAuth 2.0-приложение (локальное / тиражное в Маркете)** — *(проверено)* полный
  доступ к REST, `access_token`/`refresh_token` с истечением, нужно для UI внутри
  портала и для **мультитенантности** (одно приложение → много порталов). Если
  плагин планируется в Битрикс24.Маркет — только этот путь.

### Детализация

**Товар → `catalog.product.*`.** *(проверено)* `catalog.product.add` требует
`iblockId` (идентификатор торгового каталога; получаем через `catalog.catalog.list`)
и `name`. Базовые поля: `active` (Y/N — аналог «статуса», только при создании,
§5.6), `code` (символьный код), `weight`, `length`, `width`, `height` (double),
`measure`. Идемпотентность — `xmlId`.

**public_id → `xmlId` (External code).** *(проверено: `catalog.product.add` имеет
поле `xmlId` = «внешний код для ссылок из внешних систем».)* Это та же модель
внешнего ключа, что и `XML_ID` элемента инфоблока в Битрикс-CMS. Поиск
существующего товара — `catalog.product.list` с фильтром по `xmlId`
*(не уверен: фильтруется ли список по `xmlId` штатно и уникален ли `xmlId` в рамках
каталога — проверить, §«❓ Вопросы»)*. Это критично для §5.1 (find-or-create).

**article → артикул (свойство), НЕ `xmlId`.** Стандарт жёстко разделяет: `public_id`
(внешний ключ) ≠ `article` (артикул производителя) (§5.2). В Битрикс24 `xmlId`
занимаем под `public_id`; артикул кладём в отдельное свойство товара (или нативное
поле артикула, если оно есть в редакции) *(не уверен — проверить)*. При коллизии
артикула не перезаписывать.

**options[] → свойства товара (`catalog.productProperty`) — НЕ поля сделки CRM!**
Это **самая частая ошибка** на Битрикс24. Характеристики товара (цвет, размер,
толщина) — это **свойства торгового каталога**, а НЕ пользовательские поля CRM
(`crm.deal.userfield`, поля сделки/лида). Это разные миры: поле сделки описывает
сделку, свойство товара — товар. *(проверено: `catalog.productProperty.*` — add/
update/get/list/delete/getFields; свойства **привязаны к конкретному каталогу**,
тип свойства **неизменяем после создания**.)* Значения списков —
`catalog.productPropertyEnum.*` *(проверено, что класс упоминается в доке;
точные поля — проверить)*. Маппинг типов OneCatalog → тип свойства:
- `text` → свойство типа **«Список» (List)** + enum-значения (find-or-create
  значения по **имени**, §5.1), либо «Строка» (если ручной маппинг указал строку).
- `numeric` (`numeric_option`) → свойство типа **«Число» (Number)**.
- `boolean` (`bool_option`) → свойство-список со значениями «Да/Нет» (или строка);
  пустое/`null`/`false` → **false** (§5.6). Битрикс24 не имеет «галочки» как
  отдельного типа свойства каталога так же явно, как WooCommerce-атрибут —
  моделируем списком с двумя термами *(не уверен — проверить, есть ли нативный
  boolean-тип свойства каталога)*.

**categories[] → разделы каталога (дерево).** *(проверено)* `catalog.section.add`
(поля `iblockId`, `name`, `iblockSectionId`/`parentId` для вложенности),
`catalog.section.list`. Идемпотентность раздела — по **имени** в пределах родителя
(или по `xmlId` раздела, если задаём) (§5.1 — искать по label, не по слагу).
Дерево OneCatalog `categories[]` плоское по payload — родитель/потомок выводим из
порядка/полей, иначе кладём плоско.

**brand / country / collections[] / tags[] — нативных сущностей нет.** В CRM-каталоге
Битрикс24 НЕТ отдельных «брендов», «коллекций», «стран», «тегов» как первоклассных
объектов (в отличие от WooCommerce-таксономий/CPT). Поэтому по каждой такой сущности
— настройка **«тип объекта + цель»** (§3, §7):
- **бренд** → свойство-список товара (термин = название бренда) ИЛИ отдельный
  раздел-«папка» каталога. По умолчанию — свойство-список.
- **страна** → свойство-список. (По умолчанию выкл/свойство.)
- **коллекция** → раздел каталога ИЛИ свойство-список (выбор в настройках).
- **теги** → свойство-список (вкл/выкл; поиск термина по имени).
*(всё — не уверен в деталях привязки; первично — свойства каталога; проверить,
нужны ли HL-блоки/справочники — но в чистом Битрикс24 highload-блоков как в БУС
может не быть, см. «❓»)*.

**Медиа → картинки товара.** *(проверено)*
- Обложка → `previewPicture` (анонс) и/или `detailPicture` (детальная) прямо в
  `catalog.product.add/update`, формат — `{ fileData: [ "имя.jpg", "<base64>" ] }`
  или `{ remove: 'Y' }`.
- Галерея → `catalog.productImage.add` с `fields.productId`, `fields.type`
  (`DETAIL_PICTURE`/`PREVIEW_PICTURE`/`MORE_PHOTO`) и `fileContent: ["имя.jpg",
  "<base64>"]`; список — `catalog.productImage.list`, удаление —
  `catalog.productImage.delete`. То есть **картинки грузятся по REST как base64**,
  НЕ по URL (URL OneCatalog `…/media_files/<hash>` без расширения — качаем вручную,
  определяем MIME по содержимому, §2.3). Идемпотентность галереи — по имени файла +
  контент-ключу (§5.3); дедуп по контент-ключу из base64-префикса `media_files`.

**sizes/weight → вес + габариты.** *(проверено)* Поля `weight`, `length`, `width`,
`height` товара (double). **Конверсия единиц обязательна (§5.6):** OneCatalog отдаёт
вес в **граммах**, размеры в **миллиметрах**; Битрикс24 хранит в своих единицах
(зависит от настроек портала) — конвертировать перед записью (универсальный
конвертер: коэффициенты к базовой единице + нормализация локализованных подписей
«мм»/«г»). *(не уверен, в каких единицах Битрикс24 трактует `weight`/размеры по
умолчанию — кг/г, мм/см? проверить и привести.)*

**measurement_unit → `catalog.measure`.** *(проверено)* Единицы измерения портала —
`catalog.measure.list/get/add/getFields`; в товаре поле `measure` = ссылка на
`catalog_measure.id`. Маппинг: единицу из payload (`measurement_unit.name`: м²/шт/…)
find-or-create среди `catalog.measure` по имени/коду; полученный id → в `measure`
товара. Если единицы нет и создание недоступно — оставить дефолтную единицу портала.

**Цена → `catalog.price`/тип цены — НЕ задаём.** *(проверено, что методы
`catalog.price.*` существуют.)* **В источнике OneCatalog цены НЕТ** (§5.6 — не
синтезировать). По умолчанию цену не пишем; подстановка из своего источника — через
точку расширения (событие/фильтр нашего сервиса), не из ядра. (Замечание-рамка: для
CRM цена особенно чувствительна — её ставит менеджер/прайс, нельзя затирать
переимпортом, §5.6 «только при создании».)

**SKU / торговые предложения — не используем.** *(проверено по payload §2.2: у
OneCatalog один товар, поля вариаций нет.)* В Битрикс24 есть механизм торговых
предложений (SKU — отдельный каталог-«потомок»), но раз у источника вариаций нет —
импортируем **простой товар без SKU**. Подтвердить, что это ок для заказчика (§«❓»).

## Фоновая очередь, медиа, настройки, ЖЦ

### Архитектура (слои §4) в нашем сервисе-оркестраторе

Принципиальное отличие от WooCommerce-эталона: на Битрикс24 **нет ICML-фида и нет
клиентского cron**, которым мы могли бы управлять. Облачный портал не даёт нам
ставить системные крон-задачи. Поэтому **оркестрация и очередь живут НЕ внутри
Битрикс24, а в нашем внешнем сервисе** (тот же бэкенд, что тянет OneCatalog Wiki
API), а в портал мы только **пушим по REST**. Слои стандарта (§4) реализуем у себя:
`Api` (клиент OneCatalog) → `Units` → `Media` → `Taxonomies` (свойства/разделы) →
`Collection/Brand/CountryImporter` → `ProductImporter` (оркестратор) → `Queue` →
`Rest` (приём списка `public_id`) → `Settings` → `AdminUI`. К Битрикс24 обращается
только нижний адаптер-«клиент REST Битрикс24».

> Альтернатива «приложение внутри портала»: можно собрать OAuth-приложение Маркета
> с UI прямо в Битрикс24 и фоновой обработкой через события/агенты портала. Но
> надёжная фоновая обработка больших пакетов **снаружи** проще и переносимее
> (см. «🚫 нет клиентского cron»).

### Только API-пуш

В Битрикс24 каталог наполняется **только REST-методами** (`catalog.product.add/
update`, `catalog.section.add`, `catalog.productProperty.add`,
`catalog.productImage.add`, `catalog.measure.*`). Декларативного фид-импорта (как
ICML у RetailCRM или CSV) для нашего сценария не используем — порядок зависимостей
§5.4 соблюдаем императивно: разделы → свойства+значения → товар → `xmlId` → бренд/
страна/теги/коллекция → картинки → событие «товар импортирован» (у нас).

### Rate-limit REST Битрикс24

*(проверено в общих чертах, число уточнить)* Интенсивность ограничена **алгоритмом
«дырявого ведра» (leaky bucket)**: можно делать всплески, но не постоянный высокий
поток. Ориентир — порядка **~2 запросов/сек устойчиво, всплеск до ~50** на портал/
пользователя *(точные числа зависят от тарифа/редакции — проверить!)*. Есть
**`batch`-метод**: до **50 под-запросов в одном вызове** *(проверено)*. Практика:
группировать запись свойств/картинок/товаров в `batch`, держать backoff на ошибку
лимита (`QUERY_LIMIT_EXCEEDED`), шаг очереди ≥10 (§6). На больших каталогах
(тысячи товаров) лимит — реальный потолок по времени (см. «🚫»).

### Картинки по REST

*(проверено)* Грузим **base64** (`fileData`/`fileContent`), НЕ URL. Размер
base64-пейлоада раздувает запрос → крупные `max`-картинки лучше слать **отдельными**
вызовами `catalog.productImage.add` (не в `batch` пачкой), чтобы не упереться в
лимит размера запроса *(лимит размера тела — проверить)*. Трекинг качества (§5.3):
выбранный размер `min|middle|max` храним **у себя** (в нашей БД оркестратора, рядом
с маппингом `public_id → bitrix productId`), т.к. в Битрикс24 удобного места для
нашей меты качества может не быть; на повторном импорте — апгрейд качества заменой.
Дедуп по контент-ключу — у нас же; «общие» файлы не перекачивать.

### Настройки (§7)

Те же, что в стандарте, плюс специфика портала:
- Токен/ключ OneCatalog (`X-API-Key`), язык (`ru/en/ar/zh/kk`), шаг импорта (≥10),
  «статус» новых (`active` Y/N — только при создании).
- **Доступ к Битрикс24:** URL входящего вебхука ИЛИ OAuth-кредитансы приложения;
  `iblockId` целевого торгового каталога (или авто-выбор через `catalog.catalog.list`).
- Бренд/страна/коллекция/теги — паттерн «тип объекта (свойство-список / раздел) +
  цель»; теги вкл/выкл.
- Ручной маппинг характеристик по стабильному `specification_id` → существующее
  свойство каталога (строгий режим, §5.4); для boolean — значения терма true/false.

### Мультиязычность / i18n

- Запросы к OneCatalog с `lang` — выбранный язык товаров (§2.1).
- Битрикс24 локализован; названия свойств/разделов мы пишем уже на языке импорта.
  Если нужен мультиязычный вывод названий товара — это уже витринная задача
  (подключённый магазин/внешний сайт), CRM-каталог хранит одно название
  *(не уверен в нативной мультиязычности названий товара каталога — проверить)*.
- UI нашего админ-слоя — исходные строки EN + переводы (§9); если делаем
  приложение Маркета, у Битрикс24 свой механизм языковых файлов для UI приложения.

### Жизненный цикл (§10)

- **Вебхук (приватная интеграция)** — живёт пока действует ключ; нет срока, но утечка
  URL = доступ к порталу. Хранить секрет безопасно, давать вебхуку минимально нужные
  права (Каталог/CRM-товары).
- **OAuth-приложение Маркета** — полноценный ЖЦ: install/uninstall-обработчики,
  обновление `refresh_token`, версия приложения, опубликованный листинг. Нужно для
  тиража на много порталов.
- SemVer + CHANGELOG (Keep a Changelog) нашего сервиса/адаптера; миграция схемы
  нашего стораджа маппинга при смене ключей.

## 🚫 Сложно / невозможно (с обходами)

1. **Путаница Битрикс24 ↔ «1С-Битрикс: Управление сайтом».** Самый дорогой риск:
   написать интеграцию под инфоблоки CMS, а у клиента — облачный Битрикс24 без БУС
   (или наоборот). **Обход:** на старте однозначно зафиксировать продукт и API
   (`catalog.*` REST портала, а не `CIBlockElement` PHP-API CMS). Этот документ —
   только Битрикс24-CRM.
2. **Характеристики ≠ поля сделки.** Соблазн положить options[] в пользовательские
   поля сделки/лида — приведёт к мусору в CRM и потере характеристик на товаре.
   **Обход:** строго `catalog.productProperty.*` на товаре; поля сделки не трогаем.
3. **Нет витрины / фасетов / SEO.** CRM-каталог не имеет публичных страниц
   коллекций/брендов, умного фильтра покупателя, alt-текстов ради SEO, статусов
   публикации поста. **Обход:** эти инварианты стандарта реализуются НЕ здесь —
   либо в подключённом модуле «Интернет-магазин», либо во внешнем сайте/CMS,
   синхронизируемом с порталом. В Битрикс24 везём только то, что ложится на
   таблицу маппинга (товар/раздел/свойство/единица/картинка).
4. **Нет нативных бренд/коллекция/страна/теги.** **Обход:** настройка «тип объекта
   (свойство-список / раздел) + цель» по каждой сущности (§3, §7); по умолчанию —
   свойства-списки. Полноценные «страницы бренда» — только во внешней витрине.
5. **Rate-limit REST — потолок на больших каталогах.** Leaky-bucket (~2 rps
   устойчиво) при тысячах товаров × (свойства + N картинок) даёт большое суммарное
   время. **Обход:** `batch` (до 50 под-запросов), группировка операций, backoff на
   `QUERY_LIMIT_EXCEEDED`, фоновая очередь у нас с равномерным темпом, картинки —
   отдельными запросами вне batch. Закладывать часы на полный импорт большого
   каталога.
6. **Нет клиентского cron в облаке.** Мы не управляем планировщиком портала.
   **Обход:** очередь/оркестрация — **в нашем внешнем сервисе** (он по таймеру тянет
   OneCatalog и пушит REST), Битрикс24 — только приёмник. Если нужен запуск изнутри
   портала — события/агенты в OAuth-приложении, но это сложнее и менее переносимо.
7. **«Поля только при создании».** `active`/цена и т.п. задаём новому товару, при
   переимпорте **не затираем** (иначе теряются правки менеджера — для CRM это
   критично, там цены/статусы трогают руками). **Обход:** инвариант §5.6 — апдейтим
   только маппинг-поля (название, разделы, свойства, картинки по качеству), а
   «человеческие» поля оставляем.
8. **Динамическое создание свойств каталога.** Свойства **привязаны к конкретному
   каталогу**, тип **неизменяем** после создания *(проверено)*. Создавать на лету
   можно, но ошибки типа/каталога необратимы. **Обход:** при ВКЛ автосоздания —
   строго резолвить «найти свойство по имени → иначе создать с зафиксированным
   типом»; при сомнениях — строгий ручной маппинг (§5.4) на заранее заведённые
   свойства, чтобы не плодить кривые типы.
9. **Хранение нашей меты (качество картинок, контент-ключ дедупа, маппинг
   `public_id→productId`).** В Битрикс24 нет удобного «своего» места под служебную
   мету импортёра, как мета-поля в WP. **Обход:** хранить трекинг качества/дедупа и
   таблицу маппинга **в БД нашего сервиса-оркестратора**, а не в портале; `xmlId`
   товара использовать как мост.

## Резюме

**Основной путь.** Импортируем CRM-каталог Битрикс24 **только REST-пушем** методами
`catalog.*` (`catalog.product/section/productProperty/productPropertyEnum/measure/
productImage/price`); легаси `crm.product.*` — лишь фолбэк. Оркестрация, очередь,
трекинг качества и маппинг `public_id→productId` живут **в нашем внешнем сервисе**
(клиентского cron в облаке нет); в портал только пушим. Идемпотентность —
**`xmlId`** товара/раздела (та же модель внешнего ключа, что и `XML_ID` в Битрикс-
CMS). Аутентификация: входящий вебхук (один портал) или OAuth-приложение Маркета
(тираж). Картинки — base64 по REST; вес/размеры — с конверсией г/мм → единицы
портала; единицы — `catalog.measure`; цена — НЕ задаём (в источнике нет).

**Отличие от «1С-Битрикс: Управление сайтом» (CMS).** Это **другой продукт**:
БУС — витринная CMS с инфоблоками, шаблонами, SEO-страницами и фасетами
(для неё — `bitrix.md`); Битрикс24 — **CRM-портал** с торговым каталогом для сделок/
счетов/аналитики, без публичной витрины и фасетов. Путать их = переписывать
интеграцию.

**3–4 острых вопроса.** (1) Доступны ли `catalog.*` на тарифе/редакции клиента
(или только легаси)? (2) Уникален ли `xmlId` и фильтруется ли `catalog.product.list`
по нему (иначе find-or-create держим у себя)? (3) Характеристики кладём в **свойства
товара**, не в поля сделки — подтвердить? (4) Точные числа rate-limit/batch и
единицы веса/габаритов портала.

**3–4 «невозможно/сложно».** (1) Нет витрины/фасетов/SEO-страниц коллекций-брендов
— это слой внешнего сайта/магазина, не CRM. (2) Нет нативных бренд/коллекция/страна/
теги — везём как свойства-списки/разделы по настройке. (3) Rate-limit (leaky-bucket
~2 rps) — реальный потолок по времени на больших каталогах, спасает только `batch`
+ фоновый темп. (4) Нет клиентского cron в облаке — планировщик и очередь только
снаружи, в нашем сервисе.
