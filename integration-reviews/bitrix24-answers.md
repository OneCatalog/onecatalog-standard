## Главное про данные

🟢 **Вариаций товара НЕТ ВООБЩЕ** — один `public_id` = один готовый товар → **простой товар без торговых
предложений (SKU)** (ваш Q11 — подтверждаю авторитетно). `options[]` — характеристики-свойства, не оси
вариативности.

🟢 **Иерархия категорий ЕСТЬ — поправка к §categories.** Вы написали «дерево `categories[]` плоское по
payload → кладём плоско». Это так только для эмбеда в товаре. Справочный **`/categories/` отдаёт
`parent_category`** (id родителя или `null`). → стройте дерево `catalog.section` (`parentId`), родители
раньше детей. Битрикс24 имеет нативное дерево разделов — ложится 1:1, не плоско.

🟢 **Характеристики многозначны** — «Цвет» = 2 значения, «Помещение» = **6**. `/specifications/` отдаёт
флаг **`multitude`** (true = многозначная; boolean `false`). → свойство каталога делать **множественным**
(`isMultiple`/несколько enum-значений). Используйте `multitude`.

🟢 **`numeric` в этом инстансе не используется** (все 13 характеристик text/boolean) — код универсально.

🟢 **Цены в источнике НЕТ** (факт API); **`sizes.weight` часто `null`** (у OC.IND.1 веса нет);
`country_code` — ISO **alpha-3**; `measurement_unit` (м²/шт) — продажная единица (→ `catalog.measure`), не
габарит; всего **~13 характеристик** (свойств немного).

## Ответы на вопросы

**1. `catalog.*` vs `crm.product.*` по тарифу.** 🟡 Платформенное — **подтвердить тариф/редакцию**
(минимальный, открывающий «Торговый каталог CRM» и `catalog.*`). Контракт: `catalog.*` основной, легаси —
фолбэк. Активация деградирует, если `catalog.*` недоступен.

**2. Идемпотентность по `xmlId`.** 🟢 `public_id` → `xmlId` — верная модель (та же, что `XML_ID` в БУС).
Поиск — `catalog.product.list` фильтром по `xmlId` → найден → update, иначе add. Не по name/code/slug
(§5.1). 🟡 Уникальность `xmlId` в каталоге и фильтруемость списка по нему — **подтвердить**; если фильтр не
штатный — локальный индекс `public_id → productId` в нашем сервисе.

**3. Характеристики → свойства товара, НЕ поля сделки.** 🟢 **Подтверждаю авторитетно.** `options[]` →
`catalog.productProperty` (свойства торгового каталога), НЕ `crm.deal.userfield` (поля сделки/лида — другая
подсистема). Это та же ловушка, что properties-vs-customFields в RetailCRM / attributes в МойСкладе. Ключ
маппинга — `specification_id`, не переводимый label.

**4. Типы свойств каталога.** 🟡 Платформенное — **подтвердить набор** (Список/Строка/Число/Файл +
нативный boolean?). Контракт маппинга: `text` → Список (enum, find-or-create значений по имени) или Строка;
`numeric` → Число; `boolean` → если нативного нет, **Список с двумя термами «Да/Нет»**, **пусто/`null`/
отсутствие = false** (§5.6).

**5. Динамическое создание свойств.** 🟡 Свойства привязаны к каталогу, тип неизменяем после создания
(вы верно отметили — блокер 8). Контракт: при автосоздании строго «найти по имени → иначе создать с
зафиксированным типом»; при сомнениях — **строгий ручной маппинг** по `specification_id` на заранее
заведённые свойства (как Magento/UMI — справочники предсоздаются вне импорта). **Подтвердить политику.**

**6. Бренд/страна/коллекция/теги.** 🟢🟡 Нативных сущностей в CRM-каталоге нет → «тип объекта + цель»:
бренд → свойство-список (дефолт) или раздел; страна → свойство-список (дефолт выкл; `country_code`
alpha-3); коллекция → раздел или свойство (настройка); теги → свойство-список (выкл по умолчанию).
«Страницы бренда/коллекции» — витрина (внешний сайт/подключённый магазин), не CRM. **Подтвердить дефолты.**

**7. Цена.** 🟢 В источнике цены НЕТ — **не писать** (`catalog.price` не трогаем, §5.6). Если нужна — из
источника заказчика через событие/фильтр нашего сервиса; **только при создании, не затирать** (для CRM
критично — цены/статусы менеджер правит руками). Эталон цену не задаёт (фильтр `onecatalog_product_price`)
— тот же принцип.

**8. Единицы измерения.** 🟡 `measurement_unit` → `catalog.measure` (find-or-create по имени; создавать
недостающие через `catalog.measure.add` если доступно, иначе дефолт портала). `measurement_unit` (м²/шт) —
продажная единица, не путать с единицей габаритов (§5.6). **Подтвердить дефолт и возможность создания.**

**9. Вес/габариты — единицы.** 🟢 Источник: **вес в граммах, размеры в мм** (подпись локализована;
`sizes.*_unit`, иначе мм/г). **`weight` бывает `null`** — не выдумывать. 🟡 Единицы трактовки
`weight`/`length/width/height` портала (кг/г? мм/см/м?) — **подтвердить**; конвертер обязателен (иначе 1200
мм → 1200 см). Нативные поля `weight/length/width/height` — сильная сторона (есть, в отличие от Shopify).

**10. Картинки по REST.** 🟢🟡 base64 (`fileData`/`fileContent`), не URL (вы верно). Полезный факт:
бинарный `media_files`-URL **самодостаточен** — токен в base64-хеше, наш сервис качает его обычным GET
**без `X-API-Key`** (в эталоне `Media::sideload` так и получает max). URL без расширения → MIME по
содержимому; затем base64 в Битрикс24. Крупные max — отдельными вызовами `catalog.productImage.add` (не в
batch, чтобы не упереться в лимит тела). Обложка → `previewPicture`+`detailPicture`, галерея → `MORE_PHOTO`.
Трекинг качества/дедуп `sha1(path#size)` — в нашей БД. **Подтвердить лимит тела запроса.**

**11. SKU / торговые предложения.** 🟢 **Подтверждаю: вариаций нет → простой товар без SKU.** Механизм
торговых предложений не используем.

**12. Rate-limit / batch.** 🟡 Платформенное — **подтвердить числа** на тарифе (rps, размер «ведра»,
batch=50). Группировать в `batch`, backoff на `QUERY_LIMIT_EXCEEDED`, очередь шагом ≥10, картинки отдельно.
На больших каталогах — реальный потолок по времени (блокер 5). 🟢 Со стороны OneCatalog жёсткого лимита в
тестах не было; справочники `limit=1000`.

**13. Мультиязычность.** 🟢🟡 API отдаёт **один язык за запрос** → для ceram **`ru`, один язык**. CRM-
каталог хранит одно название; мультиязычный вывод — витринная задача (внешний сайт). Эталон тоже один язык
за прогон. **Подтвердить.**

**14. Роль Битрикс24.** 🟡 Стратегический вопрос: каталог для сделок/счетов/КП и CRM-аналитики (тогда
цена/единицы важны) или справочно. **Определить до старта.**

**15. Редакция/размещение.** 🟡 Облако/коробка; один портал (вебхук) или тираж (OAuth-приложение Маркета).
Для ceram — скорее **вебхук, один портал**. **Подтвердить.** SemVer + CHANGELOG для нашего сервиса.

**16. «Прочие» поля payload.** 🟢 `areas_per_package`, `product_shape`, `upsells/cross_sells`, alt/
описания картинок, витринные страницы/SEO — **слой внешнего сайта/подключённого магазина**, не CRM-каталог.
В нашем сервисе — через событие `product_imported` (полный payload), если потребитель есть. Ядро их не
пишет (граница §8, как mu-плагин эталона).

## Дополнительно — что сделает интеграцию лучше (золото эталона)

1. **Идемпотентность по `public_id` → `xmlId`** (+ локальный индекс, если фильтр по xmlId не штатный),
   `article` отдельно в свойство «Артикул» (только непустой; не в xmlId). Никогда не по name/slug (§5.1).
2. **Справочники — поиск по имени (label), не по слагу:** раздел по имени в пределах родителя; значение
   свойства-списка по имени; стабильный ключ маппинга характеристики — `specification_id` (в эталоне был
   баг дублей «Толщина» из транслита).
3. **Характеристики → свойства товара** (`catalog.productProperty`, НЕ поля сделки); многозначные
   (`multitude`) → множественное свойство; ручной маппинг по `specification_id`, строгий режим; тип
   свойства фиксировать (неизменяем).
4. **boolean:** `specification_option_name = null`, значение в `bool_option`; **пусто/`null`/отсутствие =
   false**; список «Да/Нет» если нет нативного boolean. (`multitude=false` у boolean.)
5. **Медиа:** URL самодостаточен (токен в хеше) → наш сервис качает (MIME по содержимому, URL без
   расширения) → base64 в `catalog.productImage`; крупные max отдельными вызовами; дедуп `sha1(path#size)`
   и трекинг качества — в нашей БД; галерея по имени файла, не по URL.
6. **Не выдумывать цену** (`catalog.price` не трогаем, create-only); `weight` бывает `null`; `active` —
   только при создании.
7. **Единицы:** конвертер г/мм → единицы портала (обязателен); `measurement_unit` (м²/шт) → нативный
   `catalog.measure`, не путать с габаритами. Нативные `weight/length/width/height` — использовать.
8. **Категории — нативное дерево `catalog.section`** из `/categories/` (`parent_category`, родители раньше
   детей). **`/collections/{slug}/` = 500 — не дёргать**, всё из эмбеда `collections[]`. **Справочники —
   `limit=1000` + сверка `meta.counts`.**
9. **Прочий payload** (`areas_per_package`, `product_shape`, alt/описания, детали бренда/коллекции,
   `upsells/cross_sells`, витринные страницы/SEO/статусы) — **слой внешнего сайта/магазина**, не CRM. В
   нашем сервисе — через событие, если потребитель есть.
10. **SemVer + CHANGELOG** (Keep a Changelog) для нашего сервиса; batch-операции; деградация §5.5 — битый
    товар/лимит логируем и пропускаем.
