# История изменений — OneCatalog Integration Standard

Формат основан на [Keep a Changelog](https://keepachangelog.com/ru/1.0.0/),
нумерация версий — по [семантическому версионированию](https://semver.org/lang/ru/).

Версионируется сам **стандарт интеграции**. Платформенные порты указывают, какой
версии стандарта они соответствуют.

## [Не выпущено] — бэклог

- Раздел про мультиязычность (общий пробел: API отдаёт один язык за запрос; модель
  «один объект + переводы» против «объект на язык» — отдельная фаза для всех портов).
- Опциональный общий PHP-core (Composer-пакет) для PHP-семейства портов
  (WooCommerce / PrestaShop / Bitrix / OpenCart / Magento): API-клиент, конвертер
  единиц, контент-ключ/дедуп, нормализация payload.

## [1.3.0] — 2026-06-25

### Добавлено
- **§10.1 «Сборка и распространение (release-артефакт)»**: каждый порт даёт готовый
  установочный архив в формате платформы (zip плагина/модуля, `.ocmod.zip`, Composer),
  скрипт сборки `build.sh`, публикацию через **GitHub Releases** с CI-сборкой по тегу
  `vX.Y.Z` и секцию «Скачать» в README. Gitflow: `dev`-сборки — **pre-release**,
  production — Release по тегу из `main`. Пункт добавлен и в чек-лист §11.

## [1.2.1] — 2026-06-25

### Исправлено (терминология)
- **Разведены 1С-Битрикс и Bitrix24** — это разные продукты (общее только ядро D7).
  Раньше «Bitrix24» в §3/§6/§12/README использовалось там, где описан 1С-Битрикс
  (инфоблоки). Теперь:
  - **1С-Битрикс: Управление сайтом** — товар = элемент инфоблока + `catalog`,
    интеграция = PHP-модуль `/bitrix/modules/` (✅ готовый порт `onecatalog-bitrix`);
  - **Bitrix24** — товар = CRM-каталог (`crm.product`), интеграция = REST/OAuth-
    приложение (🔜 отдельная будущая цель, порт под 1С-Битрикс не переносится).
- §12 дополнен отдельным блоком по Bitrix24 (REST-app) и актуализирован блок
  1С-Битрикс (нативные `MORE_PHOTO`/`TAGS`, служебная `*_meta`, public_id отдельным
  свойством — в соответствии с v1.2).

## [1.2.0] — 2026-06-25

### Изменено
- **§2.4 Picker — переписан раздел приёма сообщений** (по итогам багов в Bitrix-порте):
  явно разведены **origin виджета** (задаётся настройкой, не из `window.location`) и
  **`parentOrigin`** (обязан быть `window.location.origin`, вычисленным на КЛИЕНТЕ —
  иначе браузер блокирует `postMessage` и выбор не доходит). Приём сообщений — по
  `event.source === iframe.contentWindow` (а не по строковому `event.origin`), разбор
  `event.data`, если это JSON-строка, и **своя кнопка закрытия (× + Esc)**.
- **§3 / §7 — «нативное прежде своего»**: для справочных сущностей предпочитать
  нативное/существующее поле платформы (теги → родное поле тегов, галерея → родная),
  своё авто-поле — крайний вариант после осознанного выбора. Справочные сущности
  (коллекция/бренд/страна) **по умолчанию выключены**, включаются после выбора цели
  (дефолты в §7 приведены к этому; «своё авто-поле по умолчанию» помечено антипаттерном).

### Добавлено
- **§5.1 — инвариант «служебные значения вне видимой формы товара»**: бизнес-ключи
  (`public_id`, `article`) — видимые/редактируемые, а технические сигнатуры
  идемпотентности (медиа, цены/остатки), контент-ключи дедупа и коды сопоставления —
  в служебном мета-хранилище (отдельная таблица/мета), не как свойства товара.
- **§6 — UX ручного запуска (браузерный степпер)**: индикатор прогресса, блокировка
  повторного запуска (несколько способов запуска не должны дублировать импорт), сводка
  результатов, отмена между порциями, сохранение итога (напр. `localStorage`).

## [1.1.0] — 2026-06-21

### Добавлено
- **§13. Синхронизация цен и остатков (B2B-фид)** — новый платформо-независимый
  сценарий поверх импорта каталога (внедрён в эталоне — WooCommerce 1.8.0):
  отдельный B2B API и авторизация (`url_key` в пути + `private_key` в query);
  формат `known`/`unknown` + справочники `regions`/`warehouses`; маппинг (known по
  `public_id`, unknown по коду поставщика); стратегии цены × приоритет регионов;
  суммирование остатков по складам, дробные остатки; **change-detection (scan-and-diff)**
  как обязательный инвариант нагрузки; ненайденные (импорт через очередь / skip /
  создать+код); расписание; точки расширения для раскладки цен по регионам и остатков
  по складам в сайтовые поля; чек-лист переноса.
- §13.5 расширен: для `unknown` добавлен режим **«отстойник» (staging) — ручной отбор**
  (хранилище + страница: импортировать / игнорировать, статусы `new|imported|ignored`)
  как «шлюз» против засорения каталога; рядом с режимами skip и авто-создание.
- §13.1 уточнён под обновлённый B2B API: справочник **`data.suppliers`** вынесен на
  верхний уровень (как `regions`/`warehouses`), в оффере — скаляр `supplier_id`;
  `promo_price`/`purchasing_price` могут быть `null`. В события передаётся и `suppliers`.
- §13.4 дополнен инвариантом **устойчивости к изменению состава фида**: приоритеты по
  стабильным id, новые справочники не применяются автоматически (пометка «новое»), два
  снимка (подтверждённый/из фида), **уведомление администратора** при изменении состава
  и при ошибке выгрузки. В чек-лист §13.8 добавлены пункты про это и про журналы
  (история запусков синка / журнал очереди импорта, ручной прогон).
- Интро: стандарт явно покрывает **два сценария** — импорт каталога (§1–§12) и
  синхронизацию цен/остатков (§13).
- §10: версионирование **по вехам, по ходу** (каждая веха = свой номер сразу).
